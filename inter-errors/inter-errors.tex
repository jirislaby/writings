\begin{abstract}
  With a use of contemporary static analysis tools, many errors are found in
  source codes. Here, we introduce a mechanism to prevent recurring of
  perpetual errors which were not fixed yet or false positives which are
  likely to be present with its source code for very long time. We discuss
  what mechanisms can be used to determine such errors and demonstrate them on
  output from \st run on the Linux kernel.
\end{abstract}

\section{Introduction}
Recently, many new static analysis tools emerged (for example
\cite{coverity,findbugs,sparse}) with a potential of finding many
long-standing errors in the source code. We also contributed by our tool
called \st \cite{stanse}.

Mostly the problem with these tools is that they produce a long list of errors
with false-positives included. There is a little what can be done about that
in one specific release of sources (from the repetitive-error perspective).
But it becomes a burden when the developers should check the very same subset
of reported errors every single release of the product. In this case, when
such a tool is run on projects with short release cycle, it is possible to
recognize (a) the errors which were reported in earlier version and were not
fixed yet and (b) false positives which nobody wish to see again and again.



One of the projects with the requirements satisfied is surely the Linux
kernel. Its release cycle (between major versions, e.g.
2.6.34$\rightarrow$2.6.35) is relatively short 

\paragraph{Error Format}
Before we start with the methodology, we first have to know what the output
of the tools look like to be able to proceed to details. Even though the error
formatting practices differ tool by tool, in common they contain the source
file name where the error lays with exact line too.

The report also carries an information about what kind of error is that and
possibly what module (checker) generated that. Eventually some tools report
another fields like a line column, but this is irrelevant for further
considerations.

In the end, the entry (error) in the report list $RL$ is a tuple $E$ (e.g.
in a database as we will see later) such as:
\begin{center}
$E = (file, line, checker, error\_description) \in RL$
\end{center}
