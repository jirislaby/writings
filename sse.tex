\begin{abstract}
  \tool{} is a tool for detection of bugs described by finite state machines
  in C programs. The tool combines three well-known techniques:
  instrumentation, program slicing, and symbolic execution.  This paper
  briefly describes the approach of \tool{} including its strengths,
  weaknesses, and modifications for SV-COMP 2013. Architecture and
  installation of the tool are described as well.
\end{abstract}

%}}}

%{{{ Verification approach

\section{Verification Approach}
\label{sec:approach}

\tool{} implements our technique~\cite{sse} that combines instrumentation,
program slicing, and symbolic execution in order to detect bugs described by
finite state machines. More precisely, we instrument a given program with a
code that tracks runs of state machines representing various errorneous
behaviours. If an instrumented state machine enters an error location during
a program execution, then the original program contains a bug specified by
the machine. After instrumentation, we slice~\cite{weiser} the program to
reduce its size without affecting runs of state machines. Finally, we
symbolically execute~\cite{king} the sliced program to find bugs in the
program.

As reachability of an \texttt{ERROR} label is the only bug considered in the
SV-COMP, we have modified our instrumentor to put \texttt{assert(0)}
function calls at \texttt{ERROR} labels in the code. Given the instrumented
code, we execute \textsc{Clang} to produce an \textsc{Llvm} bitcode. This is
in turn interprocedurally sliced with respect to slicing criteria, which are
the instrumented \texttt{assert} calls. In other words we remove all the
irrelevant code except the one that has an effect on the
crash. The sliced \textsc{Llvm} bitcode is finally symbolically executed by
\textsc{Klee}~\cite{klee}. There are several possible reports \textsc{Klee}
prints on the output. It can either find a bug and report that, or finish
the computation without any report,
% (which means that the code contains no reachable \texttt{assert(0)} call), 
or terminate in some errant way (out of time, out of memory, or some
internal error for example). We map these to the demanded answers:
\emph{TRUE}, \emph{FALSE}, or \emph{UNKNOWN} respectively. This is taken
care of in a simple scripted filter.

%{{{ Software architecture

\section{Software Architecture}
\paragraph{Libraries/External Tools} For the code translation and for
operations with the \textsc{Llvm} bitcode we use
\textsc{Llvm}/\textsc{Clang} 3.1~\cite{llvm}. Symbolic executor is
\textsc{Klee}~\cite{klee}. It itself uses \textsc{Stp} constraint
solver~\cite{stp}. We obtained both \textsc{Klee} and \textsc{Stp} from GIT
and use the snapshots.

\paragraph{Software Structure and Architecture} The architecture described
in Section~\ref{sec:approach} is summarized in Figure~\ref{fig:pipeline}.
The slicer is written as a plug-in for the optimizer \texttt{opt} from the
\textsc{Llvm} suite. It is publicly available in a separate repository at
\url{https://github.com/jirislaby/LLVMSlicer/}.  For the ease of use, all the
parts of the pipeline are one by one run by a single script \texttt{runme}.

\texttt{bash} script using \texttt{sed}. The final filter as well uses
\texttt{bash} with a help of \texttt{grep}. The rest of the toolchain is
written in C++ and compiled using \texttt{gcc}.

%{{{ Discussion

\section{Discussion of Strengths and Weaknesses of the Approach}

The strength of the tool lies in high precision of the answers. In theory,
the only source of invalid answers is the slicer: it can completely remove
an infinite loop in some cases and thus an unreachable \texttt{ERROR} label
located below the loop may become reachable. However, there is no such case
in competition benchmarks. All invalid answers produced by the tool are due
to bugs in implementation.

The main weakness of the tool is a high percentage of \texttt{UNKNOWN}
results. The main source of these results follows from high computation
costs of symbolic execution of programs with loops or recursion. This
problem is relieved by slicing, but there are still many cases where the
sliced code remains complex and symbolic execution runs out of time or
memory. Other sources of \texttt{UNKNOWN} results are internal errors of
\textsc{Klee} and general limitations of SMT solving.

%{{{ Tool setup and configuration

\section{Tool Setup and Configuration}
\paragraph{Download and Installation Instructions}
\begin{itemize}
\item \emph{Requirements:} \texttt{llvm-3.1}, \texttt{clang-3.1}.
\item Download the tool at: \url{http://www.fi.muni.cz/~xslaby/symbiotic/}.
\item Change the current directory to \texttt{/opt}.
\item Untar the archive with the tool.
\item Change directory into \texttt{/opt/symbiotic}.
\item Run \texttt{./runme <file.c>} for each source file in the set.
\end{itemize}

\paragraph{Configuration Definition} None.

\paragraph{Results Reported} The results \texttt{TRUE/FALSE/UNKWNOWN} match
the competition rules.$\!\!$
