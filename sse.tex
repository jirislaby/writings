\section{Instrumentation}
\label{sec:instr}

We want to find a similar set of errors to those found by tools like
Metal~\cite{metal}. These tools use abstract interpretation as an algorithm
and automata for program behavior description. To be able to track a state of
automata using the symbolic execution, the code has to be instrumented before
the execution itself. However, the code instrumented is different to the
behavior performed by the automata in abstract interpretation on the same
code.

Abstract interpretation has to consider a set of states by each program
location and a transform function over these sets for the locations. It
is due to exploring all paths of the program at once. For the symbolic
execution it is sufficient to remember the actual state of the program path
explored. The other paths are explored separately in a different run.

To present the difference of the instrumentation before the symbolic execution
and computing abstract interpretation automata states,
Fig.~\ref{fig:instrumentation} is attached. Consider a function \texttt{foo}
where a lock A is used. Further there is a \texttt{compute} function.  Assume
that it changes the program state, especially the A state, in no way.

Abstract interpretation, while computing a fix-point, would transform states
at lines~\ref{line:lock} and~\ref{line:unlock} according to an automaton. An
example of such an automaton is depicted in Fig.~\ref{fig:automaton}. In that
specific case, line~\ref{line:lock} transforms unlocked state into locked one
and locked state into an erroneous one. The latter is eventually reported by
the tool as an error to the user.

Thanks to the simple run being evaluated at once, the lock may be only in one
state at a time. Hence we introduce an integer for each lock used. The integer
then corresponds exactly to the automaton state. I.e. all function calls to
\texttt{lock} and \texttt{unlock} are replaced by assignments to this integer
by one or zero respectively. Now it reminds to describe the way of error state
checking.

Thanks to the symbolic executor we use (cf. Section~\ref{sec:impl}), a
so-called \texttt{assert} function may be used to provoke the executor to
report a failure. The function (or macro more precisely) is used by developers
for debugging purposes. The macro takes exactly one argument. When it
evaluates to true, an error is reported. Both when a program containing the
facility is run and when executed by symbolic executor.

Thus our tool inserts an \texttt{assert} before each assignment. There the
assertion is that the integer is zero when the code is about to take a lock
and one in the opposite case.
