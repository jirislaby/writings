\maketitle

\begin{abstract}
  We introduce a novel technique for finding real errors in programs. The
  technique is based on a synergy of three well-known methods:
  metacompilation, slicing, and symbolic execution. More precisely, we
  instrument a given program with a code that tracks runs of state machines
  representing various kinds of errors. Next we slice the program to reduce
  its size without affecting runs of state machines. And then we
  symbolically execute the sliced program. Depending on the kind of symbolic
  execution, the technique can be applied as a stand-alone bug finding
  technique, or to weed out some false positives from an output of another
  bug-finding tool. We provide several examples demonstrating the practical
  applicability of our technique.
\end{abstract}

\section{Introduction}
\label{sec:introduction} \todo{H}

The title of this paper refers to two popular bug-finding techniques:
metacompilation and symbolic executions. The two techniques use completely
different principles leading to different advantages and disadvantages.

\emph{Metacompilation}~\cite{metal,pldi02} is a static analysis technique
looking for various kinds of errors specified by state machines. We explain
the technique with use of the state machine $SM(x)$ of Figure~\ref{fig:sm},
which describes errors in lock manipulation. Intuitively, the state machine
represents possible courses of states of a lock $x$ along an execution of a
program. The state of the lock is changed according to a transition of the
state machine if the execution performs a program statement syntactically
subsuming the label of the transition. We would like to decide whether there
exists any program execution where an instance of state machine $SM(x)$
assigned to some lock of the analyzed program reaches an error
state. Unfortunately, this is not feasible due to potentially unbounded
number of executions and unbounded execution length. Hence, we use static
analysis to overapproximate the set of reachable states of state machines.

Let us assume that we want to check the program of Figure~\ref{fig:copy} for
errors specified by the state machine $SM(x)$. First, we find all locks in
the program and to each lock we assign an instance of the state machine. In
our case, there is only one lock \texttt{L} and thus only one instance
$SM(\texttt{L})$. For each program location, we compute a set
overapproximating possible states of $SM(\texttt{L})$ after executions
leading to the location. Roughly speaking, we initialize the set in the
initial location to $\{U\}$ and the other sets to $\emptyset$. Then we
repeatedly update the sets according to the effect of individual program
statements until the fixed point is reached.

As we can see, the sets contain two error states: \emph{double unlock} after
the \texttt{unlock(L)} statement and \emph{return in locked state} in the
terminal location. If we analyze the computation of the sets, we can see
that the first error corresponds only to the execution going through lines
1,2,3,4,8,9,13,14. This execution path is not a real one due to the value of
\texttt{len}, which is set to 0 at line 3 and assumed to satisfy \texttt{len
  > 0} at line 13. To sum up, the first error is \emph{false positive}. The
second error corresponds to executions passing lines 1,2,3,4,5,6,7,8, then
iterating the \texttt{while}-loop and finally passing lines 13,15. All these
paths are also false positives except the one that perform zero iterations
of the \texttt{while}-loop, which is the only real execution leading to a
the only real locking error in the program.

To sum up, metacompilation is highly flexible, fast, and thus applicable on
extremely large software projects (e.g.~Linux Kernel). It examines all the
code and finds many errors. Unfortunately, some of the errors are false
positives. The main source of false positives comes from the fact that the
analysis does not work with data values. Therefore, when a branching
statement is processed, the analysis cannot decide which states of a state
machine are reachable on the positive branch and which on the negative one.
This can be illustrated on the program of Figure~\ref{fig:copy}, where the
set $\{U,L\}$ assigned to line 12 is correct, but the analysis does not have
enough information to see that the lock \texttt{L} cannot be in state U
whenever the condition on line 13 is valid.

At this point, we would like to emphasize that metacompilation actually uses
a more sophisticated algorithm enriched with many techniques for partial
elimination of false positives (see~\cite{pldi02} for details). We also note
that metacompilation uses a dedicated language called \emph{metal} for
description of state machines. The best known implementation of
metacompilation is the tool called \emph{xgcc}, but the idea of error
specification using state machines appears also in other tools,
e.g.~ESP~\cite{esp} or STANSE~\cite{stanse}.

\medskip In contrast to metacompilation, \emph{symbolic execution}~\cite{se}
analyzes each execution path separately. In contrast to standard execution,
symbolic execution replaces input data by symbols representing arbitrary
values. Executed statements then manipulate expressions over the symbols
rather than exact values. For each execution path, symbolic execution builds
a formula called \emph{path condition}, which is a necessary and sufficient
condition on input data to drive the execution along the path.  Whenever
symbolic execution reaches a branching statement, it uses the current path
condition to check if any of the branches is unfeasible. If this is the
case, the computation proceed to the other branch. Otherwise, the
computation is forked and it continues in both branches. Moreover, we add
the branching condition to the path condition on the positive branch and its
negation to the path condition on the negative branch.
% This approach ensures that the technique does not follow unfeasible
% paths. Hence, if the symbolic execution detects some error (e.g.~double
% lock) it cannot be a false positive.

The main advantage of symbolic execution is that it works only with feasible
executions and hence it does not report any false positives. A minor
disadvanate is that implementations of this technique usually detects only
few kinds of error (e.g.~violation of an assert statement). To detect a
specific kind of error, the program has to be modified to reduce the error
to one of the automatically detected errors. The main disadvantage of the
technique is its high compuation cost. In fact, symbolic execution usually
explores only a part of a program of industrial size. In particular,
programs containing loops or recursion have typically large or even infinite
number of executions and cannot be completely analyzed by symbolic
execution.

\medskip In this paper, we introduce a new technique offering flexibility of
of metacompilation and zero false positive rate of symbolic execution. The
basic idea is very simple: we use the concept of state machines to get
flexibility in error specification. Then we instrument a program to be
analyzed with a code for tracking behaviour of the state machines. The
instrumented program is then reduced using the slicing method introduced
in~\cite{W81}.\todo{Je to spravny clanek o slicingu?} The sliced program has
to meet the criterion to be equivalent to the instrumented program with
respect to reachability of error states of tracked state machines. Note that
slicing can remove big portions of the code, including cycles and function
calls. Hence, an original program with infinite number of execution paths
can be reduced to a program with a finite number of execution
paths. Finally, we executed the sliced program symbolically.

Our technique can be used in two ways according to the applied symbolic
execution tool. If we apply a symbolic executor that prefers to explore more
parts of the code (for example, it can explore only the execution iterating
each program cycle at most twice), we can use the technique as a general
bug-finding technique reporting only real errors. On the contrary, if we use
a symbolic executor exploring all execution paths, we can use our technique
for basic classification of error reports produced by other tools (e.g.~xgcc
or STANSE). For each such an error report, we can instrument the
corresponding code only with the state machine describing the reported
error. If our technique founds the same error, it is a real one. If our
technique explores all execution paths of the sliced code without detecting
the error, it is a false positive. If our technique runs out of resources,
we cannot decide whether the error is a real one or just a false positive.
We have developed a prototype implementation of our technique. Our
experimental results indicate that the technique can be applied to classify
error reports produced by STANSE applied to Linux kernel.

We emphasize the synergy of the three steps of our technique: specification
of errors with state machines (inspired by metacompilation) and
intrumentation of the original program with the code emulating the state
machines allows us to clearly specify the slicing criteria. Hence, the
sliced program contains only the code relevant to the considered
errors. Further, slicing can substantially reduce the size of the code,
which can remarkably improve performance of the symbolic
execution. 

Let us note that usage of symbolic execution instead of classic static
analysis brings us another benefit: while in metacompilation the state
machines are associated to synactic objects (e.g.~locks or pointers), here
we can associate state machines to actual values of these objects. This can
potentially lead to better precision of error description. On the other
hand, adoption of this feature can lead to unbounded number of state
machines tracked by the instrumented program, which can lower performance of
our technique. \todo{Co s timhle odstavcem?}

\section{Instrumentation}
\label{sec:instr}

We want to find a similar set of errors to those found by tools like those
using \textsc{Metal}~\cite{metal}. These tools use abstract interpretation as
an algorithm and automata for program behavior description.

Abstract interpretation loads the automata before the analysis and then fires
transitions and remembers states while computing a fix-point. Generally there
may be multiple states for each program location. It is due to exploring all
paths of the program in a single run.

With the symbolic execution, this is slightly different. It is sufficient to
remember an actual state of the automata for a program path currently
explored. Other paths are explored separately in a different runs.

There are two approaches to track the automata states. First, to implement the
functionality directly into the executor. This would be an intrusive change to
a compact code which is intended to do a dedicated job.  The second approach
is to instrument the code before the execution itself.  That results in a
separate module without touching the executor's code.

Thanks to the fact the automaton may be only in one state at a time, we can
keep the automaton state in an integer. We do not introduce such an integer
for each automaton instance\footnote{There may be multiple instances of the
automata. For example the tools create a separate instance of locking
automaton for each lock.} found in the code though. This would lead to false
alarms. Especially in C code where two pointers may point to a single object.

To avoid that, the tools leverages Andersen's points-to
analysis~\cite{andersen} and reimplements the allocator to use memory pools.
Both help all instances to be bound together properly. For example when the
parametrization of automata instance is named differently in the code
(\texttt{lock(A); B=A; unlock(B);}) or passed as a parameter with another
name.

In our implementation, there are two objects a pointer can reference. Either
a global variable or a line where allocation of the object occurs.

The latter introduces an imprecision into the analysis. An example of code
where the behavior is incorrect in depicted in Fig.~\ref{fig:invalid_ptr}. All
pointers in the list point to line~\ref{lst:alloc} where their allocation
happens.

The memory pools are created as follows:
\begin{enumerate}
\item All code locations supposed to fire a transition are found.
\item A points-to set of the automaton parameter on the found locations is
computed. Cardinalities of the sets are finite, because the count of the
pointed objects (global variables and allocation lines) is finite too.
\item All the sets are merged together to obtain all possible automaton
parameter objects. Again, the merged set is finite.
\item Memory is split into equally sized pools. Their count is equal to the
cardinality of the merged set. The pools are numbered from zero to count-1.
\end{enumerate}

When the memory is split, the allocation sites of each of the objects in the
merged set are looked up. Such allocations are superseded by a pool allocator.
It reserves a space in a specific pool. It is specified by the index of the
pool where the object belongs to. The size of the reserved space is the same
as the requested size. This approach allows for an easy reverse mapping from
pointers to (numbered) pools.

It is also needed to store automata states somewhere. For this purpose, an
array is created. Its size is designated by the count of the memory pools too.
The type of members is integral as it fits the purpose of storing the
automaton state.

Finally, it remains to generate transition functions from the input automaton.
Since the only knowledge the functions need to know is the pointer to derive a
pool index, they take it as the only argument. They involve the reverse
mapping to find the index. Then they perform a transition of the automaton
state in the array over the integer at the pool index.

Now, the instrumentation can start. First, the array is inserted into the
code. It is needed to be accessible by the transformation functions. Next, for
each location from point 1, a function call is added. The function to be
called is dependent upon the automaton specification.

An example of the transformation and pool manipulation functions is depicted
in Fig.~\ref{fig:instrumentation_stuff}. It is designed to work with locking
automaton with three states: locked, unlocked and error. Hence there is an
array with locks. There are two of them, because in the original code, there
were two (\texttt{L1} and \texttt{L2}). Further there are two automaton
manipulation functions \texttt{\_\_ISSE\_lock} and \texttt{\_\_ISSE\_unlock}.
They will be instrumented at the places where the original code called
\texttt{lock} or \texttt{unlock} respectively. Finally, there is a function
called \texttt{\_\_ISSE\_get\_memory\_pool\_index}. It serves the purpose of
computing pool indices from pointers.

looks like in Fig.~\ref{fig:copy_instrumented}. We have not described

Thanks to the symbolic executor we use (cf. Section~\ref{sec:impl}), a
so-called \texttt{assert} function may be used to provoke the executor to
report a failure when a bad state is reached. The function (or macro more
precisely) is used by developers for debugging purposes. It takes exactly one
argument. When it evaluates to true, an error is reported. Both when a program
containing the debug facility is run and when executed by the symbolic
executor we use.

Thus our tool inserts an \texttt{assert} before each assignment to the
automaton state. To be precise it means the error state of the automaton is
never reached. It is checked whether the transition which is about to be
fired would be a failure.  This approach simplifies slicing a bit. I.e.  not
all assignments are necessarily needed and might be sliced right away.

Obviously, in the example in Fig.~\ref{fig:instrumentation_stuff}, the
parameter of the \texttt{assert} macro, the assertion, is a check that the
integer is zero when the code is about to take a lock and one in the opposite
case. For the initial and final state, it is expected to be zero, i.e.
unlocked.

\section{Slicing} \label{sec:slicing} \todo{M}

Let us have a look at the instrumented program at
Figure~\ref{fig:copy_instrumented}. We can easily observe, that the main part
of the function \texttt{copy}, i.e.~the loop copying the characters, does not
affect locking states of the instrumented automata. On the other hand,
symbolic execution of such a code can be very expensive. Therefore, we use the
program slicing technique to eliminate such a code from the analysed program.
Slicing may bring considerable speed-up in the last phase of the algorithm,
where we execute the program symbolically.

We use interprocedural static slicing as proposed by Weiser in
1981~\cite{W81}. When slicing a program we first need to specify what
behavior we want to preserve. We do so by assigning a \emph{slicing criteria}
to particular lines of the program. Slicing criterion at a given program line
specifies a set of variables whose values at that line must be equal in both
original and sliced program for each input data passed to both programs.
Having slicing criteria set, we slice functions independently. Let us consider
a function with slicing criteria. We assign a set of \emph{relevant variables}
to each line of the functions. A variable is relevant, if its value has an
impact (direct or indirect) to a value of a variable in some slicing
criterion. Sets of relevant variables of the sliced function are initially
empty, except those of lines with slicing criteria (these sets contain
variables from their slicing criteria). Then, according to semantics of
program commands we propagate relevance of variables between program lines and
we insert discovered relevant variables into related sets. We apply the
propagation until sets do not increase their size. From the content of the
sets we can safely say, which program lines have no impact to the slicing
criteria and we therefore remove those lines from the body of the function.
Interprocedural slicing is about exchanging slicing criteria between functions
through call sites. When we are done slicing individual functions, we have
computed relevant variables at call sites and also at the entry locations
(lines) to the functions. So, we propagate the criteria between called and
calling functions. Then we slice the functions independently again. We repeat
this process until no \emph{new} slicing criterion is passed between
functions.

Propagation of slicing criteria between functions is mainly based on argument passing technique of the program. Nevertheless, besides argument passing we usually need to know what variables can be modified by which function. Weiser references technique~\cite{B78}, which does more then we actually need. Therefore, we have designed much more simple and satisfactory technique computing only \emph{modifies} function summaries. Modifies summary of a given function is simply a set of program variables, which are not local to the function, and which can be modified by the function. Having results from a points-to analysis, a computation of the summaries is straightforward.  An interested reader can find more details about interprocedural slicing here~\cite{W81}.

Now we show, how to slice our instrumented example program. First we need to set slicing criteria preserving values of the instrumented automata. Obviously, the array \texttt{\_\_ISSE\_lock} is the variable, whose values we want to preserve in sliced program. It only remains where to put slicing criteria. The best places for the criteria are assertions in the functions at Figure~\ref{fig:instrumentation_stuff} operating on states of the automata. Note that we do not set any slicing criterion to function \texttt{\_\_ISSE\_get\_memory\_pool\_index} since it it does not operate with the automata.

Having the criteria set the interprocedural slicing propagates another slicing criteria between functions. As variables \texttt{i} become relevant


 into the function \texttt{copy} with relevant variable \texttt{L}.

 We obviously know, values of which variables we want to preserve.

\section{Implementation and Experimental Results}
\label{sec:impl}
We decided to verify the functionality on the C code. The main reason is that
many projects with open-source license (and source code available) are written
in C. Hence there is a fair codebase available for checking.

The tool is not implemented from scratch. It is written as a module for the
LLVM framework~\cite{llvm}. This provides us with a C parser (and compiler)
for free and a documented application programming interface. Also for LLVM
a symbolic executor exists already. It is called \textsc{Klee}.

Before we can really run the executor, several steps have to be performed
first. Initially, the tool instruments the code on the C level
(Section~\ref{sec:instr}) by an LLVM module called \texttt{Prepare}. Although
the technique is universal, we narrowed it down for the sake of pilot testing
and simplicity. The set of checked properties is thus limited to locking
errors only. This comprises checks for inappropriate leaving functions with
lock held, double locks and double unlocks. We plan to augment the checked
properties set in the future.

The instrumented C code is then taken by the tool and turned into LLVM
bytecode. For this purpose a compiler from the LLVM framework called
\textsc{clang} is used. Optimizations are turned off as required by
\textsc{Klee}. Further, the generated bytecode is sliced by an LLVM
inter-procedural slicer to eliminate irrelevant code
(Section~\ref{sec:slicing}). This is done by a separate LLVM module named
\texttt{Slicer}.

Given the sliced input, the LLVM bytecode is parsed by the \texttt{Kleerer}
LLVM module. This pass creates a separate file with LLVM bytecode for each
function present in the source file. To all the created files, it also adds a
\texttt{main} function. This is where the analysis starts -- in each of the
files it calls a different function.

Finally a symbolic execution is performed by \textsc{Klee}
(Section~\ref{sec:symexe}). \todo{Describe what we do with the output\dots}

\subsection{Experiments}
\label{ssec:experiments}

The implementation described above was evaluated on a real code too. The
evaluation was performed on the Linux kernel. It is a large and well tested
piece of software. Also it is generally of high interest to be bug-free.
Otherwise systems could be indeed compromised rather easily.

All tests performed were run on a standard office machine. On a hardware
level, its configuration is a dual-core processor, 6\,GiB of memory.
Operating system installed is Linux. \textsc{Klee} is run on the instrumented
and sliced LLVM code. We specify \textsc{Klee} parameters to add an upper
bound on solver and overall time. The former is set to 3 seconds, the latter
to 30. We tried to double these times several times with no real effect.

We do not pass optimize option for \textsc{Klee} because it causes
\textsc{Klee} to crash under some circumstances. The optimization might
further speed the analysis up. The crashes are yet to be investigated.

At this early stage of the tool, the evaluation is not performed on the whole
kernel. It is because it still does not scale that much. We plan to extend
the tool to be capable of handling whole kernel. We limited our testing to a
set of files. We aimed the technique at the functions where abstract
interpretation tools reported a trace. Be it a real error or false positive.
We consulted the files with upstream developers to sort out which are in each
category.

This helped us to classify the results provided by our tool. The output of the
tool is either:
\begin{itemize}
\item confirmation of a real error -- reports an error at the same location the
abstract interpretation tool did or
\item confirmation of a false positive -- does not report anything at that
location or
\item message about unfinished computation -- due to various errors.
\end{itemize}

The last category stems from two sources. First, the tool hits a limit of
symbolic execution and times out. This happens especially when locks are
dependent upon loops. Then the loops are not sliced away and symbolic
execution does not finish in the bounded time.

Second, memory errors occur. This is due to our handling of functions input.
For each parameter of a function analyzed, a symbolic memory is allocated.
Size is determined from the parameter size. Plus, when the parameter is a
pointer, the size is multiplied by 4000. So for example 4 bytes for integer,
and 16000 for integer pointer. Further, for the pointer case, we pass a
pointer to the middle of the array (functions might dereference memory at
negative index). The idea behind is explained in~\cite{ucklee}.

Despite we use this feature, functions may touch a memory out-of-bounds this
array. Typically when a structure is passed to a function and the structure
contains another pointer. It is uninitialized and points to an invalid
memory. To remedy the inconvenience, we plan to implement the same on-demand
handling \textsc{UcKlee}~\cite{ucklee} does.

files where we are sure about the results. I.e. upstream developers confirmed
they are either error or false positive and we are able with our limited
