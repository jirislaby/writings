\maketitle

\begin{abstract}
  We introduce a novel technique finding real errors in programs. The
  technique is based on a synergy of three well-known methods:
  metacompilation, slicing, and symbolic execution. More precisely, we
  instrument a given program with a code that tracks runs of state machines
  representing various kinds of errors. Then we slice the program to reduce
  its size without affecting runs of state machines. Then we symbolically
  execute the sliced program. Depending on the kind of symbolic execution,
  the technique can be applied as a stand-alone bug finding technique, or to
  weed out some false positives of another bug-finding tool output. We
  provide several examples demonstrating practical applicability of our
  technique.
\end{abstract}

\section{Instrumentation}
\label{sec:instr}

We want to find a similar set of errors to those found by tools like those
using \textsc{Metal}~\cite{metal}. These tools use abstract interpretation as
an algorithm and automata for program behavior description.

Abstract interpretation loads the automata before the analysis and then fires
transitions and remembers states while computing a fix-point. Generally there
may be multiple states for each program location. It is due to exploring all
paths of the program in a single run.

With the symbolic execution, this is slightly different. It is sufficient to
remember an actual state of the automata for a program path currently
explored. Other paths are explored separately in a different runs.

There are two approaches to track the automata states. First, to implement the
functionality directly into the executor. This would be an intrusive change to
a compact code which is intended to do a dedicated job.  The second approach
is to instrument the code before the execution itself.  That results in a
separate module without touching the executor's code.

%\begin{figure}[htb]
%\begin{center}

%\caption{An automaton describing locking behavior}

Thanks to the fact the automaton may be only in one state at a time, we can
keep the automaton state in an integer. We do not introduce such an integer
for each automaton instance\footnote{There may be multiple instances of the
automata. For example the tools create a separate instance of locking
automaton for each lock.} found in the code though. This would lead to false
alarms. Especially in C code where two pointers may point to a single object.

To avoid that, the tools leverages Andersen's points-to
analysis~\cite{andersen} and reimplements the allocator to use memory pools.
Both help all instances to be bound together properly. For example when the
parametrization of automata instance is named differently in the code
(\texttt{lock(A); B=A; unlock(B);}) or passed as a parameter with another
name.

In our implementation, there are two objects a pointer can reference. Either
a global variable or a line where allocation of the object occurs.

The latter introduces an imprecision into the analysis. An example of code
where the behavior is incorrect in depicted in Fig.~\ref{fig:invalid_ptr}. All
pointers in the list point to line~\ref{lst:alloc} where their allocation
happens.

The memory pools are created as follows:
\begin{enumerate}
\item All code locations supposed to fire a transition are found.
\item A points-to set of the automaton parameter on the found locations is
computed. Cardinalities of the sets are finite, because the count of the
pointed objects (global variables and allocation lines) is finite too.
\item All the sets are merged together to obtain all possible automaton
parameter objects. Again, the merged set is finite.
\item Memory is split into equally sized pools. Their count is equal to the
cardinality of the merged set. The pools are numbered from zero to count-1.
\end{enumerate}

When the memory is split, the allocation sites of each of the objects in the
merged set are looked up. Such allocations are superseded by a pool allocator.
It reserves a space in a specific pool. It is specified by the index of the
pool where the object belongs to. The size of the reserved space is the same
as the requested size. This approach allows for an easy reverse mapping from
pointers to (numbered) pools.

It is also needed to store automata states somewhere. For this purpose, an
array is created. Its size is designated by the count of the memory pools too.
The type of members is integral as it fits the purpose of storing the
automaton state.

Finally, it remains to generate transition functions from the input automaton.
Since the only knowledge the functions need to know is the pointer to derive a
pool index, they take it as the only argument. They involve the reverse
mapping to find the index. Then they perform a transition of the automaton
state in the array over the integer at the pool index.

Now, the instrumentation can start. First, the array is inserted into the
code. It is needed to be accessible by the transformation functions. Next, for
each location from point 1, a function call is added. The function to be
called is dependent upon the automaton specification.

An example of the transformation and pool manipulation functions is depicted
in Fig.~\ref{fig:instrumentation_stuff}. It is designed to work with locking
automaton with three states: locked, unlocked and error. Hence there is an
array with locks. There are two of them, because in the original code, there
were two (\texttt{L1} and \texttt{L2}). Further there are two automaton
manipulation functions \texttt{\_\_ISSE\_lock} and \texttt{\_\_ISSE\_unlock}.
They will be instrumented at the places where the original code called
\texttt{lock} or \texttt{unlock} respectively. Finally, there is a function
called \texttt{\_\_ISSE\_get\_memory\_pool\_index}. It serves the purpose of
computing pool indices from pointers.\todo{ensure\_unlocked}

int __ISSE_lock[2] = { 0, 0 }; 

int __ISSE_get_memory_pool_index(const int *ptr)
{
    if (ptr == &L1) return 0;
    if (ptr == &L2) return 1;
    return -1; /* unreachable */
}

void __ISSE_ensure_unlocked(const int *L) {
    int i = __ISSE_get_memory_pool_index(L);
    assert(__ISSE_lock[i] == 0);
}

void __ISSE_lock(const int *L) {
    int i = __ISSE_get_memory_pool_index(L);
    assert(__ISSE_lock[i] == 0);
    __ISSE_lock[i] = 1;
}

void __ISSE_unlock(const int *L) {
    int i = __ISSE_get_memory_pool_index(L);
    assert(__ISSE_lock[i] == 1);
    __ISSE_lock[i] = 0;
}

Thanks to the symbolic executor we use (cf. Section~\ref{sec:impl}), a
so-called \texttt{assert} function may be used to provoke the executor to
report a failure when a bad state is reached. The function (or macro more
precisely) is used by developers for debugging purposes. It takes exactly one
argument. When it evaluates to true, an error is reported. Both when a program
containing the debug facility is run and when executed by the symbolic
executor we use.

Thus our tool inserts an \texttt{assert} before each assignment to the
automaton state. To be precise it means the error state of the automaton is
never reached. It is checked whether the following transition would be a
failure.  This approach simplifies slicing a bit. I.e. not all assignments are
necessarily needed and might be sliced right away.

In the example, the first parameter of the \texttt{assert} macro, the
assertion, is a check that the integer is zero when the code is about to take
a lock and one in the opposite case.

\section{Implementation}
\label{sec:impl}
We decided to verify the functionality on the C code. The main reason is that
many projects with open-source license (and source code available) are written
in C. Hence there is a fair codebase to test.

The tool is not implemented from scratch. It is written as a module for the
LLVM framework~\cite{llvm}. This provides us with a C parser (and compiler)
for free and a documented application programming interface. Also for LLVM
a symbolic executor exists already. It is called \textsc{Klee}.

Before we can really run the executor, several steps have to be performed
first. Initially, the tool instruments the code on the C level.  Although the
technique is universal, we narrowed it down for the sake of pilot testing and
simplicity. The set of checked properties is thus limited to locking errors
only. This comprises checks for inappropriate leaving functions with lock
held, double locks and double unlocks. We plan to augment the checked
properties set in the future.

During the instrumentation, each locking function is superseded by an
\textsc{assert} and assignment to an appropriate integer to reflect the
automaton state. This task is performed by our first LLVM module called
\texttt{Prepare}.

The instrumented C code is then taken by the tool and turned into LLVM
bytecode. For this purpose a compiler from the LLVM framework called
\textsc{clang} is used. Optimizations are turned off as required by
\textsc{Klee}. Further, the generated bytecode is sliced by an LLVM
inter-procedural slicer to eliminate irrelevant code. This is done by a
separate LLVM module named \texttt{Slicer}.

Given the sliced input, the LLVM bytecode is parsed by the \texttt{Kleerer}
LLVM module. This pass creates a separate file with LLVM bytecode for each
function present in the source file. To all the created files, it also adds a
\texttt{main} function. This is where the analysis starts -- in each of the
files it calls a different function.

\subsection{Experiments}
\label{sec:experiments}

The implementation described above was evaluated on a real code too. The
evaluation was performed on the Linux kernel. It is a large and well tested
piece of software. Also it is generally of high interest to be bug-free.
Otherwise systems could be indeed compromised rather easily.

All tests performed were run on a standard office machine. On a hardware
level, its configuration is a dual-core processor, 6\,GiB of memory.
Operating system installed is Linux. \textsc{Klee} is run on the instrumented
and sliced LLVM code. We specify \textsc{Klee} parameters to add an upper
bound on solver and overall time. The former is set to 3 seconds, the latter
to 30. We tried to double these times several times with no real effect.

We do not pass optimize option for \textsc{Klee} because it causes
\textsc{Klee} to crash under some circumstances. The optimization might
further speed the analysis up. The crashes are yet to be investigated.

At this early stage of the tool, the evaluation is not performed on the whole
kernel. It is because it still does not scale that much. We plan to extend
the tool to be capable of handling whole kernel. We limited our testing to a
set of files. We aimed the technique at the functions where abstract
interpretation tools reported a trace. Be it a real error or false positive.
We consulted the files with upstream developers to sort out which are in each
category.

This helped us to classify the results provided by our tool. The output of the
tool is either:
\begin{itemize}
\item confirmation of a real error -- reports an error at the same location the
abstract interpretation tool did or
\item confirmation of a false positive -- does not report anything at that
location or
\item message about unfinished computation -- due to various errors.
\end{itemize}

The last category stems from two sources. First, the tool hits a limit of
symbolic execution and times out. This happens especially when locks are
dependent upon loops. Then the loops are not sliced away and symbolic
execution does not finish in the bounded time.

Second, memory errors occur. This is due to our handling of functions input.
For each parameter of a function analyzed, a symbolic memory is allocated.
Size is determined from the parameter size. Plus, when the parameter is a
pointer, the size is multiplied by 4000. So for example 4 bytes for integer,
and 16000 for integer pointer. Further, for the pointer case, we pass a
pointer to the middle of the array (functions might dereference memory at
negative index). The idea behind is explained in~\cite{ucklee}.

Despite we use this feature, functions may touch a memory out-of-bounds this
array. Typically when a structure is passed to a function and the structure
contains another pointer. It is uninitialized and points to an invalid
memory. To remedy the inconvenience, we plan to implement the same on-demand
handling \textsc{UcKlee}~\cite{ucklee} does.
