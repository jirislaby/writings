\section{Instrumentation}
\label{sec:instr}

We want to find a similar set of errors to those found by tools like
Metal~\cite{metal}. These tools use abstract interpretation as an algorithm
and automata for program behavior description.  For the sake of pilot testing
and simplicity we narrowed the set of checked properties to locking errors
only. This comprises checks for inappropriate leaving functions with lock
held, double locks and double unlocks. We plan to augment the checked
properties set in the future.

To be able to track a state of automata using the symbolic execution, the code
has to be instrumented before the execution itself. However, the code being
instrumented is slightly different to the behavior performed by the automata
in abstract interpretation on the same code.

Abstract interpretation has to consider a set of states for each program
location and a transform function over these sets for the locations. It
is due to exploring all paths of the program in a single run. For the symbolic
execution it is sufficient to remember an actual state of the automata for a
program path currently explored. Other paths are explored separately in a
different run.

To present a difference between abstract interpretation automata states
computation and the instrumentation to be used with the symbolic execution,
Fig.~\ref{fig:instrumentation} is attached. Consider a function \texttt{foo}
where a lock A is used. Further there is a \texttt{compute} function.  Assume
that it changes the program state, especially the A state, in no way.

Abstract interpretation, while computing a fix-point, would transform states
at lines~\ref{line:lock} and~\ref{line:unlock} according to an automaton
describing locking. An example of such an automaton is depicted in
Fig.~\ref{fig:automaton}.  In that specific case, line~\ref{line:lock}
forces abstract interpreter to transform unlocked state into locked one and
locked state into an erroneous one. The latter is eventually reported by the
tool as an error to user.  Analogical transitions are performed on
line~\ref{line:unlock}.

Since a single run is evaluated at once while executing symbolically, the lock
may be only in one state at a time. Hence we introduce an integer for each
lock used. The integer then corresponds exactly to the automaton state.  I.e.
all function calls to \texttt{lock} and \texttt{unlock} are replaced by
assignments to this integer by one or zero respectively. So far we simulate
only the lock behavior, it reminds to describe the way error state is
revealed.

Thanks to the symbolic executor we use (cf. Section~\ref{sec:impl}), a
so-called \texttt{assert} function may be used to provoke the executor to
report a failure when a bad state is reached. The function (or macro more
precisely) is used by developers for debugging purposes. It takes exactly one
argument. When it evaluates to true, an error is reported. Both when a program
containing the debug facility is run and when executed by the symbolic
executor we use.

Thus our tool inserts an \texttt{assert} before each assignment to the lock
state. To be precise it means the error state of the automaton is never
reached. It is checked whether the following transition would be a failure.
This approach simplifies slicing a bit. I.e. not all assignments are
necessarily needed and might be sliced right away.

The first parameter of the \texttt{assert} macro, the assertion, is check that
the integer is zero when the code is about to take a lock and one in the
opposite case. This can be seen in Fig.~\ref{fig:instrumentation}, the third
column.

\paragraph{Points-To Analysis} To be precise in decision about pointers in our
tool, we implemented Andersen points-to analysis~\cite{andersen}. Primarily,
it helps us in slicing to decide if two pointers point to the same object or
not. Further it determines which functions may be called by indirect function
calls, which locks point to the same allocated object during instrumentation
phase even though the locks have a different name and so on.

\section{Implementation}
\label{sec:impl}
We decided to verify the functionality on the C code. The main reason is that
many projects with open-source license (and source code available) are written
in C. Hence there is a fair codebase to test.

The tool is not implemented from scratch. It is written as a module for the
LLVM framework~\cite{llvm}. This provides us with a C parser (and compiler)
for free and a documented application programming interface. Also for LLVM
a symbolic executor exists already. It is called \textsc{Klee}.

Before we can really run the executor, several steps have to be performed
first. Initially, the tool instruments the code on the C level. Each locking
function is superseded by an \textsc{assert} and assignment to an appropriate
integer to reflect the automaton state. This task is performed by our first
LLVM module called \texttt{Prepare}.

The instrumented C code is then taken by the tool and turned into LLVM
bytecode. For this purpose a compiler from the LLVM framework called
\textsc{clang} is used. Optimizations are turned off as required by
\textsc{Klee}. Further, the generated bytecode is sliced by an LLVM
inter-procedural slicer to eliminate irrelevant code. This is done by a
separate LLVM module \texttt{Slicer}.

Given the sliced input, the LLVM bytecode is parsed by the \texttt{Kleerer}
LLVM module. This pass creates a separate file with LLVM bytecode for each
function present in the source file. It also adds a \texttt{main} function.
This is where the analysis starts -- in each of the files it calls a different
function.

\subsection{Experiments}
\label{sec:experiments}

The implementation described above was evaluated on a real code too. The
evaluation was performed on the Linux kernel. It is a large and well tested
piece of software. Also it is generally of high interest to be bug-free.
Otherwise systems could be indeed compromised rather easily.

All tests performed were run on a standard office machine. On a hardware
level, its configuration is a dual-core processor, 6\,GiB of memory.
Operating system installed is Linux. \textsc{Klee} is run on the instrumented
and sliced LLVM code. We specify \textsc{Klee} parameters to add an upper
bound on solver and overall time. The former is set to 3 seconds, the latter
to 30. We tried to increase these times with no real effect.

We do not optimize option for \textsc{Klee} because it causes \textsc{Klee} to
crash under some circumstances. The optimization might help the analysis,
hence the crashes are yet to be investigated.

At this early stage of the tool, the evaluation is not performed on the whole
kernel. It is because it still does not scale that much. We plan to extend
the tool to be capable of handling whole kernel. We limited our testing to a
set of files. We aimed the technique at the functions where XXX\todo{what to
write here} reported a trace. Be it a real error or false positive. We
consulted the files with upstream developers to sort out which are in each
category.

This helps us to sort the results provided by our tool. The output is the tool
either:
\begin{itemize}
\item confirms a real error -- reports an error at the same location the
abstract interpretation tool did or
\item confirms a false positive -- does not report anything at that location
or
\item does not finish computation -- due to various errors.
\end{itemize}

The last category stems from two sources. First, the tool hits a limit of
symbolic execution and times out. This happens especially when locks are
dependent upon loops. Then the loops are not sliced away and symbolic
execution does not finish in the bounded time.

Second, memory errors occur. This is due to our functions input handling. For
each parameter of a function analyzed, a symbolic memory is allocated. Size is
determined as the parameter size. When the parameter is a pointer, the size is
multiplied by 4000. So for example 4 bytes for integer, and 16000 for integer
pointer. Further, for the pointer case, we pass a pointer to the middle of the
array (functions might dereference memory at negative index). The idea behind
is explained in~\cite{ucklee}.

Even if we use this feature, function may touch a memory out-of-bounds this
array. Typically when there is a pointer to a structure passed to a function
and when the structure contains another pointer. It is uninitialized. We plan
to implement the same on-demand handling \textsc{UcKlee}~\cite{ucklee} does.

sure about the results. I.e. upstream developers confirmed they are either
error or false positive and we are able with our limited knowledge to reason
