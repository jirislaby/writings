\chapter*{Abstract}
This thesis studies application of contemporary automatic bug-finding
techniques to large software projects. Large projects tend to be complex
systems so experimental tools implemented as a proof-of-concept have
difficulties to handle these projects.  There is a call for efficient
analyzers that would be capable to exercise such code from the parsing phase
to finding errors in large intermediate structures.  Also, such an analyzer
cannot generate many false alarms, otherwise the output would be unusable. In
particular we aim at the static analysis and primarily focus at techniques
like abstract interpretation and symbolic execution. Both of them were
designed in 1970's and were adopted in many tools since then. Their
applicability is increasing these days thanks to current fast processors and
the resulting ability to handle larger software.

The thesis summarizes the current state of the art of chosen techniques
implemented in static analyzers which are capable to handle large code bases
as stated above. In particular, we discuss some large code bases which can be
used for the techniques we deal with and finally choose the Linux kernel as a
test-bed. On the top of the explored state of the art, we develop our
techniques. More concretely, we present a tool \stan which is based on the
abstract interpretation and is capable to handle the Linux kernel. The tool
also tries to prune false alarms resulting from the \stan's checkers which are
based on the abstract interpretation. The false alarms pruning method is based
on a pattern checking and the method turns out to be weak. So we subsequently
try to improve the situation with a help of the symbolic execution. It is
implemented in a freely available tool called \klee.  Since the symbolic
execution requires high computation time, we slice the code by a static slicer
first.

During our experiments, we cumulated knowledge about categories of bugs usually
present in code such as the Linux kernel. We investigated it more and
developed a database of known bugs in a fixed version of the kernel. It is
publicly available and contains information about errors and false alarms
found by several bug-finding tools. This can help developers of new static
analyzers to compare their brand new tool against the database and see how
their tool stands with respect to false alarms, missed errors and of course
error hits. This section also draws a comparison of tools of which we were
able to get meaningful results on the Linux kernel.

With all the experiences we gained during the development of the previous
tools, we implemented a tool called \symbio and participated in the Software
Verification Competition 2013. \symbio is based on slicing and symbolic
execution again, but is tailored to the competition needs.  Since we
sent only a too preliminary version of the tool to the competition, the
results were not so encouraging. The tool was improved later and it gives
surprisingly better results -- we collect all possible points in some
competition categories.  We plan to compete in the 2014's competition with our
tuned tool.

Finally, we conclude the thesis with our findings and results and discuss
future possible improvements of our techniques and tools.
