\chapter*{Abstract}
This thesis studies the application of contemporary automatic bug-finding
techniques to large software projects. As they tend to be complex systems,
there is a call for efficient analyzers that can exercise such projects
without generating too many false alarms. To fulfill these requirements, this
thesis aims at the static analysis with the primary focus at techniques like
the abstract interpretation and symbolic execution.  Since their introduction
in 1970's, their applicability is increasing thanks to current fast
processors.  Only this renders the techniques usable for larger software.

The thesis summarizes the current state of the art of chosen techniques
implemented in the current static analyzers. Further, we discuss some large
code bases which can be used for the techniques and finally choose the Linux
kernel as a test-bed for them. On the top of the explored state of the art, we
develop our own techniques. We present the framework \stan which utilizes the
abstract interpretation and has no issues with handling the Linux kernel. The
framework also prunes false alarms resulting from some checkers. The pruning
method is based on the pattern matching, but it turns out to be weak. Hence we
subsequently try to improve the situation by the symbolic execution. For that
purpose we use an executor called \klee in our new tool \symbio. But since the
symbolic execution requires a high computation time, we slice the code by a
static slicer first.

During our experiments, we cumulated knowledge about categories of bugs usually
present in code such as the Linux kernel. We investigated it more and
developed a database of known bugs in a fixed version of the kernel. It is
publicly available and contains information about errors and false alarms
found by several bug-finding tools. This can help developers of new static
analyzers to compare their brand new tool against the database and see how
their tool stands with respect to false alarms, missed errors and of course
error hits.

We also participated with \symbio in the Software Verification Competition.
The tool had to be adapted to the competition needs.  Since we participated
only with a too preliminary version of the tool, the results were not so
encouraging.  Nonetheless, the tool was improved later and it gives
better results now -- we collect nearly all points in some competition
categories.  We plan to compete in the competition with our tuned tool again.

\chapter{Conclusion}
In this thesis we introduced the reader to the state of the art of some
techniques. These techniques were then implemented in our two tools in some
form. Namely, we implemented the abstract interpretation in \stan and the
slicing with the symbolic execution in \symbio. Both these tools are able to
handle large software projects like the Linux kernel. Hence we analyzed the
kernel in version 2.6.28 by the tools.  The results from the kernel were not
useful only to the kernel developers, but we also established the classified
bug database \cdb which can be used as a benchmark.  Later, we also enrolled
\symbio to the software verification competition.  Despite the results were
not so encouraging, it helped us to improve the tool a lot.

\section{Contribution}
We contributed by the freely available framework for code analyzing techniques
\stan. It allows for easy development of new code analyzing techniques. The
framework provides the needed information about analyzed code and it also
comprises code for traversing the information. There are many other useful
helpers such as false positive filters. Since \stan still produce many false
positives, we introduced a new technique to lower their occurrence. It is
based on a combination of three known techniques: instrumentation, slicing and
symbolic execution. It is an exact technique thanks to the symbolic execution
and applicable to real code thanks to slicing. We implemented the technique in
the experimental tool \symbio.

Both these techniques lead to a benchmark for code analyzers.  It serves as a
benchmark for developers of new code analyzers. They can run their tool on one
of the \cdb's projects and compare their new tool how it stands with respect
to hit and missed errors, and also how many false positives their tool
produces.


\section{Future Work}
We plan to extend \symbio so that it is capable of handling more code and
gives more exact answers in cases it failed for the last year's competition.
After that, we plan to participate in the next year's competition. We also
plan to augment \cdb to contain more projects and to extend reports for the
Linux kernel.
