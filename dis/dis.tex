\section{Introduction} \todo{weakest link on a chain; chapter cross-reffed
listing; corectness=soundness+completness+termination (Dupal)}
In the contemporary world, we are surrounded by devices making our lives
easier. It counts from modern lifts which speak to us by natural language to
supercomputers helping to treat cancer such as \emph{IBM
Watson}~\cite{watson-cancer}. Most present devices are featuring fast
processing units allowing developers to implement arbitrary functions in them.
However being written by human beings vastly, sometimes it leads to errant
behavior of the implementations. One example is the error that led to a crash
of the rocket that should launch \emph{Ariane 5} satellite to
orbit~\cite{ariane_5}.  These errors obviously cause revenue losses of
companies, \$370M in case of \emph{Ariane 5}. Moreover the later in the
development cycle an error is revealed, the more expensive it is to fix that.
The situation becomes much worse when the software is deployed at the
customer's site already. Compilers can help to warn about many primitive
errors introduced by a programmer to a program.  Nevertheless compilers are
designed to be really fast in compiling code, so their primary goal is
definitely not error finding. Hence many warnings reported by compilers are
rather by-products of performed optimizations and analyses needed for the
intermediate compilation phases. We have to emphasize that these kinds of
checks got introduced even by optimizing compilers, they were not in the
prior compilers at all.

In the early times of computer technology, programmers were forced to test
their programs other ways. Usually they, or testers if they were distinct
persons, had been running their program with diverse input to make the program
misbehave or crash, to catch program errors of course. This approach is called
\emph{dynamic program testing}. It indeed takes huge amount of time to
perform systematic testing of large projects. And yet, dynamic testing needs
obviously a runnable executable which is not available in the early phases of
development. Nevertheless, testers can come up with simple binaries testing
only finished subsystems, with so-called \emph{unit tests}.  However usually
they still do not cover all possible paths in the program. To defeat this
inconvenience, researchers introduced several approaches to find errors in
programs automatically, possibly in the early phases of development, and more
importantly in the whole program and without the urge of dynamic execution, by
\emph{static program testing}. However the more complicated static analyses
came of real use even lately, when computers become powerful enough to finish
computation of such analyses.  Until then, most the computations were
infeasible and the researcher's paper were buried in archives. Now, having the
computation power and multi-processor (thousand cores are not exceptions)
machines, we could witness rapid improvement of the situation over the
past two decades, where many new techniques emerged.\todo{coding policy helps
avoiding bugs (and also to static analyzers -- simpler code), e.g. disable
heap allocations on embedded\ldots}

Many of those were adopted or even developed by companies to make their
products safer. For example Microsoft introduced SDV toolkit~\cite{sdv} based
on \textsc{Slam} to improve situation around Windows drivers, and
\textsc{Sage}~\cite{sage} which found errors in core Windows parts like a
parser for cursor files. Other companies like Coverity, Grammatech, Klocwork,
or MathWorks emerged with their products~\cite{coverity, codesonar, klocwork,
polyspace} to offer a service to check code of their customers. Although some
of the companies also check the open source code like the Linux
kernel\footnote{\url{http://scan.coverity.com/}} or core system libraries,
developers usually have no control over the check frequency, thus have no
chance to check their code on demand.

For those reasons, some groups decided to implement their own, freely
available, checkers~\cite{stanse, cppcheck, findbugs, smatch, sparse}. They
vary in precision, speed and of course in the method used for finding bugs.
Although developers may use them at any time at their will, often they get
results with much higher false reports rate than from a commercial tool.

\subsection{Motivation}
Although many approaches exist, mostly they are implemented in some version,
but some are not applicable to real software due to high computation time
needs or sometimes even because of inability to read sources in the offshoot
of a language the projects are written in. In this thesis we aim at mapping
and investigation of current techniques used for code analysis and their
application on real, large software projects.

There exist hundreds of projects of that size fitting our needs in this case.
At the first glance it could look like we should specifically aim at kernels
of operating systems, because it may seem like a crash of a user space
application causes only pain to a user. But think about a vulnerability in
a random web browser or database system (which consist of millions lines of
code too) that could lead to stolen user's personal data.  Hence we should not
only consider kernels of operating systems, but also user space programs in
this respect.

On the other hand operating systems should generally be error-free, because
even a minor error would allow an arbitrary attacker to obtain administrator
privileges on a machine running such a vulnerable system.  Then, they can
steal whatever data they want and it is way easier to gain access to other
machines in the same network, than attacking from outside~\footnote{There are
usually no managed firewalls for the traffic flowing inside the network for
example.}.  We chose the Linux kernel in particular, because it is a large
project running on zillion machines over the world.  There are examples of
such errors like in~\cite{exploit_vmsplice, exploit_proto_ops,
exploit_userns}. Hence there is a call for techniques to possibly eliminate
all errors the kernels contain.

\paragraph{Slicing} Slicing was first described in~\cite{slicing}. It is a
technique to eliminate unneeded lines of code from a program. This is always
computed with respect to some code location and variables which we are
interested in at that place. Only the code that can affect the variables there
is left in the program. This is for example useful for debugging purposes to
understand how values are propagated to a variable. The location and variables
are called \emph{slicing criteria}.

The main two branches of slicing are \emph{dynamic} and \emph{static}. The
former considers one concrete input when computing the slice and may
potentially result in more sliced code. Static slicing on the other hand
computes the slice generally, with no input restrictions. More details about
slicing and its variations and uses can be seen in section~\ref{sec:fmics}
and in~\cite{slicing_survey}, of course.

\subsection{The Goals}
The primary goal of this thesis is to research deployment of static analysis
on large, critical, software projects. In particular, we aim specifically at
the Linux kernel in some sections. We considered contemporary approaches used
for code analysis and tried to apply them to a large code base. Based on the
results we obtain, we would like to discuss pros and cons of the techniques.
This will result either in a design of new techniques or improvement of the
contemporary ones (or both).
