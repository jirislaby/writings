\chapter{Introduction}\label{sec:intro}
In the contemporary world, we are surrounded by devices making our lives
easier. It counts from modern lifts which speak to us by natural language to
supercomputers helping to treat cancer such as \emph{IBM
Watson}~\cite{watson-cancer}. Since Moore's law~\cite{moore_law} has proven to
hold for the past four decades, most present devices are featuring quite fast
processing units thanks to the increasing count of built-in transistors. This
allows developers to implement many interesting but complex algorithms.
However being written by human beings vastly, it may lead to errant behaviour
of the implementations as they are only as strong as the weakest link in the
imaginary chain. One example is the error that led to a crash of the
\emph{Ariane~5} rocket that should launch a satellite to the orbit of the
Earth~\cite{ariane_5}. \$370M were buried in the middle of nowhere. Due to a
programmer's omission to check for an overflow in the acceleration
computation.  So these errors obviously cause companies revenue losses. It is
known the later in the development cycle an error is revealed, the more
expensive it is to fix the error~\cite{topten}. The situation becomes even
worse when the software is deployed at the customer's site or in a rocket
flying 13~000~ft above the sea level already.

Companies always have been desiring for completely error-free algorithms. But
they are out of luck as in general, there cannot be anybody to implement a
tool which would tell us whether some algorithm is error-free. If there were
one, it would be in contradiction of the \emph{Halting
problem}~\cite{halting_problem}.  However while it is impossible in theory, it
does not necessarily mean we should not at least try hard for non-general
cases when dealing with algorithms implemented in software. There, the
implementations have at least finite state space thanks to finite memory.
Even though the limited state space is still huge, companies have been always
using techniques helping to avoid and remove errors from their programs.  Even
some kind of \emph{standardization} of used coding principles like code
reviews or dictated coding style may reduce the number of introduced
errors~\cite{topten}. The reason may be that a project management imposes a
duty on developers to write simpler code. Once, the management can ban complex
constructs often leading to errors, other times, for environments like
embedded systems heap allocations are not allowed which results in weeding out
one whole category of programming errors -- memory leaks. Some companies also
started adopting standards like ISO~9001:2000~\cite{iso9000} or
MISRA~C~\cite{misra_c}. Their customers are assured that the internal
management of such company conforms to a standardized behaviour that tries to
increase quality of developed products.

Standardization cannot by definition evade programming errors completely.
Since the early times of computer technology, programmers in companies, or
testers if they were distinct persons, were forced to test the programs.
Testers were running the programs with diverse input, random inclusive, to
make the program misbehave or crash, to catch and fix program errors. This
approach is called \emph{dynamic program testing}~\cite{sw_testing}. It indeed
takes huge amount of time to perform systematic testing of large projects. And
yet, dynamic testing needs obviously a runnable executable which is not
usually available in the early phases of development. Testers can help in that
case by creating simple binaries that exercise only finished subsystems, by
so-called \emph{unit tests}. However tests, especially the manually written
ones, often do not cover all possible paths in the program. Actually, it was
shown in~\cite{klee} that a set of unit tests developed for over fifteen years
can be augmented to cover 16 more per cents of code using automatic
sophisticated techniques for tests generation. On the other hand, when an
error is reported, it is guaranteed to be genuine. Dynamic testing cannot
generate \emph{false positives}, i.e.~reports that are not real, reports about
errors that are theoretically found in the code, but in practice they do not
exist, only the analyzer is imprecise. Mostly because the analyzer ignores
context in some sort. Nonetheless, there are no false positives reported in
dynamic testing thanks to running the code in real-time with real data (if we
look aside from broken hardware misdemeanour).

\emph{Compilers} can also help to warn about many primitive errors introduced
into a program by a programmer. However compilers are designed to be really
fast in compiling code, so their primary goal is definitely not error finding.
Hence many warnings reported by compilers are rather by-products of performed
optimizations and \emph{data-flow analyses} needed for the intermediate
compilation phases. We have to emphasize that these kinds of checks got
introduced only by optimizing compilers, they were not in the prior compilers
at all. Separate projects like \textsc{Lint}~\cite{lint} or
\textsc{Dave}~\cite{dave} were used to fill this gap and were widespread at
that time instead. These projects were based on pattern matching and also on
the data-flow analysis, but mostly they were more computationally expensive
than compilers. For that price, the tools were giving more accurate results
with more reports overall. These were the first attempts of \emph{static
program analysis}~\cite{program_analysis} to tackle the problems of finding
errors in programs automatically. To tackle the problems possibly in the early
phases of development, and more importantly in the whole program and without
the urge of dynamic execution. The downside of the static techniques often is
the presence of false positives. Tools like \textsc{Lint} suggest programmers
to annotate the code, so that the tools have a clue and can thus generate less
of the false positives.

In fact, much more sophisticated techniques like~\cite{cousot,
se_king}\todo{overview in~\cite{analysis_old_survey}} were described
theoretically couple years before \textsc{Lint} was presented.  However they
came of real use even lately, when computers become powerful enough to finish
computation of such analyses in reasonable time.  Now, having the computation
power and multi-processor (thousand cores are not exceptions today) machines,
we witness a rapid improvement of the situation over the past two decades. We
could also see many new techniques emerging. Many of those were adopted or
even developed by industrial companies to make their products safer. For
example Microsoft introduced SDV toolkit~\cite{sdv} based on
\textsc{Slam}~\cite{slam_cav, slam_popl, slam_decade} and later SDV 2.0 with
\textsc{Slam2}~\cite{slam2} to improve situation around Windows drivers.
Microsoft also released \textsc{Sage}~\cite{sage} which found errors in core
Windows parts like the parser of cursor files. Other companies like Coverity,
Gimpel Software, GrammaTech, Klocwork, or MathWorks presented their
products~\cite{coverity, pclint, codesonar, klocwork, polyspace} to offer a
service to check code of their customers. Some of the companies also check
open source code like the Linux kernel or core Linux libraries. Although the
companies make the results available~\cite{coverity_scan}, developers of the
kernel or libraries usually have no control over the setup and check
frequency, thus have no chance to check their code on demand.

For those reasons, some groups decided to implement their own, freely
available, checkers, see e.g.~\cite{stanse, cppcheck, findbugs, smatch,
sparse}. They vary in precision, speed and of course in the method used for
finding bugs.  Although developers may use them at any time at their will,
often they get results with much higher false reports rate than results of a
commercial tool.

\section{Motivation}
Although many approaches exist, their implementations are often incomplete in
some sense. For example, they are not applicable to real software due to high
computation time needs (cf.~symbolic execution started being used almost 30
years after being introduced). Another problem are the languages projects are
written in. Even though there is a standard for a language, let us consider C
with the last standard issued in 1999~\cite{ansi_c}, there are many language
offshoots that ``evolved'' with respective compilers like
\textsc{Gcc}~\cite{gcc}, \textsc{Intel CC}~\cite{intel_cc}, or
\textsc{MSVC}~\cite{msvc}.  Every single of them supports a different superset
of the standard. Thus it is very hard for analyzers to implement a parser
which would be universally capable of parsing whichever program with sources
in a specified language.  In this thesis we aim at mapping and investigation
of current techniques used for code analysis and their application on real,
large software projects.

There exist hundreds of software projects of the large size which we could
consider for our purposes. We can divide them roughly into two categories,
depending on the level their code runs as they are usually several possible
privilege levels~\cite[p. 21--23]{os_concepts}.  Kernels of the operating
systems often run at the level with no limitations, i.e.~in the \emph{kernel
mode}.  There, the kernel can touch any memory in a machine and can also
change other possible resources the kernel is able to reach. User space
programs on the other hand tend to be restricted, i.e.~they run in the
\emph{user mode}. There, a process cannot change any system resource or memory
of others, if not explicitly allowed to, e.g.~by shared
memory~\cite[p. 124--126]{os_concepts} or special system calls~\cite[p.
62--73]{os_concepts}.  In particular, the privilege levels have effects on
crashes. A crash of the kernel means the machine cease production any new
actions, whereas a crash of a user space application causes mostly only pain
to a user.  So at the first glance it looks like we should specifically aim at
kernels of operating systems. But think about a vulnerability in a random web
browser or database system (which consist of millions lines of code too). It
could indeed lead to stolen user's personal data or even money from their bank
accounts. Of course there is a call for techniques to possibly eliminate all
errors the programs contain.

On the other hand operating system kernels should generally be error-free more
than the others, because even a minor error would allow an arbitrary attacker
to obtain administrator privileges on a machine running such a vulnerable
system, or simply crash it. For a \emph{denial of service} (DoS)~\cite{dos}
caused by a crash, it sometimes suffices to write a ``killer'' string to a
special file locally or even send a packet with a specific structure over the
network. Note the attacker can do it remotely from a place thousands miles far
away.  Instances of such errors are in~\cite{exploit_vmsplice,
exploit_proto_ops, exploit_userns} or \cite[p. 256]{hacking}, and more to be
found in the National Vulnerability Database~\cite{nvd}. If the attacker
succeeds beyond crashing a machine and is able to get a login there, they are
very close to stealing whatever data they want from that machine since it is
much easier to gain an administrator shell when logged on as a casual user
already.  Sometimes simply because remote administrator logins are not
allowed. And yet it is often easier to gain an access to other machines in the
same subnetwork than attacking them from the outside. One of the reasons is
that there are usually no managed firewalls for the traffic flowing among
computers inside the subnetwork~\cite{insider_attack}.

The arguments stated above persuaded us to aim at operating system kernels.
But we can choose from many. There are kernels of BSDs (NetBSD, FreeBSD,
OpenBSD, \dots), GNU Hurd, GNU Mach, Linux, Mac OS, Minix, NetWare, OS/2, Sun
OS, Solaris, Windows, and many others. Nonetheless, we limited ourselves to
the Linux kernel. The main reasons are that the kernel:
\begin{itemize}
\item is an open source,
\item is a large code base consisting of more than fifteen millions lines of
code (version 3.10),
\item is developed so fast that each version released every 2--3 months
contains over 700 0000 changed lines of code which are across more than eleven
thousand files in average (changes done for all releases between 3.5 and
3.10), and
\item is used widely, reportedly on 42\,\% of consumer computing
devices in 2012~\cite{linux_42_percent}.
\end{itemize}
Aside from the aspects of large projects stated above, the Linux kernel has
another positive aspect like it is self-contained code.

\section{Background}
This section introduces the reader into the terms we use throughout the
thesis.  Since every implemented analysis has some properties, we describe the
properties which we consider important. Such properties are usually looked at
when deciding which of the available implementations to choose for a newly
developed program, a static analyzer in particular.  There is also a small
introduction to code parsing as we deal with source codes in this thesis and
they need to be parsed into internal structures.

\subsection*{Theoretical Properties}\label{sec:theor_props}
Usually, the most important aspect for users of an analyzer is to know whether
the produced output of the analyzer makes some sense according to what is
expected, i.e.~if the results are \emph{correct} (sometimes called
\emph{sound}). Next, we usually care whether all possible errors are reported,
i.e.~whether the output is \emph{complete}. And perhaps if there will be any
output in finite time at all, if the analyzer will \emph{terminate}. Below, we
describe the three terms and provide examples of soundness and completeness
for better understanding in Figure~\ref{fig:theor_props}.

\begin{description}
\item[Soundness] When an analyzer is sound, whatever it reports is not out of
bounds of the defined problem. I.e.~if we have an analyzer and want it to tell
us which values are definitely in a variable at some point, it has to
enumerate only those which are really set there. No over-approximation is
allowed. Many analyses do not fulfil this criterion. For example to speed up
the analysis, they compute an over-approximation such that they put in
the output even items non-conforming to the original definition. This means
unsound analyses may produce false positives.
\item[Completeness] Complete analyzers have to report at least what they are
expected to. I.e.~if we have the same analyzer above which reports values
set to a variable, it needs not necessarily report only those, it can compute
some over-approximation and report more, if it is inevitable. Of course this
is for the price of unsoundness. Set in other words, incomplete analyses may
produce false negatives, i.e.~results which do not fit to the defined
behaviour of the analyzer.
\item[Termination] This is only a theoretical property as computers are
finite machines thanks to their bounded memory. So in fact, every analyzer
which tries to analyze some code will finish its computation on the finite
state space in finite time. So it will always terminate. Only theoretically,
there exist analyzers that will not terminate in general (c.f.~the Halting
problem).
\end{description}

In an ideal world, we would have sound and complete analyzers. They would
return only results without false positives and false negatives. But
computation time of such analyzers tend to be high as well as memory
consumption. This stems from the need of too much precise computation and
precise results which would have to be stored somewhere. So it is very common
to implement analyses that are only complete, but unsound. They report
everything which is possibly an error, but sometimes, we have to await
false positives. There also exist analyzers which are only sound, but
incomplete, or the third option: analyzers that are neither sound, nor
complete. The latter may help in reducing both false positives and false
negatives for the price of potentially hiding errors, i.e.~adding
false negatives.

\subsubsection{Sensitivity of Analyses}\label{sec:sensitivity}
Another interesting property of analyses is how are the analyses handling
separate commands and functions present in some code. Consider the example in
Figure~\ref{fig:lst_sens} and also consider an analysis that is watching
values stored into variables and is thus able to report this to the user. We
will use both of them to demonstrate the properties explained below.

\begin{figure}[htb]
\begin{center}
\begin{tabular}{c}
\begin{lstlisting}[language=C]
int fun(int a) {
  int x = 1;

  if (a == 2) (*@ \label{lst:sens_cond} @*)
    x = a; (*@ \label{lst:sens_xa} @*)
  x = 3; (*@ \label{lst:sens_x3} @*)
  return x; (*@ \label{lst:sens_ret} @*)
}
void fun2(void) {
  fun(5);
}
\end{lstlisting}
\end{tabular}
\end{center}
\caption{Code used for demonstration of analyses sensitivity}
\label{fig:lst_sens}
\end{figure}

Given the example above we are able to describe the following three basic
properties we usually consider when we are choosing an analysis
implementation:
\begin{itemize}
\item One implementation may take into account order of program statements,
another may completely ignore the line numbers. For our code this would mean
the analysis will report that possible values in \code{x} at
line~\ref{lst:sens_ret} is either simply 3 (the last store to \code{x}
before \code{return} proper), or with considerably less precision any of 1,
value of \code{a}, or 3 (the analysis does not care about the order, hence
takes and reports them all).  This is called \emph{flow sensitivity}, or
\emph{insensitivity} respectively.

\item An analysis may or may not remember conditions used for branching. If it
does, in our example, \code{x} is then reported to be one of 1, 2, or 3,
because \code{a} was noted to be 2 before the assignment on
line~\ref{lst:sens_xa}. In the other case, \code{x} is announced to be 1, 3,
or whatever value the analysis previously computed \code{a} can be. We have
just described \emph{path sensitivity}.

\item An analysis is called to be \emph{context-sensitive} if it considers
calling context (call-stack). In other words whether the analysis is inter- or
only intra-procedural. In our example, a context-sensitive analysis says
\code{x} can be 1 or 3 since the condition \code{a == 3} is false on
line~\ref{lst:sens_cond}. The counterpart would say \code{x} can be any
value due to the assignment on line~\ref{lst:sens_xa} (the analysis has to
assume \code{a} can contain whatever value to be complete).
\end{itemize}

\subsubsection{Types of Analyses}\label{sec:types_anal}
Analyses are also divided into sets of what information they provide. There
are two properties in this category presented below. More on those are in
section~\ref{sec:data-flow}

\begin{itemize}
\item \emph{May/must:} Sometimes we are interested whether some result given
by an analysis must hold definitely or may hold sometimes. If we get back to
the example of Figure~\ref{fig:lst_sens} and an analysis says \code{x}
\emph{must} be a constant at line~\ref{lst:sens_ret}, we can just return that
constant and eliminate all the preceding code. This would not be possible with
an answer ``maybe'' given by the \emph{may} analysis. May analysis is useful
on the other hand for example if we are interested whether a divisor in a
division may be sometimes zero. In that case we report an error.  Must
analysis will not necessarily find this bug.

\item \emph{Backward/forward:} for flow-sensitive analyses, the way of
propagation of the information is important. We may want to know at each line
of a program what are the variables used further in the code (to see whether
we need to hold its value still in a register for example). This needs going
from backwards and propagate this information to the beginning of a function
where the variable is defined. At each line we can see that information then.
On the other hand to see at each line which variables are already set to any
value, we need to propagate the information forwards.
\end{itemize}

\subsubsection{Other Views}
Of course the list above is not complete. There are many other categorizations
we do not list in here. In short, we can mention white-box versus black-box
where the algorithm either knows the internals of the checked projects or not.
For black-box, an analyzer just can start finding bugs, but will most likely
cover less code than with a white-box analyzer that has to find some knowledge
about the code first. An analogy might be seen in checking a car prior to
buying. Asking a seller for taking a ride, check whatever we see and check
whether an output from that is as expected (speedup after pressing the
throttle pedal for illustration) is black-box because we are unaware about the
internals. Asking a technician to inspect the engine of the car and check
electronic and other systems is white-box, because he has the knowledge about
the internals. There is a mid-way called grey-box testing. The technique
requires only an overview of the checked code, not full understanding of what
is going on in there.

Another angle of view might be hunting functional versus performance issues.
Once a project is working nicely with respect to the demanded functionality,
we may still want to look where we can speed the implementation up. This is
often with operating system kernels where their behaviour has to be
transparent and as fast as possible, its effects to become invisible at best.

\subsubsection{Code Parsing}
While we presented the terms above directly on programming languages
nicely readable by people, analyses are performed on an abstraction level
better understandable by computers and conveniently operated by analyzer
writers. There are several structures used in code analysis and compilation.
This section will describe the process of creating these structures from the
human readable format (programming language) and description of the created
structures proper.

First, to be able to understand text given on input, we must know its rules of
use. Every character, word or sentence have to have a predefined rule with
definition of their meaning. In natural languages this is usually written down
and marked as \emph{grammar} of that language\footnote{Not to be confused with
formal definition of grammar at the moment.}. The same holds for computer
programming languages. For each programming language, we have a set of rules,
where we define allowed characters (usually basic alphabet with parenthesis,
curly braces, and operating symbols), how words are created (what are
identifiers, reserved words etc.) and how well-defined sentences look like
(expressions, statements). This is accompanied with an explanation what each
sentence mean (semantics). So when we write for example \code{char X;} in C,
we know we have declared a variable which is capable to hold at least 8 bits
(an ASCII character in other words).

Each programming language has its more or less formally defined grammar. For
instance, for C, an ISO standard exists~\cite{ansi_c}, the same as for
C++~\cite{ansi_cpp}. This is not the case for Java, where the rules are
defined by the Sun's implementation, so it is a \emph{de facto} standard. Sun
tried twice to standardize Java, but they withdrew the drafts both
times~\cite{java_non_std}. Remember that even if a standard exists, there is
no force to push compiler implementers to adhere to that standard. This is
seen in the C family and its compilers. What is compilable by GNU's C compiler
is not necessarily understood by the Intel's compiler for example.

Having the language grammar (standard at best), we can start writing a
translators that will read source code in the human readable form on input and
produce output which is easy to work with by computers/programmers. Such
translators are called \emph{parsers}. The whole process of parsing is
depicted in Figure~\ref{fig:parsing_process}.  As we can see there are several
stages. In further text we will deal with the phases one by one. This will
happen on the basis of the C programming language.

First, the input is read by \emph{lexical analyzer} (lexer) character by
character. It consults with language grammar how proper words are formed by
characters and it starts creating so-called \emph{tokens}. Tokens are
character sequences that are marked by information whether they are text, a
number, reserved keyword (\code{int}, \code{const}, \dots), identifier
(\code{some_variable}), or any other kind defined in the grammar. We can
see how this was done on our example in Figure~\ref{fig:parsing_codes}~(b).
Keywords, identifiers and numbers are prefixed by type, braces and parenthesis
are converted to special names, the same as for operators.

A list of these tokens is then passed to the input of \emph{syntactic
analyzer} (parser). It investigates the language grammar again and looks
whether the token sequences are correct. In case they are, parsers usually
build an \emph{abstract syntax tree} (AST). There, all the irrelevant
information is forgotten (abstracted from the original language) and the rest
is stored in a tree as is depicted in Figure~\ref{fig:parsing_structs}~(a). At
this abstract point we have no clue what the original language was. Compilers
usually have separate modules for different languages, called
\emph{frontends}, that parse the input and create one pre-defined form of AST
from all diverse languages. The rest of the compiler pipeline then can be the
same for all of them and can run the same optimization for them all. The tree
is held in the compiler through the whole compilation process. It is usual
that further passes in the pipeline add more information to the nodes of this
tree. The very common one is a position of the nodes in the source code for
the debugging purposes. While ASTs hold information about the whole parsed
input, in the figure we ignore other functions, global variable declarations
and so on. There is only a subtree corresponding to our function from the
example in that figure.

We can see that we can derive any information we need about the program from
the AST. As an example we have built a \emph{control flow graph} (CFG) of that
function in Figure~\ref{fig:parsing_structs}~(b). Control flow graph of a
function is a directed graph where nodes contain statements of that function
and edges contain conditions on edges coming from branches. We can see the
control flow of \code{fun} in our example. We also refer back to the to AST
using numbers to see where the statements come from (every node or subtree in
an AST have its rule to create nodes in a CFG). The entry point of the
function is the node with no predecessors, the exit point (return from a
function) is the node with no successors and a branch is a node with more than
one successor.  It is not uncommon, for easier explanation of some algorithm,
to put statements on edges. Control flow graph is not the only structure
derived from the abstract syntax tree. We can derive symbol tables, type
definitions, perform semantic analysis to verify more than syntactic rules
etc.

It is not easy to write a correct lexer and parser by hand. Nonetheless some
projects tend to do so. \texttt{gcc} C compiler is one example of this. Over
the years the parsing part stabilized and was optimized to be fast. However
there also exist tools that help to generate lexers and parsers automatically.
Programmers just need to write down a formal grammar or otherwise defined
rules, specify tokens and push the button. The tools convert the formal
grammar to code that takes the language on input and produces an AST on the
output. Usually there are arbitrary actions allowed to be added to each node
and libraries to traverse the tree as a whole. It also depends on the type of
formal grammar preferred by the programmer.  There are various tools with
regard to this. For LL(*) grammars, one of the most known tools is
antlr~\cite{antlr4}. Flex and Bison~\cite{flex_bison} are used to generate LR
parsers. This introduction does not pretend to be complete as we do not need
more information for the purposes of this thesis; more on parsers can be
learnt from~\cite{dragon,muchnick}.

\section{Specifics of Large Projects}
In an ideal world, every code checking tool should be capable to parse and
check any project that is out there. Unfortunately, that is not true in the
real world. Tools often cannot even read source codes of a project or cannot
find any error there. This happens when the tools, or the project, are highly
specialized for example. However when a tool is ready to perform checking
already, there is still a question: where to find the files to check? Should
it just recursively crawl a specified directory and check all sources in
there? But what if some source file includes another one and the inner is not
compilable on its own as happens in the Linux kernel? There are two answers to
that question: (a) ask the user to provide a list of files, or (b) add a
support to the tool for a build system or integrated development environment
(IDE) which the project is developed in.

Having a list of files is not sufficient. Projects tend to pass special
directives to a compiler such as paths where are header files, to define some
macros on the command line, or to pre-include a specific files and so on.
Again to be able to even parse the sources, we have to ``steal'' this
information from the build system or ask a user to provide that. At this point
it starts being infeasible for a large project. A single user, it need not
even be a developer of the project, may somehow generate the file list, but
will not definitely know all the flags passed to the compiler. The tools
definitely need to \emph{support build systems}.

So at this point we know what to check and how to correctly obtain an
intermediate code structure from the source code. While the code can be
self-contained so that we have all the information we need for the analysis,
we have to bear in mind that all this information will barely fit into the
memory. In other words, when parsing large project we have to think twice how
and which information to store and where. The size also correlates with speed.
The analyzer has to be relatively fast to be able to handle thousands files.

The best code practices suggest to split functionality among smaller logical
pieces for better readability and maintainability. Programmers usually put
these pieces into separate functions. However these pieces alone are not
useful when checking for errors. They are potential source of false positives.
For example one function can lock a lock, another one unlocks it and they are
called in a sequence from some other one. The two alone would generate a
false-positive. But when the three are analyzed together, no false alarm is
reported. This drags in the need of inter-procedural variants of analyses used
by the checker. All the above are the specifics the tools have to deal with
somehow so that they are usable on large project. And this is indeed a
challenge!

\section{The Goals}
The primary goal of this thesis is to research deployment of static analysis
on large, critical, software projects. In particular, we aim specifically at
the Linux kernel in some sections. We considered contemporary approaches used
for code analysis and tried to apply them to a large code base. Based on the
results we obtain, we would like to discuss pros and cons of the techniques.
This will result either in a design of new techniques or improvement of the
contemporary ones (or both).
