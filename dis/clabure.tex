\chapter{Building Bug-database}\label{sec:clabure}
During our plays with \stan in Chapter~\ref{sec:stanse} and \symbio in
Chapter~\ref{sec:fmics}, we were slowly collecting a set of reports classified
as errors or false positives. We were pondering what to do with these reports
and imagined that there are many successful bug-finding\footnote{We
deliberately use term ``bug'' in the paper. It stems from the need of the
database to comprehend for example coding style violations. In those cases,
commonly used terms like ``failure'' or ``fault'' fail to apply.} tools based
on various program analysis methods appeared during the last ten years.  None
of them is perfect. Each tool either reports both real errors and false
positives, or it discovers only a part of real errors. We feel the necessity
of a basis for improving or evaluating the tools.  One needs to run the tool
on some source codes and then analyze the obtained bug-reports, i.e.\ classify
them as false positives or real errors, and find errors in the sources that
were missed by the tool. This work is usually tedious and time consuming,
especially when one tunes or studies performance of a tool for real software
projects. The tedious work can be avoided if suitable benchmarks,
i.e.\ programs with information about their errors, are available.

There already exist benchmark suites consisting of small synthetic programs
like \textsc{Abm}~\cite{abm}, \textsc{BegBunch}~\cite{begbunch},
\textsc{Srd}~\cite{srd}, and those published by Kratkiewicz~\cite{KRA05} and
Chatzieleftheriou and Katsaros~\cite{CHK11}. \textsc{BegBunch} does not
provide only small benchmarks. It also provides another benchmark consisting
of real-world programs. There are others to provide large benchmarks like
\textsc{BugBench}~\cite{bugbench} and \textsc{Faultbench}~\cite{HW08}. In
\textsc{Abm} and benchmarks of Kratkiewicz and Chatzieleftheriou and Katsaros,
small synthetic programs contain explicitly marked relevant program locations.
They are marked either as erroneous or safe. Nonetheless, \textsc{BegBunch}
marks only known real errors. The situation is different for benchmarks with
real-world programs. \textsc{BugBench} contains marked test cases
which trigger or do not trigger errors. \textsc{Faultbench} provides both real
errors and false positives mixed together. They are collected from
\textsc{Findbugs}~\cite{findbugs} and sorted according to priority levels
assigned to the reports by the tool. The benchmark suites discussed so far
consider different error types and provide their own error taxonomies with
exception of suites \textsc{BegBunch} and that of Chatzieleftheriou and
Katsaros. There, the error types are linked to the \emph{Common Weakness
Enumeration} (CWE)~\cite{cwe}.

As far as we know, there is no benchmark suite containing big real-life
projects with a significant list of uniformly described bug-reports
classified as real errors or false positives. Our benchmark suite should fill
this gap. We named the benchmark \cdb as an abbreviation of Classified Bug
Reports Database.

\cdb initially contained only a single project, namely the Linux kernel
2.6.28. We have collected about 1150 bug-reports of 11 error types produced by
several bug-detection tools run on the kernel. The reports have been manually
classified as either real errors or false positives by skilled programmers
with a help of Linux kernel developers. In fact, it would be sufficient to
store only real errors assuming that we know all of them (and thus we can
assume that all other bug-reports are false positives). As this assumption is
completely unrealistic for a real-life project, we store both real errors and
false positives.

Later, the database was augmented by another two projects. We added examples
from the Competition on Software Verification 2012~\cite{sv-comp12} and
2013~\cite{sv-comp13}. Adding the examples to our database was relatively
cheap as the examples are marked as false positive or real error. This is
because they need to decide if the tools hit or miss. Despite it is the same
competition over two years, thanks to it popularity, the version from 2013
brought almost ten times more examples: 286 from 2012, 2337 from 2013.
Although there are minimalistic examples in both of the sets, most examples
are extracted from the large projects such as the Linux kernel.

This chapter is based on our earlier work published in the Springer's
Lecture Notes in Computer Science~\cite{claburedb}. We presented the work at
the 14th International Conference on Verification, Model Checking, and
Abstract Interpretation.  The database can be downloaded in the
\textsc{SQLite~3} format for local use under the \emph{Open Database License
v1.0}~\cite{ODbL} or accessed via a web interface at:
\begin{center}
  \url{http://claburedb.fi.muni.cz/}
\end{center}

\section{Database Structure}\label{sec:cdb_structure}

The database is designed to accommodate various kinds of errors from diverse
projects and project versions. As projects can be written in arbitrary
programming languages, can contain very specific kinds of errors, can be
maintained by different teams, and can be interesting for distinct user
groups, we decided to have each project % and version
in a separate sub-database. A sub-database comprises information about
considered error types, bug-reports, users, tools, and relations between
them. There are two main tables in each sub-database:
\begin{description}
\item[\textsf{error\_type}] This table keeps a specification of considered
  kinds of errors. We will outline some of possible types later in
  Section~\ref{sec:cdb_error_type}. The table contains a name of the type, short
  description and a reference to the CWE if it exists (see later).
\item[\textsf{error}] Each line in this table corresponds to one
  bug-report. It is specified by the error type, location (usually file and
  line), URL for reference (to provide more information), classification
  (false positive, real error, unclassified), confirmation (an argument
  supporting the validity of the bug-report classification, e.g.~a commit ID
  of the corresponding bug-fix), user who inserted the entry, and timestamp of
  the moment of insertion.
\end{description}

\section{Database Frontend}\label{sec:cdb_frontend}
Initially, we only intended to use the \textsc{SQLite} database and only on
the command line, with use of some powerful scripts. But we realized that it
would be also nice to present the reports somehow apart from the command line.
And since we wanted to make the whole benchmark public, we implemented a
simple web interface in \perl using \emph{Common Gateway Interface} (CGI).
This was a nice proof of concept, but a terrible interface. A quest in the
form of a BSc thesis topic was put in our university system. Shortly after,
Marek Hornák was lead by the author of this PhD thesis to improve the
interface in the scope of his bachelor thesis (which is in Slovak so we do not
cite it here explicitly). He wrote a complete replacement in \textsc{Ruby on
Rails} and defended his thesis successfully.

Nonetheless the frontend was really slow for large data. Seconds to perform a
single query was unbearable. So the author of this PhD thesis rewrote the
code of the frontend, fixed some holes in it, and made it public. He also
tuned the interface part of the frontend as he has some knowledge about Web
2.0 from previous work~\cite{web2_isd}. And since the requirement for the BSc
thesis was not to support more project, this aspect was considered while
rewriting the system.  The current frontend now fully supports multiple
projects, is fast, and we believe it is safe. The frontend also allows for
downloading the database in the \textsc{SQLite} format for local use under the
\emph{Open Database License v1.0}~\cite{ODbL}.

There is an added value in the frontend. We designed it so that a user can
actually browse the code where a particular bug is. The files are not stored
in the sub-database, but they are on disk. Since a project can have a large
number of files, we implemented a tree structure for storing files. Content
of every file in the project is hashed using SHA1, which is forty hexadecimal
characters. The hash is split into two parts. The first is only two characters
and it specifies a directory, the other 38 characters are used for a file name
in this directory. For example, if the hash of a file content was
\code{12345678...abcdef}, we would store the content of the file in the
directory \code{12} as the file \code{345678...abcdef}. So we leverage the
hashing and underlying filesystem to create a two-level tree for faster
lookup. This also ensures that no files (e.g.\ from the same project with a
different version) waste space as they have the same hash. We have not
encountered a hash conflict yet. If it happens we will solve it by some known
technique to circumvent the problem. The sub-database has to hold the hash
too. It is in the \code{error} tables as \code{loc_file_hash}. The tree
structure of files is inspired by the revision control system \textsc{Git}.

On the top of that, the content of the files can be compressed by
\textsc{Lzma2} and the suffix \code{.xz} is appended to the hash above.
Despite \textsc{Lzma2} is slow for inflating (we inflate once), it is fast for
deflating and provides a very good compression ratio. Further, the \cdb
frontend supports source highlighting. Either it is precomputed, compressed
and stored with the suffix \code{-hl.xz} or done later by the frontend
on-the-fly.  So when the frontend is looking for a file to display, it first
looks for \code{hash-hl.xz} (considering the first two characters as a
directory). If there is no such file, it tries \code{hash.xz} and highlights
it if finds one. If it is does not, no file content is shown to the user.

For illustration of the frontend, Figure~\ref{fig:cdb_error} depicts a list of
seven bug-reports of type \emph{Double Resource Put} as presented in the
\cdb's web interface at \url{http://claburedb.fi.muni.cz/}. Four of these
reports are false positives (green), the other three are real errors (orange).
One of the reports is expanded, so that we can see its details and highlighted
source code with a marked error line.

\section{Current Contents of the Database}\label{sec:cdb_data}

\cdb currently contains bug-reports from three projects. The first one are
bug-reports produced for the Linux kernel 2.6.28. We have chosen Linux kernel
for several reasons: it is a big (over thirty four thousands files with nearly
fifteen millions lines of code in total), real-life, self-contained, and
open-source project with several public sources of information about errors.
Moreover, the kernel contains many distinct parts (core, hardware drivers,
filesystems, networking etc.) that cover some standard application areas of
program analysis tools. The other two projects are from the Competition on
Software Verification 2012~\cite{sv-comp12} and 2013~\cite{sv-comp13}.

\subsection{Sources of Bug-Reports}\label{sec:cdb_sources}

We used four code analysis tools to gather bug-reports for the Linux kernel:
\clang~3~\cite{clang}, \smatch~1.59~\cite{smatch}, \stan~1.2~\cite{stanse},
and one commercial tool which wanted to stay in anonymity. Note that \clang
does not natively detect locking errors in the kernel. There is only a checker
for locks used by developers of user programs. It is called
\code{experimental.unix.PthreadLock}.  Hence, we slightly modified the checker
just to understand kernel locking functions.

Another source of bug-reports in the database is the Internet. In particular,
similar to the database frontend, there was a quest for a web crawler.  Jozef
Ce\v{l}uch took the opportunity and was lead by the author of this PhD thesis
to write a web crawler and a bachelor thesis about that (it is in Slovak, so
not referenced explicitly here again). He defended the thesis thereafter.  The
web crawler was used to explore the Internet and find reported kernel bugs. It
crawled kernel and Novell bug tracking systems and many mailing lists.

Finally, \cdb comprises few bug-reports found manually. They were extracted
from the kernel commit messages while the author of this thesis was going
through some random kernel commit messages.

\subsection{Linux Kernel Error Types} \label{sec:cdb_error_type}

We have analyzed outputs of available tools applicable to the Linux kernel
and we have decided to focus on the following eleven error types. Nine of
these error types are present in the \emph{Common Weakness Enumeration}
(CWE)~\cite{cwe} and we provide their CWE identification numbers for
reference. The two remaining error types are specific for the Linux kernel
and thus they are not covered by CWE. They can be seen as a violation of the
kernel coding policy.

The list of considered error types follows. For each error type we
explicitly describe the program location associated with an error of this
type. This is to evade misinterpretation, because diverse tools can associate
the same error with different program locations. For example, an error
present at an outgoing edge of a function may be associated to the opening
brace of the function, the closing brace, or to the corresponding
\code{return} statement.

\begin{description}
\item[BUG/WARNING] (CWE 617) Developers often inject \emph{assert}s
  to their code, e.g.\ to ensure that their function is given correct input or
  the system or program is in an expected state.  For example, a destroy
  function of an object \code{obj} can contain a line
  \code{assert(!obj->active)} to check that the object to be destroyed is
  inactive already. An error occurs if a condition of some \code{assert} is
  violated.\\
  \emph{Error location:} the line with the violated assertion.
\item[Division by Zero] (CWE 369) The code contains a division
  statement and the actual value of the divisor is zero. \\
  \emph{Error location:} the line with the division.
\item[Double Lock] (CWE 764) Some lock is locked by a thread twice
  in a row and it leads to an inconsistent lock state. Note that we ignore
  double locks of semaphores as this may be their intentional application.\\
  \emph{Error location:} the line with the second call of lock.
\item[Double Unlock] (CWE 765) Some lock is unlocked by a thread
  twice in a row and it leads to an
  inconsistent lock state. Again, we ignore double unlocks of semaphores.\\
  \emph{Error location:} the line with the second call of unlock.
\item[Double Free] (CWE 415) A freeing function is called twice on
  the same address while no reassignment to the passed pointer
  occurred between the two calls. Most allocators can detect this and usually
  kill the program or system. \\
  \emph{Error location:} the line of the invalid (second) free.
\item[Memory Leak] (CWE 401) Some code inadvertently omits to free a memory
  which was allocated previously. \\
  \emph{Error location:} the line with the allocation statement.
\item[Invalid Pointer Dereference] (CWE 465) The code tries to
  access some memory, but the pointer used is invalid. It may become invalid
  in many ways. For example, the pointer may point to a released memory (known
  as \emph{dangling pointer} or \emph{use after free}) or it can be set to
  NULL (known as \emph{NULL pointer dereference}) or uninitialized. Another
  source of the problem may be accessing an array out of bounds. \\
  \emph{Error location:} the line of the dereference.
\item[Double Resource Put] (CWE 763) The code requests one copy of a
  resource (e.g.\ a structure holding hardware status) from a system, but
  there is more than one attempt to put that resource back to the
  system. Like in the example of Figure~\ref{fig:cdb_double_res}. There is
  a request for a PCI device on line~\ref{line:cdb_get} and two reclaims
  of the device on lines~\ref{line:cdb_put1} and \ref{line:cdb_put2}. The
  second is invalid of course. \\
  \emph{Error location:} the line of the second put.
\begin{figure}[htb]
  \begin{center}
  \begin{tabular}{c}
  \begin{lstlisting}
struct pci_dev *pdev;
pdev = pci_get_device(vendor, device, NULL);(*@\label{line:cdb_get}@*)

  work_with_pdev(pdev)
  pci_dev_put(pdev);(*@\label{line:cdb_put1}@*)

  \end{lstlisting}
  \end{tabular}
  \end{center}
    \caption{An example of the double resource put error.}
    \label{fig:cdb_double_res}
\end{figure}
\item[Resource Leak] (CWE 404) The code requests a resource from a
  system, but omits to reclaim that back. For example, the error would occur in
  the example of Figure~\ref{fig:cdb_double_res} if we deleted
  lines~\ref{line:cdb_put1} and \ref{line:cdb_put2} with both
  \code{pci_dev_put} calls. \\
  \emph{Error location:} the line of the request/get.
\item[Calling Function from Invalid Context] (not in CWE, Linux
  kernel specific) Some function is called at an inappropriate place or within
  an invalid context. This includes calling functions like sleep or wait
  inside spin-lock critical sections or in interrupt handlers. This is
  considered to be an error because results of such a call are undefined: the
  system may become unresponsive or may crash for instance. \\
  \emph{Error location:} the line of the inappropriate call.
\item[Leaving Function in Locked State] (not in CWE, Linux kernel
  specific) This error type originates from the kernel requirements that a
  process has to release all locks before returning control to userspace.
  This error occurs if a function has an execution path where some lock is
  locked and left locked when leaving the function. It is considered to be
  an error (violation of the kernel coding policy) as such an execution may
  lead to a deadlock on the next invocation of any function wanting to take
  the same lock. \\
  \emph{Error location:} the line of the corresponding \code{return}
  statement or closing brace of a function if there is no \code{return}.
\end{description}

Many of the types described above fit also other projects. But even if a user
of the database decides to create a new sub-database for their project, the
database is not limited to a bounded set of error types. The user still can
define arbitrary error types because error types are stored in the
sub-database too. For example for the software competition sub-databases we
have a single error type ``Reachable Error Location'' which is very similar to
``BUG/WARNING'' above.

\cdb need not be used only for gathering information about code analyzers.
It can be also used for comparing parts of the included projects. We used \cdb
to compare kernel subsystems with respect to error and false positive
occurrences. The results are depicted in Table~\ref{tab:cdb_stats}. For the
presentation purposes we excluded subsystems for which we had less than ten
reports in the database. Those were security, virtualization, or
inter-process-communication subsystems for example. If we base on the findings
of the tools we have reports from, we can see that the tools were very
unsuccessful in finding real errors in the networking subsystem while they
found relatively many false positives in there. This suggests the subsystem
contains far less errors. Further, drivers and filesystems are at the average
in the number of errors they contain (around twenty per million lines). It
raises for the sound subsystem a bit. But surprisingly, the core code
performing memory management, maintaining running system and architecture
specific was easy to find errors in. But this holds for false positives in
there too. Overall, tools produce many reports there. Continuing with false
positives, the sound subsystem is the only one where the tools found more
errors than false positives. Filesystems contain many false positives too.
According to our knowledge gained during the classification, this is caused by
complex locking rules inside filesystem code. The tools could not often cope
with a conditional call to lock and yet, located in a loop.

\section{Use of the Database}\label{sec:cdb_use}

The web frontend is easy-to-use and we think it needs no detailed explanation.
On the initial page, one chooses a project which they work later with. The
interface allows for sorting by several criteria, searching, filtering with
respect to tools and error types, showing highlighted source codes with the
line where the report is, and more.

Even though the web frontend is very practical tool to work with the data, it
is not the intended use of \cdb. The typical use is tuning and evaluation of a
bug-finding code analysis tool. To evaluate such a tool, one chooses a project
(or its part) from our benchmark suite and run the tool on these source codes.
As the second step, the sub-database of classified bug-reports corresponding
to the chosen project is downloaded from \cdb.  The installation for this use
case is easy. The \textsc{SQLite} sub-database needs only a client which is
available for many platforms, Linux, Mac OS, and Windows including.  Use of
the client is similar to other databases. It understands \emph{Structured
Query Language} (SQL) queries as many other databases. The database is opened
by \code[language=bash]{sqlite3 database.db} and it awaits SQL queries. We
provide some examples of use in Figure~\ref{fig:cdb_sql}.
\begin{figure}[htb]
  \begin{center}
  \begin{tabular}{c}
  \begin{lstlisting}[language=SQL, columns=flexible]
> SELECT loc_file, loc_line FROM error;
mm/vmalloc.c|292
net/core/skbuff.c|147
...
> SELECT COUNT(id) FROM error WHERE marking > 0 AND
  id IN (SELECT error_id FROM error_tool_rel WHERE tool_id = 1) AND
  id NOT IN (SELECT error_id FROM error_tool_rel WHERE tool_id = 2);
105
  \end{lstlisting}
  \end{tabular}
  \end{center}
    \caption{An example of SQL queries to a sub-database.}
    \label{fig:cdb_sql}
\end{figure}
First, we query all errors in the sub-database and then we count all real
errors found by tool~1 and not found by tool~2.

With the sub-database, the bug-reports produced by the newly developed tool
are compared to the downloaded data. We provide a script to automatically
compare the results to the sub-database. It is only required to convert the
tool's output to a predefined format and run the script. The other option is
to write a similar script handling the output of the tool of course.  So when
one compares the data, they immediately get a classification (real error/false
positive) of all the reports matched in the database and also information
about missed real errors if there are any. It remains to manually classify the
bug-reports unmatched with the database content. The numbers of produced false
positives, detected and missed real errors can be further statistically
processed according to methodologies suggested elsewhere~\cite{HW08,
begbunch}. They produce several metrics including accuracy and precision. The
metrics, missed errors, and false positives are valuable inputs for tuning the
tool. 

Finally, the user is kindly asked to update classification of unclassified
entries and to insert newly discovered bug-reports with their classification
into the database too.

\paragraph{Scripts Summary} We provide several scripts to use with \cdb. Here
we list them with their description. Most of them are written in \perl. They
use our common helper module for \perl \code{Helper.pm}. It makes them really
simple.
\begin{description}
\item[\textsf{cmp\_results}] It takes a sub-database and tool's reports on the
input. The input should be fed by lines in the form of \code{file\\0line},
where \code{file} is a zero-terminated string. The script takes these lines
and compares them one by one against the sub-database. Finally, it prints
statistics about the tool's reports. The results include how many errors
and false positives were matched and how many reports were missed by the tool.

\item[\textsf{fill\_hashes}] This script is used to fill the SHA1 hashes into
the sub-database as needed by the frontend (see
Section~\ref{sec:cdb_frontend}).  On the input, the script accepts a list of
files and hashes in the form \code{file\\0hash} and fills the sub-database
accordingly.

\item[\textsf{generic\_filler}] This is a configurable script that inserts
entries to the sub-database. It accepts input similar to
\textsf{cmp\_results}: \code{file\\0line\\0classification}, where
classification is a number. -1 means false positive, 0 unclassified, and 1
real error.  The script accepts many parameters to specify error type and
subtype, project and its version, tool and its version. The output of any tool
can be put into the sub-database using this script when the output is
converted to the format above.

\item[\textsf{prepare\_db}] This scripts initializes a new sub-database to be
used in \cdb. After \textsf{prepare\_db} the sub-database is ready to be
filled by one of the fillers.

\item[\textsf{stanse\_filler}] It is similar to \textsf{generic\_filler}, but
it directly parses \stan's XML report files (as are described in
Section~\ref{sec:stan_framework}) and fills them to the sub-database.

\item[\textsf{web\_crawler}] A simple web crawler that takes a list of URLs
and analyzes them for error reports. More concise and automatic crawler was
implemented by Ce\v{l}uch as we explained in Section~\ref{sec:cdb_sources}.

\end{description}

%{{{ Future Plans

\section{Conclusion and Future Plans}\label{sec:cdb_concl}

We have presented \cdb: the database of classified bug-reports. It is
intended as an open platform for sharing valuable benchmarks for developers
of program analysis tools. The database already contains a large collection
of uniformly described bug-reports produced by various tools on a large
real-life project, namely the Linux kernel 2.6.28. There are also reports
taken over from two years of a software competition.

\medskip
We plan to extend the database in several directions. Namely, we intend to
collect more bug-reports and to support more error types in the Linux kernel
project. For example, we plan to add error types connected to deadlock and
livelock.  Further, we plan to add other real-life projects. We also plan to
extend the web interface to enable developers of program analysis tools to
maintain and augment the database.

In general, the future of \cdb depends on a feedback from the program analysis
community. At this point, we would like to encourage the reader to contact us
at \href{mailto:claburedb@fi.muni.cz}{\nolinkurl{claburedb@fi.muni.cz}} if
they have any comments, suggestions (for example which projects should be
added to the database), questions, or sources of bug-reports for the Linux
kernel or other interesting projects. We would especially welcome people
willing to participate on creating and maintaining the database content for
another project.
