\chapter{Stanse}\label{sec:stanse}
At the beginning of the PhD studies, we were surveying the current known
techniques used for software checking. Of course many techniques were available
and some software projects were in the development. One of the papers to
really stir the interest was about metacompilation~\cite{metacomp}. It showed
that static analysis can efficiently find many interesting errors in
real-world code. This work eventually led to a successful commercial tool
called \textsc{Coverity Prevent}~\cite{coverity}. Over the years, several
other successful tools, like \textsc{CodeSonar}~\cite{codesonar} or
\textsc{Klocwork}~\cite{klocwork} appeared.  However, such fully-featured
tools are neither free to obtain, nor is their code available (e.g.\ for
developing new algorithms or tailoring the existing tools to specific tasks).
The existing free tools were usually severely limited in what they can do,
especially when it comes to checking real-world C code such as the Linux
kernel. These include \textsc{Uno}~\cite{uno}, \textsc{Sparse}~\cite{sparse},
and \textsc{Smatch}~\cite{smatch}.

We decided to implement an open source robust framework for two purposes:
finding errors in software and to make easier development of diverse static
analysis algorithms. We named the framework \stan. It is an abbreviation of
static analyser and was deliberately chosen not to be any English word for
easier searching on the web. To fulfill \stan's role, we decided to write it in
Java as it is relatively easy to learn and is not prone to some pointer errors
class present in other languages. We tried to design it to be fully generic to
allow for implementation of many types of analyses. An implementation of such
an algorithm within the framework is called a \emph{checker}. We implemented
some checkers too, so \stan already contains four of them. The most concise
checker was inspired by the success of \xgcc~\cite{metacomp} and its
property-description language \textsc{Metal}~\cite{metal}. This work
implements the abstract interpretation as is discussed in
Chapter~\ref{sec:ai}.

\stan is specifically developed to check software written in C. Its parser is
powerful enough to handle the code like the whole Linux kernel. So an
implemented checker can be immediately evaluated on large real-world software.
\stan has also \emph{graphical user interface} (GUI) to make the work with the
framework easier. This chapter covers the development of \stan including the
theory behind and reports of successes and failures. \stan was released and a
paper about the framework published in the Springer's Lecture Notes in
Computer Science~\cite{stanse} some time ago. We presented the paper at Annual
Doctoral Workshop on Mathematical and Engineering Methods in Computer Science
2011.  Since then, other similarly powerful frameworks were published.  We
provide \textsc{CPAchecker}~\cite{cpachecker} and
\textsc{Frama-C}~\cite{framac} as examples.

This chapter is structured as follows. Section~\ref{sec:stan_framework}

\subsection*{Configuration}

The \stan framework contains data structures capturing a configuration of an
analysis to be performed. We always need to know what source files to analyse
and by what checkers. That information we call \emph{configuration}. 

There are several ways how to tell \stan which source files to analyse. Besides
the standard possibilities (a given file, all files from a given
directory, all files listed in a list), \stan can also derive all
the necessary information from a project \code{Makefile}. In this case,
\stan also remembers compiler flags for preprocessing purposes.

The user must also specify checkers which should be run on the chosen source
files.  There can be many checkers running simultaneously in \stan.
Nonetheless when they want to share data, thy cannot proceed independently.
Checkers themselves can also be configured through their own configuration
files as can be see in Figure~\ref{fig:architecture} for \textsf{Checker N}.
The configuration can be passed to the \stan framework either via command line
arguments, or using the graphical interface.

\subsection*{Parsing Source Files}

The \stan framework can process source files written in C, more
specifically in the ISO/ANSI C99 standard~\cite{ansi_c} together with most of
the GNU C extensions~\cite{gcc}. This allows \stan to process software
projects like the Linux kernel. We also provide an experimental implementation
to support C++~\cite{ansi_cpp}.  The parsing pipeline includes preprocessing
of source files as can be seen at the bottom of Figure~\ref{fig:architecture}.
This is done by the standard GNU C preprocessor \textsc{Cpp}.  It is important
to note that we do not parse the source files in a sequence one by one as they
appear in a configuration. Since we use streaming (which we discuss later),
the pipeline is applied as on-demand for each individual source file.

The parser used in \stan is generated using the \textsc{AntLR}
tool~\cite{antlr4} from our own annotated C grammar. The reason for us to
write our own parser was that at the time we started developing
\textsc{Stanse} we could not find a free parser suitable for our purposes.
The main requirement was to parse most of the GNU C as the Linux kernel makes
heavy use of GNU C language extensions. Even though there were full compilers
like \textsc{Clang}~\cite{clang}, they are not still capable to handle the
kernel fully now. And in its current form, it still contains bugs so cannot
be used reliably. Our plan is to switch from our parser to \textsc{Clang} once
it becomes stable and feature complete, or any new compiler if there appears
one.

Of course there must be at least a parser which the is able to handle the
kernel, otherwise kernel developers could not run the kernel. They use
\gcc.  Unfortunately parser in this compiler, and most importantly its
internal structures, is not suitable for our purposes. One reason is that
control flow graph is built on the top of register transfer language (RTL) or
tree representation. Those are very hard to work with when it comes to the
code analysis. We encourage the reader to look into the GNU C compiler
internal manual~\cite{gcc_intern} where these are described.

\subsection*{Program Internal Representation}

Once the code is parsed, it is represented using \stan's \emph{internal
structures}: a call graph among functions, a control flow graph (CFG) for each
function, and an abstract syntax tree (AST) for the whole file. The AST is in
the form of XML for the ease of use. CFG nodes reference appropriate subtrees
in the AST. All these structures are depicted in the middle of
Figure~\ref{fig:architecture} as ``Internal Representation''. For developers
of new checkers, all these structures can be dumped in a textual form to a
file or in graphical form in the well-known \textsc{Graphviz} dot format.

Since we aim at large software projects consisting of hundreds or thousands of
modules (compilation units), it is often impossible in practice to store the
corresponding internal structures in the memory all at the same time. The \stan
framework therefore applies automatic \emph{streaming} of the internal
structures. This is currently performed on a module basis. Instead of parsing
all source modules in the beginning, a module is streamed in only when \stan
needs to access some internal structure belonging to the module. In other
words, the internal structures are constructed on demand, in a lazy manner.

If the memory occupied by internal structures exceeds a given limit, some
internal structures have to be freed before another module is streamed in. The
structures to be freed are selected using the least recently used (LRU)
approach: \stan discards all internal structures of the module whose
structures are not accessed for the longest time. If the discarded structure
is accessed again later, the corresponding module is streamed back in. Both
laziness of internal structures and streaming are completely invisible to
checkers. 

In the current implementation of streaming, each source file streamed out from
the memory is completely discarded. \stan does not back up already parsed
internal structures into auxiliary files before discarding. As a consequence,
when internal structures of the discarded file are needed again, \stan starts
the parsing pipeline of the file from scratch to recreate requested internal
structures. Although loading of previously parsed internal structures from
auxiliary files would speed up the streaming process, profiling of \stan's
performance on the Linux kernel has not shown streaming to be a performance
bottleneck. This could be easily changed if streaming performance becomes a
problem in the future, nonetheless.


\subsection*{Pointer Analysis in \stan}

Since each bug-finding/program analysis technique may have different
requirements regarding pointer analysis, the \stan framework currently
provides a developer with two pointer analyses:
\textit{Steensgaard}'s~\cite{steensgaard} and
\textit{Shapiro-Horowitz}'s~\cite{shapiro_horwitz}. See
Section~\ref{sec:pointer_analysis} for introduction to pointer analyses or
Hind's discussion~\cite{ptr_solved}.

\subsection*{Matching Language Constructs}

Many static analyses change their internal state only on some subset of
program expressions. For example, when finding race conditions in a
parallel program, one may only focus on expressions involving synchronisation,
while ignoring all others. The \stan framework therefore provides a
specification language for determining a set of program expressions.

The language defines a collection of \emph{patterns}. Each pattern is supposed
to identify a single specific kind of sub-trees in an AST of an analysed
program. Since the AST is in the form of the XML, a pattern itself is
therefore also an XML sub-tree of the AST. When we want to express division by
zero as a pattern in the XML format, we can use something similar to what is
depicted in Figure~\ref{fig:stan_div0}.
\begin{figure}[htb]
\begin{center}
\begin{tabular}{c}
\begin{lstlisting}[language=XML]
<binaryExpression op="/">
  <any/>
  <intConst>0</intConst>
</binaryExpression>
\end{lstlisting}
\end{tabular}
\end{center}
  \caption{An example of an XML pattern.}
  \label{fig:stan_div0}
\end{figure}
As we can see, we allow some of its nodes to be ``special'' to
define a set of possible sub-trees at that node, e.g.\ wildcards such as
\code{any}.

This is, nonetheless, not the only possible approach. For example, in the
METAL~\cite{metal} specification language, a C expression can be directly
parametrised to define a set. The solution we implemented exploits the fact
that checkers in \stan work with the AST intensively, and therefore
identifying expressions in terms of the AST is very practical.

\subsection*{Traversing Internal Representation}\label{sec:travers}

Although a checker may need to work with the internal structures in an
arbitrary way, most checkers walk through CFGs using some standard
strategy. To prevent unnecessary reimplementations, the most important and
heavily used traversal methods are implemented directly inside the framework.
With this functionality, one can implement a new checker (or its part) by
specifying:
\begin{itemize}
\item whether it should go through CFGs forwards or backwards, breadth-first
  or depth-first,
\item whether the walk-through should be performed inter-procedurally or not
  (if not, the function calls are ignored), and
\item a method (callback) to be called for each visited node in a CFG.
\end{itemize}
This makes implementation of new algorithms extremely simple.

For example, when a checker needs to implement a forward flow-sensitive
analysis, it may ask the \stan framework to traverse paths in CFGs in forward
depth-first manner. This can be implemented by a single call to a function
\code{traverseCFGToDepthForward}, which takes as an argument a CFG and a
subclass of \stan's class \code{CFGPathVisitor}. In this class the checker
defines the action which should be taken whenever a CFG node is visited
(already in the requested order). The checker implements the action in a
method \code{visit} of the subclass.

In addition, for those inter-procedural analyses which do not construct
summaries, \stan provides an automated traversal among different CFGs
according to function calls (involving automated parameters passing and
value returning). Again, this can be done using a single call to the \stan
framework.  

All this functionality described in this section is shown in
Figure~\ref{fig:architecture} as ``Code Navigating Libraries''.

\subsection*{Support for Function Summaries}\label{ssec:summaries}

Inter-procedural analyses typically build function summaries. Unfortunately,
these summaries may differ from one analysis to another. Nonetheless, quite
common part in building many summaries is passing formal and actual parameters
to call sites and mapping return values to appropriate variables. Therefore,
the \stan framework provides classes simplifying the parameter passing and
values returning for checkers. These classes also provide a conversion of a
given expression in the caller into an equal expression in the called
function. The conversion can also be requested in the opposite direction,
i.e.\ for returned values.

\subsection*{The Concept of Checkers}

In the \stan framework, a checker is an implementation of some concrete
static analysis technique. Each checker has an access to shared internal
structures of analysed source files. They are also provided with an access to
the algorithms providing navigation in those structures. This is done by an
interface among checkers, internal structures and libraries of the framework.
The interface is depicted in Figure~\ref{fig:architecture} right bellow the
checkers.

The checkers are integrated in the framework of \stan using concrete factory
design pattern. Therefore, to insert a new checker to the framework one needs
to implement generic checker interface and register it to the checkers'
factory of the framework. Then it gains a full access to the features of the
framework accessible through the discussed interface.  The process of
integration requires only three simple steps. The first step is to create a
subclass of \stan's abstract class \code{Checker}, say \code{MyChecker}.
The most important method to implement is \code{check}. This is a method where
the analysis algorithm should be implemented. The second step is to integrate
the newly created class into the framework. This means implementation of
\code{MyCheckerCreator}, a subclass of the \code{CheckerCreator} abstract
class. The final step is to register the class \code{MyCheckerCreator}. It
comprises adding a line \code{registerCheckerCreator(new MyCheckerCreator())}
at the end of \code{CheckerFactory.java}.

\subsection*{Processing Errors}

\begin{figure}[t]
  \centering
    \includegraphics[width=\textwidth]{images/screen-bug-tracing}
  \caption{Error trace browser in the \stan GUI.}
  \label{fig:bugtrace}
\end{figure}

Once a checker finds an error, it reports the error back to the framework in
the form of an annotated \emph{error trace} -- a path in the analysed code
demonstrating this error. This is performed using a provided datatype which
describes an error in the \stan framework. As once can imagine there might be
many code paths representing an error. So the question is which one to show to
the user. First, we tried to generate as many traces as possible from each
checker.  But this resulted in hundreds traces for a single reported error.
The user would be confused by so many traces, so we currently generate only
the first trace generated from most checker. Nonetheless, the functionality is
still in the framework and some checkers need and use it.

The framework yields three possibilities how to present the error traces back
to the user of \stan. First, they can be printed to the console. But as it is
only a textual output, the traces are not very usable for large projects.
Hence the GUI in \stan has a built-in trace browser which supports displaying
the traces.  There is an example of a trace in the GUI in
Figure~\ref{fig:bugtrace}. Also, a plug-in for integration \stan into the
\textsc{Eclipse} \emph{integrated development environment} (IDE) exists too.
It can be used for tracking errors directly when developing code. And finally,
the traces can be saved to an external file in the XML format. This XML file
has a wide variety of possible applications. For example, we supply a tool
transforming the XML file into an \textsc{SQLite} database. The database is
supplemented with a web interface allowing to browse errors in the database
via a web browser. Using the web browser or the built-in graphical error
browser, one can mark errors as real errors or false positives. \stan also
provides various statistics of errors like number of errors per checker,
frequency of errors of the same kind, percentage of false positives (based on
the user feedback).

Error reporting and tracing pipeline is depicted in
Figure~\ref{fig:architecture} to the right of interface and internal
representation.

\subsection*{Support for Large Projects}
As we already know, \stan supports large project internally. But it would be
unusable without some scripts to support all the functionality. In this
section we describe the scripts one by one. Most of them are written in the
programming language \perl.

\begin{description}
\item[\textsf{stcc}] \code{stcc} is provided as a replacement for a compiler
of an analyzed project. By using \code{stcc} instead of the original compiler
like \gcc, the project is still compiled by the original compiler, but also a
file list and compiler options used for the compiled files are stored aside.
These are then used as a configuration for \stan. This is effectively a
support of build systems as it is enough to set the environment variable
\code{CC=stcc} and compile the project as usual.  This script is inspired by
the \textsc{Sparse} tool~\cite{sparse}.

\item[\textsf{stanse\_run}] For even easier evaluation, we also provide users
with this script. It is basically a wrapper around \code{stcc}, build system,
and \stan. So it builds a project and the obtained configuration is passed to
\stan immediately. It also predefines configurations for checkers to be used
for the Linux kernel. This is enabled by the parameter \code{-k}. So to analyze
the kernel, one needs not do anything, but running this script.
\item[\textsf{st\_xml2sqlite3}] As mentioned earlier, we can transform the
reported errors into an \textsc{SQLite} database. This scripts takes care of
this task.
\item[\textsf{st\_xml\_bugcount}] When we are finished with classification of
the reports to real errors/false positives, this script reports statistics on
this topic.
\item[\textsf{st\_automaton}] For an easier design of automata, we provide a
script to convert them from the \stan's condifuration into 2 formats:
\textsc{Graphviz} dot and \textsc{Tikz} graph files. This script was used to
generate the large automaton described later in this chapter.
\end{description}

\section{Checkers}\label{sec:checkers}

In this section we describe the four currently available checkers. The first
three checkers need a configuration of project-specific parameters like
an enumeration of special function calls.  We provide a sample configuration for
all of them so that they can be used out of the box. Of course, the
configuration can be tailored to specific needs of other projects easily.

\subsection{\AC}
The \AC is heavily influenced by \xgcc~\cite{metacomp}. As can be seen from
the name of the checker, it employs finite state automata. They describe the
properties we want to check. These include checking locking discipline,
interrupt management, null pointer dereference, dangling pointers and many
others. Given such automaton, the checker tries to match the automata against
code execution. To perform that, the \AC needs to know when to perform a
transition. This is accomplished by specifying code patterns on the edges.
Since the checker works on the control flow graph and we have subtrees of
abstract syntax trees in there, patterns are AST subtrees too. It works
similar to pattern matching described in Section~\ref{sec:pattern_matching}:
we have an AST subtree with possible wildcards. An example of an automaton for
checking proper pairing of locks is presented in
Figure~\ref{fig:stan_automaton}.  \begin{figure}[htb] \begin{center}
\tikzstyle{stat} = [circle,thick,draw,minimum size=8mm] \tikzstyle{edge} =
[shorten <=1pt,>=stealth',semithick] \begin{tikzpicture}[node
distance=18mm,font=\sffamily] \node [stat] (ERR) {\textbf{Err}} edge
[edge,loop above] node[right, xshift=2mm, yshift=-2mm]
{\code{lock()}/\code{unlock()}} (); \node [stat] (U) [below left=of ERR] {U}
edge [edge,->] node[above left] {\code{unlock()}} (ERR); \node [stat] (L)
[below right=of ERR] {L} edge [edge,->] node[above right] {\code{lock()}}
(ERR) edge [edge,<-,bend right=20] node[above] {\code{lock()}} (U) edge
[edge,->,bend left=20] node[below] {\code{unlock()}} (U); \node [left=6mm of
U] {} edge [edge,->] node {} (U); \end{tikzpicture} \end{center} \caption{An
automaton for checking locking properties.} \label{fig:stan_automaton}
\end{figure} The automaton starts in the unlocked state (\code{U}). A
transition from that state is fired either when the checker matches a call to
\code{unlock} or \code{lock}. In the former case, the automaton is transferred
to the error state (\code{Err}) and the checker flags an error by generating
an error trace. In the other case, the checker simply transfers the automaton
to the locked state (\code{L}). The evaluation continues until a fixed point
is reached.

Compared to the implementation described by Engler et al.~\cite{metacomp,
metal}, our technique differs in several aspects. In particular, we do not use
metacompilation, automata are not input-language specific (a pattern matching
is used instead), and the inter-procedural analysis is done in the context of
a single input file.

\paragraph{Starting Functions}
Since we perform the checking on separate files, we hit a problem when trying
to employ the \AC on the Linux kernel. If we analyzed every function
with its calling subtree separately, we would obtain false positives.
So the problem is in fact which function to choose for the analysis as a
\emph{starting function}?  Or in other words, what is the \code{main} function
of the file which has no \code{main} functions?  We solved this by using the
call graph provided by the \stan framework. We start the analysis in every
root of the call graph. That is we consider as a starting function every
function which has no callers in that file. This is still not perfect as we
get false positives for cases, where the automaton would have a different
than starting state for functions called from outside of the file. But this
approach definitely helped us to evade many false positives inside the file at
least. This approach is predominantly used in the \AC, but thanks to it is
generic, it can be used in newly implemented checkers too.

\paragraph{Filtering}
Since the \AC generates many false positives, we implemented a filtering
support as a part of the checker. Similar to registering checkers to a
checker factory, \emph{false positive detectors} (FPD) can register themselves
to the \AC FPD factory. They are always given a trace and can decide its
importance including marking it as a false positive. This way we implemented
pruners like checks whether a trace contains a \code{noreturn} function, i.e.\
a function which is supposed to loop forever or crash the analyzed code. If it
is a part of an error trace, the trace is spurious as the function would cut
the trace at that point. Hence the report can be safely thrown away.

\subsection{\LC}
The \LC tries to find locking errors in analyzed code. It performs so by
accounting statistics about locked accesses to a variable. This approach is
not new and is based on a previous work~\cite{lockchecker}.  Apart of tracking
accesses to a variable, it also tracks which locks are locked while each
of the variable is accessed. Then the \LC combines the two pieces of
information together and counts statistics in how many cases each variable is
accessed while some lock is held. If the difference is proportional, an error
is reported. For instance if, some variable is changed 99 times while some
lock is held and in one case the lock is not held, this is reported as a
possible error. The boundary is currently set to 70. So if there is a variable
accessed under a lock in more than 70\% cases and once or more it is not, the
latter cases are reported as errant. Similar to the \AC, both variable
accesses and locks are specified in a configuration by patterns.

\subsection{\TC}
The \TC is aimed at checking of possible deadlocks in concurrent programs.
The technique is based on the notions of locksets~\cite{relay} and deadlock
detection by looking for cycles in \emph{resource allocation graphs} (RAG).
The \TC first tries to identify the parts of code which can run in parallel,
as different threads. This is performed by searching of functions
instantiating threads. We implicitly try functions such as
\code{pthread\_create}. This is not enough for projects like the Linux kernel.
There need not be explicit thread creation functions in modules. As some hooks
in modules can be called in parallel, the code can run concurrently
nonetheless. So on the top of list of thread creation functions, a user can
also specify manually which hooks can run concurrently.

With the knowledge of where a thread starts, the \TC builds a set of
\emph{dependency graphs} for each such thread. A dependency graph statically
represents possible locksets during one execution of a thread. Dependency
graphs are then combined and transformed into RAGs. If there is a circular
lock dependency, a RAG contains a cycle. In such case an error is reported to
the \stan framework.

\subsection{\RC}
The \RC is the simplest and also the smallest checker in \stan. Its name
correctly suggests that the its task is to search for unreachable nodes in
control flow graphs. These are then reported as warnings or errors depending
on importance. The importance stems from the type of an unreachable node. For
example superfluous semicolons (which are still statements) are less important
than an unused branch. The primary goal of the \RC is to demonstrate the
simplicity of a new checker implementation. With the help of the framework
features described in Section~\ref{sec:travers}, the code of the checker has
less than 200 lines including the mentioned error/warning classification and
comments.  Even though it is a very simple checker, it was still able to find
serious bugs in the Linux kernel. For example a superfluous semicolon can
cause unexpected unconditional returns from functions like in the code of
Figure~\ref{fig:stan_unreach}.  This is taken from the \code{ath9k} wireless
driver, file \code{drivers/net/wireless/ath/ath9k/rc.c}.

\section{Results on Linux Kernel}\label{sec:stan_results}

We have several reasons to choose the Linux kernel for testing \stan: the
kernel is a large and freely available codebase, it fully exercises most
of the features of ISO/ANSI C99~\cite{ansi_c} and GNU C extensions~\cite{gcc},
it is under constant development (there is a constant income of new bugs), and
the absence of bugs is of a great concern.  We applied \stan, together with
the four checkers described in Section~\ref{sec:checkers}, to the Linux kernel
version 2.6.28.  The \AC was configured with four automata describing the
following types of errors:
\begin{description}
\item[Incorrect pairing of functions] This includes imbalanced locking and
  reference counting errors. Both are of the same kind when a programmer
  unintentionally omits to unlock a lock on some path or decrease a reference
  which was previously increased.
\item[Resource or memory leaks] These are usual errors when working with
  dynamic memory allocations and similar resources. Some path unintentionally
  omits to free the resources which were requested earlier.
\item[Errors in pointer manipulation] Often, when manipulating pointers,
  programmers, screw a pointer and for example try to dereference the null
  pointer or dangling pointers which point to already freed memory.
\item[Deadlocks] We specifically aimed at kernel specific deadlocks caused by
  sleeping inside spinlock critical sections or interrupt handlers. This is
  not allowed in the kernel and the results of such code are undefined
%  -- it often leads to a deadlock.
\end{description}

The running time of \stan on a common desktop machine with two 2.5\,GHz
cores and 4\,GiB of memory was under two hours. The memory usage of the Java
process oscillated between 400 and 1000\,MiB. The number of reports by the
checkers is presented in Table~\ref{tab:results_stan}.

Let us note that \RC actually found 751 errors, but 720 of them are of low
importance (including 696 superfluous semicolons). Even though they are real,
we omitted them from the statistics.

We have manually analysed the reports and tried to classify them as real
errors or false positives. We classified all except 31 reports found by \AC.
There, we are not able to decide whether it is a real error or not in a short
time. The numbers of real errors, false positives and the ratio of real errors
to all classified errors can be also found in Table~\ref{tab:results_stan}.
The overall ratio of real errors to all classified errors is not high: 22.8\%.
Nonetheless, \stan in the current version does not have any thorough false
positive filtering technique, which may be implemented in the future. Finally,
note that the checkers do not produce any false negatives assuming there is no
bug in the checkers' implementation. The reason is that all the checkers
implement may analyses, over-approximating the set of error behaviours.

More than 130 of the 172 real errors were reported to kernel developers and
fixed in the following kernel releases. The rest was either discovered by
independent checkers and reported by someone else or the incorrect code
disappeared from the kernel completely before we finished our evaluation of
the reports. Interestingly, some of the reported errors remained undiscovered
for more than seven years. For illustration, see our report at
\url{http://lkml.org/lkml/2009/3/11/380}. The whole list of reports generated
by \stan on that particular kernel version can be found in our database which
we describe in Chapter~\ref{sec:clabure}.

\subsection{Important Errors Found by \stan}\label{sec:stan_imp_errors}
Although checkers currently implemented in \stan are based on widely known
techniques, running them on the Linux kernel helped to uncover several
important bugs. It is despite the fact that some industrial tools check the
kernel periodically. We presented one error reported by the \RC earlier. In
the text below we present two typical errors discovered by \stan, using two
other checkers.

\subsubsection{\AC}
Many errors found by the \AC trigger only under specific conditions. It is
caused by the fact that the standard paths are usually dynamically tested.
Nonetheless some of the specific conditions may be established by a user
rather easily.  Consider the code excerpt in Figure~\ref{fig:stan_ac_bug}
taken from the Linux kernel 2.6.28. The excerpt is extracted from the
\code{set_lock_status} function in file
\code{drivers/pci/hotplug/pciehp_core.c}. This is a driver which adds a
support for the PCI Express hot-plug.
\begin{figure}[htb]
  \begin{center}
  \begin{tabular}{c}
  \begin{lstlisting}
mutex_lock(&slot->ctrl->crit_sect);(*@\label{line:stan_rc_lock}@*)
/* has it been >1 sec since our last toggle? */
if ((get_seconds() - slot->last_emi_toggle) < 1)(*@\label{line:stan_rc_if}@*)
	return -EINVAL;(*@\label{line:stan_rc_ret}@*)
  \end{lstlisting}
  \end{tabular}
  \end{center}
    \caption{An error found by the \AC.}
    \label{fig:stan_ac_bug}
\end{figure}
In the figure, note that the call to the \code{mutex_lock} function on
line~\ref{line:stan_rc_lock} is followed by a conditional statement. The code
returns immediately on line~\ref{line:stan_rc_ret} in case the condition
holds. The problem is the omitted call to \code{mutex_unlock} in that branch.
If this branch is hit and the lock is to be taken again later, we will
encounter a deadlock. In fact, the deadlock could be easily triggered by a
user. It is sufficient to write \code{1} to any of the
\code{/sys/bus/pci/slots/.../lock} files twice within a second.  This deadlock
was confirmed and fixed in the kernel 2.6.29 by the commit ``PCI hotplug: fix
lock imbalance in pciehp''. By that commit, the omitted call to
\code{mutex_unlock} was added.

\subsubsection{\TC}
%http://lkml.org/lkml/2009/4/14/527
Here, we present an example of a non-trivial error found by the \TC. The error
could not be found by the \AC as it is not thread-aware. The error described
here is from the Linux kernel 2.6.28. It is in the enterprise cryptographic
filesystem \textsc{eCryptfs}, more specifically from its file
\code{fs/ecryptfs/messaging.c}. In the file, there are three locks which we
are interested in. One is local per context: \code{msg_ctx->mux}. The other
two are global locks: \code{ecryptfs_daemon_hash_mux} and
\code{ecryptfs_msg_ctx_lists_mux}.

Let us denote lock dependencies as a binary relation where the first component
depends on the second. For example the call \code{lock(A)} followed by the
call \code{lock(B)} means a dependency of B on A, and we write B $\rightarrow$
A. So in Figure~\ref{fig:stan_tc_bug1}, we see two calls to locking functions
on lines~\ref{line:ecrypt_mux1} and~\ref{line:ecrypt_hmux1}. They produce a
dependency \code{ecryptfs_daemon_hash_mux} $\rightarrow$ \code{msg_ctx->mux}.

	...
	mutex_unlock(&ecryptfs_daemon_hash_mux);
	...
unlock:

out:

function \code{ecryptfs_send_message_locked} is called from

\section{Discussion}\label{sec:stan_discuss}
A lot of work is needed to be done in the area of automatic false alarm
filtering and error importance classification. As we know from
Section~\ref{sec:stan_results}, the \AC was used for checking the Linux kernel
and it produced hundreds of reports. We also know that many of them (81.7\%)
were false positives. In this section we discuss where these false reports
reports usually stem from.

In the first place, it is the analyzed code complexity that causes headache to
many developers of static analyzers. Analyzing simple and short sources is
easy as was presented in many experimental tools. When it comes to real-world
code with many inter-dependencies among files and complex inter-procedural
flows of checked properties, experimental tools often stop working.  We also
started naively with simple configurations for the \AC, similar to automaton
for locking of Figure~\ref{fig:stan_automaton}. It turned out to be too naive
as we ended up with tens thousands of reports with no useful information. We
had to tune the automaton. The result is an automaton which, when its height
fits on an A4 paper oriented landscape, would need 28 sheets for its width. A
very simplified variant of the automaton is in
Figure~\ref{fig:stan_big_automaton}.
\begin{figure}[htb]
  \begin{center}
  \input{stanse_big_automaton}
  \end{center}
    \caption{A simplified automaton used for checking the kernel locking.}
    \label{fig:stan_big_automaton}
\end{figure}
It neither contains loops on final states, nor transitions as they make the
picture totally unreadable. Nonetheless the transitions are easily
conceivable. We mark locked states by a suffix \code{L} and unlocked states by
a suffix \code{U}. The more \code{L}'s, the more nested a lock is. Finally, we
have several lock categories like anonymous locks \code{A}. The picture is not
complete also because we omitted parametrizations of states. \stan allows us
to parametrize nodes and transitions to support named locks and model
situations like nested locking (\code{lock(A); lock(B); unlock(B);
unlock(A);}) without losing precision.

After months of tuning of the configurations we finally got some meaningful
results. There finally started appearing real errors like those described
earlier in Section~\ref{sec:stan_imp_errors}. But there were still also false
positives stemming from the limits of the abstract interpretation which the
\AC is based upon. Let us demonstrate a typical false positive which we took
again from the analysis of the Linux kernel 2.6.28, file
\code{drivers/net/ns83820.c}. In particular, the checker flagged some reports
in the function \code{rx_refill} of Figure~\ref{fig:stan_typical_fp}.
\begin{figure}[htb]
  \begin{center}
  \begin{tabular}{c}
  \begin{lstlisting}

  unsigned i;
  unsigned long flags = 0;
...
  if (gfp == GFP_ATOMIC)
    spin_lock_irqsave(..., flags);(*@\label{line:stan_lock}@*)
  for (i=0; i<NR_RX_DESC; i++) {
...
  }
  if (gfp == GFP_ATOMIC)
    spin_unlock_irqrestore(..., flags);

  return i ? 0 : -ENOMEM;

over the control flow.  Since the abstract interpretation is not
path-sensitive, the \AC generates reports exactly about these two false
positives.

We tried to evade false positives like these by searching for similar patterns
along paths. In this particular case, we would extract that both dotted and
dashed paths go once through the true branch of a condition and once through
the false branch while the condition is the same. But this needed non-trivial
effort as the paths usually start only after the first condition. In our
code of Figure~\ref{fig:stan_typical_fp}, by the lock on
line~\ref{line:stan_lock}. One solution to this problem is to construct a path
condition over a path since the starting point of the analysis. Then use a
solver to check validity of the path condition. And the report is not created
unless the path can be validated. The other approach is more heavy-weight: to
run the symbolic execution through the code and see whether it reports the
same error.  We tried this and the output of this experiment is described
throughout Chapter~\ref{sec:fmics}.

\section{Conclusions}\label{sec:stan_concl}

\stan is a free Java-based framework design for simple and efficient
implementation of error-finding algorithms based on the static analysis. The
framework can process large-scale software projects written in ISO/ANSI
C99~\cite{ansi_c}, together with most the GNU C extensions~\cite{gcc}. There
is also an experimental support for C++.
\stan
does not currently use any new techniques -- its novelty comes from the fact
that (to our best knowledge) there is no other open-source framework with
comparable applicability and efficiency. We note that more than 130 errors
found by \stan have been reported to and confirmed by Linux kernel developers
already.

We also discussed the weaknesses of the implemented analyses. The \stan's
major downside are false stemming from the nature of the implementation of the
abstract interpretation. We suggested few solutions to filter them out, but
were not much successful, as the false positive ratio is still high. The
following Chapter~\ref{sec:fmics} tries harder to beat the false positives.
