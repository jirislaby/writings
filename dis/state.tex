\chapter{State of the Art}\label{sec:state}
This section is deserved to description and current state of techniques for
code analysis referred to later in the text. To be able to do that, we will
first describe the techniques proper.

\paragraph{Static Analysis} Under the term of static analysis we understand
analyzing code without directly executing it on a processor. I.e.~to perform
exactly the opposite of dynamic analysis or testing where a program is
executed and either automatically fed by some input in some form or tested
manually by testers. This has the nice side-effect that the implementation
need not be finished when we want to analyze it statically. So the static
analysis is not as limited as the dynamic one, it can be used whenever we want
in the development cycle. Static analysis is thus used in several fields of
the computer science. \emph{Optimizing compilers} can eliminate unused code,
propagate constants where variables would have to be left otherwise, or remove
implicitly added runtime out-of-bounds checks from the code if it knows the
bounds (as it happens in Java). \emph{Debuggers} can limit themselves only to
code that is relevant to variables a programmer is interested in, or print
possible sources of values currently stored in a tracked variable.
\emph{Program tests} can be generated based on the analysis of a program to
cover more code paths. And of course static analysis is a prerequisite for
\emph{static analyzers} to exist in the first place.

Moreover if the static analysis is well-defined according to some rules
defined e.g.~by Cousot's framework, as we will see later, we are able to
decide whether it is sound, complete and whether it terminates. Without those
properties, optimizing compilers would generate invalid code for example.

\section{Pattern Matching}\label{sec:pattern_matching}
The simplest automatic analysis we can think of is to find some predefined
patterns in code.  We can take an instance of a regular expression pattern
matcher like \texttt{grep}, give it a pattern ``/ 0'' and see if we did that
stupid zero division error somewhere in the code. However this primitive
approach may be a source of false-positives. This happens when developers
comment their code with comments similar to ``/ 0 happened here''.  But there
of course exist more robust techniques based on pattern matching. They, in
some way, parse the code (which also removes comments like the above) and do
pattern matching on some internal structure~\cite{findbugs}. If we recall the

a checker which would match AST subtrees. Then to match division by zero

\section{Data-flow Analysis}\label{sec:data-flow}
One of the basic static analyses is \emph{data-flow analysis}. It collects
information about possible values set in variables at specified points in
code. This is usually computed by stabilizing sets of equations, usually by
computing a fixed point of the equations:

\begin{dfn}[Fixed point]
A \emph{fixed point} of a function $f : D \rightarrow D$ is an element $x \in
D$ such that $f(x) = x$. The set $fp(f)$ of fixed points of $f$ is $\{x \in D
\mid f(x) = x \}$.
\end{dfn}

In the data-flow analysis, we have a pair of equations for each statement $s$:

\begin{equation} \label{eq:data_flow}
\begin{aligned}
out_s & = trans_s(in_s) \\
in_s & = join_{p\in pred_s}(out_p)
\end{aligned}
\end{equation}

We define transfer function $trans$ to transform input set $in_s$ of a
statement $s$ to the output set $out_s$ according to the effect of $s$ in a
particular analysis. The join operator $join$ takes output sets of all
predecessors and combines it together. It is usually set intersection or
union depending whether the analysis is to be may or must. And depending
whether an analysis is forwards or backwards (cf.
Section~\ref{sec:types_anal}), the $in$ set is bound to an incoming or
outgoing control flow graph edge respectively. Vice versa for the $out$ set.
These two sets are the result of the analysis before and after applying the
effect of the statement.

One of the basic data-flow analyses is \emph{reaching definitions}. At some
point in code, we are interested in all statements where some variable was set
and was not re-set during the control flow to our point. With this
information we can decide whether some variable was set already at the point
it is used. If it is not, we should report about an uninitialized access. Let
us demonstrate that on the following example. There, the reaching definition
set at line~\ref{lst:reach_use} is a set of the two labels
$\{\ref{lst:reach_def1}, \ref{lst:reach_def2} \}$, where we define \code{a}.
Note that line~\ref{lst:reach_def0} does not belong there because \code{a}
was re-set later on all paths to line~\ref{lst:reach_use}.
\begin{center}
\begin{tabular}{c}
\begin{lstlisting}[language=C]
a = 1; (*@ \label{lst:reach_def0} @*)
if (x)
  a = 3; (*@ \label{lst:reach_def1} @*)
else
  a = 4; (*@ \label{lst:reach_def2} @*)
b = a; (*@ \label{lst:reach_use} @*)
\end{lstlisting}
\end{tabular}
\end{center}

Formally, let us denote the set of statements as $S$. To compute reaching
definitions we define join operator as set union ($\bigcup$) and the transfer
function as follows:
\begin{align*}
trans_s(in) & = gen_s \cup (in - kill_s) \\
gen_s & =
\begin{cases}
\{s\}     & \text{$s$ is an assignment} \\
\emptyset & \text{otherwise}
\end{cases} \\
kill_s & =
\begin{cases}
\{x \in S \mid \text{$x$ is $a = b$} \wedge x \neq s \} & \text{$s$ is an
assignment $a = c$} \\
\emptyset & \text{otherwise}
\end{cases} \\
\end{align*}
In other words we generate a statement when a variable is set there and we
kill all other assignment statements which set the very same variable at any
other point.  Finally, we compute a fixpoint of the
equations~\ref{eq:data_flow} to have reaching definitions on each entry and
exit of a statement ($in_s$, $out_s$).

\section{Abstract Interpretation}\label{sec:ai}
The data-flow analysis is only a limited tool for deciding some properties
about the analysis. There was a call to formalize the process which would
for example allow for convenient proofs of algorithms' correctness and
soundness.  \emph{Abstract interpretation} is a technique that has a history
stretching back to the 1970s. Back then, it was described by Cousot\&Cousot in
\cite{cousot}. It is designed to allow users of the formalism to decide
whether their analysis is correct, if it terminates, to formulate more
precision levels of an analysis consistently and more.  With this formalism,
we can also interpret data-flow analyses as abstract interpretations, as was
shown in~\cite{ai_intro}.

As the name of the analysis suggests, there is an abstraction of how values
are handled in an analysed program. Let us show this on a demonstrative
example. Consider a program in Figure~\ref{fig:lst_ai_intro} and an analysis
that should decide whether \code{i} is positive at the end (let us ignore
overflows at this moment).

First, consider an ``abstraction'' that abstracts the code in no way, i.e.~we
take the concrete evaluation. There are only two lines in the code that store
something to \code{i}: lines~\ref{lst:ai_intro_set1} and
\ref{lst:ai_intro_set2}. The former simply sets \code{i} to \code{3}, so
we just remember that. The latter, increases \code{i} repeatedly, so we have
to remember that \code{i} can be anything from $\{6,12,24,\dots\}$ there. To
be able to decide if a variable is positive or negative when analyzing the
program concretely, we would need infinite time and memory to store all
possible values from all possible runs. Nonetheless, if we only remember
abstract information such as ``this number is positive/negative'', and define
how arithmetic operations behave on those, we need to store only a
single value.

To work with the abstract values, we define two functions: \emph{abstraction}
denoted $\alpha$ and \emph{concretization} denoted $\gamma$, both accepting
one parameter.  $\alpha$ returns an abstract value corresponding to the
concrete value on input and $\gamma$ does the opposite.  In
Figure~\ref{fig:ai_alpha_omega}, we provide a definition of the two functions
to work for our positive/negative analysis, where $\oplus$ is used to denote
the positive abstract value and $\ominus$ for negative. Further, the same as
for the data-flow analysis, during the abstract interpretation, we compute a
fixed point. The fixed point is computed to stabilise the abstract values at
each statement and often it starts with values set to $\bot$. The values are
changed by a transform function that changes the state according to the effect
of a statement. We have two possibilities to define the transform function as
can be seen in Figure~\ref{fig:ai_square}. We can simply define a transform
function for abstract values $F_a$ to obtain a new abstract state
corresponding to the effect of a statement applied to the original state. The
other option for more complex statements, is to first jump back to the
concrete values, perform the transformation on the concrete values using a
concrete transform function $F_c$ and switch back to the abstract values.
$F_a$ is then defined as $F_a = \alpha \circ F_c \circ \gamma$. Per definition
(\cite{cousot}), the resulting abstract value must be less than or equal to
the one computed by a direct $F_a$.

We know how to define the arithmetic multiplication in this case precisely on
the abstract level as is shown in Table~\ref{tab:ai_multiplication}. With
these definitions, we can continue interpreting our example of
Figure~\ref{fig:lst_ai_intro}. The first set to \code{i} happens on
line~\ref{lst:ai_intro_set1}. There, \code{3} is abstracted as $\oplus$, so
\code{i} is set to $\oplus$ there too.  Another set happens on
line~\ref{lst:ai_intro_set2} where we multiply \code{i} by \code{2}. \code{2}
is abstracted as $\oplus$ and since we know \code{i} is $\oplus$ at this
point, the multiplication results, according to
Table~\ref{tab:ai_multiplication} in \code{i} being still $\oplus$. Note that
for this simple analysis, the only values we need to remember are one of
$\ominus$, $\oplus$, or possibly \textit{unset} in case no store was processed
yet. \textit{unset} is mostly referred to as $\bot$. Further, if the body of
the loop is changed to be \code{i *= -2}, what would be the abstract value of
\code{i} after joining $\oplus$ from odd iterations with $\ominus$ from even
iterations? The same as in the data-flow analysis case, we have a kind of join
operator which can in this case say \code{i} is \textit{unknown}. Note that
\textit{unknown} is usually referred to as $\top$.  So in fact, when we are
computing a real abstract interpretation of a program, the behaviour tables
(Table~\ref{tab:ai_multiplication}) are extended to contain also $\bot$ and
$\top$. Obviously, multiplication of $\bot$ and any other value $x$ from the
table results in $x$ as the first initialization happened. For $\top$, the
join operator usually returns $\top$ for two reasons. First, the
transformation function shall be monotone (\cite{cousot} and \cite[p.
348--349]{kleene}). Second, hopping from \textit{unknown} back to some of the
``known'' values would break soundness of the analysis.

Now, let us demonstrate the interpretation of the code of
Figure~\ref{fig:lst_ai_intro} with the modified while-loop body which
multiplies \code{i} by \code{-2}. The computation of a fixed point is depicted
in Table~\ref{tab:ai_computation} as it goes over time until a fixed point is
reached in the third iteration. We see that the result is rather imprecise, we
cannot make any assumption about \code{i} when it is returned from the
function. The abstraction was apparently too coarse and the interpretation
does not help much.

We are still missing a formalism to represent the values and design a monotone
function over the values effectively.  In the abstract interpretation this is
solved by using complete lattices~\cite{lattice1,lattice2}. They are used to
represent both concrete and abstract values and they give us the possibility
to use them as a basis effectively. Thanks to work presented
in~\cite{tarski}, we can compute fixed points of monotone functions working on
lattices. Further, lattice operators can be used when we are interpreting a
program.  For example, when a branching occurs, the join operator needed when
joining the branches simply takes the values present on the outputs of both
branches and performs lattice join ($\sqcup$) of the values in that lattice.
Since an abstract interpretation works on two complete lattices, one for
concrete, one for abstract values, $\alpha$ and $\gamma$ are used to move from
one to the other.  In fact the pair $(\alpha,\, \gamma)$ forms a \emph{Galois
connection}~\cite{cousot_galois} between the lattices and so Galois
connection can be used as a tool for designing abstractions.  Now, let us
formally define a complete lattice and show some examples of lattices.

\begin{dfn}[Complete lattice]
A \emph{complete lattice} $(L,\, \leq,\, \bot,\, \top,\, \sqcup,\, \sqcap)$ is
a partially ordered set $(L,\, \leq)$ where every set $X \subseteq L$ has a
least upper bound (supremum) $\sqcup X$ and a greatest lower bound (infimum)
$\sqcap X$. Namely, $L$ has a least element $\bot = \sqcup \emptyset$ and a
greatest element $\top = \sqcap L$.
\end{dfn}

We were working so far with a positive/negative analysis. To represent values
in that abstraction as a complete lattice, we naturally define the lattice as
follows.  $\oplus$ is incomparable to $\ominus$ as switching from one to the
other would be pointless in our case when joining branches.  $\bot$ is less
than both since we start with \textit{unset} values and want to go upper in
the lattice when any of the abstract values is set.  Finally, $\top$ is
greater than both to catch the case of \textit{unknown} when branches bring
different values.  A Hasse diagram for such a lattice is the one of
Figure~\ref{fig:posneg}~(a). We may want a more precise abstraction
represented by the lattice in Figure~\ref{fig:posneg}~(b).  There, we also
remember whether the value of a variable is 0.  This can become useful if we
have a multiplication or division in the code.  For multiplication, we can
propagate the zero constant, for division, we can tell whether we are dividing
by zero for example.  Extending $(\alpha, \gamma)$ from the previous example
is straightforward in this case.

We were also working with concrete ``abstraction'' at the beginning of this
section. First, let us have a look at a lattice of an abstraction where we
remember only a single value of a variable. This is useful for analyses like
constant propagation as we can propagate only a single value. Such a lattice
is depicted in Figure~\ref{fig:lat_concrete}.  Naturally, joining branches
where values are different gives \textit{unknown}.  However, our concrete
interpretation was computing with all possibles sets of values. We need to
augment the above lattice to a power set $\mathcal{P}(\mathbb{Z})$ using a set
inclusion as the order. The complete lattice is defined as
$(\mathcal{P}(\mathbb{Z}),\, \subseteq,\, \bot,\, \top,\, \sqcup,\, \sqcap)$,
where $\bot=\emptyset$, $\top=\mathbb{Z}$, $\sqcup=\cup$, and $\sqcap=\cap$. A
part of a Hasse diagram is demonstrated in Figure~\ref{fig:lat_concrete_set}.

There is a difference to be spotted between these two. Although they both
contain an infinite number of elements, the latter lattice has infinite
height. This is exactly the source of the need of infinite time and memory for
this interpretation when analyzing particular programs with loops. The
Kleene's sequence is not necessarily finite there, so the computation of a
fixed point may not finish. There is a mechanism suggested in~\cite{cousot} to
overcome the problem. It is a \emph{widening} operator $\triangledown: L_a
\times L_a \rightarrow L_a$ defined such that:
\begin{itemize}
\item $\forall x,y \in L_a,\, x \leq_a x \triangledown y$ and $y \leq_a x
\triangledown y$, and
\item the sequence $x_0 = \bot$, $x_1 = F_a(x_0)$, \dots, $x_{n+1}=x_n
\triangledown y_n$ for some $y_n \in L_a$ is not strictly increasing.
\end{itemize}
That informally means, that the sequence is sped up by widening, so that it
converges eventually. For the example with powerset lattice, we can define the
widening operator to limit the number of iteration to some constant and wide
the set to $\top$ if it reaches the limit. The opposite of the widening
operator is \emph{narrowing} operator $\vartriangle: L_a \times L_a
\rightarrow L_a$ and is defined similarly as:
\begin{itemize}
\item $\forall x,y \in L_a,\, x \leq_a y \Rightarrow x \leq_a x \vartriangle y
\leq_a y$, and
\item the sequence $x_0 = \bot$, $x_1 = x_0 \vartriangle y_0$, \dots,
$x_{n+1}=x_n \vartriangle y_n$ for some $y_0,\dots,y_n \in L_a$ is not
strictly decreasing.
\end{itemize}
Narrowing can help to make the abstract values more precise. When our widening
operator returned $\top$, we can narrow it back to some abstract value which
we know will never cause an increase. This is not so conceivable for number
interpretation, but see~\cite{ai_widening, cousot_galois} where the operators
are demonstrated on intervals and described in more detail.

Intervals are not the only shapes to represent abstract values of numbers.
Octagons, elipses, polyhedra and other shapes are used too~\cite{ai_octagons,
ai_polyhedra, astree}. Several abstractions can be also combined to
communicate with each other and improve thus accuracy or any other aspect of
the analysis~\cite{ai_product, astree, clousot}. Strings~\cite{ai_strings1,
ai_strings2} and foating point numbers~\cite{ai_float1} can be abstracted
too.  Further reading on the abstract interpretation is
in~\cite{cousot_progress}, where is described the progress in the area of the
abstract interpretation over 25 years. A not so usual use of the abstract
interpretation can be found in~\cite{ai_unusual1, ai_unusual2, ai_unusual3}.

\section{Symbolic Execution}\label{sec:se}
In the previous chapter we described the abstract interpretation which is a
framework for fast, but many times imprecise analyses. As a result we often
get a pile of false positives on the output of an abstract analyzer. There
also exist techniques which cannot formally generate false positives, but are
often slower than abstract interpreters. One of such techniques is the
\emph{symbolic execution}. The technique was introduced in 1970's and
described by several authors concurrently~\cite{se_king, se_select,
se_dissect, se_clarke}. The symbolic execution represents program inputs by
symbolic values instead of trying to analyze a program many times with
concrete, initialized, data. The program is then executed and program
expressions are manipulated including the symbolic values. Let us demonstrate
this on an example of Figure~\ref{fig:se_example}. The example is two-fold.

First, there is a read from the user. How long it would take to exercise the
program to get to the true branch of the \code{if}s with dynamic testing?
With \code{int} being usual 32 bits, there is only one successful attempt out
of $2^{32}$ overall. Of course, with a static analyzer based on the data-flow
analysis or abstract interpretation, those true branches would be analyzed
quite quickly.  This brings us to the second point nonetheless. Many complete
analyzers generate a report about division by zero possible at
line~\ref{line:se_ex_div}. This is indeed a false positive because \code{b} is
set to \code{1} on line~\ref{line:se_ex_set} in case \code{a} is
\code{0x12345678}. The source of the false positive is that the analyzers do
not often remember conditions on branchings. Some of them do a post processing
on reports to eliminate such reports as we will see in
Chapter~\ref{sec:stanse}.

int fun(void) {
  int a = read(); (*@\label{line:se_ex_sset}@*)
  int b = 0;

  if (a == 0x12345678) (*@\label{line:se_ex_if1}@*)
    b = 1; (*@\label{line:se_ex_set}@*)

The symbolic execution tackles this problem. All inputs from the user are
replaced by \emph{symbols}. For a variable \code{a}, we denote the symbol
corresponding to \code{a} as $\sym{a}$. Such symbols are allowed to be
assigned to variables like constants and other expressions. In our example, we
would have \code{int a = }~$\sym{a}$\code{;}. Obviously we have to teach the
symbolic executor to deal with symbols in arithmetic operations and branching
statements. In the first place, we need to have a storage. Symbolic executors
contain a \emph{symbolic memory} $\theta$ which assigns to each variable its
current symbolic value. So after the line~\ref{line:se_ex_sset}, $\theta(a) =
\sym{a}$. To be precise in execution of the code, the executor also has to
remember the branches which it chose to jump to at branching points. This is
stored in a \emph{path condition} $\varphi$. It is a first-order
formula~\cite{fol} representing a necessary and sufficient condition over
symbols to drive the execution along the currently executed path to the
actually evaluated point. $\varphi$ is set initially to $\true$ and is
evaluated and potentially updated at every branch. This happens on
lines~\ref{line:se_ex_if1} and~\ref{line:se_ex_if2} in our example. On
line~\ref{line:se_ex_if1}, we check, whether $\varphi$ implies validity of the
condition instantiated by the current state of $\theta$. More concretely, the
executor checks whether either of the following two implications hold (at most
one~\cite{se_king}):
\begin{align}
\varphi & \rightarrow \theta(a) = 0x12345678 \label{equ:se_fi} \\
\varphi & \rightarrow \theta(a) \neq 0x12345678 \label{equ:se_se}
\end{align}
There are three possibilities that can happen:
\begin{description}
\item[Implication \ref{equ:se_fi} is valid] The executor simply continues
execution by jumping directly to the \emph{true} branch.
\item[Implication \ref{equ:se_se} is valid] The executor simply continues
execution by jumping directly to the \emph{false} branch.
\item[Neither is valid] This means the executor has no concrete idea about the
values to prove validity. So to retain soundness, it has to fork the execution
of the whole program to explore both branches. In that case, the path
condition $\varphi$ has to be augmented in the respective branches. For the
true branch, $\varphi \wedge \theta(a) = 0x12345678$ is a new path condition,
for the other branch $\varphi \wedge \theta(a) \neq 0x12345678$.
\end{description}
In our example, on line~\ref{line:se_ex_if1} neither implication is valid, as
$\theta(a)=\sym{a}$ which cannot prove validity of $\theta(a) = 0x12345678$.
The situation is different on line~\ref{line:se_ex_if2} nonetheless. There, we
can get into two states, by the two previous forked executions. In one, the
path condition contains $\theta(a) = 0x12345678$ and symbolic memory is
holding \code{1} for \code{b}. In the other $\theta(a) \neq 0x12345678$ and
$\theta(b) = 0$. From the first one, the executor can induce
Implication~\ref{equ:se_fi}, so it continues by jumping into the true branch.
And since $\theta(b) = 1$, no division by zero can happen. Similarly for the
second fork: the executor jumps to the false branch of the \code{if}, i.e.~it
skips the \code{if} completely.

The only remaining piece of information needed to perform the symbolic
execution is the program location which is currently evaluated by the
executor.  Then we define the \emph{program state} as a triple $(l,\,
\theta,\, \varphi)$, where $l$ is the current program location in a control
flow graph. The executor works with these triples when executing a program
and updates the members as described above. The execution is usually depicted
as a \emph{symbolic execution tree} due to the forks. Nodes of the tree are
labeled by program states computed during the execution.  Edges of the tree
correspond to transitions between program states labeling their end nodes.
For demonstration, we have a control flow graph of the function from
Figure~\ref{fig:lst_ai_intro} in Figure~\ref{fig:se_intro}~(a). A symbolic
execution tree of the control flow graph is depicted in
Figure~\ref{fig:se_intro}~(b).  For readability of figures, nodes are not
labeled by full program states, but only by the current location in the
control flow graph. In addition, we label branching edges by instances of the
corresponding branching conditions in the flow graph. These labels allow us to
reconstruct the path condition for each node in the tree: it is the
conjunction of labels of all edges along the path from the root to the
particular node. Note that contents of symbolic memories are not depicted in
the figure.

We have not discussed performance of the symbolic execution yet. There are
two main sources of potential performance bottlenecks. First, symbolic
executors use \emph{Satisfiability Modulo Theory} solvers~\cite{smt} to reason
about validity of formulae like~\ref{equ:se_fi} and \ref{equ:se_se}.  Whereas
performance of individual solvers vary, complex queries are not easy to answer
for any solver.  So it usually takes a long time for solvers before they
answer such queries. Second, the symbolic execution suffers from the
\emph{path explosion problem}, i.e.~an exponential growth of paths that need
to be explored. Every branching and in the code potentially multiplies the
possible number of paths through the program by two. It is even worse for
loops, where the symbolic execution tree can become huge. So the larger
program, the exponentially larger symbolic execution tree to be explored yet.
In fact the tree can be also infinite, e.g.~when analyzing infinite loops.
Hence, for real-world programs, the symbolic execution is barely usable
without incorporating other techniques to reduce the count of explored paths.
Some tools try to tackle this issue by traversing more paths at once to
substantially reduce the size of the execution tree~\cite{cse_atva}. Others,
like in~\cite{pex_ddse, G07}, compose symbolic executions from several
functions using summaries which results in an inter-procedural analysis. This
approach limits the need of execution to separate functions only. \cite{rwset,
klee, exe}~are based on recording of already seen behavior and pruning its
repetition. Tools also limit the tree depth by a hard-coded constant, to
obtain finite tree for the price of incompleteness~\cite{ucklee, se_limited1}.
Some works aim at guessing of the number of loops needed to reach some target
location to help analyzing large programs with many nested
loops~\cite{partial_loop}.

Many modern tools combine the symbolic execution with dynamic (concrete)
analysis. This combination of analyses is called \emph{concolic
execution}. Tools implementing concolic execution achieve better scalability
and higher coverage of analyzed code~\cite{dart, active_props, cute, exe,
pex_ddse}.  Programs are run dynamically first with random input.  While a
program is running, path conditions are stored along the executed path
directed by the radnom input. When the dynamic run finishes, the symbolic
execution is then run. It is given a new constrains on the input. The
constraints are built using solver by negating some pieces of the path
condition collected from the dynamic run. The symbolic run generates new input
this way. The program is ran with these inputs again which can cover more
paths than the first dynamic run.  This process repeats until all paths i nthe
program are covered or the analyzer is terminated. Let us consider the example
of Figure~\ref{fig:lst_dart}.  Suppose that it is hard to compute an inversion
of the \code{hash} function.  It may take ages till proper values for \code{x}
and \code{y} are generated to trigger the assertion. But using concolic
execution, the first random input can be \code{3} for \code{x} and \code{4}
for \code{y}. Suppose $hash(4)=235$, during the dynamic execution, we obtain
$x=3 \wedge hash(y)=235$ as a path condition.  Now, the solver can easily
generate a new input by fixing \code{y} to \code{4} and setting \code{x} to
\code{235}.  With the next dynamic execution, the \code{assert} will be
trigerred. \textsc{Check 'n' Crash}~\cite{check_crash} does the opposite:
first it runs coarse symbolic execution to find a suspicious location in code,
then the tool generates a test and runs the program dynamically to verify the
test case.

symbolic\todo{mam tady rozepsat ty techniky vic do detailu?}
debugging~\cite{se_king}, the symbolic execution is currently widely used for
generating unit tests which cover more paths in code than tests written
manually~\cite{se_generate_test1, se_generate_test2, se_generate_test3,
se_generate_test4, se_generate_test5, exe, klee}. Offutt and
Seaman~\cite{se_generate_test2} use the symbolic execution also for re-writing
program internal variables as functions of the input values and then generates
tests using constraint-solving. The symbolic execution is usable as a partial
correctness prover~\cite{se_corr1, se_corr2}, as a reverse engineering
engine~\cite{se_reverse}, or for refactoring software to a one which is more
specialized~\cite{se_spec}, The symbolic execution can be extended to handle
heap~\cite{se_heap, se_gener}, summarize effects of libraries~\cite{se_libs},
or to extract dependencies of outputs on inputs~\cite{se_output1}.  The
symbolic execution was used to find bugs in aircraft control and nuclear power
systems~\cite{se_critical} or filesystems by generating malicious
disks~\cite{exe_fs}. Even though, some do not explicitly mention the symbolic
execution, they in fact implement it, to find errors in memory
management~\cite{prefix} for example.  There are also overviews where is a
summarization of some techniques we mention above~\cite{se_review88,
se_review09, se_review13}. We use the symbolic execution in our tool described

\section{Program Slicing}\label{sec:slicing}
As was described in the previous section, the symbolic execution is prone to
slowness when analyzing programs with loops or many branchings. We described
some techniques how to cope with this problem in that section too. But in case
we are interested only whether some code locations in a program are reachable,
another approach can be used in an analyzer to help the symbolic execution.
The analyzer can simply remove all the code which does not affect the flow to
those code locations in any way.  This is exactly a task that can be performed
by \emph{program slicing}. This is a technique that was described by Weiser in
his PhD thesis in 1979~\cite{slicing_phd} and later also
published~\cite{slicing}. As the slicing leaves only the code that really
affects particular code locations, the slicing may bring a considerable
speed-up to techniques working with code in general, not only to the symbolic
execution.  That way, analyzers do not need to evaluate every path that is in
the code, only those relevant to some code locations, e.g.\ lines with
potential bugs.

\newcommand{\newsym}[1]{\ensuremath\triangleright #1 \triangleleft}

When we want to slice a program we have to specify what behaviour we want to
preserve in the program first. In the slicing, this is performed by defining
\emph{slicing criteria}. These are chosen lines of the program accompanied
with variables we are interested in at those lines. Note there is no
limitation on the variables like whether they are set or used on those lines.
Such defined criteria ensure that values of variables at those lines must be
equal in both original and sliced program for any execution. We can slightly
loosen the requirement of ``any path'' to specified execution paths as we will
see later nonetheless.  According to the original
algorithm~\cite{slicing_phd}, when a slicer is given the slicing criteria, it
processes functions in a program independently as follows. The slicer starts
with some function and assigns a set of \emph{relevant variables} to each line
of the function. Relevant variables sets are filled initially by variables
from slicing criteria when the line has assigned some criteria or the sets are
left empty otherwise.  Further, according to the semantics of program
commands, slicers propagate relevant variables among function lines and insert
discovered relevant variables into sets on corresponding lines:
\begin{itemize}
\item All relevant variables from a successive line are copied to the current
line's set except for those which are set on the current line. This is a
simple information \emph{propagation}.
\item If the current line sets some variable which is relevant at a successive
line, the slicer adds all variables which the current line references.
I.e.\ the slicer performs computation of \emph{direct dependency}.
\item If the current line is a branching (or loop head) and some line in the
branch (or loop body) sets a relevant variable, the slicer adds all referenced
variables to the current line's relevant set. This is \emph{indirect
dependency} and serves the purpose of leaving in place branches and loops
which indirectly affect slicing criteria.
\end{itemize}
The slicers continue evaluation the function repeatedly until the sets stop
increasing in size, i.e.\ a fixed point is reached.  From the content of the
sets and the program semantics we can safely claim, which program lines have
no impact on the initial slicing criteria. Those are exactly the lines, where
intersection between variables that are set on that line and relevant
variables on the same line is empty. Slicers also have to leave lines with
branches or loop heads that affect some unsliced lines from the previous
sentence. The rest can be removed from the body of the function. Let us
demonstrate the computation on the example of Figure~\ref{fig:lst_ai_intro}.
Suppose we want to slice the program according to line 5 and variable \code{i}
there. The computation of relevant sets denoted as $R_{5,i}$ is demonstrated
in Table~\ref{fig:tab_slicing}. We stamp newly added variables in the concrete
iteration of relevant sets computation by $\newsym{~}$ in the table. We also
note the rule which applied to add the variable: the variables are marked with
one of C, P, D, I in superscript. It means their source is slicing criteria
(C), propagation rule (P), or direct (D) and indirect (I) dependency. The
result of the slicing is that we cannot slice anything from the program,
because \code{i} on line~5 depends on all lines, including the function
parameter \code{int x} on line~1.

The process described above is only intra-procedural. To extend the slicer
to be inter-procedural, we have to change the slicer to pass the relevant
variables among functions. That is down to a called function at every call
instruction and back up from the called function. As relevant variables
propagate from the bottom up, the slicer has to pass the variables from a call
site to return points of the called function and when the slicer is done with
propagation of the variables through the whole function to its beginning, the
variables are transferred back to the call instruction. Of course the slicer
has to take care of actual with formal parameters binding and of the
relationship between return value and the variable the result is stored into.
The same as above, the process repeats until a fixed point is reached.
Inter-procedural slicers were discussed by Horwitz et al. and
Gallagher~\cite{slic_inter1, slic_inter2}.

Originally, there were only slicers implementing the technique as described
above. These are \emph{static} slicers and they slice such that any possible
input to a program is handled the same as in the original code. But slicing
is often used for debugging purposes~\cite{slicing_debugging} and there, we
are usually interested in a value of some particular variable only when some
specific input was passed to the program.  Imagine we have a bug in the
program and know what was passed on input. We also know that a variable
printed on the output has an invalid value. The slicer can be therefore
limited to this input, in particular, the input need not trigger visiting all
bodies of branches or loops, so this allows for much more code removal than
static slicing.  This approach is called \emph{dynamic}
slicing~\cite{slicing_dyn}. Note the slicer has to look at conditions and
track a path condition and this is usually slow. So this obviously results in
a drawback: computing a dynamic slice is more computationally expensive than
computing static slices. Since control flow graphs are used to compute slices,
there appeared techniques to overcome the problem by pruning of the control
flow graphs. They can significantly decrease the computation
time~\cite{slic_prunecfg1, slic_prunecfg2}, but the result is less precise.
\emph{Hybrid} slicing tries to tackle this problem by combining the dynamic
slicing with dynamic runs.  Using breakpoints, slicers record which code
locations are visited with which input and compute the dynamic slice
according to the locations.  It is both faster and less space consuming than
the dynamic slicing.  The dynamic slicing can be improved such that it
works with more than one input~\cite{slic_quasi, slic_union}, which can be
generalized as describing input as constraints~\cite{slic_constraint,
slic_constraint_bw}.

There is also an orthogonal method to the static slicing.  We may be
interested in all program statements which \emph{are affected} by some
variable at some line. Since the standard slicing is sometimes called
\emph{backward} slicing, the orthogonal approach was named \emph{forward}
slicing~\cite{slic_forw} and is used for dead code removal for example. The
forward and backward slicing can be generalized as
\emph{chopping}~\cite{slic_chopping}. The chopping requires two distinct sets
of slicing criteria: one about what a variable affects (forward slicing), the
other about what is affected by that variable (backward slicing). The slicing
can be abstracted a bit more~\cite{slic_abstract}, such that it can answer
which values in a variable affect statements in the slicing criterion.

When taking over and reusing a library, a programmer can reduce the library
code size using slicers. Programmers just need to specify the parts of the
library which are used, and the library is sliced~\cite{slic_interface}.
The authors suggest also use in the reverse engineering -- slicing closed code
by some guessed criteria helps recognition tools to find patterns and reverse
engineer the code. Further, some variables like temporaries or iterators are
unwanted in slices when debugging. They can be disabled to appear in the
slice~\cite{slic_stop}. Similar is the \emph{barrier}
slicing~\cite{slic_barrier} which wants us to define a set of statements that
should be ignored by a slicer. This is done transitively, no further
statements starting at the barrier are put into the slice. So barriers are
often at interfaces of massively used, and therefore likely error-free,
libraries.  The slicing can be used also for finding bugs using \emph{dicing}:
two slices are compared, where one has to be a correct slice (a variable has a
good value) and one incorrect (a variable has a bad value). Intersection of
the slices are lines of code which are likely correct. The bug is in the
others of the incorrect slice. Slicers can work also on the semantic level, to
not preserve syntax, but leave semantics with respect to
criteria~\cite{slic_amorphous}. Slicers can reason about paths in a
program~\cite{slic_path} such as ``why this statement on this path is not
executed?'' Finally, the slicing can be used for model checking~\cite{slic_mc}
and for concurrent programs too~\cite{slic_concur} and the slicing can be used
for model checking~\cite{slic_mc}.  More details about slicing and its
variations and uses can be seen in various surveys and
reviews~\cite{slic_overview95, slic_overview96, slic_overview98,
slic_overview01a, slic_overview01b, slic_overview05, slic_overview12}.

\section{Model Checking}
This is a bit different technique from static analysis we are dealing in this
thesis with. It does not check the program proper, but instead tries to prove
some properties on a model of the program. The model is represented by some
finite state machine and the properties are a specified behaviour which has to
be satisfied in time. For example a property can be ,,This message has to come
as a reply in the future.'' Properties are described by logic formulas,
usually in the temporal logic like LTL or CTL (see more about temporal logics
in~\cite{temporal}). Correctness of the system is then established, if every
run of the system corresponds to a particular formula. If it does not for some
case, a counterexample consisting of system states is generated. Obviously,
even if no counterexample is generated, the system still might not be correct
in case its model or properties checked are incorrect.

These technique was spread even after introduction of \emph{symbolic model
checking}~\cite{mc_symbolic1,mc_symbolic2}. It started representing sets of
states by boolean functions. For example if we have two 1-bit wide variables
\texttt{a} and \texttt{b} and their values are always the same (1 and 1 or 0
and 0 pairwise), we can represent it by a formula \texttt{a} $\Leftrightarrow$
\texttt{b} instead of storing all the values. The formulas can be represented
in a space-saving way by \emph{Reduced Ordered Binary Decision
Diagrams}~\cite{bdd} (ROBDD or simply BDD) were introduced to represent the
set of states. Despite the exponential complexity still persists (due to path
explosion problem), BDDs helped to limit the memory use a lot. Both symbolic
model checking and BDDs allowed more complex systems to be checked on the same
hardware (cf.~\cite{mc_symbolic1} again).

In parallel to BDDs-based model checking, \emph{bounded model checking}
exists. Instead of exploring the whole state space of the system model, there
is a predefined constant \textit{k} which determines the maximum length of
execution to be explored. If it explores all these \textit{k}-prefixes, either
\textit{k} is incremented and the checking continues, or the model checker is
stopped. It can have two reasons: the user set the top bound for \textit{k}
(the analysis is of course incomplete in that case), or the execution is
infeasible, because it runs out of resources. Model checking and bounded model
checking are orthogonal techniques, it was shown in~\cite{bmc_survey} that
both of them reason about different class of properties. (Although there can
be an intersection.)

Due to path explosion problem and slow computers in the past, its use was
limited rather to check small reactive systems in industry like processors or
automotive control systems. Later, there appeared both less and more
successful tools to check large software projects. For example model checker
\textsc{Blast}~\cite{blast} failed first due to inability to reason about
pointers in the Linux kernel drivers~\cite{blast_kernel}. Later, it was shown,
that \textsc{Blast} needs an instrumented code and then works better.  So it
is currently successfully used in the Linux Driver Verification
toolkit~\cite{ldv} to verify Linux device drivers.  Similarly, Microsoft
started adopting their model checker \textsc{SLAM} in the driver verifier
\textsc{SDV}~\cite{sdv} for checking Windows drivers and they apparently
succeeded as they require every driver to be checked by this toolkit with
predefined temporal rules. There are over 160 rules which are checked before a
driver is accepted by Microsoft. \textsc{SDV} is a proprietary software with
rules tailored to Windows drivers. Therefore it cannot be used for checking
open source projects. So Clarke et al.~\cite{cbmc} adapted Microsoft's
approach using \textsc{CBMC} model checker and defined at least seven rules to
check Linux kernel drivers. The rules were later extended by Post and
K\"{u}chlin~\cite{slicx}. \textsc{DDverify} verifier~\cite{ddverify} and
\textsc{Saturn}~\cite{saturn} were used for automated verification of the
Linux kernel drivers. \textsc{Undertaker}~\cite{undertaker} is a model checker
for the Linux kernel configuration system. It serves the purpose of validating
dependencies in the \texttt{Kconfig} configuration files.

As this thesis does not deal with model checking, curious readers are pointed
to a survey~\cite{mc_survey} and a book~\cite{mc_book} about model checking.
There is also a survey about bounded model checking~\cite{bmc_survey}
available.

\section{Tools}\label{sec:soa_tools}
There are many successful tools based on static analysis. Unfortunately for
some of them, especially the industrial one, it is not really known what
technique they are based on. We will describe them in a separate paragraph.
The rest of the tools is categorized according to the technique used. Even
though some of them are combination of more. In this overview we mostly limit
ourselves to analyzers of C code.

\subsection*{Pattern Matching}
We already mentioned the tools like \textsc{grep} which are used by developers
to find diverse errant behaviour. Nonetheless, there are much more
sophisticated tools.  One of the most known tools based on pattern matching is
\textsc{FindBugs}~\cite{findbugs}. Despite it is intended to find errors in
Java programs, we mention it here, because it became very popular. This
happened thanks to the ease of use, hundreds of error types it can find,
scalability and precision. It is able to prioritize the reports according to
their weight so that it usually puts real errors on the top of the list in
favor of false reports.

\textsc{Coccinelle}~\cite{coccinelle} is another pattern matcher, better to
write a semantic patcher. It uses a special language SmPL for specifying a
semantic patch that allows for matching against a control flow graph nodes and
changing them arbitrarily. The graph is then converted back to the source
code.  \textsc{Coccinelle} is used mainly for modifying code to use newer
interfaces like replacing two calls to \code{kmalloc} and \code{memset}
functions by a single call to \code{kzalloc} which does the same as the two
together.  Since it is able to match the control flow graph, it can be used
for error pattern matching too~\cite{wysiwib}. As an example, we can pick
missing checks for allocations: matching a call to an allocation function not
followed by an \code{if} to check the return value. If this construct is
found, it can be replaced to be the allocation plus an added check.

\subsection*{Data-flow Analyzers}
As we slightly mentioned in the introduction, compilers contain data-flow
analysis and generate warnings or errors when they find some suspicious code.
When compilers were still not so evolved, other analyzers were used to fulfill
this role. One of the most known known data-flow analyzers at that time was
\textsc{Lint}~\cite{lint}. It was capable of finding errors like division by
zero, constant conditions which always evaluate either to true or false, or
use of uninitialized variables. Later, \textsc{Lint} resulted in some clones
like \textsc{LCLint} which was later renamed to \textsc{Splint}.
\textsc{LCLint} was augmented to catch errors like buffer overflows and it
also manages to work with code annotations. So the users can enable more
checks if they annotate the checked code, otherwise false positive rate is
unbearable.  Nonetheless \textsc{Splint} is not developed for more than a
decade due to obvious reasons: the errors it reports are reported by modern
compilers nowadays.

\subsection*{Abstract Interpreters}
Engler et al. brought the abstract interpretation to the attention in 2000 by
a technique based on meta compilation~\cite{metacomp}. There, program
developers specify properties of the checked system in \textsc{Metal}
language~\cite{metal}, compile it into an internal representation and the tool
applies the specification on all executable paths in a particular function.
Violation of the specification on a path is then
reported as an error. The compiler used for \textsc{Metal} compilation is a
modified GNU C compiler (\textsc{gcc}) 3.1.1 called \textsc{xgcc}. It
redefines \textsc{gcc}'s internal code representation, data types and applies
the analysis on a such modified structure. Programming languages supported by
\textsc{xgcc} are C and C++.

\subsection*{Industrial Tools}
Only a little information is known about industrial tools. It is due to
secrets the companies hold to not let their competitors know. What can be
found in the reference materials, the tools often use combination of the
techniques we have already described. They work on the top of various pattern
matching, data-flow analyses, and some maybe use the abstract interpretation.
Even though the internals are not known exactly, we at least try to summarize
the tools and the little about what their providers make available about them.
All the tools below are capable to handle C, some of them C++ and other
languages like C\# or Java.

The data-flow analyzer \textsc{Lint} was an inspiration for Gimpel's
\textsc{PC Lint/FlexeLint}~\cite{pclint} and Cleanscape's
\textsc{LintPlus}~\cite{lintplus}. The same as their predecessor, they aim at
finding suspicious code constructs. MathWorks'
\textsc{PolySpace}~\cite{polyspace} was one of the first abstract interpreters
capable of handling real code. But according to the authors, the tool's upper
limit is still approximately 50 thousand lines of code that can be analyzed.
The tool was inspired by the crash of Ariane~5 (see introduction). Another
abstract interpreter which can handle large scale programs is AbsInt's
\textsc{Astree}~\cite{astree}. It is sound and can be also tuned to produce no
false positives. \textsc{Astree} also combines several abstractions to
perform more accurate analysis~\cite{astree_combine}.

\textsc{xgcc} was a predecessor of Coverity's
\textsc{Prevent}~\cite{coverity}, and \textsc{Metal} for Coverity's
\textsc{Extend}. \textsc{Prevent} is run regularly to check the Linux kernel
and the results are provided to kernel developers.  KlocWork's
\textsc{Insight}~\cite{klocwork} and GrammaTech's
\textsc{CodeSonar}~\cite{codesonar} are specialized to finding security and
reliability issues. \textsc{Insight} was originally intended as an optimizer
for C code.

Microsoft developed many tools based on symbolic execution.
Even though \textsc{PREfix}~\cite{prefix} explicitly does not mention it uses
the symbolic execution, it is used there to find errors in memory management
for example. \textsc{PREfix} has no problems analyzing large projects.
\textsc{Pex}~\cite{pex} is a symbolic executor intended for checking .NET web
applications. They developed also a fuzz tester \textsc{SAGE}~\cite{sage}. It
is a symbolic executor working directly on the x86 assembly code. It is
therefore able to work also with libraries whose source code was not made
public.  \textsc{YOGI}~\cite{yogi} is a tool which combines symbolic execution
with dynamic execution. It was experimentally used to replace
\textsc{SLAM}~\cite{slam_decade} in SDV~\cite{sdv} with comparable better
results. Microsoft also released \textsc{PREfast}, a tool that contains a set
of rules that a code should fulfill and with a use of annotations, it checks
code against the rules. Its main use is to disallows developers to commit
changes that contain certain class of bugs.

\subsection{Summary of the Linux Kernel Checkers}
Some of the tools above can analyze kernels or only drivers of operating
systems. In this sections we specifically sum those which are capable to
handle the Linux kernel.

One of the basic ones is based on pattern matching
and every developer should use it before sending patches to the kernel. It is
named obviously after what it is intended to do as
\textsc{checkpatch}~\cite{checkpatch}. It only checks for patterns like if
somebody introduces a whitespace to the code or if the code is properly
indented. Some better analysis can be performed by a semantic parser called
\sparse~\cite{sparse}. It is still a tool working with some sort of patterns
matched against the parse tree.  It was written by the founder of the kernel
and is tailored exclusively to the kernel as the checks have only a little
sense elsewhere.  The checks include basic intra-procedural checks for proper
locking, use of zero instead of \code{NULL}, or extended type checking using
the annotated code in the kernel. Yet another pattern matcher is Dan
Carpenter's source matcher \textsc{Smatch}~\cite{smatch}. Formerly, it was a
set of Perl programming languages scripts. They were later turned into C code
and \sparse was used as a base. \textsc{Smatch} added many more checks
regarding the Linux kernel like doing direct memory accesses on a stack or
passing incorrect arguments to picked functions.  These four tools are
extensively used by the kernel developers and errors reported are getting
fixed constantly~\cite{linux_faults}.
