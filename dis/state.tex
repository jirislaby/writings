\chapter{State of the Art}\label{sec:state}
This section is describes the current state of the art of the code analysis.
We especially describe techniques which are referred to later in the text.  In
this overview we mostly limit ourselves to analyzers of C code.

\paragraph{Static Analysis} In our understanding, the term \emph{static
analysis} is analyzing code without directly executing it on a processor.
In other words to perform exactly the opposite of the dynamic analysis where a
program is executed and either automatically fed by some input in some form or
tested manually by testers. This approach has the nice side-effect that the
implementation need not be finished when we want to analyze it statically.  So
the static analysis is not as limited as the dynamic one, it can be used
whenever we want in the development cycle. Static analysis is thus used in
several fields of the computer science.
\begin{itemize}
\item \emph{Optimizing compilers} can eliminate unused code, propagate
constants where variables would have to be left otherwise, or remove
implicitly added runtime out-of-bounds checks from the code if it knows the
bounds (as it happens in Java).
\item \emph{Debuggers} can limit themselves only to code that is relevant to
variables a programmer is interested in, or print possible sources of the
value currently stored in a tracked variable.
\item \emph{Program tests} can be generated based on the analysis
of a program to cover more code paths.
\item \emph{Static analyzers} can exist in the first place of course.
\end{itemize}

Moreover if the static analysis is well-defined according to some rules
defined e.g.\ by Cousot's framework~\cite{cousot} described later, we are able
to decide whether the analysis is sound, complete, and whether it terminates.
This is useful for example when building an optimizing compiler to prove it
does not generate invalid code, i.e.\ it is sound.

\section{Pattern Matching}\label{sec:pattern_matching}
The simplest automatic analysis we can think of is to find some predefined
patterns in code of a program.  We can take an instance of a regular
expression pattern matcher like \texttt{grep}, give it a pattern ``/ 0'' and
see if we did zero division error somewhere in the code. However this
primitive approach may be a source of false reports. This happens when
developers comment their code with comments similar to ``/ 0 happened here''.
But there of course exist more robust techniques based on pattern matching.
They, in some way, parse the code (which also removes the problem with the
comments like the above) and do pattern matching on some internal
structure. If we recall the abstract syntax tree from
Figure~\ref{fig:intro_parsing_ast}, we can design a checker which would
match AST subtrees for instance. Then to match division by zero errors, we
simply try to find a subtree in an AST like of Figure~\ref{fig:ast_to_match}.
The left leaf is left intentionally blank to express a wildcard.

One of the most known tools based on the pattern matching is
\textsc{FindBugs}~\cite{findbugs}. Despite it is intended to find errors in
Java programs, we mention it here, because it became very popular. This
happened thanks to the ease of use, hundreds of error types it can find,
scalability and precision. It is able to prioritize the reports according to
their weight so that it usually puts real errors on the top of the list in
favor of false reports.

\textsc{Coccinelle}~\cite{coccinelle} is another pattern matcher, better to
write a semantic patcher. It uses a special language SmPL for declaring a
semantic patch. It is utilized for matching against control flow graph nodes
and changing them arbitrarily. The graph is then converted back to source
code.  \textsc{Coccinelle} is used mainly for modifying code to use newer
interfaces like replacing two calls to \code{kmalloc} and \code{memset}
functions by a single call to \code{kzalloc} which does the same as the two
together.  Since it is able to match the control flow graph, it can be used
for error pattern matching too~\cite{wysiwib}. As an example, we can pick
missing checks for allocations: a call to an allocation function with no
condition to check the return value afterwards. If such a call is found, it
can be replaced by the allocation with the check.

\section{Data-flow Analysis}\label{sec:data-flow}
One of the basic static analyses is \emph{data-flow analysis}. It collects
information about possible values of variables at specified points in code. We
usually analyze each program line separately to see what are the effects of
the line and then put this information to a transfer equation of what holds
before and after the execution of the line. Next, we stabilize this set of of
equations, usually by computing the least fixed point of the equations. Such
computed analysis can be utilized for various code optimizations or code
analysis.

Let us demonstrate data-flow analysis on an example and then generalize that
approach. We chose one of the basic data-flow analyses: \emph{reaching
definitions}~\cite[Section 9.2.4]{dragon}. At some point $p$ in code, we are
interested in all assignments where some variable $\var{a}$ was set and was
not re-set during the control flow to the point $p$. This lets us reason about
many questions like whether a variable was set already before it is used or if
we should report an uninitialized access. We will show the analysis on the
code which is in the second column of Table~\ref{tab:state_df_eq}.
Intuitively, the reaching definition set on line~\ref{lst:state_rd_ref} is a
set of four labels $\{\ref{lst:state_rd_def1}, \ref{lst:state_rd_def2},
\ref{lst:state_rd_def3}, \ref{lst:state_rd_def4}\}$, where we define \code{a}
and \code{b}, because all of them can reach line~\ref{lst:state_rd_ref}.
%Note that line~\ref{lst:reach_def0} does not belong there because \code{a}
%was re-set later on both paths to line~\ref{lst:reach_use}.

constructing the equations and computing their fixed point. We can simply hold
the reaching definitions in the transform equations.  That is, initially at
the beginning of a function, the first equation holds an empty set. Then the
equation is gradually transformed according to statements and the definitions

transform equation mangles the contents coming from predecessors only on
assignment statements.  Other equations simply copy data from predecessors.
On assignments, the equation takes the set of statements from the
predecessors, removes all statements that set the same variable and adds

definitions from both branches. The transform equations $f_s$ for each

The equation is in the form of $f_s = gen_s \cup (pred_s \setminus kill_s)$.
$gen_s$ is a set $\{s\}$ that \emph{generates} $s$ if it is an assignment,
otherwise $gen_s$ is empty.  $pred_s$ is a union of $f_{s-1}$ for each

\begin{dfn}[Fixed point]\label{def:fixpoint}
A \emph{fixed point} of a function $f : D \rightarrow D$ is an element $x \in
D$ such that $f(x) = x$. The set $fp(f)$ of fixed points of $f$ is $\{x \in D
\mid f(x) = x \}$.
\end{dfn}

Now we generalize the approach. In the data-flow analysis, we have a pair of
equations for each statement $s$ of a program:
\begin{equation} \label{eq:data_flow}
\begin{aligned}
out_s & = trans_s(in_s) \\
in_s & = join_{p\in pred_s}(out_p)
\end{aligned}
\end{equation}
We define a transfer function $trans_s$ to transform input set $in_s$ of a
statement $s$ to the output set $out_s$ according to the effect of $s$ in a
particular analysis. The join operator $join$ takes output sets of all
predecessors and combines it together. It is usually a set intersection or
union depending whether the analysis is may or must (cf.\
Section~\ref{sec:types_anal}). Further, whether an analysis is forwards
or backwards (Section~\ref{sec:types_anal}), the $in$ set is bound to an
incoming or outgoing control flow graph edge respectively. Analogously for the
$out$ set.  These two sets are the \emph{result} of the analysis before and
after applying the effect of the statement.

For the reaching definitions example above, we define the join operator as a
set union ($\bigcup$) and the transfer function in this notation to hold
statement line numbers as follows ($S$ is the set of all statements in the
analysed program):
\begin{align*}
trans_s(in) & = gen_s \cup (in - kill_s) \\
gen_s & =
\begin{cases}
\{s\}     & \text{$s$ is an assignment} \\
\emptyset & \text{otherwise}
\end{cases} \\
kill_s & =
\begin{cases}
\{x \in S \mid x \text{ assigns to \code{a}} \} & s \text{ assigns to
\code{a}} \\
\emptyset & \text{otherwise}
\end{cases} \\
\end{align*}

For curious readers, they are defined in every superior literature about
compilers~\cite{dragon, muchnick}. Thanks to the simplicity, the data-flow
analysis is also used in many tools including checkers.  The very common place
of extensive use are compilers. Apart from generating code, they generate
warnings or errors when some suspicious code is found. A good example is
\clang~\cite{clang}. It is a C/C++ compiler from the \llvm toolkit~\cite{llvm}
and contains also a static checker framework.  There are checkers for finding
errors in memory handling, locking, or pointer operations. Nonetheless mostly
the results are imprecise and contain many false reports.
\textsc{Cppcheck}~\cite{cppcheck} works on the same principles so it checks
for very similar set of errors.  When compilers were still not so evolved,
other data-flow analyzers were used to fulfill this role. One of the most
known analyzers at that time was \textsc{Lint}~\cite{lint}. It was capable of
finding errors like division by zero, constant conditions which always
evaluate either to true or false, or use of uninitialized variables.  Later,
\textsc{Lint} resulted in some clones like \textsc{LCLint}~\cite{lclint} which
were augmented to catch errors like buffer overflows. \textsc{LCLint} was
later renamed to \textsc{Splint}.  These tools mostly have a support for
annotations so more speculative checks can be enabled. In that case, the code
has to be heavily annotated, otherwise the false positive rate is unbearable.
Nonetheless \textsc{Splint} is not developed for more than a decade due to
obvious reasons: the most errors it reports are reported by modern compilers
nowadays.  There exist also analyzers that are not so powerful to handle large
code bases or require high user intervention to annotate the checked code. As
an example we give \textsc{Aspect}~\cite{aspect} which compares a program
specification of input-output behaviour provided by a user to data-flow models
extracted from the code. This was only a short introduction to the data-flow
analysis, more information and many examples are available in the
literature~\cite{dragon, muchnick}.



\section{Abstract Interpretation}\label{sec:ai}\todo{transfer vs

The data-flow analysis is only a limited tool. It only collects information
about data and how they propagate through the control flow of a program.
Therefore it is not as useful for designing more complex analyses which are
usual for code checking. There was a call to formalize the process for a
convenient design of more complex analyses. Cousots' work on this topic
stretches back to the 1970s. Back then, they described a framework called
\emph{abstract interpretation}~\cite{cousot}. It is designed to allow users of
the formalism to easily design an analysis, decide whether it is correct and
sound, to formulate more precision levels of the analysis consistently and
more.  The formalism was designed in a way that we can also interpret analyses
like data-flow as an abstract interpretation~\cite{ai_intro}.

As the name of the analysis suggests, the data domains used in analyzed code
are replaced by abstract ones. The abstract domain is a \emph{complete
lattice}~\cite{lattice1,lattice2} as defined in
Definition~\ref{def:colattice}. Since the program works on a new (abstract)
domain, we have to provide a new semantics of the program over the domain.
There are no limits, the semantics can be arbitrary. We define a
\emph{monotone function} for all instructions in the analyzed code to
transform the program states over the instructions.  Thanks to the work of
Tarski~\cite{tarski} and Kleene~\cite[Pages 348--349]{kleene}, we can always
compute fixed points of monotone functions which transform states over
lattices.  Further, using lattices is effective. Their operators join and meet
can be used when an analyzer is interpreting a program.  For example, when a
branching
occurs, the branch join operator simply takes the values present on the
outputs of both branches and performs the lattice join ($\sqcup$).

\begin{dfn}[Complete lattice]\label{def:colattice}
A \emph{complete lattice} $(L,\, \leq,\, \bot,\, \top,\, \sqcup,\, \sqcap)$ is
a partially ordered set $(L,\, \leq)$ where every set $X \subseteq L$ has a
least upper bound (supremum) $\sqcup X$ and a greatest lower bound (infimum)
$\sqcap X$. In particular, $L$ has a least element $\bot = \sqcup \emptyset$
and a greatest element $\top = \sqcap L$.
\end{dfn}

Let us demonstrate the abstract interpretation on an example the same as the
data-flow analysis and then generalize it again. We take the same example as
is in Table~\ref{tab:state_df_eq} and reachable definitions analysis again.
First, we have to define the abstract set which the abstract interpretation
works with.  Since the analysis works with lines where is an assignment and
there can be more reachable definitions assigned to a single line, the complete
lattice will consists of a powerset of all assignments in the program:
$A=\{1,2,4,5\}$.  The abstract domain is therefore lattice $L_a =
(\mathcal{P}(A),\, \subseteq,\, \emptyset,\, A,\, \cap,\, \cup)$. Now we would
have to define the transform function over the lattice. But it is the same as
in Table~\ref{tab:state_df_eq}. So computing the least fixed point is the same
as is shown in Table~\ref{tab:state_df_fp} too.

As we see, the abstract interpretation is defined by the complete lattice,
and a transform function. Formally, it is a tuple $I=\langle L,\, \circ,\,
\leq,\, \top,\, \bot,\, F\rangle$, where $(L,\, \leq,\, \bot,\, \top,\,
\sqcup,\, \sqcap)$ is a complete lattice, $\circ$ is either supremum
($\sqcup$) or infimum ($\sqcap$) operator, and $F$ is a monotone function
defining the interpretation on a program. The definition can be loosened to
require only a semi-lattice, as we need only one operator.

It is not yet obvious how to define the transform function of a program. We
can define a transform function $F_a$ working directly on the abstract domain.
It obtains a new abstract state corresponding to the effect of a statement
applied to the original state. The same as we showed in the example, this is
usually defined on an intuitive level as a composition of simple
transformations performed on single statements of a program. For complex
analyses it need not be easy to design such $F_a$, so there is also another
option.  First, we define two functions: \emph{abstraction} denoted $\alpha$
and \emph{concretization} denoted $\gamma$, both accepting one parameter.
$\alpha$ returns an abstract value corresponding to the concrete value on
input and $\gamma$ does the opposite. Now, we can jump back to the concrete
values, perform the transformation on the concrete values using a concrete
transform function $F_c$ and switch back to the abstract values.  $F_a$ is
then defined as $F_a = \alpha \circ F_c \circ \gamma$. To retain soundness of
the analysis, the resulting abstract value must be less than or equal to the
one computed by a direct $F_a$. We depict these two options in
Figure~\ref{fig:state_ai_square}.

Both are used for abstracting numbers in programs. One is capable to remember
only a single value of a variable and is therefore useful for analyses like
constant propagation~\cite[Chapter 9.4]{dragon}. The other lattice uses
intervals to remember possible bounds of some variable's value.  This is very
useful domain especially when analyzing bounded loops. When indices, and
probably other variables in the body of the loop are bounded, we can run some
optimizations like loop unrolling~\cite[page 735]{dragon}.

There is a difference to be spotted between these two lattices. Although they
both contain an infinite number of elements (in theory), the latter lattice
has infinite height.  The Kleene's sequence is not necessarily finite there,
so the computation of a fixed point may not finish. There is a mechanism
suggested by Cousots~\cite{cousot} to overcome the problem. It is a
\emph{widening} operator $\triangledown$. Informally, the sequence is sped up
by widening, so that it converges eventually.  Imagine we have an intervals
lattice and an infinite loop where we increment a variable that was initially
set to zero. The infinite sequence of intervals computed for that variable
would be: $[0, 0],\, [0,1],\, \dots,\, [0,\infty]$. We can define the widening
operator to set one of the interval bounds to infinity after some time to
force the sequence finite: $[0, 0],\, [0,1],\, [0,\infty]$. The opposite of
the widening operator is a \emph{narrowing} operator $\vartriangle$. The
narrowing can help to make the abstract values more precise. When our widening
operator put an $\infty$ in an interval, we can narrow it back to some
abstract value which we know will never cause an increase. This is of course
not possible for our simple loop, but the reader is pointed to Cortesi and
Zanioli~\cite{ai_widening} or Cousots~\cite{cousot_galois} who demonstrate
widening and narrowing on more concise examples.

Intervals are not the only shapes to represent abstract values of numbers.
Octagons, elipses, polyhedra and other shapes are used too~\cite{ai_octagons,
ai_polyhedra, astree}. Several abstractions can be also combined to
communicate with each other and improve thus accuracy or any other aspect of
the analysis~\cite{ai_product}. This was utilized in tools like
\textsc{Astree}~\cite{astree} and \textsc{Clousot}~\cite{clousot}.
Strings~\cite{ai_strings1, ai_strings2} and foating point
numbers~\cite{ai_float1} can be abstracted too.  Further reading on the
abstract interpretation is in an overview~\cite{cousot_progress}, where is the
progress in the area of the abstract interpretation over 25 years. A not so
usual use of the abstract interpretation can be found too~\cite{ai_unusual1,
ai_unusual2, ai_unusual3}.  Engler et al.\ brought the abstract interpretation
to the attention in 2000 by a technique based on meta
compilation~\cite{metacomp}. There, program developers specify properties of
the checked system in \textsc{Metal} language~\cite{metal}, compile it into an
internal representation and the tool applies the specification on all
executable paths in a particular function.  Violation of the specification on
a path is then reported as an error.  On the similar base is built
\stan~\cite{stanse} which is discussed in Chapter~\ref{sec:stanse}. It is
uncertain, whether some industrial tool adopts the abstract interpretation
too, see the discussion in Section~\ref{sec:industrial}.

The correctness and soundness (cf.\ Section~\ref{sec:theor_props}) of an
analyzer based on the abstract interpretation is very prone to selection of
the abstract domain. While many employed abstractions are sound (produce all
possible errors), they are not so often complete (false positive rate is
high). For example, this was demonstrated on several abstractions of numbers
which we discussed above~\cite{ai_octagons}.




\section{Symbolic Execution}\label{sec:se}
In the previous section we described the abstract interpretation which is a
framework for fast, but typically imprecise analyses. As a result of
incomplete abstract analyzers, we often get a pile of false positives on their
output.  There also exist techniques which cannot formally generate false
positives, but are often slower than abstract interpreters or sometimes even
unsound.  One of such techniques is the \emph{symbolic execution}. The
technique was introduced in 1970's and described by several authors
concurrently~\cite{se_king, se_select, se_dissect, se_clarke}. The symbolic
execution represents program inputs by symbols. Therefore it does not need to
analyze the program many times with concrete, initialized, data. The program
is then executed and program expressions are manipulated including the
symbolic values. Let us demonstrate this on an example of
Figure~\ref{fig:se_example}. The example is two-fold.
\begin{figure}[htb]
\begin{center}
\begin{tabular}{c}
\begin{lstlisting}
int fun(void) {
  int a = read() + 1; (*@\label{line:se_ex_sset}@*)
  int b = 0;

  if (a == 0x12345678) (*@\label{line:se_ex_if1}@*)
    b = 1; (*@\label{line:se_ex_set}@*)

  if (a == 0x12345678) (*@\label{line:se_ex_if2}@*)
    a = 10 / b; (*@\label{line:se_ex_div}@*)

  return a;
}
\end{lstlisting}
\end{tabular}
\end{center}
  \caption{Example to demonstrate the symbolic execution.}
  \label{fig:se_example}
\end{figure}

First, there is a read from the user. How long it would take to exercise the
program to get to the true branch of the \code{if}s with dynamic testing?
With \code{int} being usual 32 bits, there is only one successful attempt out
of $2^{32}$ overall. Of course, with a static analyzer based on the data-flow
analysis or abstract interpretation, those true branches would be analyzed
quite quickly.  This brings us to the second point nonetheless. Many complete
analyzers generate a report about division by zero possible at
line~\ref{line:se_ex_div}. This is indeed a false positive because \code{b} is
set to \code{1} on line~\ref{line:se_ex_set} in case \code{a} is
\code{0x12345678}. The source of the false positive is that the analyzers do
not often remember conditions on branchings.

The symbolic execution tackles this problem. All inputs from the user are
replaced by \emph{symbols}. For a variable \code{a}, we denote the symbol
corresponding to \code{a} as $\sym{a}$. We allow symbols to be assigned to
variables similar to constants and other expressions. In our example, the
assignment on line~\ref{line:se_ex_sset} would set $\sym{a}+1$ to \code{a}.
Obviously we have to teach the symbolic executor to deal with symbols in
arithmetic operations and branching statements.  In the first place, the
symbolic executor needs to remember the currently evaluated program location
$l$ in a control flow graph.  Next, we need a storage. Symbolic executors
contain a \emph{symbolic memory} $\theta$ which assigns to each variable its
current symbolic value. So after evaluating the statement on
line~\ref{line:se_ex_sset}, $\theta(\var{a}) = \sym{a}+1$. To be precise in
the execution of code, the executor also has to remember the branches which it
chose to jump to at branching points. This information is stored in a
\emph{path condition} $\varphi$. It is a first-order formula~\cite{fol}
representing a necessary and sufficient condition over symbols to drive the
execution along the currently executed path to the actually evaluated point.
$\varphi$ is set initially to $\true$ and is evaluated and potentially updated
at every branch.  This happens on lines~\ref{line:se_ex_if1}
and~\ref{line:se_ex_if2} in our example. On line~\ref{line:se_ex_if1}, we
check whether $\varphi$ implies validity of the condition instantiated by the
current state of $\theta$. More concretely, the executor checks whether either
of the following two implications hold (at most one~\cite{se_king}):
\begin{align}
\varphi & \rightarrow \left( \theta(\var{a}) = \var{0x12345678} \right) \label{equ:se_fi} \\
\varphi & \rightarrow \left( \theta(\var{a}) \neq \var{0x12345678} \right) \label{equ:se_se}
\end{align}
There are three possibilities that can happen:
\begin{description}
\item[Implication \ref{equ:se_fi} is valid] The executor simply continues
execution by jumping directly to the \emph{true} branch.
\item[Implication \ref{equ:se_se} is valid] The executor simply continues
execution by jumping directly to the \emph{false} branch.
\item[Neither is valid] This means that both branches are possible, depending
on the symbolic input.  So to retain soundness, the symbolic executor has to
fork the execution of the whole program to explore both branches. In that
case, the path condition $\varphi$ has to be augmented in the respective
branches. For the true branch, $\varphi \wedge (\sym{a}+\var{1} =
\var{0x12345678})$ is a new path condition, for the other branch $\varphi
\wedge (\sym{a}+\var{1} \neq \var{0x12345678})$.
\end{description}
In our example, on line~\ref{line:se_ex_if1} neither implication is valid, as
$\theta(\var{a})=\sym{a}+1$ cannot prove validity of $\theta(\var{a}) =
\var{0x12345678}$. So the execution forks.  The situation is different on
line~\ref{line:se_ex_if2} nonetheless. There, we can get into two states, by
the two previous forked executions. In one, the path condition contains
$\theta(\var{a}) = \var{0x12345678}$ and symbolic memory is holding \code{1}
for \code{b}. In the other $\theta(\var{a}) \neq \var{0x12345678}$ and
$\theta(\var{b}) = 0$. From the first one, the executor can induce
Implication~\ref{equ:se_fi}, so it continues by jumping into the true branch.
And since $\theta(\var{b}) = 1$, no division by zero can happen.  Similarly
for the second fork: the executor jumps to the false branch of the \code{if},
i.e.~it skips the \code{if} completely. Note that this is one of the very
cases where other types of analyses like the abstract interpretation produce
false positives.

The three information above constitute the \emph{program state}. It is a
triple $(l,\, \theta,\, \varphi)$.  The executor works with these triples when
executing a program and updates the members accordingly as described above.
Due to the forks, the execution of a program is usually depicted as a
\emph{symbolic execution tree}. Nodes of the tree are labeled by program
states computed during the execution.  Edges of the tree correspond to
transitions between program states labeling their end nodes.

We have not discussed performance of the symbolic execution yet. There are
two main sources of potential performance bottlenecks. First, symbolic
executors use \emph{Satisfiability Modulo Theory} solvers~\cite{smt} to reason
about validity of formulae like~\ref{equ:se_fi} and \ref{equ:se_se}.  Whereas
performance of individual solvers vary, complex queries are not easy to answer
for any solver.  So it usually takes a long time for solvers before they
answer such queries. Second, the symbolic execution suffers from the
\emph{path explosion problem}, i.e.\ an exponential growth of paths that need
to be explored. Every branching and in the code potentially multiplies the
possible number of paths through the program by two. It is even worse for
loops, where the symbolic execution tree can become huge. So the larger
program, the exponentially larger symbolic execution tree to be explored yet.
In fact the tree can be also infinite, e.g.\ when analyzing infinite loops.
Hence, for real-world programs, the symbolic execution is barely usable
without incorporating other techniques to reduce the count of explored paths.
Some tools try to tackle this issue by traversing more paths at once to
substantially reduce the size of the execution tree~\cite{cse_atva}. Others,
like in~\cite{pex_ddse, G07}, compose symbolic executions of several
functions using summaries which results in an inter-procedural analysis. This
approach limits the need of execution to separate functions only. \cite{rwset,
klee, exe}~are based on recording of already seen behavior and pruning its
repetition. Tools also limit the tree depth by a hard-coded constant, to
obtain finite tree for the price of incompleteness~\cite{ucklee, se_limited1}.
Some works aim at guessing of the number of loops needed to reach some target
location to help analyzing large programs with many nested
loops~\cite{partial_loop}.

Many modern tools combine the symbolic execution with dynamic (concrete)
analysis. This combination of analyses is called \emph{concolic
execution}. Tools implementing concolic execution achieve better scalability
and higher coverage of analyzed code~\cite{dart, active_props, cute, exe,
pex_ddse}.  Programs are run dynamically first with random input.  While a
program is running, path conditions are stored along the executed path
directed by the radnom input. When the dynamic run finishes, the symbolic
execution is then run. It is given a new constrains on the input. The
constraints are built using solver by negating some pieces of the path
condition collected from the dynamic run. The symbolic run generates new input
this way. The program is ran with these inputs again which can cover more
paths than the first dynamic run.  This process repeats until all paths i nthe
program are covered or the analyzer is terminated. Let us consider the example
of Figure~\ref{fig:lst_dart}.  Suppose that it is hard to compute an inversion
of the \code{hash} function.  It may take ages till proper values for \code{x}
and \code{y} are generated to trigger the assertion. But using concolic
execution, the first random input can be \code{3} for \code{x} and \code{4}
for \code{y}. Suppose $hash(4)=235$, during the dynamic execution, we obtain
$x=3 \wedge hash(y)=235$ as a path condition.  Now, the solver can easily
generate a new input by fixing \code{y} to \code{4} and setting \code{x} to
\code{235}.  With the next dynamic execution, the \code{assert} will be
trigerred. \textsc{Check 'n' Crash}~\cite{check_crash} does the opposite:
first it runs coarse symbolic execution to find a suspicious location in code,
then the tool generates a test and runs the program dynamically to verify the
test case.

Despite King originally introduced the symbolic execution for symbolic
debugging~\cite{se_king}, the symbolic execution is currently widely used for
generating unit tests which cover more paths in code than tests written
manually~\cite{se_generate_test1, se_generate_test2, se_generate_test3,
se_generate_test4, se_generate_test5, exe, klee}. Offutt and
Seaman~\cite{se_generate_test2} use the symbolic execution also for re-writing
program internal variables as functions of the input values and then generates
tests using constraint-solving. The symbolic execution is usable as a partial
correctness prover~\cite{se_corr1, se_corr2}, as a reverse engineering
engine~\cite{se_reverse}, or for refactoring software to a one which is more
specialized~\cite{se_spec}, The symbolic execution can be extended to handle
heap~\cite{se_heap, se_gener}, summarize effects of libraries~\cite{se_libs},
or to extract dependencies of outputs on inputs~\cite{se_output1}.  The
symbolic execution was used to find bugs in aircraft control and nuclear power
systems~\cite{se_critical} or filesystems by generating malicious
disks~\cite{exe_fs}. Even though, some do not explicitly mention the symbolic
execution, they in fact implement it, to find errors in memory
management~\cite{prefix} for example.  There are also overviews where is a
summarization of some techniques we mention above~\cite{se_review88,
se_review09, se_review13}. We use the symbolic execution in our tool described
in Chapters~\ref{sec:fmics} and~\ref{sec:svcomp}.

\paragraph*{Tools}

\textsc{PathFinder-SE}~\cite{pathfinder_se} is an extension to the model
checker \textsc{PathFinder} which enables the model checker to perform the
symbolic execution.  They further improved the tool to handle e.g.\ concurrent
programs, heap structures and data.

\textsc{IntScope}~\cite{intscope} decompiles x86 binaries to an intermediate
representation form and runs the symbolic execution to find integer overflows.

\paragraph{Concolic Executors}
\textsc{PathCrawler}~\cite{pathcrawler} was the first executor to combine
symbolic and dynamic execution. It does not use concrete values in the
symbolic execution to simplify complex constraints nonetheless. This was first
performed only by \textsc{Dart}~\cite{dart}.  The primary goal is to execute
\emph{all}, or at least as much as possible feasible program paths, in a
systematic way. It starts dynamically with random input and symbolically
tracks branching conditions.  The symbolic execution and a solver is then used
to change the input so that it traverses another paths when executed
dynamically again. This happens until the code is covered fully or an user
stops the process.  \textsc{Dart} can propagate concrete values from the
dynamic runs to the symbolic run when the solver is incapable to reason about
some values.  \textsc{Dart} gives up the symbolic execution and switches to
standard random testing when pointer constrains are used. This was improved in
another executor \textsc{Cute}~\cite{cute}. It handles pointers better and is
able to analyze concurrent programs. So except it generates the inputs
systematically, it also makes diverse schedule plans of individually spawn
threads.  \textsc{Cute} cannot still handle symbolic offsets of pointers.

\paragraph{Egt and Derivatives}
At the same time as \textsc{Dart} and \textsc{Cute} emerged, Cadar et al.\
came up with \textsc{Egt}~\cite{egt} for generating test cases from symbolic
inputs.  \textsc{Egt} was a predecessor of \textsc{Exe}~\cite{exe} which
finally introduced a precise handling of pointers with symbolic offsets.
\textsc{Exe} was later completely redesigned and the new tool
\textsc{Klee}~\cite{klee} set to execute the \llvm code. \textsc{Klee}
optimizes solver queries, has better search heuristics, and other improvements
like a file system model to simulate a filesystem without really involving it.
This allowed for an analysis of \textsc{Coreutils}, the basic set of utilities
most Linux users work with and to actually find errors there.  \textsc{Klee}
was used also for checking a tool equivalence. It took two implementations of
the same and symbolically compared the results. Several deviations from the
standard behaviour were found that way. This was later improved more in
\textsc{UC-Klee}~\cite{ucklee}. First, it compares memory contents of both
implementations. Second, it makes the memory symbolic when accessed
(on-demand). Third, it checks that both implementations behave the same with
respect to error conditions like division by zero. \textsc{Klee} and
\textsc{UC-Klee} are put into concolic executors since it allows to invoke
library functions dynamically.

Since any of the concolic execution tools is prone to selected strategy that
decides which states are explored next, they were studied and compared in
\textsc{Crest}~\cite{crest}.  It is a tool very close to \textsc{Cute} in what
it performs, except it is freely available, source codes inclusive.

Instead of generating only a single input, \textsc{Sage}~\cite{sage} defines
the term \emph{generational search} to generate as many inputs as possible at
once. This is unimportant for cases, where the whole state space is
explored. Nonetheless, this is not the case as real programs cannot be covered
fully. So generating more input often helps more coverage. The same as above,
it records branching conditions from the first dynamic run, negates all of
them one by one and asks a solver for a solution.  For each positive reply the
solver gives, \textsc{Sage} generates a separate input. The whole tool does
not use any intermediate language, it symbolically executes directly the x86
binary code. So it can be used also in cases where source code is not known.
The whole approach in \textsc{Sage} is called \emph{automated whitebox fuzz
testing}. \textsc{SmartFuzz}~\cite{smartfuzz} works very similarly, but the
set of errors the tool aims at is limited only to integer errors.
\textsc{SmartFuzz} captures signed/unsigned conversions for example. The
generated inputs are filtered, so more eager heuristics that allow more inputs
generation are allowed.

None of the above support automatic bi-directional conversion across the
boundary of the symbolic and concrete execution. This was one of the two
novelties in \textsc{S${}^2$e}~\cite{s2e}: \emph{selective symbolic
execution}. It was published by some of the authors earlier to solve the
problem of path explosion in libraries and system calls~\cite{se_selective}.
As we are mainly interested only in some parts of libraries, the symbolic
execution does not fork and only the execution is forked at branches when it
gets back to our code from the library. Note this is somewhat similar to the
barrier slicing. The second novelty is a true concolic execution because it
runs the \textsc{Klee} symbolic executor and a modified \textsc{Qemu} virtual
machine in parallel. Those two communicate with each other and exchange
information, especially the symbolic state is updated from the dynamic
execution and concrete values instantiated from the symbolic run whenever
needed. Converters in both directions from x86 code to \llvm have to be in
place to have both executions running. \textsc{S${}^2$e} was used to to find
errors in Windows network drivers, and allegedly it was the first one to
employ the symbolic execution for performance analysis too.
\textsc{S${}^2$e}'s parallel engined works on the top of \textsc{Ddt${}^+$},
which is a reimplementation of a closed source tool \textsc{Ddt}~\cite{ddt}. A
very similar to \textsc{S${}^2$e}, but less generic framework is
\textsc{BitBlaze}~\cite{bitblaze}. It analyses binary files and creates
intermediate code with a reduced instruction set. This code is executed
symbolically.  Concurrently, the analyzed program is executed dynamically by a
component that is capable of monitoring and binary instrumentation. These two
are glued together by a third component that ensures communication between
the first two. One of unusual uses is generating exploits by a comparison of
unpatched and patched versions.  \textsc{BitBlaze} was used for checking HTTP
and NTP protocol servers~\cite{bitblaze_net}.
\textsc{Bitscope}~\cite{bitscope} is built on the same cornerstone as
\textsc{BitBlaze}, but provides understanding of dependencies between program
inputs and outputs. Both were used for recognizing malware in software.
Another true concolic executor is \textsc{Sjpf}~\cite{sjpf} stemming from
\textsc{PathFinder-SE}~\cite{pathfinder_se}. The difference is that it
symbolically executes the Java bytecode, therefore performing the symbolic
execution more directly. Nevertheless its downside is that it still does not
track constraints when switching from symbolic execution to concrete as the
ones above.

\section{Program Slicing}\label{sec:slicing}
As was described in the previous section, the symbolic execution is prone to
slowness when analyzing programs with loops or many branchings. We described
some techniques how to cope with this problem in that section too. But in case
we are interested only whether some code locations in a program are reachable,
another approach can be used in an analyzer to help the symbolic execution.
The analyzer can simply remove all the code which does not affect the flow to
those code locations in any way.  This is exactly a task that can be performed
by \emph{program slicing}. This is a technique that was described by Weiser in
his PhD thesis in 1979~\cite{slicing_phd} and later also
published~\cite{slicing}. As the slicing leaves only the code that really
affects particular code locations, the slicing may bring a considerable
speed-up to techniques working with code in general, not only to the symbolic
execution.  That way, analyzers do not need to evaluate every path that is in
the code, only those relevant to some code locations, e.g.\ lines with
potential bugs.

\newcommand{\newsym}[1]{\ensuremath\triangleright #1 \triangleleft}

When we want to slice a program we have to specify what behaviour we want to
preserve in the program first. In the slicing, this is performed by defining
\emph{slicing criteria}. These are chosen lines of the program accompanied
with variables we are interested in at those lines. Note there is no
limitation on the variables like whether they are set or used on those lines.
Such defined criteria ensure that values of variables at those lines must be
equal in both original and sliced program for any execution. We can slightly
loosen the requirement of ``any path'' to specified execution paths as we will
see later nonetheless.  According to the original
algorithm~\cite{slicing_phd}, when a slicer is given the slicing criteria, it
processes functions in a program independently as follows. The slicer starts
with some function and assigns a set of \emph{relevant variables} to each line
of the function. Relevant variables sets are filled initially by variables
from slicing criteria when the line has assigned some criteria or the sets are
left empty otherwise.  Further, according to the semantics of program
commands, slicers propagate relevant variables among function lines and insert
discovered relevant variables into sets on corresponding lines:
\begin{itemize}
\item All relevant variables from a successive line are copied to the current
line's set except for those which are set on the current line. This is a
simple information \emph{propagation}.
\item If the current line sets some variable which is relevant at a successive
line, the slicer adds all variables which the current line references.
I.e.\ the slicer performs computation of \emph{direct dependency}.
\item If the current line is a branching (or loop head) and some line in the
branch (or loop body) sets a relevant variable, the slicer adds all referenced
variables to the current line's relevant set. This is \emph{indirect
dependency} and serves the purpose of leaving in place branches and loops
which indirectly affect slicing criteria.
\end{itemize}
The slicers continue evaluation the function repeatedly until the sets stop
increasing in size, i.e.\ a fixed point is reached.  From the content of the
sets and the program semantics we can safely claim, which program lines have
no impact on the initial slicing criteria. Those are exactly the lines, where
intersection between variables that are set on that line and relevant
variables on the same line is empty. Slicers also have to leave lines with
branches or loop heads that affect some unsliced lines from the previous
sentence. The rest can be removed from the body of the function. Let us
demonstrate the computation on the example of Figure~\ref{fig:lst_ai_intro}.
Suppose we want to slice the program according to line 5 and variable \code{i}
there. The computation of relevant sets denoted as $R_{5,i}$ is demonstrated
in Table~\ref{fig:tab_slicing}. We stamp newly added variables in the concrete
iteration of relevant sets computation by $\newsym{~}$ in the table. We also
note the rule which applied to add the variable: the variables are marked with
one of C, P, D, I in superscript. It means their source is slicing criteria
(C), propagation rule (P), or direct (D) and indirect (I) dependency. The
result of the slicing is that we cannot slice anything from the program,
because \code{i} on line~5 depends on all lines, including the function
parameter \code{int x} on line~1.

The process described above is only intra-procedural. To extend the slicer
to be inter-procedural, we have to change the slicer to pass the relevant
variables among functions. That is down to a called function at every call
instruction and back up from the called function. As relevant variables
propagate from the bottom up, the slicer has to pass the variables from a call
site to return points of the called function and when the slicer is done with
propagation of the variables through the whole function to its beginning, the
variables are transferred back to the call instruction. Of course the slicer
has to take care of actual with formal parameters binding and of the
relationship between return value and the variable the result is stored into.
The same as above, the process repeats until a fixed point is reached.
Inter-procedural slicers were discussed by Horwitz et al. and
Gallagher~\cite{slic_inter1, slic_inter2}.

Originally, there were only slicers implementing the technique as described
above. These are \emph{static} slicers and they slice such that any possible
input to a program is handled the same as in the original code. But slicing
is often used for debugging purposes~\cite{slicing_debugging} and there, we
are usually interested in a value of some particular variable only when some
specific input was passed to the program.  Imagine we have a bug in the
program and know what was passed on input. We also know that a variable
printed on the output has an invalid value. The slicer can be therefore
limited to this input, in particular, the input need not trigger visiting all
bodies of branches or loops, so this allows for much more code removal than
static slicing.  This approach is called \emph{dynamic}
slicing~\cite{slicing_dyn}. Note the slicer has to look at conditions and
track a path condition and this is usually slow. So this obviously results in
a drawback: computing a dynamic slice is more computationally expensive than
computing static slices. Since control flow graphs are used to compute slices,
there appeared techniques to overcome the problem by pruning of the control
flow graphs. They can significantly decrease the computation
time~\cite{slic_prunecfg1, slic_prunecfg2}, but the result is less precise.
\emph{Hybrid} slicing tries to tackle this problem by combining the dynamic
slicing with dynamic runs.  Using breakpoints, slicers record which code
locations are visited with which input and compute the dynamic slice
according to the locations.  It is both faster and less space consuming than
the dynamic slicing.  The dynamic slicing can be improved such that it
works with more than one input~\cite{slic_quasi, slic_union}, which can be
generalized as describing input as constraints~\cite{slic_constraint,
slic_constraint_bw}.

There is also an orthogonal method to the static slicing.  We may be
interested in all program statements which \emph{are affected} by some
variable at some line. Since the standard slicing is sometimes called
\emph{backward} slicing, the orthogonal approach was named \emph{forward}
slicing~\cite{slic_forw} and is used for dead code removal for example. The
forward and backward slicing can be generalized as
\emph{chopping}~\cite{slic_chopping}. The chopping requires two distinct sets
of slicing criteria: one about what a variable affects (forward slicing), the
other about what is affected by that variable (backward slicing). The slicing
can be abstracted a bit more~\cite{slic_abstract}, such that it can answer
which values in a variable affect statements in the slicing criterion.

When taking over and reusing a library, a programmer can reduce the library
code size using slicers. Programmers just need to specify the parts of the
library which are used, and the library is sliced~\cite{slic_interface}.
The authors suggest also use in the reverse engineering -- slicing closed code
by some guessed criteria helps recognition tools to find patterns and reverse
engineer the code. Further, some variables like temporaries or iterators are
unwanted in slices when debugging. They can be disabled to appear in the
slice~\cite{slic_stop}. Similar is the \emph{barrier}
slicing~\cite{slic_barrier} which wants us to define a set of statements that
should be ignored by a slicer. This is done transitively, no further
statements starting at the barrier are put into the slice. So barriers are
often at interfaces of massively used, and therefore likely error-free,
libraries.  The slicing can be used also for finding bugs using \emph{dicing}:
two slices are compared, where one has to be a correct slice (a variable has a
good value) and one incorrect (a variable has a bad value). Intersection of
the slices are lines of code which are likely correct. The bug is in the
others of the incorrect slice. Slicers can work also on the semantic level, to
not preserve syntax, but leave semantics with respect to
criteria~\cite{slic_amorphous}. Slicers can reason about paths in a
program~\cite{slic_path} such as ``why this statement on this path is not
executed?'' Finally, the slicing can be used for model checking~\cite{slic_mc}
and for concurrent programs too~\cite{slic_concur} and the slicing can be used
for model checking~\cite{slic_mc}.  More details about slicing and its
variations and uses can be seen in various surveys and
reviews~\cite{slic_overview95, slic_overview96, slic_overview98,
slic_overview01a, slic_overview01b, slic_overview05, slic_overview12}.

\section{Model Checking}
This is a bit different technique from static analysis we are dealing in this
thesis with. It does not check the program proper, but instead tries to prove
some properties on a model of the program. The model is represented by some
finite state machine and the properties are a specified behaviour which has to
be satisfied in time. For example a property can be ``This message has to come
as a reply in the future.'' Properties are described by logic formulas,
usually in the temporal logic like LTL or CTL (see more about temporal logics
in {\O}hrstr{\o}m and Hasle's book~\cite{temporal}). Correctness of the system
is then established, if every run of the system corresponds to a particular
formula. If it does not for some case, a counterexample consisting of system
states is generated. Obviously, even if no counterexample is generated, the
system still might not be correct in case its model or checked properties are
incorrect. And yet, according to Ball et al., developing a thorough and
precise model may take years~\cite{mc_years}.

There are two basic approaches to model checkers depending on how the states
are stored: \emph{explicit} and \emph{symbolic}. Explicit model checkers
enumerate the states one by one as was implemented e.g.\ in
\textsc{Divine}~\cite{divine}. Symbolic model
checking~\cite{mc_symbolic1,mc_symbolic2} started representing sets of states
by boolean functions. For example if we have two 1-bit variables \code{a} and
\code{b} and their values are always the same (1 and 1 or 0 and 0 pairwise),
we can represent it by a formula \code{a} $\Leftrightarrow$ \code{b} instead
of storing all the values. The formulas can be represented in a space-saving
way by \emph{Reduced Ordered Binary Decision Diagrams}~\cite{bdd} (ROBDD or
simply BDD).  Despite the exponential complexity still persists (due to the
path explosion problem), BDDs helped to limit the memory use a lot.  Both the
symbolic model checking and BDDs allowed more complex systems to be checked on
the same hardware (cf.~\cite{mc_symbolic1} again).  Symbolic model checking
was later improved to defer the path explosion problem more by techniques like
\textsc{Cegar}~\cite{cegar}. \textsc{Cegar} starts with an abstraction of the
model and makes it more precise whenever a \emph{spurious} counterexample is
produced.  And only in that part of the model which is too abstract. Examples
of symbolic model checkers are \textsc{NuSmv}~\cite{nusmv} or
\textsc{Bebop}~\cite{bebop}.

In parallel to the BDDs-based model checking, \emph{bounded model checking}
exists. Instead of exploring the whole state space of the system model, there
is a predefined constant \textit{k} which determines the maximum length of
execution to be explored. If a bounded model checker explores all these
\textit{k}-prefixes, either \textit{k} is incremented and the checking
continues, or the model checker is stopped. It can have two reasons: the user
set the top bound for \textit{k} (the analysis is of course incomplete in that
case), or the execution is infeasible, because it runs out of resources. Few
representatives are \textsc{Esbmc}~\cite{esbmc} and
\textsc{Llbmc}~\cite{llbmc}. Model checking and bounded model checking are
orthogonal techniques, it was shown in~\cite{bmc_survey} that both of them
reason about different class of properties. (Nonetheless there can be an
intersection.)

Due to the path explosion problem and relatively slow computers in the past,
use of the model checking was limited rather to check small reactive systems
in industry like manufacturers of processors and automotive control systems.
Later, there started appearing more successful tools to check large software
projects. For example model checker \textsc{Blast}~\cite{blast} failed first
due to inability to reason about pointers in the Linux kernel
drivers~\cite{blast_kernel}. It was shown later, that \textsc{Blast} needs an
instrumented code and then works significantly better.  So it is currently
successfully used in the Linux Driver Verification toolkit~\cite{ldv} to
verify Linux device drivers. \textsc{Blast} was also extended to handle
concurrent data accesses~\cite{blast_concur}.  \textsc{Spin}~\cite{spin} is
aware of concurrent data accesses too. So \textsc{Spin} checked a model of the
Linux common filesystem layer~\cite{spin_vfs}. Earlier, some individual file
systems which use this layer were checked by Yang et al.\
too~\cite{mc_kernel_fs}. Other subsystems like the networking stack were
also checked~\cite{mc_kernel_ip}.  \textsc{DDverify} verifier~\cite{ddverify}
and \textsc{Saturn}~\cite{saturn} were used for automated verification of the
Linux kernel drivers.  \textsc{Undertaker}~\cite{undertaker} is a model
checker for the Linux kernel configuration system. It validates dependencies
in the \texttt{Kconfig} configuration files.  There are also model checkers
aimed specifically at the Linux kernel like \textsc{Predator}~\cite{predator}.

Similar to the attempts on the Linux kernel, Microsoft started adopting their
model checker \textsc{Slam}~\cite{slam_cav, slam_popl, slam_decade} and
\textsc{Slam 2}~\cite{slam2} in their driver verifier \textsc{Sdv}~\cite{sdv}
for checking Windows drivers. They apparently succeeded as they require every
driver to be checked by this toolkit with over 160 predefined temporal rules
before the driver is accepted by Microsoft.  \textsc{Sdv} is a proprietary
software with rules tailored to Windows drivers.  Therefore it cannot be used
for checking open source projects. So Clarke et al.~\cite{cbmc} adapted
Microsoft's approach using \textsc{Cbmc} model checker and defined at least
seven rules to check Linux kernel drivers. The rules were later extended by
Post and K\"{u}chlin~\cite{slicx}.  There exist also successful tools for
modelling and verifying Java programs like
\textsc{PathFinder}~\cite{pathfinder}.

As this thesis does not deal with the model checking, curious readers are
pointed to a survey~\cite{mc_survey} and a book~\cite{mc_book} about model
checking.  There is also a survey about bounded model
checking~\cite{bmc_survey} available.

\section{Industrial Tools}\label{sec:industrial}
There are many successful tools based on static analysis. Unfortunately for
some of them, it is not really known what technique they are based on.  It is
due to secrets the companies hold to not let their competitors know. What can
be found in the reference materials, the tools often use combination of the
techniques we have already described. They work on the top of various pattern
matching, data-flow analyses, and some maybe use the abstract interpretation.
Even though the internals are not known exactly, we at least try to summarize
the tools and the little about what their providers make available about them.
All the tools below are capable to handle C, some of them C++ and other
languages like C\# or Java.

The data-flow analyzer \textsc{Lint} was an inspiration for Gimpel's
\textsc{PC Lint/FlexeLint}~\cite{pclint} and Cleanscape's
\textsc{LintPlus}~\cite{lintplus}. The same as their predecessor, they aim at
finding suspicious code constructs. MathWorks'
\textsc{PolySpace}~\cite{polyspace} was one of the first abstract interpreters
capable of handling real code. But according to the authors, the tool's upper
limit is still approximately 50 thousand lines of code that can be analyzed.
The tool was inspired by the crash of Ariane~5 (see introduction). Another
abstract interpreter which can handle large scale programs is AbsInt's
\textsc{Astree}~\cite{astree}. It can be tuned to be sound, i.e.\ to produce
no false positives.  \textsc{Astree} also combines several abstractions to
perform more accurate analysis~\cite{astree_combine}.

\textsc{xgcc} was a predecessor of Coverity's
\textsc{Prevent}~\cite{coverity}, and \textsc{Metal} for Coverity's
\textsc{Extend}. \textsc{Prevent} is run regularly to check the Linux kernel
and the results are provided to kernel developers.  KlocWork's
\textsc{Insight}~\cite{klocwork} and GrammaTech's
\textsc{CodeSonar}~\cite{codesonar} are specialized to finding security and
reliability issues. \textsc{Insight} was originally intended as an optimizer
for C code. IBM's \textsc{Apollo}~\cite{apollo} aims at checking web
applications, \textsc{PHP} scripts more precisely.

Microsoft developed many tools based on the symbolic execution.  Even though
\textsc{PREfix}~\cite{prefix} explicitly does not mention it, it is used there
to find errors in memory management for example. \textsc{PREfix} has no
problems analyzing large projects thanks to computation of function summaries,
but needs user intervention to model library functions. This was improved in
\textsc{Archer} of Xie et al.~\cite{archer}. A similar use as IBM's
\textsc{Apollo} above has \textsc{Pex}~\cite{pex}. It is a symbolic executor
intended for checking .NET web applications. They developed also a fuzz tester
\textsc{SAGE}~\cite{sage}.  It is a symbolic executor working directly on the
x86 assembly code. It is therefore able to work also with libraries whose
source code was not made public.  \textsc{YOGI}~\cite{yogi} is a tool which
combines symbolic execution with dynamic execution. It was experimentally used
to replace \textsc{SLAM}~\cite{slam_decade} in \textsc{Sdv}~\cite{sdv} with
comparable better results. Microsoft also released \textsc{PREfast}, a tool
that contains a set of rules that a code should fulfill and with a use of
annotations, it checks code against the rules. Its main use is to disallows
developers to commit changes that contain certain classes of errors such as
inter-procedural integer overflows.

\section{Summary of the Linux Kernel Checkers}
Some of the tools above can analyze kernels or only drivers of operating
systems. In this sections we specifically sum those which are capable to
handle the Linux kernel.

One of the basic ones is based on the pattern matching and every developer
is supposed to use it before sending patches to the kernel maintainers. It is
named obviously after what it is intended to do as
\textsc{checkpatch}~\cite{checkpatch}. It only checks for patterns like if
somebody introduces a whitespace to the code or if the code is properly
indented. Some better analysis can be performed by a semantic parser called
\sparse~\cite{sparse}. It is still a tool working with some sort of patterns
matched against the parse tree.  It was written by the founder of the kernel
and is tailored exclusively to the kernel as the checks have only a little
sense elsewhere.  The checks include basic intra-procedural checks for proper
locking, use of zero instead of \code{NULL}, or extended type checking using
the annotated code in the kernel. Yet another pattern matcher is Dan
Carpenter's source matcher \textsc{Smatch}~\cite{smatch}. Formerly, it was a
set of Perl programming languages scripts. They were later turned into C code
and \sparse was used as a base. \textsc{Smatch} added many more checks
regarding the Linux kernel like doing direct memory accesses on a stack or
passing incorrect arguments to picked functions. Even though Coverity's
\textsc{Prevent} is an industrial tool, it is constantly checking the kernel
and reports are made available. The reports are of diverse kinds like dead
code, null pointer dereferences, use before test and so on. Many of them are
getting fixed. These four tools are extensively used by the kernel developers
and errors reported are getting fixed constantly~\cite{linux_faults}.

There are also tools that can be used, but are not used that massively by so
many developers. These include \clang~\cite{clang} and
\textsc{Cppcheck}~\cite{cppcheck}.  Historically, \textsc{xgcc} was used to
find errors in the kernel~\cite{metal_kernel}, mainly for errors like if there
are proper checks of untrusted data coming from the network or an user to the
kernel. It found over one hundred of such errors. As \stan~\cite{stanse} is a
tool based on a similar technique, it was used the same way and found over one
hundred another errors.

There are also projects based on model checking which are successful to check
the Linux kernel. \textsc{Ldv}~\cite{ldv} is one of those. The same as
\textsc{Prevent}'s reports, \textsc{Ldv}'s reports are sent to the subsystem
maintainers so that they can look at them and fix them.
