\chapter{State of the Art}\label{sec:state}
This section describes the current state of the art of the code analysis.  We
especially describe techniques which are referred to later in the text.  In
this overview we mostly limit ourselves to analyzers of C code.

\paragraph{Static Analysis} In our understanding, the term \emph{static
analysis} is analyzing code without directly executing it on a processor.
In other words, it performs exactly the opposite of the dynamic analysis where a
program is executed and either automatically fed by some input in some form or
tested manually by testers. The static analysis approach has a nice
side-effect that a checked program need not be finished when we want to
analyze it statically.  So the static analysis is not as limited as the
dynamic one, it can be used whenever we want in the development cycle. Static
analysis is standardly used in several fields of the computer science aside
from the code checking:
\begin{itemize}
\item \emph{Optimizing compilers} use results of the static analysis to
eliminate unused code, propagate constants where variables would have to be
used otherwise, or to remove implicitly added runtime out-of-bounds checks if
the compiler knows that the bounds are never exceeded (this is implemented in
a Java compiler).

\item \emph{Debuggers} use results of the static analysis to limit themselves
only to code that is relevant to variables a programmer is interested in.
Debuggers can also print possible sources of the value currently stored in a
tracked variable for example.

\item \emph{Program tests} can be generated using the static analysis to cover
more code. The analysis can establish some knowledge about the code and help
to find inputs that exercise more code.

\end{itemize}

Moreover, if the static analysis is well-defined according to some rules
defined e.g.\ by Cousot's framework~\cite{cousot} described later, we are able
to decide whether the analysis is sound, complete, and whether it terminates.
This is useful for example when building an optimizing compiler to prove it
does not generate invalid code.

\section{Pattern Matching}\label{sec:pattern_matching}
The simplest automatic analysis we can think of is to find some predefined
patterns in code of a program.  We can take an instance of a regular
expression pattern matcher like \texttt{grep}, give it a pattern ``/ 0'' and
see if we did a primitive zero division error somewhere in the code. But this
approach may be a source of false reports. This happens when developers
comment their code with comments similar to ``/ 0 happened here''.  But there
of course exist more robust techniques based on pattern matching.  They, in
some way, parse the code (which also removes the problem with the comments
like the one above) and do pattern matching on some internal structure. If we
recall the abstract syntax tree from Figure~\ref{fig:intro_parsing_ast}, we
can design a checker which would match AST subtrees for instance. Then to
match division by zero errors, we simply try to find a subtree in an AST like
of Figure~\ref{fig:ast_to_match}.  The left leaf is left intentionally blank
to express a wildcard.

    level distance = 1cm}]

	    child { node [ast] {} }
	    child { node [ast] {\code{0}} }
	  ;
    \end{tikzpicture}
\end{center}
    \caption{A subtree of an AST to match for a primitive division by zero.}
  \label{fig:ast_to_match}

One of the most known tools based on the pattern matching is
\fbugs~\cite{findbugs}. Despite it is intended to find errors in Java
programs, we mention it here, because it became very popular. This happened
thanks to the ease of use, hundreds of error types it can find, scalability
and precision. It is able to prioritize the reports according to their weight
so that it usually puts real errors on the top of the list in favor of false
reports.

\textsc{Coccinelle}~\cite{coccinelle} is another pattern matcher. The authors
call it rather a semantic patcher, because it can understand the semantics of
the checked code. It uses a special language SmPL for declaring a semantic
patch. The patch is utilized for matching against control flow graph nodes and
changing them arbitrarily. The graph is then converted back to source code.
\textsc{Coccinelle} is used mainly for modifying code to use newer interfaces
like replacing two calls to \code{kmalloc} and \code{memset} functions by a
single call to \code{kzalloc} which does the same as the two together.  Since
it is able to match the control flow graph, it can be used for error pattern
matching too~\cite{wysiwib}. As an example, we can pick missing checks for
allocations: a call to an allocation function with no condition to check the
return value afterwards.

\section{Data-flow Analysis}\label{sec:data-flow}
One of the basic static analyses is \emph{data-flow analysis}. It collects
information about some data flowing through points in code. The data can be
some information about values of variables, about statements etc. We usually
analyze each program line separately to see what are the effects of the line
and then put this information to a transfer equation. It describes relations
between what holds before and after the execution of a particular line. Next,
we stabilize this set of equations, usually by computing the fixed
point of the equations. The fixed point is either the least or greatest
solution to the equations, depending on the performed analysis. Data computed
using the data-flow analysis can be utilized for various code optimizations or
program analysis.

Let us demonstrate the data-flow analysis on an example and then generalize
that approach. We chose one of the basic data-flow analyses: \emph{reaching
definitions}~\cite[Section 9.2.4]{dragon}. At some point $p$ in code, we are
interested in all assignments where some variable $\var{x}$ was set and was
not re-set during the control flow to the point $p$. This lets us reason about
many questions like whether a variable can be replaced by a constant as an
optimization.  We show the analysis on the code which is in the second
column of Table~\ref{tab:state_df_eq}.  Intuitively, the reaching definition
set on line~\ref{lst:state_rd_ref} is a set of four labels
$\{\ref{lst:state_rd_def1},\, \ref{lst:state_rd_def2},\,
\ref{lst:state_rd_def3},\, \ref{lst:state_rd_def4}\}$, where we define
\code{a} and \code{b}, because all of those definitions can reach
line~\ref{lst:state_rd_ref}.
%Note that line~\ref{lst:reach_def0} does not belong there because \code{a}
%was re-set later on both paths to line~\ref{lst:reach_use}.

Now we derive the information algorithmically using the data-flow analysis by
constructing the transfer equations and computing their fixed point. In the
case of the reaching definitions analysis, we can simply hold the definitions
in the equations.  That is, initially at the beginning of a function, the
equation there holds an empty set. Then this empty set is gradually
transformed according to statements and the definitions are put into the set.
Since we need to propagate information about definitions, there are two
types of propagations:
\begin{enumerate}
\item On statements which are assignments, the contents of incoming sets is
mangled. The equation takes the set of statements from the predecessors,
removes all statements that assign to the same variable as the current
assignment and adds itself. If there are more predecessors, we use the set
union as we want to propagate definitions from both branches.

\item Equations for other statements simply copy data from predecessors.

\end{enumerate}

We can notice, that every equation is of this form:
\begin{equation*}
f_s = gen_s \cup (pred_s \setminus kill_s)
\end{equation*}
where $gen_s$ is the set $\{s\}$ that \emph{generates} $s$ if it is an
assignment, otherwise $gen_s$ is empty for that particular equation.  $pred_s$
is a union of $f_{s'}$ for each predecessor $s'$.  Finally, if $s$ is an
assignment to \code{x}, $kill_s$ is a set that contains other assignments to
\code{x}.  Otherwise $kill_s$ is empty.

Now, we would like to compute the least fixed point of the equation. We can
use Definition~\ref{def:fixpoint} for that purpose, but we need a single
composite function $F$. We achieve that by rewriting the transform equations
to functions taking as parameters all others functions. For all $x \in [0,
7]$, the functions look like $f_x=F_x(f_0,\, f_1,\, \dots,\, f_8)$. The
composite function is then $F=\lambda x . (F_0(x),\, F_1(x),\, \dots,\,
F_7(x))$. The fixed point computation for our example of
Table~\ref{tab:state_df_eq} is shown in Table~\ref{tab:state_df_fp}.

\begin{dfn}[Fixed point]\label{def:fixpoint}
A \emph{fixed point} of a function $F : D \rightarrow D$ is an element $x \in
D$ such that $F(x) = x$. The set $fp(f)$ of fixed points of $F$ is $\{x \in D
\mid F(x) = x \}$.
\end{dfn}

There is another requirement to ensure a fixed point exists and that some
data-flow analysis finds one and terminates. The function $F$ has to be
monotone and has to work on an ordered set, lattice in particular.  Since we
have not defined lattices yet, this is described in more detail later in
Section~\ref{sec:ai}. The requirement stems from works of Kleene~\cite[pages
348--349]{kleene} and Tarski~\cite{tarski}.

Now we generalize the approach of the data-flow analysis. In the data-flow
analysis, we have a pair of equations for each statement $s$ of a program:
\begin{equation} \label{eq:data_flow}
\begin{aligned}
out_s & = trans_s(in_s) \\
in_s & = join_{p\in pred_s}(out_p)
\end{aligned}
\end{equation}
We define a transfer function $trans_s$ to transform the input set $in_s$ of a
statement $s$ to the output set $out_s$ according to the effect of $s$ in a
particular analysis. The join operator $join$ takes output sets from all
previous nodes $pred_s$ and combines them together. The previous edges are
either predecessors or successors, depending whether an analysis is forward or
backward (Section~\ref{sec:types_anal}). For forward analysis, $in_s$ set is
bound to an incoming control flow graph edge(s) of the node $s$ and $out_s$
set to the outgoing one(s).  Other way round for a backward analysis. Further,
$join$ is usually a set union or intersection depending whether the analysis
is may or must (cf.\ Section~\ref{sec:types_anal}).

For the reaching definitions example above, we define the join operator as
the set union ($\bigcup$) and the transfer function in this notation to hold
statement line numbers as follows ($S$ is the set of all statements in the
analysed program):
\begin{align*}
trans_s(in) & = gen_s \cup (in - kill_s) \\
gen_s & =
\begin{cases}
\{s\}     & \text{$s$ is an assignment} \\
\emptyset & \text{otherwise}
\end{cases} \\
kill_s & =
\begin{cases}
\{x \in S \mid x \text{ assigns to \code{a}} \} & s \text{ is an assignment}
  \\
& \text{(let } s \text{ assign to \code{a})} \\
\emptyset & \text{otherwise}
\end{cases} \\
\end{align*}

These are the most known analyses which we believe the reader knows. If not,
they are defined in every superior literature about compilers~\cite{dragon,

The simplicity is also a reason, that the data-flow analysis is used in
many tools including checkers.  The very common place of extensive use are
compilers. Apart from generating code, they generate warnings or errors when
some suspicious code is found. A good example is \clang~\cite{clang}. It is a
C/C++ compiler from the \llvm toolkit~\cite{llvm} and contains also a static
checker framework.  There are checkers for finding errors in memory handling,
locking, or pointer operations. Sadly, the results are mostly imprecise
and contain many false reports.  \textsc{Cppcheck}~\cite{cppcheck} works on
the same principles so it checks for very similar set of errors. Earlier,
while compilers were still not so evolved, other data-flow analyzers were used
to fulfill this role. One of the most known analyzers at that time was
\textsc{Lint}~\cite{lint}. It was capable of finding errors like division by
zero, constant conditions which always evaluate either to true or false, or
use of uninitialized variables.  Later, \textsc{Lint} resulted in some clones
like \textsc{LCLint}~\cite{lclint} which were augmented to catch errors like
buffer overflows. \textsc{LCLint} was later renamed to \textsc{Splint}.  These
tools mostly have a support for annotations so more speculative checks can be
enabled. In that case, the code has to be heavily annotated, otherwise the
false positive rate is unbearable.  Nonetheless, the development of
\textsc{Splint} stagnated over the past decade due to obvious reasons: the
most errors and warnings \textsc{Splint} was used to report are reported by
modern compilers nowadays. It was also thanks to the work of Reps et
al.~\cite{dataflow_large}. They showed that it is possible to compute the
data-flow sets precisely for the whole program inter-procedurally in
polynomial time. They presented that on finding possibly uninitialized
variables.  There exist also analyzers that are not so powerful to handle
large code bases or require high user intervention to annotate the checked
code. We give as an example \textsc{Aspect}~\cite{aspect}, which compares a
program specification of input-output behaviour provided by a user to
data-flow models extracted from the code. This section is only a short
introduction to the data-flow analysis, more information and many examples are
available in the literature~\cite{dragon, muchnick}.



\section{Abstract Interpretation}\label{sec:ai}
The data-flow analysis is only a limited tool. It only collects information
about data and how they propagate through the control flow of a program.
Therefore it is not as useful for designing more complex analyses which are
usual for code checking and possibly elsewhere. There was a call to formalize
the process for a convenient design of more complex analyses. Cousots' work on
this topic stretches back to the 1970s. Back then, they described a framework
called \emph{abstract interpretation}~\cite{cousot}. It is designed to allow
users of the formalism to easily design an analysis, decide whether it is
correct and sound, to formulate more precision levels of the analysis
consistently and more.  The formalism was designed in a way that we can also
interpret analyses like the data-flow as an abstract
interpretation~\cite{ai_intro}.

As the name of the analysis suggests, the data domains used in analyzed code
are replaced by abstract ones. The abstract domain is a \emph{complete
lattice}~\cite{lattice1,lattice2} as defined in
Definition~\ref{def:colattice}. Since the analyzed program is re-defined to
work on a new (abstract) domain, we have to provide a new semantics of the
program over the domain.  There are no limits, the semantics can be arbitrary.
We define a \emph{monotone function} for all instructions in the analyzed code
to transform the program states over the instructions.  Thanks to the work of
Tarski~\cite{tarski} and Kleene~\cite[pages 348--349]{kleene}, we can always
compute fixed points of monotone functions which transfer states over
lattices.  Further, using lattices is effective. Their operators join and meet
can be used when an analyzer is interpreting a program.  For example, when a
branching occurs, the branch join operator simply takes the values present on
the outputs of both branches and performs the lattice join ($\sqcup$).

\begin{dfn}[Complete lattice]\label{def:colattice}
A \emph{complete lattice} $(L,\, \leq,\, \bot,\, \top,\, \sqcup,\, \sqcap)$ is
a partially ordered set $(L,\, \leq)$ where every set $X \subseteq L$ has a
least upper bound (supremum) $\sqcup X$ and a greatest lower bound (infimum)
$\sqcap X$. In particular, $L$ has a least element $\bot = \sqcup \emptyset$
and a greatest element $\top = \sqcap L$.
\end{dfn}

Let us demonstrate the abstract interpretation on an example, the same we did
for the data-flow analysis and then generalize it again. We take the same
example as is in Table~\ref{tab:state_df_eq} and reachable definitions
analysis again.  First, we have to define the abstract domain which the
abstract interpretation works with.  Since the analysis works with lines where
is an assignment and there can be more reachable definitions assigned to a
single line, the complete lattice will consists of a powerset of all
assignments in the program: $A=\{1,2,4,5\}$.  The abstract domain is therefore
the lattice $L_a = (\mathcal{P}(A),\, \subseteq,\, \emptyset,\, A,\, \cap,\,
\cup)$. Now we would have to define the transfer function over the lattice.
But it is the same as in Table~\ref{tab:state_df_eq}. So computing the least
fixed point is the same as is shown in Table~\ref{tab:state_df_fp} too.

As we see, the abstract interpretation is defined by the complete lattice,
and a transfer function. Formally, it is a tuple $I=\langle L,\, \circ,\,
\leq,\, \top,\, \bot,\, F\rangle$, where $(L,\, \leq,\, \bot,\, \top,\,
\sqcup,\, \sqcap)$ is a complete lattice, $\circ$ is either supremum
($\sqcup$) or infimum ($\sqcap$) lattice operator, and $F$ is a monotone
function defining the interpretation of a program. Obviously, the definition
can be loosened to require only a semi-lattice, as we need only one operator.

It is not yet obvious how to define the transfer function of a program. We can
define a transfer function $F_a$ working directly on the abstract domain.  It
obtains a new abstract state with respect to the original state and counting
in the effect of a program which is evaluated. Simply the same as we showed in
the example of reaching definitions, where the transfer function is defined on
an intuitive level as a composition of simple transfer functions performed on
single statements of a program.  For complex analyses it need not be so easy
to design such $F_a$, so there is also another option.  First, we define two
functions: \emph{abstraction} denoted $\alpha$ and \emph{concretization}
denoted $\gamma$, both accepting one parameter.  $\alpha$ returns an abstract
value corresponding to the concrete value on input and $\gamma$ does the
opposite. Now, we can jump back to the concrete values, perform the
transformation on the concrete values using a concrete transfer function $F_c$
and switch back to the abstract values.  $F_a$ is then defined as $F_a =
\alpha \circ F_c \circ \gamma$. To retain soundness of the analysis, the
resulting abstract value must be less than or equal to the one computed by a
direct $F_a$. We depict these two options in Figure~\ref{fig:state_ai_square}.

Now consider two lattices in Figures~\ref{fig:state_lat_concrete}
and~\ref{fig:state_lat_int}. Both are used for abstracting numbers in
programs. One is capable to remember only a single value of a variable and is
therefore useful for analyses like constant propagation~\cite[Chapter
9.4]{dragon}. The other lattice uses intervals to remember possible bounds of
some variable's value.  This is very useful domain especially when analyzing
bounded loops. When indices, and probably other variables in the body of the
loop are bounded, we can run some optimizations like loop unrolling~\cite[page
735]{dragon}.

There is a difference to be spotted between these two lattices. Although they
both contain an infinite number of elements (in theory), the latter lattice
has infinite height.  The Kleene's sequence is not necessarily finite there,
so the computation of a fixed point may not finish. There is a mechanism
suggested by Cousots~\cite{cousot} to overcome the problem. It is a
\emph{widening} operator $\triangledown$. Informally, the sequence is sped up
by widening, so that it converges eventually.  Imagine we have an intervals
lattice and an infinite loop where we increment a variable that was initially
set to zero. The infinite sequence of intervals computed for that variable
would be: $[0, 0],\, [0,1],\, \dots,\, [0,\infty]$. We can define the widening
operator to set one of the interval bounds to infinity after some time to
force the sequence finite: $[0, 0],\, [0,1],\, [0,\infty]$. The opposite of
the widening operator is a \emph{narrowing} operator $\vartriangle$. The
narrowing can help to make the abstract values more precise. When our widening
operator put an $\infty$ in an interval, we can narrow it back to some
abstract value which we know will never cause an increase. This is of course
not possible for our simple loop, but the reader is pointed to Cortesi and
Zanioli~\cite{ai_widening} or Cousots~\cite{cousot_galois} who demonstrate
widening and narrowing on more concise examples.

Intervals are not the only shapes to represent abstract values of numbers.
Octagons, elipses, polyhedra and other shapes are used too~\cite{ai_octagons,
ai_polyhedra, astree}. Several abstractions can be also combined to
communicate with each other and improve thus accuracy or any other aspect of
the analysis~\cite{ai_product}. This was utilized in tools like
\textsc{Astree}~\cite{astree} and \textsc{Clousot}~\cite{clousot}.
Strings~\cite{ai_strings1, ai_strings2} and foating point
numbers~\cite{ai_float1} can be abstracted too.  Further reading on the
abstract interpretation is in an overview~\cite{cousot_progress}, where is the
progress in the area of the abstract interpretation over 25 years. A not so
usual use of the abstract interpretation can be found too~\cite{ai_unusual1,
ai_unusual2, ai_unusual3}.  Engler et al.\ brought the abstract interpretation
to the attention in 2000 by a technique based on meta compilation in
\xgcc~\cite{metacomp}. There, program developers specify properties of the
checked system in \textsc{Metal} language~\cite{metal}, compile it into an
internal representation and the tool applies the specification on all
executable paths in a particular function.  Violation of the specification on
a path is then reported as an error.  \xgcc suffers from false positives
despite the use of false positive suppression algorithms like killing
variables and expressions, synonyms, false path pruning, and others.  Besides
the suppression algorithms, error reports from the tool are further ranked
according to their probability of being real errors. There are generic and
statistical ranking algorithms ordering error reports.  An extension
introduced later~\cite{lockchecker} provides an automatic inference of some
temporal properties based on statistical analysis of assumed programmer's
beliefs.  On the similar base as \xgcc is built \stan~\cite{stanse} which is
discussed in Chapter~\ref{sec:stanse}.  It is uncertain, whether some
industrial tool adopts the abstract interpretation too, see the discussion in
Section~\ref{sec:industrial}.

The correctness and soundness (cf.\ Section~\ref{sec:theor_props}) of an
analyzer based on the abstract interpretation is very prone to selection of
the abstract domain. While many employed abstractions are sound (produce all
possible errors), they are not so often complete (false positive rate is
high). For example, this was demonstrated on several abstractions of numbers
which we discussed above~\cite{ai_octagons}.




\section{Symbolic Execution}\label{sec:se}
In the previous section we described the abstract interpretation which is a
framework for fast, but typically imprecise analyses. As a result of
incomplete abstract analyzers, we often get a pile of false positives on their
output.  There also exist techniques which cannot formally generate false
positives, but are often slower than abstract interpreters or sometimes even
unsound.  One of such techniques is the \emph{symbolic execution}. The
technique was introduced in 1970's and described by several authors
concurrently~\cite{se_king, se_select, se_dissect, se_clarke}. The symbolic
execution represents program inputs by symbols. Therefore it does not need to
analyze the program many times with concrete, initialized, data. The program
is then executed and program expressions are manipulated including the
symbolic values. Let us demonstrate this on an example of
Figure~\ref{fig:se_example}. The example is two-fold.
\begin{figure}[htb]
\begin{center}
\begin{tabular}{c}
\begin{lstlisting}
int fun(void) {
  int a = read() + 1; (*@\label{line:se_ex_sset}@*)
  int b = 0;

  if (a == 0x12345678) (*@\label{line:se_ex_if1}@*)
    b = 1; (*@\label{line:se_ex_set}@*)

  if (a == 0x12345678) (*@\label{line:se_ex_if2}@*)
    a = 10 / b; (*@\label{line:se_ex_div}@*)

  return a;
}
\end{lstlisting}
\end{tabular}
\end{center}
  \caption{Example to demonstrate the symbolic execution.}
  \label{fig:se_example}
\end{figure}

First, there is a read from the user. How long it would take to exercise the
program to get to the true branch of the \code{if}s with dynamic testing?
With \code{int} being usual 32 bits, there is only one successful attempt out
of $2^{32}$ overall. Of course, with a static analyzer based on the data-flow
analysis or abstract interpretation, those true branches would be analyzed
quite quickly.  This brings us to the second point nonetheless. Many complete
analyzers generate a report about division by zero possible at
line~\ref{line:se_ex_div}. This is indeed a false positive because \code{b} is
set to \code{1} on line~\ref{line:se_ex_set} in case \code{a} is
\code{0x12345678}. The source of the false positive is that the analyzers do
not often remember conditions on branchings.

The symbolic execution tackles this problem. All inputs from the user are
replaced by \emph{symbols}. For a variable \code{a}, we denote the symbol
corresponding to \code{a} as $\sym{a}$. We allow symbols to be assigned to
variables similar to constants and other expressions. In our example, the
assignment on line~\ref{line:se_ex_sset} would set $\sym{a}+1$ to \code{a}.
Obviously we have to teach the symbolic executor to deal with symbols in
arithmetic operations and branching statements.  In the first place, the
symbolic executor needs to remember the currently evaluated program location
$l$ in a control flow graph.  Next, we need a storage. Symbolic executors
contain a \emph{symbolic memory} $\theta$ which assigns to each variable its
current symbolic value. So after evaluating the statement on
line~\ref{line:se_ex_sset}, $\theta(\var{a}) = \sym{a}+1$. To be precise in
the execution of code, the executor also has to remember the branches which it
chose to jump to at branching points. This information is stored in a
\emph{path condition} $\varphi$. It is a first-order formula~\cite{fol}
representing a necessary and sufficient condition over symbols to drive the
execution along the currently executed path to the actually evaluated point.
$\varphi$ is set initially to $\true$ and is evaluated and potentially updated
at every branch.  This happens on lines~\ref{line:se_ex_if1}
and~\ref{line:se_ex_if2} in our example. On line~\ref{line:se_ex_if1}, we
check whether $\varphi$ implies validity of the condition instantiated by the
current state of $\theta$. More concretely, the executor checks whether either
of the following two implications hold (at most one~\cite{se_king}):
\begin{align}
\varphi & \rightarrow \left( \theta(\var{a}) = \var{0x12345678} \right) \label{equ:se_fi} \\
\varphi & \rightarrow \left( \theta(\var{a}) \neq \var{0x12345678} \right) \label{equ:se_se}
\end{align}
There are three possibilities that can happen:
\begin{description}
\item[Implication \ref{equ:se_fi} is valid] The executor simply continues
execution by jumping directly to the \emph{true} branch.
\item[Implication \ref{equ:se_se} is valid] The executor simply continues
execution by jumping directly to the \emph{false} branch.
\item[Neither is valid] This means that both branches are possible, depending
on the symbolic input.  So to retain soundness, the symbolic executor has to
fork the execution of the whole program to explore both branches. In that
case, the path condition $\varphi$ has to be augmented in the respective
branches. For the true branch, $\varphi \wedge (\sym{a}+\var{1} =
\var{0x12345678})$ is a new path condition, for the other branch $\varphi
\wedge (\sym{a}+\var{1} \neq \var{0x12345678})$.
\end{description}
In our example, on line~\ref{line:se_ex_if1} neither implication is valid, as
$\theta(\var{a})=\sym{a}+1$ cannot prove validity of $\theta(\var{a}) =
\var{0x12345678}$. So the execution forks.  The situation is different on
line~\ref{line:se_ex_if2} nonetheless. There, we can get into two states, by
the two previous forked executions. In one, the path condition contains
$\theta(\var{a}) = \var{0x12345678}$ and symbolic memory is holding \code{1}
for \code{b}. In the other $\theta(\var{a}) \neq \var{0x12345678}$ and
$\theta(\var{b}) = 0$. From the first one, the executor can induce
Implication~\ref{equ:se_fi}, so it continues by jumping into the true branch.
And since $\theta(\var{b}) = 1$, no division by zero can happen.  Similarly
for the second fork: the executor jumps to the false branch of the \code{if},
i.e.~it skips the \code{if} completely. Note that this is one of the very
cases where other types of analyses like the abstract interpretation produce
false positives.

The three information above constitute the \emph{program state}. It is a
triple $(l,\, \theta,\, \varphi)$.  The executor works with these triples when
executing a program and updates the members accordingly as described above.
Due to the forks, the execution of a program is usually depicted as a
\emph{symbolic execution tree}. Nodes of the tree are labeled by program
states computed during the execution.  Edges of the tree correspond to
transitions between program states labeling their end nodes.  For
demonstration, we use the code from the data-flow example as was in the second
column of Table~\ref{tab:state_df_eq}. A control flow graph of the example is
shown in Figure~\ref{fig:state_se_cfg}.

We have not discussed performance of the symbolic execution yet. There are
two main sources of potential performance bottlenecks. First, symbolic
executors use \emph{Satisfiability Modulo Theory} solvers~\cite{smt} to reason
about validity of formulae like~\ref{equ:se_fi} and \ref{equ:se_se}.  Whereas
performance of individual solvers vary, complex queries are not easy to answer
for any solver. So it usually takes a long time for solvers before they
answer such queries. And worse, some queries can be also out of decidable
theories.  Second, the symbolic execution suffers from the \emph{path
explosion problem}, i.e.\ an exponential growth of paths that need to be
explored. Every branching in the code potentially multiplies the possible
number of paths through the program by two. It is even worse for loops, where
the symbolic execution tree can become huge or infinite, e.g.\ when analyzing
loops.  Hence, for real-world programs, the symbolic execution is barely
usable without incorporating other techniques to reduce the count of explored
paths.  Some tools try to tackle this issue by traversing more paths at once
to substantially reduce the size of the execution tree~\cite{cse_atva}.
Others~\cite{pex_ddse, G07}, compose symbolic executions of several functions
using summaries which results in an inter-procedural analysis. This
approach limits the need of execution to separate functions only.
\textsc{Exe}~\cite{exe}, \textsc{Klee}~\cite{klee}, and
\textsc{RWset}~\cite{rwset} are based on recording of already seen behavior
and pruning its repetition. Tools like Girgis'~\cite{se_limited} and
\ucklee~\cite{ucklee} also limit the tree depth by a hard-coded
constant, to obtain finite tree for the price of unsoundness.  Some works aim
at guessing the number of loops needed to reach some target location to help
analyzing large programs with many nested loops~\cite{partial_loop}.

Despite King originally introduced the symbolic execution for symbolic
debugging~\cite{se_king}, the symbolic execution is currently widely used for
generating unit tests which cover more paths in code than tests written
manually~\cite{se_generate_test1, se_generate_test2, se_generate_test3,
se_generate_test4, se_generate_test5, exe, klee}. Offutt and
Seaman~\cite{se_generate_test2} use the symbolic execution also for re-writing
program internal variables as functions of the input values and then generate
tests using constraint-solving. The symbolic execution is usable as a partial
correctness prover~\cite{se_corr1, se_corr2}, as a reverse engineering
engine~\cite{se_reverse}, or for refactoring software to a one which is more
specialized~\cite{se_spec}. It can be extended to handle heap~\cite{se_heap,
se_gener}, summarize effects of libraries~\cite{se_libs}, or to extract
dependencies of outputs on inputs~\cite{se_output1}. It was used to find bugs
in aircraft control and nuclear power systems~\cite{se_critical} or
filesystems by generating malicious disks~\cite{exe_fs}. Even though, some do
not explicitly mention the symbolic execution, they in fact implement it, to
find errors in memory management~\cite{prefix} for example.
\textsc{PathFinder-SE}~\cite{pathfinder_se} is an extension to the model
checker \textsc{PathFinder} which enables the model checker to perform the
symbolic execution.  They further improved the tool to handle e.g.\ concurrent
programs, heap structures and data.  \textsc{IntScope}~\cite{intscope}
decompiles x86 binaries to an intermediate representation form and runs the
symbolic execution to find integer overflows.

\subsection{Concolic Execution}
Many modern tools combine the symbolic execution with dynamic (concrete)
analysis. This combination of analyses is called \emph{concolic
execution}. Tools implementing concolic execution achieve better scalability
and higher coverage of analyzed code.  \textsc{PathCrawler}~\cite{pathcrawler}
was the first executor to combine symbolic and dynamic execution. It does not
use concrete values in the symbolic execution to simplify complex constraints
nonetheless. This was first performed only by \textsc{Dart}~\cite{dart}.  The
primary goal is to execute \emph{all}, or at least as much as possible
feasible program paths, in a systematic way. It starts dynamically with random
input and symbolically tracks branching conditions.  The symbolic execution
and a solver is then used to change the input so that it traverses another
paths when executed dynamically again. This happens until the code is covered
fully or an user stops the process.  \textsc{Dart} can propagate concrete
values from the dynamic runs to the symbolic run when the solver is incapable
to reason about some values. Especially, when pointer constrains are used,
\textsc{Dart} gives up the symbolic execution and switches to standard random
testing. This was improved in another executor \textsc{Cute}~\cite{cute}. It
handles pointers better and is able to analyze concurrent programs. So except
it generates the inputs systematically, it also makes diverse schedule plans
of individually spawn threads.  \textsc{Cute} cannot still handle symbolic
offsets of pointers. Very close tool to \textsc{Cute} is
\textsc{Crest}~\cite{crest}, except it is freely available, source
codes inclusive. Since every concolic execution tool is prone to selected
strategy that decides which states are explored next, the authors studied and
compared the strategies too.

\paragraph{\textsc{Exe} and Derivatives}
At the same time as \textsc{Dart} and \textsc{Cute} emerged, Cadar et al.\
came up with \textsc{Egt}~\cite{egt} for generating test cases from symbolic
inputs.  \textsc{Egt} was a predecessor of \textsc{Exe}~\cite{exe} which
finally introduced a precise handling of pointers with symbolic offsets.
\textsc{Exe} was later completely redesigned and the new tool
\textsc{Klee}~\cite{klee} was put to execute \llvm code. \textsc{Klee}
optimizes solver queries, has better search heuristics, and other improvements
like a model to simulate a filesystem without really involving it.  This
allowed for a successful error finding in \textsc{Coreutils}, the basic set of
utilities most Linux users work with. \textsc{Klee} can be used also for
checking a tool equivalence. It takes two implementations of the same and
symbolically compares the results. Several deviations from the standard
behaviour of some utilities were found that way. This was later improved more
in \ucklee~\cite{ucklee}. First, it compares memory contents of both
implementations. Second, it makes the memory symbolic when accessed
(on-demand). Finally, it checks that both implementations behave the same with
respect to error conditions like division by zero. \textsc{Klee} and
\textsc{UC-Klee} are put into concolic executors since they allow to invoke
library functions dynamically.

Instead of generating only a single input, \textsc{Sage}~\cite{active_props,
sage} defines the term \emph{generational search} to generate as many inputs
as possible in a single symbolic run. This is unimportant for cases, where the
whole state space is explored. But this is not the case as real programs are
barely covered fully. So generating more inputs often helps more coverage. As
it is a concolic executor, it records branching conditions from the first
dynamic run.  But it negates \emph{all} of them one by one and asks a solver
for a solution.  For each positive reply the solver gives, \textsc{Sage}
generates a separate input. The whole tool does not use any intermediate
language, it symbolically executes directly the x86 binary code. So it can be
used also in cases where source code is not known.  The whole approach in
\textsc{Sage} is called \emph{automated whitebox fuzz testing}.
\textsc{SmartFuzz}~\cite{smartfuzz} works very similarly, but the set of
errors the tool aims at is limited only to integer errors. That is,
\textsc{SmartFuzz} captures signed/unsigned conversions for example. The
generated inputs are filtered, so more eager heuristics that allow more inputs
generation are allowed.

None of the above support automatic bi-directional conversion across the
boundary of the symbolic and concrete execution. This was one of the two
novelties in \textsc{S${}^2$e}~\cite{s2e}: \emph{selective symbolic
execution}. It was published by some of the authors earlier to solve the
problem of path explosion in libraries and system calls~\cite{se_selective}.
As we are mainly interested only in some parts of the analyzed code, the
symbolic execution does not fork in, let us say, libraries. Only when the
execution gets back to our code from the libraries, it can fork again.  The
second novelty is a true concolic execution because it runs the \textsc{Klee}
symbolic executor and a modified \textsc{Qemu} virtual machine in parallel.
Those two communicate with each other and exchange information, especially the
symbolic state is updated from the dynamic execution and concrete values
instantiated from the symbolic run whenever needed. Converters in both
directions from x86 code to \llvm have to be in place to have both executions
running.  \textsc{S${}^2$e} was used to to find errors in Windows network
drivers, and allegedly it was the first one to employ the symbolic execution
for performance analysis too.  \textsc{S${}^2$e}'s parallel engine works on
the top of \textsc{Ddt${}^+$}, which is a reimplementation of a closed source
tool \textsc{Ddt}~\cite{ddt}. A very similar to \textsc{S${}^2$e}, but less
generic framework is \textsc{BitBlaze}~\cite{bitblaze}. It analyses binary
files and creates intermediate code with a reduced instruction set. That is
executed symbolically.  One of unusual uses is generating exploits by a
comparison of unpatched and patched versions.  \textsc{BitBlaze} was used for
checking HTTP and NTP protocol servers~\cite{bitblaze_net}.
\textsc{Bitscope}~\cite{bitscope} is built on the same cornerstone as
\textsc{BitBlaze}, but provides understanding of dependencies between program
inputs and outputs. Both were used for recognizing malware in software.
Another true concolic executor is \textsc{Sjpf}~\cite{sjpf} stemming from
\textsc{PathFinder-SE}~\cite{pathfinder_se}. The difference between them is
that \textsc{Sjpf} symbolically executes the Java bytecode, therefore performs
the symbolic execution more directly. Nevertheless its downside is that it
still does not track constraints in both directions as the ones above, only
when switching from concrete execution to symbolic.

\textsc{Check 'n' Crash}~\cite{check_crash} does quite the opposite to
\textsc{Cute}. First, it runs coarse symbolic execution to find a suspicious
location in code, then the tool generates a test case and runs the program
dynamically to verify the test.  There are also overviews where is a
summarization of some symbolic execution techniques we mention
above~\cite{se_review88, se_review09, se_review10, se_review13}. And we also
employ the symbolic execution in our tool described in
Chapters~\ref{sec:fmics} and~\ref{sec:svcomp}.

\section{Program Slicing}\label{sec:slicing}
As was described in the previous section, the symbolic execution is prone to
slowness when analyzes programs with loops or many branchings. We mentioned
some techniques to cope with this problem in that section too, but have not
enumerated them all of course. There is another approach that can be used in
an analyzer to help the symbolic execution. The requirement is to have a set
of code locations, e.g.\ lines with potential bugs. In that case, the analyzer
can simply remove all the code which does not affect the flow to those code
locations in any way and the analyzer can evaluate only the relevant part.
This is exactly a task that can be performed by \emph{program slicing}.  This
is a technique that was described by Weiser in his PhD thesis in
1979~\cite{slicing_phd} and later also published as an article~\cite{slicing}.
As the slicing leaves only the code that really affects particular code
locations, the slicing may bring a considerable speed-up to techniques working
with code in general, not only to the symbolic execution.

\newcommand{\newsym}[1]{\ensuremath\boldsymbol{\mathsf{#1}}}

When we want to slice a program we have to specify what behaviour we want to
preserve in the program first. In the slicing, this is performed by defining
\emph{slicing criteria}. These are chosen lines of the program accompanied
with variables we are interested in at those lines. Note that there is no
limitation on the variables like whether they are set or used on those lines.
Such defined criteria ensure that values of variables at those lines must be
equal in both original and sliced program for any execution. We can slightly
loosen the requirement of ``any execution'' to ``a specified execution'' as we
will see later nonetheless.  Thereafter, when a slicer is given the slicing
criteria, it processes functions in a program independently, in an
intra-procedural way. The slicer starts with some function and assigns a set
of \emph{relevant variables} to each line of the function. For lines, where
are some slicing criteria, relevant variables are filled by variables from the
criteria. Otherwise the relevant sets are left empty.  Further, according to
the semantics of program commands, the slicer propagates relevant variables
among function lines and inserts discovered variables into relevant sets on
corresponding lines. This happens as follows:
\begin{itemize}
\item All relevant variables from a successive line are copied to the current
line's set except for those which are set on the current line. This is a
simple information \emph{propagation}.
\item If the current line sets some variable which is relevant on the next
line, the slicer adds all variables which the current line references.  I.e.\
the slicer performs computation of a \emph{direct dependency}.
\item If the current line is a branching node (or loop head) and some line in
the branch (or loop body) sets a relevant variable, the slicer adds all
variables referenced by the current line to the current line's relevant set.
This is \emph{indirect dependency} and serves the purpose of leaving in place
branches and loops which indirectly affect slicing criteria.
\end{itemize}
The slicer continues the evaluation of the function statements repeatedly
until the sets stop increasing in size, i.e.\ a fixed point is reached.  From
the content of the sets and the program semantics we can safely claim, which
program lines have no impact on the initial slicing criteria.
Those are exactly the lines, where intersection between variables that are set
on that line and relevant variables on the \emph{successive} line(s) is empty.
Slicers also have to leave lines with branches or loop heads that affect some
unsliced lines from the previous sentence. The rest can be removed from the
body of the function.  Let us demonstrate the computation on the example of
Table~\ref{tab:state_df_eq}.  Suppose we want to slice the program according
to line~\ref{lst:state_sl_crit} and variable \code{a} there. The computation of

result of the slicing is that we cannot slice anything from the program,
because \code{a} on line~\ref{lst:state_sl_crit} depends on all lines. As we
can see, the intersection of relevant and set variables is never empty.

To extend the slicer to be inter-procedural, we have to change the slicer to
pass the relevant variables among functions: down to a called function at
every \emph{call} instruction and back up from the called function. As relevant
variables propagate from the bottom up, the slicer has to pass the variables
from a call site to return points of the called function and when the slicer
is done with the transfer of the variables through the whole function to its
beginning, the variables are transferred back to the call instruction. Of
course the slicer has to take care of actual with formal parameters binding
and of the relationship between return value and the variable where the result
is stored into. Finally, we have to ensure that the set variables of the call
instruction proper are those which the called function modifies, so that the
call instruction is not sliced.  The same as above, the process repeats until
a fixed point is reached.  Inter-procedural slicers were discussed by Horwitz
et al.~\cite{slic_inter1} and Gallagher~\cite{slic_inter2}.

Originally, there were only slicers implementing the technique as described
above. These are \emph{static} slicers and they slice such that any possible
input to a program is handled the same as in the original code. But slicing
is often used for debugging purposes~\cite{slicing_debugging} and there, we
are usually interested in a value of some particular variable only when some
specific input was passed to the program.  Imagine we have a bug in the
program and know what was passed on the input. We also know that a variable
printed on the output has an invalid value. The slicer can be therefore
limited to this input. In particular, the input need not trigger visiting all
bodies of branches or loops, so this allows for much more code removal than
static slicing.  This approach is called \emph{dynamic}
slicing~\cite{slicing_dyn}. Note the slicer has to look at conditions and
track a path condition and this is usually slow. So this obviously results in
a drawback: computing a dynamic slice is more computationally expensive than
computing static slices. Since control flow graphs are used to compute slices
(not lines like in the original algorithm), there appeared techniques to
overcome the problem by pruning of the control flow graphs. They can
significantly decrease the computation time~\cite{slic_prunecfg1,
slic_prunecfg2}, but the result is less precise.  \emph{Hybrid} slicing tries
to tackle this problem by combining the dynamic slicing with dynamic runs.
Using breakpoints, slicers record which code locations are visited with which
input and compute the dynamic slice according to the locations.  It is both
faster and less space consuming than the dynamic slicing.  The dynamic slicing
can be improved such that it works with more than one input~\cite{slic_quasi,
slic_union}, which can be generalized as describing input as
constraints~\cite{slic_constraint, slic_constraint_bw}.

There is also an orthogonal method to the static slicing.  We may be
interested in all program statements which \emph{are affected} by some
variable at some line. Since the standard slicing is sometimes called
\emph{backward} slicing, the orthogonal approach was named \emph{forward}
slicing~\cite{slic_forw} and is used for dead code removal for example. The
forward and backward slicing can be generalized as
\emph{chopping}~\cite{slic_chopping}. The chopping requires two distinct sets
of slicing criteria: one about what a variable affects (forward slicing), the
other about what affects that variable (backward slicing). The slicing can be
abstracted a bit more~\cite{slic_abstract}, such that it can answer what
values in a variable affect statements from the slicing criteria.

When taking over and reusing a library, a programmer can reduce the library
code size using slicers. Programmers just need to specify the parts of the
library which are used, and the library is sliced~\cite{slic_interface}.
The authors suggest also use in the reverse engineering -- slicing closed code
by some guessed criteria helps recognition tools to find patterns and reverse
engineer the code. Further, some variables like temporaries or iterators are
unwanted in slices when debugging. They can be disabled to appear in the
slice~\cite{slic_stop}. Similar is the \emph{barrier}
slicing~\cite{slic_barrier}. Barriers are statements which should be ignored
by a slicer including their successors.  So barriers are often at interfaces
of massively used, and therefore likely error-free libraries.  The slicing can
be also used for finding errors using \emph{dicing}. We have to know two
locations in code. On one location, there is a variable which has a correct
value. On the other there is a variable with a wrong value. We slice the code
separately with those two variables and locations as criteria and the slices
are compared. Intersection of the slices are lines of code which are likely
correct. The error has to be in the rest of the incorrect slice. Slicers can
work also on the semantic level, to not preserve syntax, but leave semantics
with respect to criteria~\cite{slic_amorphous}. The slicing can reason about
some paths in a program~\cite{slic_path} such as why a statement on a
particular path is not executed? It can be used for proving
correctness~\cite{slic_mc} and for concurrent programs too~\cite{slic_concur}.
And finally, Weiser suggested to use the slicing for transfomation of a
program to a couple of parallel programs: slices can run concurrently on
separate processor without shared memory~\cite{slicing}.  More details about
slicing and its variations and uses can be seen in various surveys and
reviews~\cite{slic_overview95, slic_overview96, slic_overview98,
slic_overview01a, slic_overview01b, slic_overview05, slic_overview12}.

\section{Model Checking}
This is a bit different technique from static analysis we are dealing in this
thesis with. It does not check the program proper, but instead tries to prove
some properties on a model of the program. The model is represented by some
finite state machine and the properties are a specified behaviour which has to
be satisfied in time. For example a property can be ``This message has to come
as a reply in the future.'' Properties are described by logic formulas,
usually in the temporal logic like LTL or CTL (see more about temporal logics
in {\O}hrstr{\o}m and Hasle's book~\cite{temporal}). Correctness of the system
is then established, if every run of the system corresponds to a particular
formula. If it does not for some case, a counterexample consisting of system
states is generated. Obviously, even if no counterexample is generated, the
system still might not be correct in case its model or checked properties are
incorrect. And yet, according to Ball et al., developing a thorough and
precise model may take years~\cite{mc_years}.

There are two basic approaches to model checkers depending on how the states
are stored: \emph{explicit} and \emph{symbolic}. Explicit model checkers
enumerate the states one by one as was implemented e.g.\ in
\textsc{Divine}~\cite{divine}. Symbolic model
checking~\cite{mc_symbolic1,mc_symbolic2} started representing sets of states
by boolean functions. For example if we have two 1-bit variables \code{a} and
\code{b} and their values are always the same (1 and 1 or 0 and 0 pairwise),
we can represent it by a formula \code{a} $\Leftrightarrow$ \code{b} instead
of storing all the values. The formulas can be represented in a space-saving
way by \emph{Reduced Ordered Binary Decision Diagrams}~\cite{bdd} (ROBDD or
simply BDD).  Despite the exponential complexity still persists (due to the
path explosion problem), BDDs helped to limit the memory use a lot.  Both the
symbolic model checking and BDDs allowed more complex systems to be checked on
the same hardware (cf.~\cite{mc_symbolic1} again).  Symbolic model checking
was later improved to defer the path explosion problem more by techniques like
\textsc{Cegar}~\cite{cegar}. \textsc{Cegar} starts with an abstraction of the
model and makes it more precise whenever a \emph{spurious} counterexample is
produced.  And only in that part of the model which is too abstract. Examples
of symbolic model checkers are \textsc{NuSmv}~\cite{nusmv} or
\textsc{Bebop}~\cite{bebop}.

In parallel to the BDDs-based model checking, \emph{bounded model checking}
exists. Instead of exploring the whole state space of the system model, there
is a predefined constant \textit{k} which determines the maximum length of
execution to be explored. If a bounded model checker explores all these
\textit{k}-prefixes, either \textit{k} is incremented and the checking
continues, or the model checker is stopped. It can have two reasons: the user
set the top bound for \textit{k} (the analysis is of course incomplete in that
case), or the execution is infeasible, because it runs out of resources. Few
representatives are \textsc{Esbmc}~\cite{esbmc} and
\textsc{Llbmc}~\cite{llbmc}. Model checking and bounded model checking are
orthogonal techniques, it was shown in~\cite{bmc_survey} that both of them
reason about different class of properties. (Nonetheless there can be an
intersection.)

Due to the path explosion problem and relatively slow computers in the past,
use of the model checking was limited rather to check small reactive systems
in industry like manufacturers of processors and automotive control systems.
Later, there started appearing more successful tools to check large software
projects. For example model checker \textsc{Blast}~\cite{blast} failed first
due to inability to reason about pointers in the Linux kernel
drivers~\cite{blast_kernel}. It was shown later, that \textsc{Blast} needs an
instrumented code and then works significantly better.  So it is currently
successfully used in the Linux Driver Verification toolkit~\cite{ldv} to
verify Linux device drivers. \textsc{Blast} was also extended to handle
concurrent data accesses~\cite{blast_concur}.  \textsc{Spin}~\cite{spin} is
aware of concurrent data accesses too. So \textsc{Spin} checked a model of the
Linux common filesystem layer~\cite{spin_vfs}. Earlier, some individual file
systems which use this layer were checked by Yang et al.\
too~\cite{mc_kernel_fs}. Other subsystems like the networking stack were
also checked~\cite{mc_kernel_ip}.  \textsc{DDverify} verifier~\cite{ddverify}
and \textsc{Saturn}~\cite{saturn} were used for automated verification of the
Linux kernel drivers.  \textsc{Undertaker}~\cite{undertaker} is a model
checker for the Linux kernel configuration system. It validates dependencies
in the \texttt{Kconfig} configuration files.  There are also model checkers
aimed specifically at the Linux kernel like \textsc{Predator}~\cite{predator}.

Similar to the attempts on the Linux kernel, Microsoft started adopting their
model checker \textsc{Slam}~\cite{slam_cav, slam_popl, slam_decade} and
\textsc{Slam 2}~\cite{slam2} in their driver verifier \textsc{Sdv}~\cite{sdv}
for checking Windows drivers. They apparently succeeded as they require every
driver to be checked by this toolkit with over 160 predefined temporal rules
before the driver is accepted by Microsoft.  \textsc{Sdv} is a proprietary
software with rules tailored to Windows drivers.  Therefore it cannot be used
for checking open source projects. So Clarke et al.~\cite{cbmc} adapted
Microsoft's approach using \textsc{Cbmc} model checker and defined at least
seven rules to check Linux kernel drivers. The rules were later extended by
Post and K\"{u}chlin~\cite{slicx}.  There exist also successful tools for
modelling and verifying Java programs like
\textsc{PathFinder}~\cite{pathfinder}.

As this thesis does not deal with the model checking, curious readers are
pointed to a survey~\cite{mc_survey} and a book~\cite{mc_book} about model
checking.  There is also a survey about bounded model
checking~\cite{bmc_survey} available. To the best of our knowledge, there are
no later surveys or books covering the topics in that scale as these three do.

\section{Industrial Tools}\label{sec:industrial}
There are many successful industrial tools based on the static analysis.
Unfortunately for some of them, it is not really known what technique they are
based on.  It is due to secrets the companies want to hold to not let their
competitors find out. From what can be found in the reference materials, the
tools often use combination of the techniques we have already described. They
work on the top of various pattern matching, data-flow analyses, and some
maybe use the abstract interpretation.  Even though the internals are not
known exactly, we at least try to summarize the tools and the little which
their providers make available.  Prevalently, the tools below are capable to
handle C, some of them C++ and other languages like C\# or Java.

The data-flow analyzer \textsc{Lint} was an inspiration for Gimpel's
\textsc{PC Lint/FlexeLint}~\cite{pclint} and Cleanscape's
\textsc{LintPlus}~\cite{lintplus}. The same as their predecessor, they aim at
finding suspicious code constructs. MathWorks'
\textsc{PolySpace}~\cite{polyspace} was one of the first abstract interpreters
capable of handling real code. But according to the authors, the tool's upper
limit is still approximately fifty thousand lines of code that can be
analyzed.  The tool was inspired by the crash of Ariane~5 (see the
introduction).  Another abstract interpreter which can handle large scale
programs is AbsInt's \textsc{Astree}~\cite{astree}. It can be tuned to be
sound, i.e.\ to produce no false positives.  \textsc{Astree} also combines
several abstractions to perform more accurate analysis~\cite{astree_combine}.

\xgcc was a predecessor of Coverity's \textsc{Prevent}~\cite{coverity}, and
\textsc{Metal} for Coverity's \textsc{Extend}. \textsc{Prevent} is run
regularly to check the Linux kernel and the results are provided to kernel
developers.  KlocWork's \textsc{Insight}~\cite{klocwork} and GrammaTech's
\textsc{CodeSonar}~\cite{codesonar} are specialized to finding security and
reliability issues. \textsc{Insight} was originally intended as an optimizer
for C code. IBM's \textsc{Apollo}~\cite{apollo} aims at checking web
applications, \textsc{PHP} scripts more precisely.

Microsoft developed many tools based on the symbolic execution.  Even though
\textsc{PREfix}~\cite{prefix} explicitly does not mention it, it is used there
to find errors in memory management for example. \textsc{PREfix} has no
problems analyzing large projects thanks to computation of function summaries,
but needs user intervention to model library functions. This was improved in
\textsc{Archer} of Xie et al.~\cite{archer}. A similar use as IBM's
\textsc{Apollo} above has \textsc{Pex}~\cite{pex}. It is a symbolic executor
intended for checking .NET web applications. They developed also a fuzz tester
\textsc{SAGE}~\cite{sage}.  It is a symbolic executor working directly on the
x86 assembly code. It is therefore able to work also with libraries whose
source code was not made public.  \textsc{YOGI}~\cite{yogi} is a tool which
combines symbolic execution with dynamic execution. It was experimentally used
to replace \textsc{SLAM}~\cite{slam_decade} in \textsc{Sdv}~\cite{sdv} with
comparable better results. Microsoft also released \textsc{PREfast}, a tool
that contains a set of rules that a code should fulfill and with a use of
annotations, it checks code against the rules. Its main use is to disallows
developers to commit changes that contain certain classes of errors such as
inter-procedural integer overflows.

\section{Summary of the Linux Kernel Checkers}
Some of the tools above can analyze kernels or only drivers of operating
systems. In this sections we specifically sum those which are capable to
handle the Linux kernel. We discuss or use many of them in the following
chapters, hence it makes sense to mention them here explicitly.

One of the basic ones is based on the pattern matching and every developer
is supposed to use it before sending patches to the kernel maintainers. It is
named obviously after what it is intended to do as
\textsc{checkpatch}~\cite{checkpatch}. It only checks for patterns like if
somebody introduces a whitespace to the code or if the code is properly
indented. Some better analysis can be performed by a semantic parser called
\sparse~\cite{sparse}. It is still a tool working with some sort of patterns
matched against the parse tree.  It was written by the founder of the kernel
and is tailored exclusively to the kernel as the checks have only a little
sense elsewhere.  The checks include basic intra-procedural checks for proper
locking, use of zero instead of \code{NULL}, or extended type checking using
the annotated code in the kernel. Yet another pattern matcher is Dan
Carpenter's source matcher \textsc{Smatch}~\cite{smatch}. Formerly, it was a
set of \perl programming languages scripts. They were later turned into C code
and \sparse was used as a base. \textsc{Smatch} added many more checks
regarding the Linux kernel like doing direct memory accesses on a stack or
passing incorrect arguments to picked functions. Even though Coverity's
\textsc{Prevent} is an industrial tool, it is constantly checking the kernel
and reports are made available. The reports are of diverse kinds like dead
code, null pointer dereferences, use before test and so on. Many of them are
getting fixed. These four tools are extensively used by the kernel developers
and errors reported are getting fixed constantly~\cite{linux_faults}.

There are also tools that can be used, but are not used that massively by so
many developers. These include \clang~\cite{clang} and
\textsc{Cppcheck}~\cite{cppcheck}.  Historically, \xgcc was used to find
errors in the kernel~\cite{metal_kernel}, mainly for errors like if there are
proper checks of untrusted data coming from the network or an user to the
kernel. It found over one hundred of such errors. As \stan~\cite{stanse} is a
tool based on a similar technique, it was used the same way and found over one
hundred another errors.

There are also projects based on model checking which are successful to check
the Linux kernel. \textsc{Ldv}~\cite{ldv} is one of those. The same as
\textsc{Prevent}'s reports, \textsc{Ldv}'s reports are sent to the subsystem
maintainers so that they can look at them and fix them.
