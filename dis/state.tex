\chapter{State of the Art}\label{sec:state}
This section is deserved to description and current state of techniques for
code analysis referred to later in the text. To be able to do that, we will
first describe the techniques proper.

\paragraph{Static Analysis} Under the term of static analysis we understand
analyzing code without directly executing it on a processor. I.e.~to perform
exactly the opposite of dynamic analysis or testing where a program is
executed and either automatically fed by some input in some form or tested
manually by testers. This has the nice side-effect that the implementation
need not be finished when we want to analyze it statically. So the static
analysis is not as limited as the dynamic one, it can be used whenever we want
in the development cycle. Static analysis is thus used in several fields of
the computer science. \emph{Optimizing compilers} can eliminate unused code,
propagate constants where variables would have to be left otherwise, or remove
implicitly added runtime out-of-bounds checks from the code if it knows the
bounds (as it happens in Java). \emph{Debuggers} can limit themselves only to
code that is relevant to variables a programmer is interested in, or print
possible sources of values currently stored in a tracked variable.
\emph{Program tests} can be generated based on the analysis of a program to
cover more code paths. And of course static analysis is a prerequisite for
\emph{static analyzers} to exist in the first place.

Moreover if the static analysis is well-defined according to some rules
defined e.g.~by Cousot's framework, as we will see later, we are able to
decide whether it is sound, complete and whether it terminates. Without those
properties, optimizing compilers would generate invalid code for example.

\section{Pattern Matching}\label{sec:pattern_matching}
The simplest automatic analysis we can think of is to find some predefined
patterns in code.  We can take an instance of a regular expression pattern
matcher like \texttt{grep}, give it a pattern ``/ 0'' and see if we did that
stupid zero division error somewhere in the code. However this primitive
approach may be a source of false-positives. This happens when developers
comment their code with comments similar to ``/ 0 happened here''.  But there
of course exist more robust techniques based on pattern matching. They, in
some way, parse the code (which also removes comments like the above) and do
pattern matching on some internal structure~\cite{findbugs}. If we recall the

a checker which would match AST subtrees. Then to match division by zero

\section{Data-flow Analysis}\label{sec:data-flow}
One of the basic static analyses is \emph{data-flow analysis}. It collects
information about possible values set in variables at specified points in
code. This is usually computed by stabilizing sets of equations, usually by
computing a fixed point of the equations:

\begin{dfn}[Fixed point]
A \emph{fixed point} of a function $f : D \rightarrow D$ is an element $x \in
D$ such that $f(x) = x$. The set $fp(f)$ of fixed points of $f$ is $\{x \in D
\mid f(x) = x \}$.
\end{dfn}

In the data-flow analysis, we have a pair of equations for each statement $s$:

\begin{equation} \label{eq:data_flow}
\begin{aligned}
out_s & = trans_s(in_s) \\
in_s & = join_{p\in pred_s}(out_p)
\end{aligned}
\end{equation}

We define transfer function $trans$ to transform input set $in_s$ of a
statement $s$ to the output set $out_s$ according to the effect of $s$ in a
particular analysis. The join operator $join$ takes output sets of all
predecessors and combines it together. It is usually set intersection or
union depending whether the analysis is to be may or must. And depending
whether an analysis is forwards or backwards (cf.
Section~\ref{sec:types_anal}), the $in$ set is bound to an incoming or
outgoing control flow graph edge respectively. Vice versa for the $out$ set.
These two sets are the result of the analysis before and after applying the
effect of the statement.

One of the basic data-flow analyses is \emph{reaching definitions}. At some
point in code, we are interested in all statements where some variable was set
and was not re-set during the control flow to our point. With this
information we can decide whether some variable was set already at the point
it is used. If it is not, we should report about an uninitialized access. Let
us demonstrate that on the following example. There, the reaching definition
set at line~\ref{lst:reach_use} is a set of the two labels
$\{\ref{lst:reach_def1}, \ref{lst:reach_def2} \}$, where we define \code{a}.
Note that line~\ref{lst:reach_def0} does not belong there because \code{a}
was re-set later on all paths to line~\ref{lst:reach_use}.
\begin{center}
\begin{tabular}{c}
\begin{lstlisting}[language=C]
a = 1; (*@ \label{lst:reach_def0} @*)
if (x)
  a = 3; (*@ \label{lst:reach_def1} @*)
else
  a = 4; (*@ \label{lst:reach_def2} @*)
b = a; (*@ \label{lst:reach_use} @*)
\end{lstlisting}
\end{tabular}
\end{center}

Formally, let us denote the set of statements as $S$. To compute reaching
definitions we define join operator as set union ($\bigcup$) and the transfer
function as follows:
\begin{align*}
trans_s(in) & = gen_s \cup (in - kill_s) \\
gen_s & = 
\begin{cases}
\{s\}     & \text{$s$ is an assignment} \\
\emptyset & \text{otherwise}
\end{cases} \\
kill_s & =
\begin{cases}
\{x \in S \mid \text{$x$ is $a = b$} \wedge x \neq s \} & \text{$s$ is an
assignment $a = c$} \\
\emptyset & \text{otherwise}
\end{cases} \\
\end{align*}
In other words we generate a statement when a variable is set there and we
kill all other assignment statements which set the very same variable at any
other point.  Finally, we compute a fixpoint of the
equations~\ref{eq:data_flow} to have reaching definitions on each entry and
exit of a statement ($in_s$, $out_s$).

\section{Abstract Interpretation}\label{sec:ai}
The data-flow analysis is only a limited tool for deciding some properties
about the analysis. There was a call to formalize the process which would
for example allow for convenient proofs of algorithms' correctness and
soundness.  \emph{Abstract interpretation} is a technique that has a history
stretching back to the 1970s. Back then, it was described by Cousot\&Cousot in
\cite{cousot}. It is designed to allow users of the formalism to decide
whether their analysis is correct, if it terminates, to formulate more
precision levels of an analysis consistently and more.  With this formalism,
we can also interpret data-flow analyses as abstract interpretations, as was
shown in~\cite{ai_intro}.

As the name of the analysis suggests, there is an abstraction of how values
are handled in an analysed program. Let us show this on a demonstrative
example. Consider a program in Figure~\ref{fig:lst_ai_intro} and an analysis
that should decide whether \code{i} is positive at the end (let us ignore
overflows at this moment).

First, consider an ``abstraction'' that abstracts the code in no way, i.e.~we
take the concrete evaluation. There are only two lines in the code that store
something to \code{i}: lines~\ref{lst:ai_intro_set1} and
\ref{lst:ai_intro_set2}. The former simply sets \code{i} to \code{3}, so
we just remember that. The latter, increases \code{i} repeatedly, so we have
to remember that \code{i} can be anything from $\{6,12,24,\dots\}$ there. To
be able to decide if a variable is positive or negative when analyzing the
program concretely, we would need infinite time and memory to store all
possible values from all possible runs. Nonetheless, if we only remember
abstract information such as ``this number is positive/negative'', and define
how arithmetic operations behave on those, we need to store only a
single value.

To work with the abstract values, we define two functions: \emph{abstraction}
denoted $\alpha$ and \emph{concretization} denoted $\gamma$, both accepting
one parameter.  $\alpha$ returns an abstract value corresponding to the
concrete value on input and $\gamma$ does the opposite.  In
Figure~\ref{fig:ai_alpha_omega}, we provide a definition of the two functions
to work for our positive/negative analysis, where $\oplus$ is used to denote
the positive abstract value and $\ominus$ for negative. Further, the same as
for the data-flow analysis, during the abstract interpretation, we compute a
fixed point. The fixed point is computed to stabilise the abstract values at
each statement and often it starts with values set to $\bot$. The values are
changed by a transform function that changes the state according to the effect
of a statement. We have two possibilities to define the transform function as
can be seen in Figure~\ref{fig:ai_square}. We can simply define a transform
function for abstract values $F_a$ to obtain a new abstract state
corresponding to the effect of a statement applied to the original state. The
other option for more complex statements, is to first jump back to the
concrete values, perform the transformation on the concrete values using a
concrete transform function $F_c$ and switch back to the abstract values.
$F_a$ is then defined as $F_a = \alpha \circ F_c \circ \gamma$. Per definition
(\cite{cousot}), the resulting abstract value must be less than or equal to
the one computed by a direct $F_a$.

We know how to define the arithmetic multiplication in this case precisely on
the abstract level as is shown in Table~\ref{tab:ai_multiplication}. With
these definitions, we can continue interpreting our example of
Figure~\ref{fig:lst_ai_intro}. The first set to \code{i} happens on
line~\ref{lst:ai_intro_set1}. There, \code{3} is abstracted as $\oplus$, so
\code{i} is set to $\oplus$ there too.  Another set happens on
line~\ref{lst:ai_intro_set2} where we multiply \code{i} by \code{2}. \code{2}
is abstracted as $\oplus$ and since we know \code{i} is $\oplus$ at this
point, the multiplication results, according to
Table~\ref{tab:ai_multiplication} in \code{i} being still $\oplus$. Note that
for this simple analysis, the only values we need to remember are one of
$\ominus$, $\oplus$, or possibly \textit{unset} in case no store was processed
yet. \textit{unset} is mostly referred to as $\bot$. Further, if the body of
the loop is changed to be \code{i *= -2}, what would be the abstract value of
\code{i} after joining $\oplus$ from odd iterations with $\ominus$ from even
iterations? The same as in the data-flow analysis case, we have a kind of join
operator which can in this case say \code{i} is \textit{unknown}. Note that
\textit{unknown} is usually referred to as $\top$.  So in fact, when we are
computing a real abstract interpretation of a program, the behaviour tables
(Table~\ref{tab:ai_multiplication}) are extended to contain also $\bot$ and
$\top$. Obviously, multiplication of $\bot$ and any other value $x$ from the
table results in $x$ as the first initialization happened. For $\top$, the
join operator usually returns $\top$ for two reasons. First, the
transformation function shall be monotone (\cite{cousot} and \cite[p.
348--349]{kleene}). Second, hopping from \textit{unknown} back to some of the
``known'' values would break soundness of the analysis.

Now, let us demonstrate the interpretation of the code of
Figure~\ref{fig:lst_ai_intro} with the modified while-loop body which
multiplies \code{i} by \code{-2}. The computation of a fixed point is depicted
in Table~\ref{tab:ai_computation} as it goes over time until a fixed point is
reached in the third iteration. We see that the result is rather imprecise, we
cannot make any assumption about \code{i} when it is returned from the
function. The abstraction was apparently too coarse and the interpretation
does not help much.

We are still missing a formalism to represent the values and design a monotone
function over the values effectively.  In the abstract interpretation this is
solved by using complete lattices~\cite{lattice1,lattice2}. They are used to
represent both concrete and abstract values and they give us the possibility
to use them as a basis effectively. Thanks to work presented
in~\cite{tarski}, we can compute fixed points of monotone functions working on
lattices. Further, lattice operators can be used when we are interpreting a
program.  For example, when a branching occurs, the join operator needed when
joining the branches simply takes the values present on the outputs of both
branches and performs lattice join ($\sqcup$) of the values in that lattice.
Since an abstract interpretation works on two complete lattices, one for
concrete, one for abstract values, $\alpha$ and $\gamma$ are used to move from
one to the other.  In fact the pair $(\alpha,\, \gamma)$ forms a \emph{Galois
connection}~\cite{cousot_galois} between the lattices and so Galois
connection can be used as a tool for designing abstractions.  Now, let us
formally define a complete lattice and show some examples of lattices.

\begin{dfn}[Complete lattice]
A \emph{complete lattice} $(L,\, \leq,\, \bot,\, \top,\, \sqcup,\, \sqcap)$ is
a partially ordered set $(L,\, \leq)$ where every set $X \subseteq L$ has a
least upper bound (supremum) $\sqcup X$ and a greatest lower bound (infimum)
$\sqcap X$. Namely, $L$ has a least element $\bot = \sqcup \emptyset$ and a
greatest element $\top = \sqcap L$.
\end{dfn}

We were working so far with a positive/negative analysis. To represent values
in that abstraction as a complete lattice, we naturally define the lattice as
follows.  $\oplus$ is incomparable to $\ominus$ as switching from one to the
other would be pointless in our case when joining branches.  $\bot$ is less
than both since we start with \textit{unset} values and want to go upper in
the lattice when any of the abstract values is set.  Finally, $\top$ is
greater than both to catch the case of \textit{unknown} when branches bring
different values.  A Hasse diagram for such a lattice is the one of
Figure~\ref{fig:posneg}~(a). We may want a more precise abstraction
represented by the lattice in Figure~\ref{fig:posneg}~(b).  There, we also
remember whether the value of a variable is 0.  This can become useful if we
have a multiplication or division in the code.  For multiplication, we can
propagate the zero constant, for division, we can tell whether we are dividing
by zero for example.  Extending $(\alpha, \gamma)$ from the previous example
is straightforward in this case.

We were also working with concrete ``abstraction'' at the beginning of this
section. First, let us have a look at a lattice of an abstraction where we
remember only a single value of a variable. This is useful for analyses like
constant propagation as we can propagate only a single value. Such a lattice
is depicted in Figure~\ref{fig:lat_concrete}.  Naturally, joining branches
where values are different gives \textit{unknown}.  However, our concrete
interpretation was computing with all possibles sets of values. We need to
augment the above lattice to a power set $\mathcal{P}(\mathbb{Z})$ using a set
inclusion as the order. The complete lattice is defined as
$(\mathcal{P}(\mathbb{Z}),\, \subseteq,\, \bot,\, \top,\, \sqcup,\, \sqcap)$,
where $\bot=\emptyset$, $\top=\mathbb{Z}$, $\sqcup=\cup$, and $\sqcap=\cap$. A
part of a Hasse diagram is demonstrated in Figure~\ref{fig:lat_concrete_set}.

There is a difference to be spotted between these two. Although they both
contain an infinite number of elements, the latter lattice has infinite
height. This is exactly the source of the need of infinite time and memory for
this interpretation when analyzing particular programs with loops. The
Kleene's sequence is not necessarily finite there, so the computation of a
fixed point may not finish. There is a mechanism suggested in~\cite{cousot} to
overcome the problem. It is a \emph{widening} operator $\triangledown: L_a
\times L_a \rightarrow L_a$ defined such that:
\begin{itemize}
\item $\forall x,y \in L_a,\, x \leq_a x \triangledown y$ and $y \leq_a x
\triangledown y$, and
\item the sequence $x_0 = \bot$, $x_1 = F_a(x_0)$, \dots, $x_{n+1}=x_n
\triangledown y_n$ for some $y_n \in L_a$ is not strictly increasing. 
\end{itemize}
That informally means, that the sequence is sped up by widening, so that it
converges eventually. For the example with powerset lattice, we can define the
widening operator to limit the number of iteration to some constant and wide
the set to $\top$ if it reaches the limit. The opposite of the widening
operator is \emph{narrowing} operator $\vartriangle: L_a \times L_a
\rightarrow L_a$ and is defined similarly as:
\begin{itemize}
\item $\forall x,y \in L_a,\, x \leq_a y \Rightarrow x \leq_a x \vartriangle y
\leq_a y$, and
\item the sequence $x_0 = \bot$, $x_1 = x_0 \vartriangle y_0$, \dots,
$x_{n+1}=x_n \vartriangle y_n$ for some $y_0,\dots,y_n \in L_a$ is not
strictly decreasing. 
\end{itemize}
Narrowing can help to make the abstract values more precise. When our widening
operator returned $\top$, we can narrow it back to some abstract value which
we know will never cause an increase. This is not so conceivable for number
interpretation, but see~\cite{ai_widening, cousot_galois} where the operators
are demonstrated on intervals and described in more detail.

Intervals are not the only shapes to represent abstract values of numbers.
Octagons, elipses, polyhedra and other shapes are used too~\cite{ai_octagons,
ai_polyhedra, astree}. Several abstractions can be also combined to
communicate with each other and improve thus accuracy or any other aspect of

\section{Symbolic Execution}\label{sec:se}
In the previous chapter we described the abstract interpretation which is a
framework for fast, but many times imprecise analyses. As a result we often
get a pile of false positives on the output of an abstract analyzer. There
also exist techniques which cannot formally generate false positives, but are
often slower than abstract interpreters. One of such techniques is the
\emph{symbolic execution}. The technique was introduced in 1970's and
described by several authors concurrently~\cite{se_king, se_select,
se_dissect, se_clarke}. The symbolic execution represents program inputs by
symbolic values instead of trying to analyze a program many times with
concrete, initialized, data. The program is then executed and program
expressions are manipulated including the symbolic values. Let us demonstrate
this on an example of Figure~\ref{fig:se_example}. The example is two-fold.

First, there is a read from the user. How long it would take to exercise the
program to get to the true branch of the \code{if}s with dynamic testing?
With \code{int} being usual 32 bits, there is only one successful attempt out
of $2^{32}$ overall. Of course, with a static analyzer based on the data-flow
analysis or abstract interpretation, those true branches would be analyzed
quite quickly.  This brings us to the second point nonetheless. Many complete
analyzers generate a report about division by zero possible at
line~\ref{line:se_ex_div}. This is indeed a false positive because \code{b} is
set to \code{1} on line~\ref{line:se_ex_set} in case \code{a} is
\code{0x12345678}. The source of the false positive is that the analyzers do
not often remember conditions on branching. Some of them do a post processing
on reports to eliminate such reports as we will see in
Chapter~\ref{sec:stanse}.

\begin{figure}[htb]
\begin{center}
\begin{tabular}{c}
\begin{lstlisting}
int fun(void) {
  int a = read(); (*@\label{line:se_ex_sset}@*)
  int b = 0;

  if (a == 0x12345678) (*@\label{line:se_ex_if1}@*)
    b = 1; (*@\label{line:se_ex_set}@*)

  if (a == 0x12345678) (*@\label{line:se_ex_if2}@*)
    a = 10 / b; (*@\label{line:se_ex_div}@*)

  return a;
}
\end{lstlisting}
\end{tabular}
\end{center}
  \caption{Example to demonstrate the symbolic execution.}
  \label{fig:se_example}
\end{figure}

The symbolic execution tackles this problem. All inputs from the user are
replaced by \emph{symbols}. For a variable \code{a}, we denote the symbol
corresponding to \code{a} as $\sym{a}$. Such symbols are allowed to be
assigned to variables like constants and other expressions. In our example, we
would have \code{int a = }~$\sym{a}$\code{;}. Obviously we have to teach the
symbolic executor to deal with symbols in arithmetic operations and branching
statements. In the first place, we need to have a storage. Symbolic executors
contain a \emph{symbolic memory} $\theta$ which assigns to each variable its
current symbolic value. So after the line~\ref{line:se_ex_sset}, $\theta(a) =
\sym{a}$. To be precise in execution of the code, the executor also has to
remember the branches which it chose to jump to at branching points. This is
stored in a \emph{path condition} $\varphi$. It is a first-order
formula~\cite{fol} representing a necessary and sufficient condition over
symbols to drive the execution along the currently executed path to the
actually evaluated point. $\varphi$ is set initially to $\true$ and is
evaluated and potentially updated at every branch. This happens on
lines~\ref{line:se_ex_if1} and~\ref{line:se_ex_if2} in our example. On
line~\ref{line:se_ex_if1}, we check, whether $\varphi$ implies validity of the
condition instantiated by the current state of $\theta$. More concretely, the
executor checks whether either of the following two implications hold (at most
one~\cite{se_king}):
\begin{align}
\varphi & \rightarrow \theta(a) = 0x12345678 \label{equ:se_fi} \\
\varphi & \rightarrow \theta(a) \neq 0x12345678 \label{equ:se_se}
\end{align}
There are three possibilities that can happen:
\begin{description}
\item[Implication \ref{equ:se_fi} is valid] The executor simply continues
execution by jumping directly to the \emph{true} branch.
\item[Implication \ref{equ:se_se} is valid] The executor simply continues
execution by jumping directly to the \emph{false} branch.
\item[Neither is valid] This means the executor has no concrete idea about the
values to prove validity. So to retain soundness, it has to fork the execution
of the whole program to explore both branches. In that case, the path
condition $\varphi$ has to be augmented in the respective branches. For the
true branch, $\varphi \wedge \theta(a) = 0x12345678$ is a new path condition,
for the other branch $\varphi \wedge \theta(a) \neq 0x12345678$.
\end{description}
In our example, on line~\ref{line:se_ex_if1} neither is valid, as
$\theta(a)=\sym{a}$ which cannot prove validity of $\theta(a) = 0x12345678$.

The only remaining piece of information needed to perform the symbolic
execution is the program location which is currently evaluated by the
executor.  Then we define the \emph{program state} as a triple $(l,\,
\theta,\, \varphi)$, where $l$ is the current program location in a control
flow graph.

\section{Model Checking} \todo{Honzo, toto prosim precti, jestli neblabolim.}
This is a bit different technique from static analysis we are dealing in this
thesis with. It does not check the program proper, but instead tries to prove
some properties on a model of the program. The model is represented by some
finite state machine and the properties are a specified behaviour which has to
be satisfied in time. For example a property can be ,,This message has to come
as a reply in the future.'' Properties are described by logic formulas,
usually in the temporal logic like LTL or CTL (see more about temporal logics
in~\cite{temporal}). Correctness of the system is then established, if every
run of the system corresponds to a particular formula. If it does not for some
case, a counterexample consisting of system states is generated. Obviously,
even if no counterexample is generated, the system still might not be correct
in case its model or properties checked are incorrect.

These technique was spread even after introduction of \emph{symbolic model
checking}~\cite{mc_symbolic1,mc_symbolic2}. It started representing sets of
states by boolean functions. For example if we have two 1-bit wide variables
\texttt{a} and \texttt{b} and their values are always the same (1 and 1 or 0
and 0 pairwise), we can represent it by a formula \texttt{a} $\Leftrightarrow$
\texttt{b} instead of storing all the values. The formulas can be represented
in a space-saving way by \emph{Reduced Ordered Binary Decision
Diagrams}~\cite{bdd} (ROBDD or simply BDD) were introduced to represent the
set of states. Despite the exponential complexity still persists (due to path
explosion problem), BDDs helped to limit the memory use a lot. Both symbolic
model checking and BDDs allowed more complex systems to be checked on the same
hardware (cf.~\cite{mc_symbolic1} again).

In parallel to BDDs-based model checking, \emph{bounded model checking}
exists. Instead of exploring the whole state space of the system model, there
is a predefined constant \textit{k} which determines the maximum length of
execution to be explored. If it explores all these \textit{k}-prefixes, either
\textit{k} is incremented and the checking continues, or the model checker is
stopped. It can have two reasons: the user set the top bound for \textit{k}
(the analysis is of course incomplete in that case), or the execution is
infeasible, because it runs out of resources. Model checking and bounded model
checking are orthogonal techniques, it was shown in~\cite{bmc_survey} that
both of them reason about different class of properties. (Although there can
be an intersection.)

Due to path explosion problem and slow computers in the past, its use was
limited rather to check small reactive systems in industry like processors or
automotive control systems. Later, there appeared both less and more
successful tools to check large software projects. For example model checker
\textsc{Blast}~\cite{blast} failed due to inability to reason about pointers
in the Linux kernel drivers~\cite{blast_kernel}. On the other hand Microsoft
started adopting their model checker \textsc{SLAM} and driver
verifier~\cite{sdv} for checking Windows drivers and they apparently succeeded
as they require every driver to be checked by this toolkit with predefined
temporal rules before it is accepted.

As this thesis does not deal with this type of analysis, curious readers are
pointed to~\cite{mc_survey} and~\cite{mc_book} about model checking and
to~\cite{bmc_survey} about bounded model checking.

The same as for other analyses, points-to analysis have always one of  the

\paragraph{Field-sensitivity} In languages allowing structured constructs, like
\code{struct} in C, we have two options: to consider each of the member
of such an object or ignore them. Obviously ignoring that information leads to
faster analysis, however imprecisions occur. The imprecisions are two-fold:
each member itself can be a pointer and should have its own points-to set, and
knowing exactly to which member of a structure a pointer points-to is indeed
useful information. On the other hand, when we want more precise analysis,
there are two steps, with different precision. First, \emph{field-based
sensitivity} have only one global information about structure members, per
each member name. So when we have two totally different structures \code{a}
and \code{b} containing a member \code{void *x}, this analysis considers
them to be the same and updates \code{x}'s points-to set any time
\code{a.x} or \code{b.x} is set to point to a new object. The other
analysis, the true \emph{field-sensitive} one, properly handles members of
structures in natural sense, but for a higher computation price, of course.
