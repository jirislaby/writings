\begin{abstract}
  We have been using our own implementation of DWARF unwinder in every SUSE
  release in the past decade. First, it makes the kernel faster, second, it
  prunes the backtraces from false positives. However, this is obviously an
  out-of-tree solution. Given the progress in the live patching area in
  upstream, there is a call for some reliable unwinder in upstream too. This
  talk describes the path of our unwinder to upstream, its rejection, proposed
  solution and final SLE15 implementation.
\end{abstract}

\section{Introduction} \label{sec:intro}

\textit{WARNING}. If the report produced by the kernel did not contain
information what functions were visited up to the point of failure, such
report would be helpless for developers. This piece of information is called a
\emph{call trace} and an example is depicted in Fig.~\ref{fig:call_trace}.

\subsection{Frames and (Optional) Frame Pointers} \label{sec:fp}
There are many approaches on where and how to extract a call trace entries.
In the kernel, we use the fact that every \codeasm{call} to a function
implicitly stores a \emph{return address} to the stack before yielding the
execution to the function and every \code{return} reads and implicitly removes
this address.  So every call is said to create a \emph{stack frame} for a
particular function call. And every \code{return} destroys it.  There is an
example of code and stack in Fig.~\ref{fig:stack_frame}. On the left side,
there are three functions which cause the program to crash by \code{abort}.
When that happens, the possible state of the stack is depicted on the right
side.

More precisely, when \code{fun1} is called from \code{main}, the address of
the instruction right after the \codeasm{call} is stored on the stack (to know
exactly where to return) and \code{fun1} now continues the execution.
Traditionally, the first thing a function does is storing the caller's
base/frame pointer (\code{BP}) to the stack (for later restore) and making
\code{BP} to point to \code{SP} -- to the current ``bottom'' of the stack,
i.e.\ the current frame.  This happens in a so-called \emph{prologue}.  Further,
given \code{fun1} also contains a local variable \code{xx}, which is to be
allocated on stack, this is done too. Then there is a call to \code{fun2} and
the steps are repeated until \code{abort}'s body causes the program to exit.

If \code{fun2} ever returned, the \code{fun1}'s function body would be over
and code at the very end of the function (\emph{epilogue}) would set \code{SP}
to back to the value in \code{BP} like it was after the prologue.  Finally,
the function would restore the old (caller's) value of \code{BP} and invoke
\code{return} -- pop the return address off the stack and jump there.

Conclusively, \code{BP} is somewhat static inside a function and changes only
by \codeasm{call}s and \code{return}s. Exactly, \code{BP} points to the
current frame plus 2 entries: return address and saved \code{BP}. This renders
\code{BP} to be only a \emph{convenient} way of restoring \code{SP} in the
epilogue.  If the code properly tracks what it performs with stack throughout
the function body, it can easily restore the stack state (\code{SP}) in the
epilogue based only on the tracked operations.  And pushes to and pops from
the stack are relatively easy to count, especially when the code is generated
by a compiler. Hence, when the code is compiled by \gcc with optimizations
turned on, by default, \gcc does not use \code{BP} as a frame/base pointer
storage.  This behaviour can be disabled by \code{-fno-omit-frame-pointer}, if
needed, however.

Now, consider that that both \code{SP} and \code{BP} are usually stored in
registers. On \xesh, these are \codeasm{rsp} and \codeasm{rbp}
(\codeasm{esp}/\codeasm{ebp} on \xesl), respectively. If we avoid using
\code{BP} for stack tracking, we free up one general purpose register for
arbitrary use.  This indeed leads to a speed-up, see
Section~\ref{sec:suse_predwarf} for more information.

\subsection{Tail calls} \label{sec:tail_calls}
When the code in Fig.~\ref{fig:stack_frame} is compiled by an optimizing
compiler, there are not two actual \codeasm{call}s and two \code{return}s.
\code{main} still calls \code{fun1} the usual way. But since the invocation of
\code{fun2} is the last thing to do in \code{fun1}, \code{fun1} will not do
this by \codeasm{call}, but by \code{jump}. This means that no new stack frame
is created for \code{fun2}. Therefore, \code{return} in \code{fun2} will pop
the return address from the \code{fun1}'s stack frame, i.e.\ returns directly
to \code{main}. This is very common in the code and we have to take this into
account when doing call traces.

\subsection{Unwinders} \label{sec:unwinders}
Now that we know the stack contains stack frames with return addresses,
stored base pointers (sometimes), and other potentially useful information, we
can make use of it. The tools to extract stack contents and build call traces
are called \emph{stack unwinders}. More on the techniques used for unwinding
is presented in Section~\ref{sec:unwinding}.

\subsection{Use of Call Traces and Our Motivation} \label{sec:uses}
Crashes (incl.\ BUGs and WARNINGs) are obvious users of unwinders. They print
out the information so that developers can debug the crash. There are also
other users like \textsc{KMemLeak} (kernel memory leak detector),
\textsc{KMemCheck} (kernel memory accesses checker), or \textsc{LockDep}.
They all do use the call traces to store information about locations where
memory accesses, allocations, and locks take place respectively. This
information is sooner or later reported to the user if some problem is
detected by them.

We can consider the all above as debugging purposes. There is one more not so
obvious use of unwinders: \textit{Kernel Live Patching} (KLP). Although we
started providing live patching since SLE12, we have not been using stack
traces for anything until the upcoming release: SLE15. With SLE15, we are very
likely to provide the upstream solution which evolved in the meantime. And it
utilizes call traces to check if a process is somewhere in a function that is
to be patched. Including those up to the call chain towards userspace. Hence
we are very interested in fast and precise unwinding too.

\section{Unwinding} \label{sec:unwinding}
All unwinders described in this section work similarly. They somehow scan the
stack and return a call trace. It is important to note, that there is not a
single stack in the kernel. Whenever an interrupt or exception occurs, the
stack is switched to the intterupt, NMI, exception, or other stack. So in
fact, not only a single stack is scanned, but all of them are scanned up to
the point where userspace invoked the kernel e.g.\ by syscall. Nevertheless,
we consider a single stack in the text below for simplicity.

\subsection{Simple Approach} \label{sec:unw_guess}
The simplest method used in the kernel is to scan the stack by 8-byte chunks
and put every value which is a function address to a call trace. A function
address is easy to be identified in the kernel. It is an address in
well-defined ranges: kernel code, modules, and other minor code. There is a
helper to decide this. It is called \code{\_\_kernel\_text\_address}. And
given the stack contains return addresses and whole stack is scanned, the
functions will be surely part of the resulting call trace. And of course with
other random material found on the stack which by a chance fits in the kernel
code space.

This approach is used whenever all of the sophisticated methods below are
unavailable/disabled. Since this is the most simple and naive unwinder, it is
also the fastest one.  But least precise, of course. The implementation can be
found in \file{arch/x86/kernel/unwind\_guess.c}.

\subsection{Frame Pointers Approach} \label{sec:unw_fp}
When base/frame pointers are enabled (\code{CONFIG\_FRAME\_POINTER}), they can
be used to identify the start of stack frames and to locate a return address
in them precisely. Since a pointer to the previous stack frame is right next
to the return address, the unwinder uses it to extract another return address
from there. This repeats until the bottom of the stack is reached.

There is a run-time penalty stemming from enabled frame pointers. Functions
have to contain prologues and epilogues to maintain the frame pointer
(\code{BP}). It means (a) longer execution time needed, and (b) more
instruction cache misses due to larger code.  The overall slowdown measured is
up to 10\,\%~\cite{fp_slowdown} -- depending on the load.  The implementation
is in \file{arch/x86/kernel/unwind\_frame.c} and is still relatively simple.

\subsection{DWARF Approach} \label{sec:unw_dw}
There was ongoing effort to use unwinding in Kernel Live Patching to conclude
no task is in a patched function. While there were sucessfull attempts to make
frame-pointer-based unwinding reliable, it still drags in the run-time
overhead described earlier. Hence, it was also proposed by us to use
DWARF-based unwinding~\cite{use_dwarf}.

DWARF works on a completely different basis. It needs no additional in-code
operations.  Instead, when a stack is to be unwound, this is performed solely
on the out-of-band data basis.


\subsection{Rewrite of Unwinding -- the New API} \label{sec:rewrite}
While we (at SUSE) have been using the DWARF unwinder plugged into the
upstream code for a long time, the interface was not nice and maintaining
another unwinder was quite hard. Therefore, the unwinding in the upstream
kernel was completely redesigned in 4.9 by commit \commit{7c7900f89770}
(``x86/unwind: Add new unwind interface and
implementations'')~\cite{git_unw_rewrite}. It allows for quite easy switching
from one unwinder to another. And this cleaned-up API also prepared the field
for easy use by KLP too. So now, there are three kinds of call traces from the
new API's point of view:
\begin{enumerate}
  \item In case of crash, \textit{BUG}, and so on, a call trace is dumped to
    the log. It should contain all available data from the stack. So similar
    to the Guess unwinder in Section~\ref{sec:unw_guess}, everything what is
    on the stack and is a kernel code address is printed.  This happens
    despite more precise unwinders might be available in the kernel at that
    moment. Nevertheless, if they are, the entries reported by the more
    precise unwinder are not accompanied by the `\code{?}' mark. So in most
    cases, developers may considered entries with the `\code{?}' mark as
    irrelevant.  The reason for this behaviour is that unwinders might have
    troubles to unwind for unpredictable reasons, while we still want to dump
    at least some call trace. This is invoked via \code{dump\_stack} and
    derivatives.
  \item \textsc{LockDep}, \textsc{KMemLeak}, \textsc{FTrace}, and similar
    tools save the stack trace for their use. Unlike for the previous kind,
    these traces contain only entries which are reported by the unwinder.
    This kind of call traces is obtained by \code{save\_stack\_trace}.
  \item Kernel Live Patching demands only reliable (precise) call traces.  But
    unwinders sometimes put imprecise entries into call traces in the previous
    kind. So here, when the unwinder is unsure about any entry, whole call
    trace shall be discarded and an error returned instead of imprecise call
    trace.  A new function \code{save\_stack\_trace\_tsk\_reliable} was
    introduced for this purpose.
\end{enumerate}

In the new API, there is a common ``core'' for stack unwinding.  Then, there
are implementations of the respective unwinders. The current API allows only
one unwinder to be enabled. And the enabled unwinder has to define only a few
functions. The functions and their meanings are as follows:
\begin{itemize}
  \item \code{\_\_unwind\_start} -- initialize the unwinder along with its
    internal state. This is a preparation for unwinding. Usually, the state is
    set up according to the state of the process. It can be the currently
    running process, current process with pre-defined values of registers
    (program counter, stack pointer, etc.), or a sleeping process. Tasks
    running on other CPUs are usually ignored as the call stack changes very
    quickly and also it is not easy to grab call stack of such process.
  \item \code{unwind\_next\_frame} -- unwind to the next frame. This is called
    repeatedly until enough frames are unwound or the bottom of the stack is
    reached. This functions does what was described in previous sections about
    respective unwinders.
  \item \code{unwind\_get\_return\_address} -- extract the return address from
    the current frame and return it.
  \item \code{unwind\_get\_return\_address\_ptr} -- return pointer to the
    location of the return address. This is used only for comparison with
    the current stack pointer when doing \code{dump\_stack} to the log. If
    they match, the entry is printed as reliable (without the `\code{?}'
    mark).
  \item \code{unwind\_get\_entry\_regs} -- implementations should return
    \code{pt\_regs} structure describing registers. It is dumped to the log.
    This is useful for example when there was an interrupt and we want to dump
    the state of registers before the interrupt happened.
\end{itemize}
