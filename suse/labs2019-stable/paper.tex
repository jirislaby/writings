\begin{abstract}
  Here, we provide a short summary of the past and current habits of merging
  stable patches into the SUSE kernel trees. Starting with the sorted series
  section, the policy undergoes a change, so we will sum up what is happening
  in SLE15 now and what is our plan for future releases.
\end{abstract}

\section{Introduction} \label{sec:intro}
SUSE is maintaining kernels for their products in a git tree~\cite{git_suse}.
Each product has a dedicated branch called after it. Every such branch is
based on some picked upstream kernel version which is immutable over time. On
the top of it, the branches contain from tens (master and stable branches) to
tens of thousands (SLE, Leap) of patches we maintain in them. To be able to
apply the patches -- to define the order -- we also store their list in a file
called \file{series.conf}.

In master and stable branches, these patches are placed into
\file{series.conf} in an order depending on a person who did the backport
(cf.\ Fig.\ \ref{fig:series_stable}). The patches are divided into so called
\emph{sections} there. There are sections for filesystems, x86, TTY, sound,
and alike. For tens of patches it is quite bearable to maintain the patches in
this form. And we also used to use this schema for SLE before SLE 12 SP2.
However, lately, we switched to a \emph{sorted section} (see Fig.\
\ref{fig:series_sle15}).

Sorted section is a section in \file{series.conf} where all the patches are
sorted according to their appearance in the upstream kernel. This makes lives
of developers easier by the order of magnitude. The main reason is that there
are no interdependencies between patches which would be placed in different
sections. These dependencies created sometimes large conflicts when applying
patches. Therefore, it also made the patches to differ from the upstream ones.
This made the situation even worse with more and more patches coming (and
differing from upstream).

One exception from this sorted-section rule are stable patches. In upstream,
for each kernel release, some later minor releases with non-intrusive fixes
are released. So for example, for a released kernel 4.12, we have 4.12.1,
4.12.2, etc. These stable releases are composed of many patches and we add
them in a single-commit-per-file manner (like any other patch). For all
branches master, stable, SLE, and Leap, we put these stable patches to the
very front of the series file. Their order is defined by the stable tree.
Obviously, this creates exactly the same interdependencies as non-sorted
sections described earlier. Note that other (non-sorted) sections were left in
place. They are used for patches which were not made upstream yet. One example
is \file{setuid-dumpable-wrongdir} patch which is present in both stable and
SLE15 as can be seen in Figures\ \ref{fig:series_stable} and
\ref{fig:series_sle15}.

SLE15 and SLE15-SP1 is based on 4.12. But only fourteen stable versions were
released for 4.12. These were put to the SLE15 branch to the front of its
series file. After that, we only add some patches collected from the upstream
stable list along with those reported by our tools like
git-fixes~\cite{git_fixes}.  All these are put to the sorted section, despite
they have the same origin and reasoning as stable patches before. In other
words, we continue collecting stable patches, but we do put them to the sorted
section, not to the front of series file.

\section{Proposal} \label{sec:proposal}
SLE15-SP2 will be based on a fresh upstream kernel. Likely on 5.3. Here, we
propose that every single stable patch that is released in any later 5.3.x
stable release would be placed to the sorted section too. So we will stop
accepting stable releases as they are, and will take only SHA identifiers of
the commits. Upstream patches with these SHAs will be taken instead. And they
will be put to the sorted sections appropriately. This will further reduce the
burden as interferences with other patches will be quite lower.

\section{Discussion} \label{sec:discussion}
With the proposal above, there are few open questions which we discuss here.

\subsection{Vanilla flavour}
For every SLE or Leap, we also release a so-called vanilla kernel
(named kernel-vanilla). It is a kernel released by upstream. So for SLE15,
that would be pure 4.12.14 with few additional patches required for proper
packaging. This flavour can be useful for customers. If they find a bug in our
kernel, we can ask them whether it is an upstream issue by checking the
presence of the bug in kernel-vanilla. If it is not there, then it is a bug in
our later backports, e.g. in the sorted section.
