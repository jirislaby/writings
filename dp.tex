\begin{ThesisAbstract}
  In this thesis we are concerned in fast transfers from hardware to computer
  memory and vice versa. Above all we specialize to scatter-gather direct
  memory access transfers from/to hardware through ring buffers. Also we
  discuss other transfer methods implemented in other projects and consider
  using them here. Finally we implement the method and test it with all
  available tools and describe the testing process.
  
\end{ThesisAbstract}

\chapter{Introduction}
  To even boot a computer, it's necessary to proceed many transfers inside the
  hardware among its components. It's a basic function which must work,
  however there are several methods of how to achieve that. Especially
  the fast ones are very important in computer science which results in higher
  overall throughput of the system.

  There are several computer science areas, where fast transfers are very
  important. Beside the computer-inside transfers among CPU, disks,
  DVD writers or main memory, we must take in account also the outside ones,
  such as moving huge data amounts from/to external disks, displays or data
  cards in cellphones for example, all over buses with specific data formats.
  Outstanding speed is important especially in the area of network
  communication; backbone network routers or monitoring probes are very
  dependent on how fast data transfer can be achieved in the end.

  Nowadays the equipment of operating system can do fast network transfers
  through well-known interfaces, but there exist several methods which are
  not so common, but might provide yet another significant improvement. For
  instance nowadays mainly used technique of \emph{direct memory access} in
  cooperation with ring buffer data structure and memory mapping (so-called
  \emph{mmap}) might move big chunks of data without even processor
  assistance from network adapter towards application in userspace and vice
  versa without any real data copying. This results in short transaction
  times.

  \section{Devices interconnection}
  \label{chap.inter}
  The first thing to explain is how data flows from devices to processor and
  particularly which types of buses are on the route from a CPU to a device.
  Data path may be seen on figure \ref{fig.dev_inter}.

  \begin{figure}[ht]
    \begin{center}
      \includegraphics[width=6cm]{images/data_flow}
      \caption{Devices interconnection}
      \label{fig.dev_inter}
    \end{center}
  \end{figure}

  We are concerned especially in \emph{peripheral component interconnect}
  (PCI) devices, because both \com\ card families are PCI based. Each such
  device is physically connected to any PCI bus via PCI slot (bottom left).
  PCI bus is attached to \emph{southbridge} (bottom center), it communicates
  with \emph{northbridge} (middle) over internal bus and it's finally wired
  with processor via \emph{front-side bus} (FSB, top center). Nowadays
  it's still more common, that all southbridge, northbridge and memory
  controller are implemented as proper PCI devices directly connected to the
  main PCI bus.

  3 main signal groups are usually on each bus:
  \begin{list}{-}{}
    \item \textbf{control bus} -- it's supposed to send controlling signals
      and synchronize objects which are connected to it
    \item \textbf{address bus} -- that is address of memory which is requested
      to read from or write to
    \item \textbf{data bus} -- data itself which flows from/to address
      specified on the address bus
  \end{list}

  While FSB (on pentium and newer), PCI-X and PCI express has 64-bits wide
  address and data buses (PCI minimizes signal count by multiplexing these two
  into one by first sending address and then data), legacy PCI ones are
  32-bit only. This shortcoming is important when we will send 64-bit numbers
  over PCI to older \com\ cards in section \ref{chap.hwsetup}.
  Further detailed PCI information are in \cite{pcispec} and \cite{pciespec}.

  \section{DMA and Bus Mastering}
  \label{chap.dma}
  \emph{Direct memory access} known as DMA is a method widely used to
  speed up transfers in computer engineering. It's based on technique called
  \emph{bus mastering}. A device which wants to move data from/to memory
  becomes a bus master and no other device or even processor can't touch
  address and data bus. While being a master, it sets address on an address
  bus and sends or latches data on a data bus. When there are more than one
  data phases after address phase on multiplexed bus, the transfer is done in
  so-called \emph{burst mode}.
  
  On older machines, there was only one device called DMA controller (DMAC),
  which could become a master. Nowadays each PCI device (see also Section
  \ref{chap.pci}) may become a master, so that it can access memory directly
  by its own DMAC\footnote{Note that there is no common PCI DMAC, so if the
  device doesn't have one, it doesn't support DMA either.}. DMAC count is not
  bounded to only one per device as we will see implementation of \com\ 
  hardware in section \ref{chap.netcope}. Unit which takes care of honest bus
  accesses by more than one DMACs is called \emph{Arbiter}.

  \subsection{Virtual memory}
  With only physical memory addressing problems arose early. Applications
  needed
  continuous big chunks of memory, but it was impossible to find such long
  block after a while system was running. It was caused by process called
  (external) fragmentation. Other tasks held memory blocks and there wasn't
  enough space in between them to return. See figure \ref{fig.fragmentation}
  for illustration.
  
  Task 1 holds two memory blocks of size 2 units. Task 2
  allocated one block three units wide. Now, if any of the two wants to
  allocate block of size 4, allocation will fail, although there is
  enough free space in sum.

  \begin{figure}[h]
    \begin{center}
      \includegraphics[width=7cm]{images/fragmentation}
      \caption{Memory fragmentation illustration}
      \label{fig.fragmentation}
    \end{center}
  \end{figure}

  Many algorithms were introduced to avoid this behavior, some of them are
  described in \cite{knuth_art} including buddy system used in Linux
  kernel for in-kernel allocations.
  
  \subsubsection{Paging}
  To solve the situation with external fragmentation, method named
  \emph{paging} was introduced. Linear address space is divided into pages
  of fixed size\footnote{4096 bytes on x86 with page size extension
  disabled} that might be mapped to physical memory or disks as swapped pages.

  CPU accesses physical memory by virtual addresses. This is not possible
  without translating a virtual address into the physical one. Transparent
  translation is done by \emph{memory management unit} (MMU). Some modern
  mainboards add another unit called IOMMU between device and main memory
  path, so that devices address by logical address too. It allows creating a
  virtual space even for devices, but it's not widely spread so far
  so \com\ card developers must not take it for granted.

  Virtual address itself is usually divided into few parts and each of them is
  offset in some page table. On 32-bit x86 with extensions disabled, 2 such
  tables exist and each table contains entries/pointers to next table or
  to physical page in the last table respectively.
%  If the entry in either
%  table is marked as non-present, \emph{page fault} (exception handled by
%  operating system) is thrown. OS must either put (map) the requested page in
%  or kill the program which references illegal address.
  On figure \ref{fig.vaddr} is shown, how is logical address split. CR3 is
  register in x86 processors containing location of global page directory
  depicted on the left side.
  \begin{figure}[h]
    \begin{center}
      \includegraphics[width=9cm]{images/vaddr}
      \caption{Virtual address format}
      \label{fig.vaddr}
    \end{center}
  \end{figure}

  This schema allows us to allocate huge contiguous memory blocks without a
  need of having contiguous physical memory. In fact there is no need to have
  as much physical memory as requested. Some memory may be, as written above,
  backed by a page stored on disk (swapped out) or even have no "physical"
  page mapped to it thanks to demand paging.

  On 32-bit, we are able to use up to 3 gibibytes\footnote{SI unit for
  $2^{30}$} of virtual space per process. This limit comes from 1:3 of
  kernel to user address space split -- the remaining addressable
  1\,GiB is used as kernelspace. On 64-bit, 48-bit virtual addresses are
  supported by most current processors, up to $2^{47}$ may be used in
  userspace, kernel is mapped in top $2^{47}$ bytes due to 64-bit sign
  extension (see \cite{ia3a} for reference) and the space between them is an
  unused memory hole, which might be addressed even after newer processor
  family will be available.

  Consequences of all these constraints are considered later in section
  \ref{chap.netcope}, because we cannot, as we mentioned earlier allocate
  too much contiguous physical space and virtual memory is not unbounded
  too, especially on 32-bit machines.

  \subsection{Memory map}
  As we are able to set (and unset) present bit in an entry of each table
  MMU traverses and an exception is thrown when the bit is not set when the
  particular table entry is accessed, we can hook a page fault handler to this
  path which will easily install missing pages.

  This can be used for \emph{memory mapping} (MMAP). We register a page fault
  function by OS interface and when user application maps some space and
  accesses it, we may set up there any pages we want. According to exact
  address which was the root cause of the fault, we compute a number of page
  which should be in that particular memory place and install it there.
  Application sees this as a transparent process and it accesses the data from
  the installed page since that moment.

  \subsubsection{Memory mapped I/O}
  Memory map is sometimes inappropriately confused with \emph{memory mapped
  I/O} (MMIO). In the latter, we are able to access device memory space in
  kernel by virtual address. It's achieved by mapping this I/O space into
  kernelspace. For example \fpga\ memory space of a \com\ card may be referred
  by a virtual address after being mapped this way. \\[1em]
  \emph{Note: We are even able to access the \fpga\ from
  userspace by memory map fault handler, but it is a separate mapping and has
  nothing to do with MMIO.}

  \subsubsection{DMA interface}
  Referring back to section \ref{chap.dma} we already know, that PCI
  multiplexes address and data bus into one wiring. To become a master on
  this bus, the device asks arbiter for a band and after it gets start signal
  on control bus, it starts sending. First, address is sent, then data part.
  If there is more data to send, burst mode is used until arbiter emits stop
  signal. After that signal, the device must finish the transfer of near few
  bytes in the cache line\footnote{its size is configurable through config
  space} and acknowledge stopping. Next device gets start signal from arbiter
  at this moment and starts data transfer. The ancestral device may ask for a
  band again unless all data have not been sent.

  The transfer is theoretically limited by the other device (including main
  memory) boundary\footnote{it is 4096 bytes on main memory when extensions
  disabled}. It must not
  exceed this limit, the transfer is interrupted by the other side otherwise.
  Transfers of 1 KiB per one burst are not uncommon in \com\ case.

  \subsection{Character device}
  There exist several ways how to pass data from userspace to kernelspace in
  Linux. Most of them are based on moving data through special
  files\footnote{most of communication channels are files, even socket
  creating system call returns a file descriptor passed to standard read and
  write functions} such as sockets, block and character devices.
  
  We need a mechanism which allows reads byte by byte, memory map, some control
  message passing for controlling the driver and application and direct
  driver-application data flow. Unlike
  block devices which might be read only by blocks with other restrictions
  and sockets which goes through Linux networking stack, character ones
  satisfies all our prerequisites and so we use it. Here comes a bare
  introduction to char subsystem internals, information about API (including
  source code examples) are available in \cite{ldd}, chapter~3 and about how
  it works in \cite{ulk}.

  \subsubsection{Device nodes}
  \label{chap.devnodes}
  We use character device approach on both \com\ driver parts -- for
  combosix core driver and in both versions of szedata we have a separated
  namespace combosix*, szedata* and newly szedataII*, where * stands for any
  so far random number.
  
  A node in \emph{/dev} directory must be created for each such device. They
  used to be created manually by userspace programs \emph{MAKEDEV} and
  \emph{mknod}, although this was not perfect because of the need of every
  single node creation and so some dynamic principles described later in
  this section were introduced.

  The existing node might then be opened and used by userspace
  application and on the other end handled by the driver which created it.
  Which handlers exactly are necessary in implementation will be discussed in
  section \ref{chap.code} after considering operations done in specification.

  \subsubsection{udev}
  \label{chap.udev}
  Userspace device (udev) is a dynamic device node creation system
  with support of running external programs depending on defined configuration
  rules. It's a successor of devfs, a filesystem implemented directly in
  kernel and mounted on \emph{/dev}. Udev mounts only a temporary filesystem
  (tmpfs) on the same directory, but from its name, it does all its work in
  userspace by listening on socket for userspace events (uevents) from kernel
  and invoking mknod, modprobe\footnote{kernel module insertion program} and
  other programs.

  One particular userspace rule may look like the following one which creates
  a symbolic
  link \emph{/dev/input/event15} for Microsoft (vendor ID 0x045e) Laser Mouse
  6000 (device ID 0x00f0).
\begin{verbatim}
KERNEL=="event*", SUBSYSTEM=="input", \
        ATTRS{idVendor}=="045e",      \
        ATTRS{idProduct}=="00f0",     \
        SYMLINK="input/event15"
\end{verbatim}
  This is the point, where IDs read from PCI configuration space are sent by
  kernel in uevent and might be matched in rules file by udev.
  
  As device node names are random so far and we want numbers in names to
  correspond with each other, e.g. to have combosix6 and szedataII6 for one
  concrete card in PCI slot 6, we need to write similar rules to generate
  persistent and consistent naming.

  Two separate files are needed to achieve that, one another with persistent
  rules will be created on the fly. Shell script which augments the persistent
  rules and a rules file for matching the \com\ cards which runs the 
  script are in detail described in Appendix \ref{app.udev}.

  \subsubsection{Use case diagram}
  We'll start with use case diagram, because it's usually a starting point for
  other diagrams. We create a module szedata2 in which we put use cases for
  each task we will have and OS, hardware and application roles outside the
  module.

  The only OS's task is to insert and remove the module to/from system, we
  only set up this use cases which allocate or free resources needed for
  module to run. HW's only task is to inform the module that the transfer has
  been completed. This is done by timeout or position reached registers
  described already in section \ref{chap.netcope}.

  Application will have much more use cases. First, we need to initialize the
  device after opening the device node, next the application wants to get
  access to the data and need to control the device somehow and finally
  cleanup on application's close.

  All the operations must be atomic due to concurrent access, this is not
  mentioned neither in the specification nor further, because we take this for
  granted.

  \paragraph{Open and initialization} is done in the following steps. If this
  is the first open, we must allocate space for ring buffers and their
  descriptors
  and set the device to initial state including device's ring buffers setup.
  Next step is to map this memory to userspace. This is done by invokation of
  mmap by application. Now, application has all the needed information, it
  tells the driver which interfaces it wants to use and starts the device.

  \paragraph{Data processing} includes locking (asking for) data area either
  to read from or write to, do the processing and unlocking. Overall two
  method of read operation raised:
  \begin{list}{-}{}
    \item \textbf{load balancing} -- several readers exist and data are divided
      among them. They all get distinct data and do the same operation on
      them. It's very similar to \emph{single program multiple data} in
      processors theory.
    \item \textbf{multiprocessing} -- all readers want all data to do
      different tasks with them. We can approach this to \emph{multiple
      program single data}.
  \end{list}
  We implement only the latter so far, but it's possible that we augment the
  implementation to the former too, see \ref{chap.future}.

  For outgoing direction, there is only one possibility which hardware permits.
  Only one writer at a time is allowed, we take this in account even
  while designing the implementation. There is a plan to extend this behaviour
  to allow multiple writers too, it has not been finished.

  \paragraph{Close and deinitialization} is done only after all application
  have been already closed. After such last close we first stop the device
  and then free all the allocated resources and mappings.

  \subsubsection{Sequence diagram}
  These are usually used for depicting how events happen during some time
  period. The topmost event is done first and then all events below it
  towards the bottom one. XXX seqd citation needed XXX

  We create 2 sequence diagrams. One for receiving and one for sending data.
  They are slightly different in how application locks and unlocks data.

  \paragraph{RX sequence diagram} starts when open is called by an
  application. We need to allocate descriptors for ring buffer, the ring
  buffer itself if we didn't do it on module insertion (this is the first
  optional part in the diagram). Next we \emph{mmap} a page with status
  information
  and space with ring buffers, choose which interfaces the application wants
  to read from and emits start control message which results in start and end
  pointer registers setup if it wasn't already done by some previous and still
  running application and passing start value to control register.

  A loop follows in which the application periodically calls lock, processes
  data if any available and unlocks the area. After it finishes its work
  (interrupts the loop), it calls stop which in return stops the device if it
  was the last user, invokes \emph{unmap} on all earlier mapped
  resources and closes the device node. If we allocate the ring buffer in the
  open, we will free it now and return back to the userspace, where the caller
  might exit. When the application ends unexpectedly (e.g. crashes), we must
  do the cleanup work too. The convenience of the OS is that it will unmap
  all spaces
  call the close itself, so that we see this the same. The only exception is
  that any control message won't be sent in such case, so we must check in
  close if the device is stopped already or not.

  \paragraph{TX sequence diagram} differs only in processing between lock and
  unlock phase.
  It gets locked area of some bytes, which it must fill, not process and there
  might be only one application which might have locked area at a time.

  \subsubsection{libsze2}
  Although there is no library for szedata v. 1, we consider to have one for
  szedata2. The main point to have it is not to propagate the kernel
  interface directly to the user, to simplify the interface and make it
  abstract.
  
  The library functions might be an exact copy of use cases from the
  specification, since the use case alone means exactly that it is a
  separate sequence of commands which might not be further divided. If we
  encapsulate each use case's contents in a library call, we reach the
  aims from previous paragraph. Above all, if we create a shared library,
  we'll get easy to update part (module) of each application.
