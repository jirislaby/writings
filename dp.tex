\begin{ThesisAbstract}
  In this thesis we are concerned in fast transfers from hardware to computer
  programs and vice versa. Above all we specialize to coherent direct
  memory access transfers from/to hardware through ring buffers. Also we
  discuss other transfer methods implemented in other projects and consider
  using them here. Finally we implement the method and test it with all
  available tools, describe the testing process and compare with already
  working products.
  
\end{ThesisAbstract}

\chapter{Introduction}
  To even boot a computer, it's necessary to proceed many transfers inside the
  hardware among its components. It's a basic function which must work,
  however there are several methods of how to achieve that. Especially
  the fast ones are very important in computer science which results in higher
  overall throughput of the system itself and also in faster communication with
  other ones.

  \subsection{PCI}
  \label{chap.pci}
  We are concerned especially in \emph{peripheral component interconnect}
  (PCI) devices, because both \com\ card families are PCI based. Each such
  device is physically connected to any PCI bus via PCI slot.
  PCI bus is attached to \emph{southbridge}, it communicates
  with \emph{northbridge} over internal bus and it's finally wired
  with processor via \emph{front-side bus} (FSB) and with memory. Nowadays
  it's still more common, that all southbridge, northbridge and memory
  controller are implemented as proper PCI devices directly connected to the
  main PCI bus.
  
  Older PCI buses have a \emph{control bus} supposed to send controlling
  signals and synchronize objects which are connected to it and multiplexed
  \emph{address} and \emph{data buses}. Device starts transmission by sending
  address, then follow data
  on the same wires. On the other hand, PCI express uses packets for
  communication over serial physical layer, it further must contain hubs and
  switches for packet transmission. Data and addresses (might be both 32 and
  64-bit) are part of packets, each request and reply is a packet here. 

  Legacy PCI has 32-bit wide address/data bus. This shortcoming affects
  sending 64-bit numbers over the
  bus to older \com\ cards as we will see in section \ref{chap.hwsetup}.
  Further detailed PCI information are in \cite{pcispec} (PCI, PCI-X) and
  \cite{pciespec} (PCI express).

  \subsubsection{Configuration space}
  Every modern bus supports a mechanism to autoconfigure devices connected
  to it. Also each PCI device has its configuration space, where all the needed
  information are stored or may be set to. This space contains information
  about who developed the device and which model it is. These data are useful
  especially in driver area, where the drivers want to bind only to specific
  subset of devices.

  In our scope of interest are device, vendor and their subsystem variant
  IDs contained in this space, because these are mostly only identifiers
  designating one concrete device family. It is helpful in userspace scripts
  to create persistently named nodes in chapter \ref{chap.udev}.

  \subsubsection{Linux PCI support}
  PCI is well supported in Linux since it became a standard. Concrete API is
  in a scope of chapter~12 of \cite{ldd}. In short, driver's hook is called
  whenever subsystem finds suitable PCI device, initialization is done by
  functions with pci\_ prefix, especially device must be enabled and spaces
  remapped to be usable. Then the device is ready to work. Deinitialization
  and device switch off is done in driver's remove hook.

  Transfers are done by processor by standard read and writes over the
  remapped space, unit by unit. On 32-bit unit is up to 4 bytes, on 64-bit up
  to 8 bytes. These transfers are not too fast and need processor attention.

  \section{DMA and Bus Mastering}
  \label{chap.dma}
  \emph{Direct memory access} known as DMA is a method widely used to
  speed up transfers in computer engineering. It's based on technique called
  \emph{bus mastering}. A device which wants to move data from/to memory
  becomes a bus master and no other device or even processor can't touch
  address and data bus. While being a master, it sets address on an address
  bus and sends or latches data on a data bus. When there are more than one
  data phases after address phase on multiplexed bus, the transfer is done in
  so-called \emph{burst mode}.
  
  On older machines, there was only one device called \emph{DMA controller}
  (DMAC),
  which could become a master. Nowadays each PCI device (see also Section
  \ref{chap.pci}) may become a master, so that it can access memory directly
  by its own DMAC\footnote{Note that there is no common PCI DMAC, so if the
  device doesn't have one, it doesn't support DMA either.}.
  As we will see in chapter~\ref{chap.netcope}, their count is not limited to
  one per PCI device. Unit which takes care of honest bus
  accesses by more than one DMAC is called \emph{Arbiter}.

  \subsection{PCI DMA interface}
  To become a master on legacy PCI bus, the device asks arbiter for a band and
  after it gets start signal on control bus, it starts sending. First,
  address is sent, then data part.
  If there is more data to send, burst mode is used until arbiter emits stop
  signal. After that signal, the device must finish the transfer of near few
  bytes in the cache line\footnote{its size is configurable through config
  space} and acknowledge stopping. Next device gets start signal from arbiter
  at this moment and starts data transfer. The ancestral device may ask for a
  band again unless all data have not been sent. This method is implemented in
  PCI and PCI-X versions of \com\ cards.

  PCI express is a bit different, it asks the other device to get some amount
  of data in request packet and the device fills reply packet with the data.
  Sending is similar, the source device fills the data
  directly in the request and gets back completion packet. This is obviously
  implemented in PCI express based \com.

  The transfer is theoretically limited by the other device (including main
  memory) boundary\footnote{it is 4096 bytes on main memory when extensions
  disabled}. It must not
  exceed this limit, the transfer is interrupted by the other side otherwise.
  Transfers of 1\,KiB per one transfer are not uncommon in \com\ case.

  \subsection{Linux DMA interface}
  Linux has three groups of DMA APIs. ISA/LPC, PCI and generic. ISA/LPC is used
  only for old 8237 chips located on old ISA buses and on todays LPC buses,
  neither one is in our scope of interest. See
  \emph{Documentation/DMA-ISA-LPC.txt} from Linux kernel sources for usage
  explanation.

  \paragraph{PCI interface} This interface supports coherent (also called
  consistent) and non-coherent DMA mappings.
  We always must decide which one we should use. Non-coherent mappings are
  often mapped only for a transfer and then unmapped and underlying buffer
  freed. This is method widely used for socket buffers which are
  thrown away after
  successful transmission. We will probably use this interface for exactly
  this purpose out of scope of this thesis, see chapter~\ref{chap.future}.

  The other group of coherent are mappings usually XXX what? XXX
  mapped at the driver load or device start and unmapped and freed even after
  device stop. Its domain are DMA ring buffers, firmwares used by some
  devices and executed from main memory while device work and so on. This
  mapping hence perfectly fits our needs and will use it,
  see~\ref{chap.netcope}.

  \paragraph{Generic interface} It is actually an abstraction of PCI interface.
  It's not bounded to any PCI device, it may be used for non-PCI devices,
  however it obviously doesn't support all the PCI features.

  \section{Memory management}
  \label{chap.mm}
  This chapter is a short introduction into memory management as used not
  only in Linux. Memory management in details on Intel processors is described
  in \cite{ia1} and also in \cite{gorman} and \cite{ulk}, chapter~2.

  \subsection{Physical memory}
  In the usual meaning we call main \emph{random access memory} as a physical
  memory and we usually\footnote{It's possible not to have this kind of memory
  by a processor in special computers like \emph{NUMA} (non-uniform memory
  access).} have one or more of them plugged in the motherboard. These can be
  addressed by several methods; Linux implements \emph{Flat mode}, where
  each byte of memory is simply addressed by \emph{linear address}. First
  memory byte has address 0, on 32-bit without any extensions the last
  possible byte (if there is that much physical memory) has address
  $2^{32}-1$.

  \subsection{Virtual memory}
  With only physical memory addressing problems arose early. Applications
  needed
  continuous big chunks of memory, but it was impossible to find such long
  block after a while system was running. It was caused by process called
  (external) fragmentation. Other tasks held memory blocks and there wasn't
  enough space in between them to return.

%  enough free space in sum.

  Many algorithms were introduced to avoid this behavior, some of them are
  described in \cite{knuth_art} including buddy system used in Linux
  kernel for in-kernel allocations of larger blocks.
  
  \subsubsection{Paging}
  However, the algorithms only lowers the risk of fragmentation
  being hit, we would still have problems with allocations bigger than some
  constant specific for each allocation mechanism.  
  To really solve the situation with external fragmentation, method named
  \emph{paging} was introduced. Linear address space is divided into pages
  of fixed size\footnote{4096 bytes on x86 with page size extension
  disabled} that might be mapped to physical memory or disks as swapped pages.

  CPU accesses physical memory by virtual addresses with help of \emph{memory
  management unit} (MMU) which does transparent translation to physical
  address. Some modern
  mainboards add another unit called IOMMU between device and main memory
  path, so that devices address by logical address too. It allows creating a
  virtual space even for devices, but it's not widely spread so far
  so \com\ card developers must not take it for granted.

%  offset in some page table. On 32-bit x86 with extensions disabled, 2 such
%  tables exist and each table contains entries/pointers to next table or
%  to physical page in the last table respectively.
%  If the entry in either
%  table is marked as non-present, \emph{page fault} (exception handled by
%  operating system) is thrown. OS must either put (map) the requested page in
%  or kill the program which references illegal address.
%  On figure \ref{fig.vaddr} is shown, how is logical address split. CR3 is
%  register in x86 processors containing location of global page directory
%  depicted on the left side.
%  \begin{figure}[htb]
%    \begin{center}
%      \includegraphics[width=9cm]{images/vaddr}
%      \caption{Virtual address format}
%      \label{fig.vaddr}
%    \end{center}
%  \end{figure}

  This schema allows us to allocate huge contiguous memory blocks without a
  need of having contiguous physical memory. In fact there is no need to have
  as much physical memory as requested. Some memory may be, as written above,
  backed by a page stored on disk (swapped out) or even have no "physical"
  page mapped to it thanks to demand paging.

  Although 64-bit kernels are able to address up to $2^{47}=128$\,TiB mapped
  to the bottom of address space on modern processors (kernel is mapped to
  the top of the same size thank to 64-bit sign extension, see
  \cite{ia3a} for reference, with
  so far unused hole between them), 32-bit ones are bounded to nowadays real
  limit of 3\,GiB of virtual space per process. The remaining addressable
  1\,GiB is used as kernelspace. This will be reflected while estimating
  size of buffers needed for userspace applications along with
  physical memory constraints for device memory buffers.

  \paragraph{Linux memory interface} To obtain a buffer smaller than a page
  is used \emph{kmalloc} and \emph{kzalloc} returning a zeroed space. They are
  calls of slab allocator (see \cite{gorman}, chapter~8). Freeing is provided
  by \emph{kfree}.
  
  For larger physically contiguous buffers (consequent pages)
  \emph{get\_free\_page} family should be used -- variants for zeroed page(s),
  for single page, for functions which returns \emph{struct page} pointer
  instead of logical address. Freeing is done by \emph{free\_page}. This
  page allocations might be theoretically used for DMA, its physical addresses
  could be gained by \emph{page\_to\_phys} (\emph{struct page}) or
  \emph{virt\_to\_phys} (logical address), however this is not clean, because
  the returned address needn't be device addressable. We should call
  \emph{dma\_alloc\_coherent} to get an addressable page. It gets
  \emph{struct device} as one of parameters where DMA capabilities are
  written down and so it knows which addresses are OK to return. Possibly
  wrapper \emph{pci\_alloc\_consistent} might be used to pass \emph{struct
  pci\_dev} directly. \emph{dma\_free\_coherent} and
  \emph{pci\_free\_consistent} serves as cleanups.

  \emph{vmalloc} is for in-kernel virtual contiguous memory allocations.
  Another wrappers like \emph{vmalloc\_32}, \emph{vmalloc\_user} and
  \emph{vmalloc\_32\_user} exist for allocating buffers addressable by 32-bit
  addresses and for those intended to be used in userspace as of 2.6.18 --
  underlying pages are marked as user and zeroed not to leak any information.
  \emph{vfree} frees all these allocations.

  For the convenience of \com\ PCI devices, we use
  \emph{pci\_alloc\_consistent} and \emph{pci\_free\_consistent} for DMA
  allocations, \emph{vmalloc\_32\_user} for buffer to be mapped to userspace
  and obviously \emph{kmalloc} for driver state structures.

  \subsection{Memory map}
  As we are able to set (and unset) page present bit in structures
  MMU traverses and an exception is thrown when the bit is not set while any
  of the structures is accessed, we can hook a page fault handler to this
  path which would install missing pages.

  This can be used for \emph{memory mapping} (MMAP). We register a page fault
  function by OS interface and when user application maps some space and
  accesses it, we may set up there any pages we want. According to exact
  address which was the root cause of the fault, we compute a number of page
  which should be in that particular memory place and install it there.
  Application sees this as a transparent process and it accesses the data from
  the installed page since that moment.

  \paragraph{Linux mmap interface}
  Many incompatible interfaces were introduced and removed meanwhile Linux
  2.6 development process. First versions support remapping of some kind of
  memory only by implementing real
  fault handler called \emph{nopage}. This hook is called for every page fault
  and must install the requested page (offset is passed in as a parameter) or
  fail to program be killed. \emph{nopage} hook was changed to \emph{fault}
  with slightly modified prototype in 2.6.23.

  Not to implement \emph{fault} or \emph{nopage} handler for each mapped
  space, helpers exist. For remapping I/O space, \emph{io\_remap\_page\_range}
  was used in some variants, because it changed prototype time by time, then
  \emph{io\_remap\_pfn\_range} was implemented and it's still in use.
  While for remapping kernel memory obtained by page allocation mechanism
  (i.e. memory which is physically contiguous),
  \emph{remap\_pfn\_range} may be used, a helper called
  \emph{remap\_vmalloc\_range} for virtual memory exists even since 2.6.18 and
  it remaps only \emph{vmalloc\_*user} buffers. We'll use all these mentioned
  interfaces, depending on which kernel we compile for, to be backward 
  compatible and to have all the functionality (and simpleness for later
  kernels).

  \subsubsection{Memory mapped I/O}
  Memory map is sometimes inappropriately confused with \emph{memory mapped
  I/O} (MMIO). We are able to access device memory space in kernel by virtual
  address by MMIO. It's achieved by mapping I/O space into kernelspace.
  For example \fpga\ memory space of a \com\ card may be referred
  by a virtual address after being mapped this way.

  \paragraph{MMIO interface} Memory mapped I/O is done by invoking
  \emph{ioremap} or \emph{pci\_iomap} for PCI devices. Then, for the former,
  \emph{read} and \emph{write} are used for device space access. The latter
  interface is
  slower, because it calls \emph{ioremap} and all consequent read and writes
  (functions \emph{ioread}, \emph{iowrite}) are done through calling back
  \emph{read} and \emph{write} or \emph{in} and \emph{out} depending on what
  kind is the underlying mapping of. In our case we know, it's mapped memory,
  so that we use \emph{ioremap}/\emph{iounmap} and \emph{read} and
  \emph{write} functions.

  We refer to \cite{ldd}, chapter~9 and 15 for further reading about accessing
  device I/O and memory mapping.

  \section{Character device}
  While we are able to pass data from device to memory available only to kernel
  via DMA so far, we still need to put the data to userspace application
  somehow. Several ways how to do it exist in Linux.
  Most of them are based on moving data through special
  files\footnote{most of communication channels are files, even socket
  creating system call returns a file descriptor passed to standard read and
  write functions} such as sockets, block and character devices.

  The basic and simplest idea is to \textbf{copy} the data by a processor
  from kernel buffer to an user one. It's easy to implement and least
  error-prone, on the other hand it's one of the slowest known methods and as
  we need to read high bandwidth data in userspace, it's unusable in our case.
  Memory map suits in such situation, this eliminates use of both sockets and
  block devices, since neither of them can be mapped, furthermore socket data
  flow through networking stack. The only remaining are character devices,
  introduction to char subsystem internals follows further.

  \subsection{Device nodes}
  \label{chap.devnodes}
  As as entry point from userspace to kernelspace serves \emph{device nodes}.
  The existing node in \emph{/dev} might be opened and used by userspace
  application,
  control is handled by the driver which registered it. It depends on what
  features are implemented in the driver and what controls are defined as an
  interface. Handlers for szedata2 will be discussed in section
  \ref{chap.code} after review of operations done in specification.

  If the driver is loaded and requested node does not exist yet, it must be
  created. It used to be created manually by userspace program \emph{mknod} or
  wrapper scripts like \emph{MAKEDEV}, however this was not perfect
  because of the need of every single node creation and so some dynamic
  principles described later in this section were introduced.

  \subsection{udev}
  \label{chap.udev}
  Userspace device (\emph{udev}) is a dynamic device node creation mechanism
  with support of running external programs depending on defined configuration
  rules. It's a successor of \emph{devfs}, a filesystem implemented directly
  in kernel and mounted on \emph{/dev}. Udev mounts only a temporary
  filesystem (tmpfs) on the same directory, but from its name, it does all its
  work in userspace by listening on socket for userspace events
  (\emph{uevents}) from kernel and invoking \emph{mknod},
  \emph{modprobe}\footnote{kernel module insertion program} and other
  programs.

  For instance following example shows a node creation in \emph{/dev/combosix/}
  for \com\ device matched by IDs read from PCI configuration space which were
  sent by kernel in uevent. \vspace{0.5em}
%Node name is a number sent by the driver.
\begin{verbatim}
KERNEL=="combosix*", ATTRS{idVendor}=="18ec" \
    ATTRS{idProduct}=="c058", NAME="combosix/%n"
\end{verbatim}
\vspace{0.5em}
  
  As device node names are random so far -- it depends only on kernel PCI cards
  enumeration (which may vary) -- and we want numbers in names to
  correspond with each other, e.g. to have \emph{combosix6} and
  \emph{szedataII6} for one
  concrete card in PCI slot 6, we need to write similar rules to generate
  persistent and consistent naming.

  Two separate files are needed to achieve that, one another with persistent
  rules will be created on the fly. Shell script which augments the persistent

  \subsection{Character device Linux interface}
  First step to support character devices by a driver is to allocate a
  region of devices. This was done by \emph{register\_chrdev\_region} in the
  past since regions were statically assigned among certain drivers, nowadays
  is used a variant for dynamic registration \emph{register\_chrdev} which
  registers 256 nodes immediately or \emph{alloc\_chrdev\_region} which
  registers none, but keeps a track of a region being virtually used. These
  two give a registered region descriptor as a return value which is then
  passed to \emph{unregister\_chrdev} in cleanup path.

  Each character device has its own file operations structure with hooks to be
  called on each particular event (\emph{open}, \emph{close},
  \emph{mmap}\ldots). Entries left blank (except \emph{open} and \emph{close})
  means that this concrete function is unsupported.
  While we pass this structure directly to \emph{register\_chrdev},
  \emph{alloc\_chrdev\_region} needs further attention, each incoming hardware
  device must be added by \emph{cdev\_add} with operations as second
  parameter. \emph{cdev\_del} is used for removal.

  Although that's all what is needed from Linux kernel view, this is not
  enough to support \emph{udev} node creation. Each driver creates a class by
  \emph{class\_create}, frees up by \emph{class\_destroy}. Then each device is
  created by \emph{device\_create} and after this call \emph{udev} receives a
  signal to create a node. Node is deleted after \emph{device\_destroy}. Both
  these functions are created in class namespace, hence the class must be
  unconditionally created before and freed after \emph{device\_*} calls.
  This API evolved even in 2.6 Linux kernel and is probably most unstable one
  in a scope of this thesis. Many function variants appeared, many changed
  prototypes and then disappeared during 2.6. Functions like
  \emph{class\_simple\_create}, \emph{class\_device\_create} and others will
  be mentioned in back-porting chapter~\ref{chap.backport}.

  We should mention also misc character devices. This interface allows adding
  of only one character device and is mostly implemented for single devices
  like timers or access to e.g. virtualization resources and hence doesn't
  suit to our needs. We choose \emph{alloc\_chrdev\_region} method, since it's
  recommended and \emph{device\_create} as well. Further
  character device resources are available in \cite{ldd}, chapter~3 and in
  \cite{ulk}.

  \subsection{szedata2}
  Szedata2 is one of designs, the abbreviation stands for \emph{straight
  zero copy data} in the meaning of avoid copying data from memory to the
  application and vice versa. The name was chosen after its predecessor
  szedata v. 1.

  \subsubsection{Design specification}
  Both szedata versions operate in \emph{FPGA} on \ncope\ architecture.
  Version 1 uses \emph{PowerPC} as a DMA controller -- program running
  on that processor used to transfer data through scatter-gather DMA method
  (described in \cite{ldd}, chapter~15) over memory chunk pool.
  In the first phase, the data transferred at one turn contained only one
  packet of constant length. While \emph{PowerPC} was working with higher
  rates in this mode, we found out, that it's a bottleneck of the whole
  architecture, its control bus was too slow to achieve higher transfer rates.
  
  We tried to improve this behaviour by aggregating more data packets into one
  transfer block and we redefined data structure too -- data are now in packet
  of variable length with modified header in comparison to the older one.
  We had to redesign also the program running on the card a bit and define the
  packet structure in the specification. The header includes two sizes:
  segment size and header size. The former is size of whole segment, i.e.
  size of header, padding and data in sum, while the latter is size of optional
  part of segment containing some control information. This control data are
  placed directly after the header, optional padding follows and finally the
  packet contents with start aligned to 8-byte
  boundary (thanks to padding). We refer to this structure as a \emph{packet},
  its format is shown on figure \ref{fig.netcope_pkt}.

  \begin{figure}[hbt]
    \begin{center}
      \includegraphics[width=10cm]{images/packet}
      \caption{szedata variable length packet}
      \label{fig.netcope_pkt}
    \end{center}
  \end{figure}

  Despite all improvements, this was still unsuitable due to grown
  complexity and still the transfers couldn't be done at estimated
  (theoretically computed) speeds.
  We decided to redesign whole design architecture and fork szedata project
  into the version 2. \emph{PowerPC} will be no longer used for DMA transfers,
  it'll be fully disconnected. Four separate DMA controllers for each
  interface will be implemented directly in the design instead. By that step
  we eliminate the impact of slow control bus, but we still have problems
  with data structures, the aggregation is too complex.

  After digging through some specifications such as Intel gigabit Ethernet
  cards \cite{intel_gbe}, drivers e.g. for Broadcom 43 wireless cards
  and info about projects such as Endace DAG, we considered a ring buffer
  structure as an underlying structure. This was the breakpoint to start
  szedata2 Liberouter project.

  First, we needed to create a design specification. Let's consider 10\,Gb/s
  data flow and userspace task latency of 50\,ms. We need a buffer of size
  at least $\frac{10\cdot10^{3}\cdot0,05}{8}\doteq60$\,MiB per interface not
  to throw incoming data away. But as we noted in chapter \ref{chap.mm}, we
  cannot allocate that much contiguous physical memory. This made us to split
  the buffer on the page basis and create a mechanism to cope with this.

  The idea is to allocate yet another memory for descriptor page with entries
  pointing to the pages of the interspersed ring buffer. We took in account
  64-bit systems, so that the pointers are represented by 8 bytes each. Since
  we allocate the descriptor memory on a page basis too, it allows us to have
  only $\frac{4096}{8}=512$ pointers to ring pages
  which gives $512\cdot4096=2$\,MiB sized ring buffer. This is not enough, so
  we introduced two kinds of descriptor entries. One is still a direct pointer
  to a ring buffer page, the other one is a pointer to the next descriptor
  table. To distinguish them, we take to advantage of all page addresses being
  zeroed in low 12~bits (see \cite{gorman}, chapter~3) -- descriptor pointers
  are logically or-ed with~1. Such situation is depicted on figure
  \ref{fig.ring}.

  \begin{figure}[htb]
    \begin{center}
      \includegraphics[width=9cm]{images/ring_struct}
      \caption{szedata2 ring buffer structure}
      \label{fig.ring}
    \end{center}
  \end{figure}

  \paragraph{Registers and address space}
  The last thing we need to cover before implementation is how we declare
  communication channel between driver and newly specified design. For the
  convenience of MMIO, we define design address space as a memory area. Every
  DMA engine has its own subspace with its controlling register. Up to 16
  engines are supported for future purposes, we use only 4 so far. Each space
  consist of these register groups:

  \begin{enumerate}
    \item pointer to the first page with descriptors
    \item start, end and size registers -- ring buffer state pointers, it's
      head, tail and size of whole buffer
    \item control and status registers -- used to control the design and check
      actual state
    \item interrupt registers -- timeout, position or both may be used for
      interrupt generation; position is triggered when one of ring buffer
      registers reaches value stored in position register
  \end{enumerate}

  As the DMA engine needs to have information about ring buffer before it
  starts working, registers groups 1. and 2. shall be set prior to sending
  start command to control register 3. Interrupt registers 4. are optional,
  they may be set and changed on the fly. Concrete steps of hardware
  initialization and deinitialization will be described in the driver
  specification in chapter~\ref{chap.spec}.

  After successfully launch of the engine, we only need to update tail
  pointers for receive direction after we process data and head for transmit
  after we write data to the ring buffer. Any other processing is implemented
  in the engine and we needn't to care.

  \section{Requirements}
  We decided to use some methods yet in previous chapters, here we
  summarize them and add some other requirements we must
  ensure our implementation will satisfy:
  \begin{itemize}
    \item multiple ring buffer support (per interface and direction)
    \item correct ring buffers operations (transmit head must not exceed tail)
    \item ring buffers and their descriptors setup in hardware
    \item hardware setup (another initialization, start, stop)
    \item export/remap data through memory map
    \item some controlling mechanism to communicate with application
    \item multiple applications support, multiple access to the data
    \item speed
    \item as highest as possible code correctness (successful testing)
    \item documentation (the more well commented code the better)
    \item abstraction (userspace library would be good)
    \item last but not the least: clean code (satisfying kernel coding style)
  \end{itemize}

  The only thing worth to discuss is how and when we should allocate space
  for a ring buffer. Since the impact to the system might be significant due
  to large amount of memory requirements, we must consider following two
  variants. Either we can allocate the space right after the driver loads or
  alternatively even after first application opens the corresponding character
  device node.

  The former variant assures fast application startup, but we might risk that
  we'll never use the allocated memory just because no user opens the node.
  Since the driver is about to be used in environments where the application
  is expected to be executed if the driver was loaded yet, we proceed with
  this choice by default. Otherwise we would stall on every application start
  end exit potentially exposed to start-up failures due to momental memory
  exhaustion. In spite of all the threats, somebody might want the latter
  behaviour, so we'll let user to choose by adding an option for this.

  \section{Models and design}
  \label{chap.spec}
  As we mentioned earlier, we use UML for modelling. Many UML tools are
  available for Linux, we chose one named \emph{Bouml}\footnote{
  Bouml is a free (under the terms of GPL) multiplatform project with support
  of UML including version 2. See its homepages at \url{http://bouml.free.fr/}
  for more info.}. It can do most of UML
  defined diagrams, however we'll use only \emph{use case} and
  \emph{sequence} diagrams. Complete diagrams and their documentation are in
  Appendix~\ref{app.models}, this chapter introduces the way how we got
  those diagrams as a result.

  \subsection{Use case diagram}
  Starting point is use case diagram. We create only one use case module
  which will correspond to our driver and so we name it the same -- szedata2.
  Actors choice is simple, we have an \emph{OS}, \emph{hardware} and an
  \emph{application}. OS only \emph{inserts} and \emph{removes} the driver to
  and from the kernel and furthermore
  since there are no asynchronous actions which hardware can do except
  optional interrupts, the only use case regarding hardware is
  \emph{interrupt}.
  
  Insert and remove takes care of initialization, the module may allocate and
  also free ring buffers depending on allocate-on-modprobe option in these two
  use cases. HW interrupt use case only wakes up applications waiting for
  this event.

  Application is more complicated actor, it opens (and initializes) the
  device, process data (and communicate) and finally closes (and stops) it.
  All the operations must be atomic due to concurrent access, this is not
  mentioned neither in the specification nor further, because we take this for
  granted. Fine grained application use cases description follows.

  \subsubsection{Open and initialization} We already know operations order from
  chapter~\ref{chap.netcope} that shall be done before using the device. We
  create one use case as a starting point and place following steps in it:
  \begin{enumerate}
    \item \emph{open} system call $\rightarrow$ initialize buffers if it was
      first open (not to break yet started hardware), we use reference counter
      to assure only one start of the device
    \item \emph{mmap} system call $\rightarrow$ memory map buffers
    \item \emph{select/subscribe} command $\rightarrow$ select which
      interfaces are about to be used
    \item \emph{start} command $\rightarrow$ initialize and start selected
      hardware (allow only one start per interface throughout all
      applications)
  \end{enumerate}
  Even though \emph{open} and \emph{mmap} are different calls from the driver
  view, application sees them as one basic operation, so we leave them in one
  use case. On the other hand, we find out, that we want to allow the
  application to select and start interfaces on the fly, so we must place
  points 3 and 4 in separate use cases which gives three initialization use
  cases in the end.
 
  \subsubsection{Data processing} Design supports full-duplex communication, so
  we split processing further in two categories -- receive (RX) and
  transmit (TX) and describe them separately.

  \paragraph{RX} Since hardware is started and it does RX processing fully
  standalone, we only check buffer tail pointer for changes in loop.
  This means calling kernel and getting back area with data (or no data if
  no change). After finished processing on those data, we free them
  up to tell the driver to move the tail. This gives us two separate use cases
  \emph{get\_data} and \emph{release\_data}, since the processing is done in
  application.

  After we gained access to the data, two receive processing methods raised
  altogether:
  \begin{itemize}
    \item \textbf{load balancing} -- several readers exist and data are divided
      among them. They all get distinct data and do the same operation on
      them. It's very similar to \emph{single program multiple
      data}\footnote{see \url{http://en.wikipedia.org/wiki/SPMD} for
      reference} in processors theory.
    \item \textbf{multiprocessing} -- all readers want all data to do
      different tasks with them. We can approach this to \emph{multiple
      program single data}.
  \end{itemize}
  We implement only the latter so far, but it's possible that we augment the
  implementation to the former too, see future work in
  chapter~\ref{chap.future}.

  \paragraph{TX} Outbound direction is similar in data processing --
  application asks for area to write to (gets current transmit head), writes
  data and issues command to move the head. It corresponds to the two created
  use cases \emph{request\_space} and \emph{put\_data}. It differs in access
  to the buffer, because all applications shares one TX
  ring buffer and hence only one may access it at a time not to create holes
  of unused space\footnote{There is a plan to extend this behaviour to allow
  multiple writers too, however it has not been finished yet.}. We cope with
  this by setting up proper prerequisites in the use case descriptions.

  \subsubsection{Close and deinitialization} After open reference count drops
  to zero we do steps from initialization in reverse order. \emph{close\_node}
  use case consist of \emph{hardware stop} command, \emph{unmap} and
  allocated memory cleanup. We might alternatively consider placing
  \emph{hardware stop} command in separate use case to allow stopping out of
  line, anyway we must turn off the hardware on last close if the use case
  wasn't invoked before.

  \subsection{Sequence diagrams}
  We model sequence diagrams to show up how events happen during time. The
  topmost event is done first and then all events below it towards the bottom
  one. We create two depicting again RX and TX, they will slightly differ in
  how application locks and unlocks data.

  \paragraph{RX sequence diagram} We specify better application code flow
  here. We establish three lifelines, \textbf{application}, \textbf{module}
  and \textbf{hardware} which obviously match use case actors.
  
  The work starts by \emph{open} message from \textbf{application} to
  \textbf{module}. The diagram copies actions described in use case
  initialization section above, so it nonsense to repeat the steps again. We
  just augment the use case specification further by splitting single use case
  steps into multiple messages here. For instance \emph{mmap} from use case is
  here as two messages: \emph{mmap status page} and \emph{mmap RX space}.
  That's because we try to be as much exact as possible to directly turn these
  information into the real code in the next chapter.
  
  \paragraph{TX sequence diagram} We reflect difference of RX and TX showed
  before in use case modeling here by not conditioning head move on
  \emph{put\_data} message and we also change all RX occurrences to TX and
  invoke proper messages, indeed. There is no other difference to consider
  from model view.

  \subsection{Combo6 code base}
  We have evolved base common to all Liberouter projects not to reinvent the
  wheel for each design. These base drivers are responsible for whole card
  initialization, an interrupt hook setup, loading firmware/design,
  controlling combosix device node which allows access to all spaces of the
  card and few other tasks. Since we have two types of \com\ cards, two
  card-specific drivers exist: \emph{combo6} and \emph{combo6x}; they
  are used by the \emph{combo6core} module which consist of common parts of
  both. Since this module provides encapsulation, design drivers needn't care
  about which particular card driver is driving the device, it uses
  \emph{combo6core} interface to call the proper one, so
  that the access is fully transparent.

  One more convenience is exported info in \emph{sysfs}\footnote{sysfs on
  wikipedia: \url{http://en.wikipedia.org/wiki/Sysfs}} and
  \emph{procfs}\footnote{procfs on wikipedia:
  \url{http://en.wikipedia.org/wiki/Procfs}} filesystems done by the central
  driver. This info may be further used by application to recognize module
  version or tune some values and parameters.

  Design driver should define \emph{combo6\_device\_ops} structure with name,
  card type, IDs which it supports and hooks for attach, detach and interrupt
  and pass it to \emph{combo6\_device\_register} to register it by the core
  module. Deregistration is then done by \emph{combo6\_device\_unregister}
  call. Our structure looks like this code:
\begin{lstlisting}[language=C,label=lst.c6dops,name=lst.c6dops]
static struct combo6_device_ops szedata2x_ops = {
	.owner = THIS_MODULE,
	.name = "szedata2", /* name of this driver */
	.dhw = DHW_COMBO6X, /* combo6x card only */ (*@\label{lstl.c6dops_4}@*)
	.exclusive = 1, /* only one attached driver */
	.intfs = 2, /* limit interfaces */
	.id_lsw	= 0x41c10300, /* from id */
	.id_hsw = 0x41c103ff, /* to id */
	.id_text = "NIC_NetCOPE", /* design name */
	.attach	= szedata2_hw_attach,
	.detach	= szedata2_hw_detach,
	.interrupt = szedata2_hw_interrupt,
};
\end{lstlisting}
  Some entry values need to be explained:
  \begin{itemize}
    \item \emph{owner} is set not to allow in-use modules removal, core driver
      adds 1 to reference count of the module and kernel disallows the module
      removal
    \item \emph{intfs} are limited to 2 of total 4, since the testing design
      supports only 2 interfaces so far, whereas the hardware reports 4
      present
    \item \emph{id\_*} entries are taken directly from the szedata2 design
  \end{itemize}
  We can see, that the structure also contains designator of combo6x card on
  the line~\ref{lstl.c6dops_4}.
  We must set that entry and that means creating two such structures to
  support both \com\ and \comx\ cards.

  \subsection{Implementation}
  We used term command in the specification many times, but we haven't
  explained command passing mechanism. Although \emph{ioctl} system call was
  uttered as deprecated long time ago, nothing better was introduced so far
  and it's still widely used even in new projects. Hence we'll use it for
  commands passing too.

  We split the implementation into two separate files. One is for
  non-hardware dependent part (\emph{szedata2.c} and \emph{szedata2.h}) and
  one with the rest depending on the hardware (\emph{hw.c} and \emph{hw.h}).
  The former will consist of ring buffer computations, initialization and DMA
  allocations and also character device node operations (incl. \emph{ioctls}
  and \emph{mmap}). The latter will contain the structure from previous
  section and registration to the \emph{combo6} base and all ring buffer
  operations done on the hardware (get and set head/tail pointers, ring size
  and descriptors setting, etc.) and last but not the least invocation of DMA
  allocation routines from the former file.
  
  We can conveniently use this split
  for testing and early development, because we hadn't working design while
  writing the driver. We'll create dummy \emph{hw.c} with few timers and fake
  ring operations and develop szedata2 part as specified and test it by
  that dummy module. We'll get properly working and tested driver by evoking
  corner cases by this technique. More of this is in the testing
  chapter~\ref{chap.testing}.

  \subsubsection{Structures}
  As a first thing let's settle down data structures passed through the
  system. Many of them raised through the modeling and module specification,
  we summarize them here.

  Since 64-bit operations are expensive and non-atomic on 32-bit machines,
  we try to minimize 64-bit computations as much as possible. In fact we need
  only an mmap offset to be 64-bit to allow spaces be bigger in sum than
  4\,GiB on 64-bit or 32-bit PAE enabled machines. See following structure
  descriptions and notes about 64-bit needs.

  \paragraph{Application-driver interface}
  We decided to pass all read-only information in status page which is
  allocated per application on its first open. It's depicted on the top left
  on the figure~\ref{fig.status_page}.

  \begin{figure}[htb]
    \begin{center}
      \includegraphics[width=12cm]{images/status_page}
      \caption{\emph{mmap} status page structure}
      \label{fig.status_page}
    \end{center}
  \end{figure}

  First, from the specification, status page shall contain information about
  spaces
  to be mmaped. According to the manual page of mmap and Linux kernel
  requirements, only two parameters must be known -- offset and size. As
  mentioned earlier, offset is 64-bit value and so the size is. Eleven
  (indexed by 0 through 10) such values might be stored in the page, however
  we use only
  two\footnote{Note that it's not space wasting, page size is usually 4\,KiB
  and it's enough space to reserve some entries for future purposes.}, one for
  RX space, one for TX. These offsets point to so-called \emph{spaces}
  (on the right side) which might consist (only ring buffer spaces) of
  \emph{areas}. \emph{Area} is directly the ring buffer corresponding to one
  particular hardware interface in the \com\ hardware case, however we do not
  call it interface ring buffer, because area-interface one-to-one mapping
  needn't hold if we recall,
  that the part which assures the mapping is hardware independent and might be
  used for any hardware.

  The application then remaps all desired spaces indexed by \emph{SZE2\_MMIO}
  constants and may poll for their status in \emph{area descriptors} (another
  status page entries). It's a structure with 64-bit offset and 32-bit size
  pointing directly to the area and flags which denotes if area/ring buffer
  wrap occurred and where is the next \emph{area descriptor} for the wrapped
  data.

  There are also sanity checks (magic numbers), version number and set values
  for e.g. poll in the status page, it's not so important so we let reader to
  investigate exact descriptor structure. Status page is named as \emph{struct
  sze2\_instance\_info} in the source code.

  To not poll every area we use commands to get a number of area which is
  ready to be read or write to. Four \emph{ioctls} corresponding to use cases
  exist for this purpose. RX lock and unlock are easy to define. We only ask
  for areas we want (bitmap in one parameter) and unlock all areas without
  a parameter.
  TX direction needs complicated passing of parameters. Beside the area we are
  interested in, we need to pass a size too. We define \emph{struct
  sze2\_tx\_lock} containing these two entries, unlock has same contents, only
  the name is \emph{struct sze2\_tx\_unlock}. All these sizes are sufficient
  to be 32-bit, since we don't expect somebody to read or write chunks bigger
  than 4\,GiB.

  The remaining is the area subscription done by \emph{ioctl} too. It contains
  array of two members (RX and TX) with info about areas to subscribe and poll
  threshold (minimal ready-to-process size which resumes application) used for
  waiting in the driver via \emph{poll} system call.
  Documentation in Appendix~\ref{app.doxygen} deals with this interface.

  \paragraph{Intradriver interface}
  We've declared already, that we will split the driver into two parts.
  Interface between them is straightforward. Interrupt, tail and head pointers
  passing is needed in one direction, registration and DMA allocation in the
  other one. This interface is described in the documentation contained in
  Appendix~\ref{app.kdoc}.

  \subsubsection{Hardware independent part}
  As we don't have fully working design yet, we start with design
  independent part. First we code module \emph{init} and \emph{exit}
  functions. We create a \emph{class} and reserve character device region
  (with at most \emph{SZEDATA2\_MAX\_MINORS}\footnote{currently the value is
  8, but it may be altered on compile time, since it's a macro}) as described
  in chapter~\ref{chap.devnodes} and store a number which we got from it to
  use later in \emph{exit} function when unregistering character device.

  We see, that we don't allocate any buffers or any other data, so we must
  export hooks for events when device comes and leaves, even than we can
  create and destroy any structures. These operations are let to the hardware
  specific part, because it knows how many areas are needed and how large ring
  buffers should be allocated.
  
  Registration of incoming device is split into two parts -- allocation
  (\emph{alloc\_szedata2}) and registration (\emph{szedata2\_register})
  itself. After calling the former, callbacks shall be set up and eventually
  DMA allocations must be done now (or alternatively even on the first
  open). If we didn't the split, we might end up with kernel panic, because
  concrete character device is set in the register path and some so far zeroed
  callback invoked. \emph{szedata2\_destroy} is created for cleanup purposes.
  We must take in account situations when latter register function
  hasn't been called at all. This is achieved by setting one bit in the
  register function and test for this bit in the destroy one.

  DMA allocation is done through PCI interface described in
  chapter~\ref{chap.dma}. Allocation is done per page basis (recall
  chapter~\ref{chap.netcope}), so if we consider four interfaces, both RX and
  TX directions and 50\,MiB sized ring buffers, we have
  $\frac{4\cdot2\cdot50\cdot2^{20}}{4096}\doteq100000$ pages for 4096\,B pages.
  We need to store physical and virtual address for each page (to be able to
  work with the buffer and also free it later), worst case are on 64-bit
  machines, where both numbers are 8-bytes, we further have
  $100000\cdot2\cdot8=1600000$. This information might be stored in
  $\frac{1600000}{4096}\doteq390$ pages. That's a number very close to
  $2^{MAX\_ORDER}$\footnote{\emph{MAX\_ORDER} is maximum allowed power of 2
  to allocate via \emph{get\_free\_page}, it used to be between 5 and 20
  depending on architecture and configuration}, so that we would end up in
  failure if we didn't use virtual memory (\emph{vmalloc}) for those
  descriptors.

  Even if we're allocating ring buffer page after page, we might face
  failures. It's because of how allocations work in memory management
  subsystem. For implementation in Linux 2.4 see~\cite{gorman}, 2.6 differs
  slightly, but it's the same in principle. Most of inactive pages is freed
  even under high pressure, i.e. priority is high. The priority is raised by
  each failure and the higher the priority is the more pages are tried to free
  (also the used ones might be swapped out if they belong to userspace). This
  means we should try to allocate pages which failed at least several times
  to raise the priority with some delay between the tries to let swapper move
  some pages to the swap device. See \emph{szedata2\_alloc\_dmaspace} function
  for final implementation.

  We passed also \emph{struct file\_operations} when registering concrete
  character device which contains \emph{open}, \emph{release} (close),
  \emph{mmap}, \emph{poll} and \emph{ioctl} hooks.

  \subsubsection{Back-porting to older kernels}
  \label{chap.backport}
  Since we want the driver to be used on many configurations, we have to
  support all Linux~2.6 kernel in the best case. This is not possible without
  help of a programmer. Several APIs have changed during 2.6 as we mentioned
  in \emph{Linux interface} sections earlier.

  For the convenience of modularism, we needn't to care of IRQ handler or
  MMIO mapping changes, since we caught the changes in the \emph{combo6}
  modules already. We created a file named \emph{kocompat.h} where
  compatibilities are solved by testing for kernel version and using proper
  interface. However we must solve \emph{mmap} and \emph{class} changes for
  all drivers, because change in them is too new and unresolved.
  
  While we were porting drivers from one versioning system to another, we
  created two trees with drivers. One is \emph{kernel} and there are
  sourcecodes prepared for inclusion in mainline kernel and the another is
  \emph{build}. This is where we locally build the drivers and where another
  compatibility stuff lays. It has the same hierarchy and file names except
  the files contain only \emph{\#include} line to "load" the contents of the
  real file from \emph{kernel} subtree. We'll use this split for compatibility
  handling, because this issues need to be solved in the \emph{build} tree.

  \paragraph{Mmap} \emph{.fault} handler was introduced and previously used
  \emph{.nopage} was removed.

  Why we should do this? Macros used (irq handler vs. 2.6.19, mmap vs.
  remap\_pfn\_range, remap\_page\_range), workarounds (class+class\_simple

  \subsubsection{libsze2}
  Even though there is no library for szedata v. 1, we consider to have one for
  szedata2. The main point to have it is not to propagate the kernel
  interface directly to the user, to simplify the interface and make it
  abstract.
  
  The library functions might be an exact copy of use cases from the
  specification, since the use case alone means exactly that it is a
  separate sequence of commands which might not be further divided. If we
  encapsulate each use case's contents in a library call, we reach the
  aims from previous paragraph. Above all, if we create a shared library,
  we'll get easy to update part (module) of each application.
