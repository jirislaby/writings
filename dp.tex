\begin{ThesisAbstract}
  In this thesis we are concerned in fast transfers from hardware to computer
  memory and vice versa. Above all we specialize to scatter-gather direct
  memory access transfers from/to hardware through ring buffers. Also we
  discuss other transfer methods implemented in other projects and consider
  using them here. Finally we implement the method and test it with all
  available tools and describe the testing process.
  
\end{ThesisAbstract}

\chapter{Introduction}
  To even boot a computer, it's necessary to proceed many transfers inside the
  hardware among its components. It's a basic function which must work,
  however there are several methods of how to achieve that. Especially
  the fast ones are very important in computer science which results in higher
  overall throughput of the system.

  There are several computer science areas, where fast transfers are very
  important. Beside the computer-inside transfers among CPU, disks,
  DVD writers or main memory, we must take in account also the outside ones,
  such as moving huge data amounts from/to external disks, displays or data
  cards in cellphones for example, all over buses with specific data formats.
  Outstanding speed is important especially in the area of network
  communication; backbone network routers or monitoring probes are very
  dependent on how fast data transfer can be achieved in the end.

  Nowadays the equipment of operating system can do fast network transfers
  through well-known interfaces, but there exist several methods which are
  not so common, but might provide yet another significant improvement. For
  instance nowadays mainly used technique of \emph{direct memory access} in
  cooperation with ring buffer data structure and memory mapping (so-called
  \emph{mmap}) might move big chunks of data without even processor
  assistance from network adapter towards application in userspace and vice
  versa without any real data copying. This results in short transaction
  times.

  \section{Devices interconnection}
  \label{chap.inter}
  The first thing to explain is how data flows from devices to processor and
  particularly which types of buses are on the route from the CPU to a device.
  Data path may be seen on figure \ref{fig.dev_inter}.

  \begin{figure}[ht]
    \begin{center}
      \includegraphics[width=8cm]{images/data_flow}
      \caption{Devices interconnection}
      \label{fig.dev_inter}
    \end{center}
  \end{figure}

  Each PCI device is physically connected to any PCI bus via PCI slot. Other
  buses (might be another PCI) can be connected through PCI-to-$<$the
  bus$>$ bridge. PCI bus is attached to southbridge (called ICH\footnote{I/O
  controller hub} by Intel), it communicates with northbridge
  (MCH\footnote{memory controller hub}) over internal bus and it's finally
  wired with processor via front-side bus (FSB). Sometimes all southbridge,
  northbridge and memory controller are implemented as a proper PCI devices
  directly connected to main PCI bus.

  Usually 3 main group of signals are on each bus:
  \begin{list}{*}{}
    \item \textbf{control bus} -- it's supposed to send signals and synchronize
      object which is connected to
    \item \textbf{address bus} -- that is address of memory which is requested
      to read from or write to
    \item \textbf{data bus} -- data itself which flows from/to address
      specified on address bus
  \end{list}

  PCI has data and address bus multiplexed, so that it minimizes signals
  count. Also it's important to know, that while FSB has 64-bits wide address
  bus,
  standard PCI buses are 32-bit only. We must take this in account if we want
  to send 64-bit numbers over PCI to older \com\ cards. PCI-X and PCI-express
  buses are natively 64-bit long, so this is not a problem for some newer
  \comx\ cards.

  \section{DMA and Bus Mastering}
  Direct memory access known as abbreviation DMA is a method widely used to
  speed up transfers in computer engineering. It's based on technique called
  bus mastering. A device which wants to move data from/to memory becomes bus
  master and no other device or even processor can't touch address and data
  bus described in section \ref{chap.inter}. While being a master, it sets
  address on an address bus and sends or latches data on a data bus.
  
  On older machines, there was only one such device called DMA controller
  (DMAC). Nowadays each PCI device (see also Section \ref{chap.pci}) may
  become a master, so that it can access memory directly by its own DMAC.
  Note that there is no common PCI DMAC, so if the device doesn't have one, it
  doesn't support DMA either. DMAC count is not bounded to only
  one as we will see in chapter \ref{chap.netcope}. Unit which takes care of
  honest bus accesses if there is more than one bus master is called
  \emph{Arbiter}.

  \section{Memory management}
  This chapter is a short introduction into memory management as used not
  only in Linux. Memory management in details on Intel processors is described
  in \cite{ia1} and also in \cite{gorman}.
  \subsection{Physical memory}
  In the usual meaning we call main \emph{random access memory} as physical
  memory. Although it is possible to not have this kind of memory by a
  processor in special computers like \emph{NUMA} (non-uniform memory access),
  we usually have one or more
  memories plugged in motherboard. There are several addressing methods how
  processor can access data on it.

  \subsubsection{Real-address mode}
  In the early Intel 8086 processors, this mode was used. For the
  compatibility reasons, this mode is available on todays CPUs too, and is
  used mostly only in the early booting stage.

  Physical memory is divided into 16 64-kilobyte sized segments, so that up
  to $2^{20}$ bytes might be addressed. The rest of memory is unused.

  \subsubsection{Segmentation mode}
  Similar to previous model, but it allows segments to be up to $2^{32}$ in
  size and their maximal count is 16383 on 32-bit. Application sees them as
  independent address spaces potentially of different types. This might be
  used to avoid stack or heap overflows and code or data damage, because
  each of them can reside in separate segment.

  \subsubsection{Flat mode}
  Each byte of memory is simply addressed by \emph{linear address}. First
  memory byte has address 0, on 32-bit without any extensions the last
  possible byte (if there is that much physical memory) has address
  $2^{32}-1$.

  \subsection{Virtual memory}
  With only physical memory addressing problems arose early. Applications
  needed
  continuous big chunks of memory, but it was impossible to find such long
  block after a while system was running. It was caused by process called
  (external) fragmentation. Other tasks held memory blocks and there wasn't
  enough space in between them to return. See figure \ref{fig.fragmentation}
  for illustration.
  
  Task 1 holds two memory blocks of size 2 units. Task 2
  allocated one block three units wide. Now, if any of the two wants to
  allocate block of size 4, allocation will fail, although there is
  enough free space in sum.

  \begin{figure}[h]
    \begin{center}
      \includegraphics[width=8cm]{images/fragmentation}
      \caption{Memory fragmentation illustration}
      \label{fig.fragmentation}
    \end{center}
  \end{figure}

  Many algorithms were introduced to avoid this behavior, some of them are
  described in Knuth\cite{knuth_art} including buddy system used in Linux
  kernel for in-kernel allocations. We will see consequences of this
  constraints later in chapter \ref{chap.netcope}.

  \subsubsection{Paging}
  To solve the situation with external fragmentation, method named
  \emph{paging} was introduced. Linear address space is divided into pages
  of fixed size\footnote{4096 bytes on Intel with page size extension (PSE)
  disabled} that might be mapped to physical memory or disks as swap.

  CPU addresses physical memory by virtual addresses. This is not possible
  without translating virtual address into the physical (either in segmented
  or flat mode) one. Transparent translation is done in \emph{memory
  management unit} (MMU).

  Virtual address is usually divided into several parts and each of them is
  offset in some table. On Intel 32-bit with \emph{page address extension}
  (PAE) and PSE disabled, 3 such tables exist and each table contains
  entries/pointers to next table or
  to physical page in the last table respectively. If the entry in either
  table is marked as non-present, page fault, which is exception handled by
  operating system, is thrown. OS must either put (map) the requested page in
  or kill the program which references illegal address.

  On figure \ref{fig.vaddr} is shown, how is logical address split on 32-bit
  Intel with extensions disabled. CR3 is register in these processors
  containing location of global page directory depicted on the left side.
  \begin{figure}[h]
    \begin{center}
      \includegraphics[width=10cm]{images/vaddr}
      \caption{Virtual address format}
      \label{fig.vaddr}
    \end{center}
  \end{figure}

  This schema allows us to allocate huge contiguous memory blocks without a
  need of having contiguous physical memory. In fact there is no need to have
  as same physical memory as requested. Some memory may be, as written above,
  backed by a page stored on disk (swapped out) or even have no "physical"
  page mapped to it -- Linux uses demand paging, so that page is physically
  backed up even after accessing it (page fault handler takes care of this).

  On 32-bit, we are able to use up
  to 3 gibibytes\footnote{SI unit for $2^{30}$} of virtual memory per process.
  This limit comes from 1:3 of kernelspace to userspace split of process
  virtual address space. On 64-bit, 48-bit virtual addresses are supported by
  most current processors, up to $2^{47}$ may be used in userspace, kernel is
  mapped in top $2^{47}$ bytes due to 64-bit sign extension (see \cite{ia3a}
  for reference) and the space between them is an unused memory hole, which
  might be addressed even by newer processors.

  We will consider these facts while deciding how much memory we should
  allocate for the hardware at maximum.

KERNEL!="combosix*", GOTO="cesnet_generator_end"(*@\label{lstl.udevrules_1}@*)

  Since we want to match only additions of combosix devices, rules at line

\begin{lstlisting}[name=lst.udevrules]
ENV{CS}="$kernel"
ENV{MATCHPATH}="$devpath"
\end{lstlisting}

\begin{lstlisting}[name=lst.udevrules]
IMPORT{PROGRAM}="csdevname" (*@\label{lstl.udevrules_11}@*)

LABEL="cesnet_generator_end" (*@\label{lstl.udevrules_12}@*)
\end{lstlisting}
  And finally invoke the script and set up the new name if the script changed

  \paragraph{csdevname script}
  Shell script \textbf{csdevname} called from line \ref{lstl.udevrules_11}
  above follows:
\lstset{language=bash}
\begin{lstlisting}[label=lst.udevscript,name=lst.udevscript]
RULES_FILE='/etc/udev/rules.d/70-cesnet.rules'

. /lib/udev/rule_generator.functions(*@\label{lstl.udevscript_1}@*)
\end{lstlisting}
  This loads udev predefined functions used later. RULES\_FILE is the
  name of file, where we store our persistent naming. Variable name is
  predefined (and used in the script sourced at line \ref{lstl.udevscript_1})
  and must not be changed.
\begin{lstlisting}[name=lst.udevscript]
cs_name_taken() {
	local value="$(find_all_rules 'NAME=' $CS)"
	if [ "$value" ]; then return 0; else return 1; fi
}

find_next_available() {
	raw_find_next_available "$(find_all_rules 'NAME=' "$1")"
}
\end{lstlisting}
  Functions \textbf{cs\_name\_taken} and \textbf{find\_next\_available} are
  for finding available names for new coming device. Both uses functions
  defined in the sourced script from line \ref{lstl.udevscript_1}, that allows
  be them as compact as possible.
\begin{lstlisting}[name=lst.udevscript]
write_rule() {
	local match="$1"
	local name="$2"
	local comment="$3"
	{
		[ "$comment" ] && echo "# $comment"
		echo "SUBSYSTEM==\"combosix\", ACTION==\"add\"$match, NAME=\"$name\""
	} >> $RULES_FILE
}
\end{lstlisting}
  \textbf{write\_rule} takes 3 parameters -- under which condition this rule
  is triggered, static node name and potentially a comment to this rule. The
  result is written to previously defined \textbf{RULES\_FILE}.
\begin{lstlisting}[name=lst.udevscript]
lock_rules_file(*@\label{lstl.udevscript_27}@*)
choose_rules_file(*@\label{lstl.udevscript_29}@*)
\end{lstlisting}
  At lines \ref{lstl.udevscript_27} and \ref{lstl.udevscript_29} we lock up
  the rules file to prevent race
  conditions (if two writers occurred, it would result in unpredictable
  behavior) and decide in which file will be the new rules written (since
  usual location might be still read-only while booting -- \emph{udev} will
  then move the temporary file to the proper directory after it becomes
  writable).
\begin{lstlisting}[name=lst.udevscript]
MATCHPATH=${MATCHPATH%/*/*}
if [ "$MATCHPATH" ]; then
	match="$match, DEVPATH==\"$MATCHPATH/*\""
fi

if [ -z "$match" ]; then
	echo "missing valid match" >&2
	unlock_rules_file
	exit 1
fi
\end{lstlisting}
  Now we define matching rule passed to \textbf{write\_rule} later (line
  \ref{lstl.udevscript_49}). Sanity check of the value is done here and if it
  fails we return error.
\begin{lstlisting}[name=lst.udevscript]
if [ "$CS_NAME" ]; then
	COMMENT="$COMMENT (custom name)"
	if [ "$CS_NAME" != "$CS" ]; then
		CS=$CS_NAME;
		echo "CS_NEW=$CS"
	fi
else
	base=${CS%%[0-9]*}
	if cs_name_taken; then
		CS="$base$(find_next_available "$base[0-9]*")"
		echo "CS_NEW=$CS"
	fi
fi
\end{lstlisting}
  Here the name, which is returned to rules (\textbf{CS\_NEW}), is set. Logic
  is simple, either the one passed by external application is used or a new
  one is found (first unused).
\begin{lstlisting}[name=lst.udevscript]
write_rule "$match" "$CS" "$COMMENT"(*@\label{lstl.udevscript_49}@*)

unlock_rules_file
\end{lstlisting}
  Finally the file with static naming is augmented and unlocked.

  Since this moment, if the device come in the system again later, even in
  other order, system will ever create a node with the same name stored in
  the rules file.
