\begin{ThesisAbstract}
  In this thesis we are concerned in fast transfers from hardware to computer
  memory and vice versa. Above all we specialize to scatter-gather direct
  memory access transfers from/to hardware through ring buffers. Also we
  discuss other transfer methods implemented in other projects and consider
  using them here. Finally we implement the method and test it with all
  available tools and describe the testing process.
  
\end{ThesisAbstract}

\chapter{Introduction}
  To even boot a computer, it's necessary to proceed many transfers inside the
  hardware among its components. It's a basic function which must work,
  however there are several methods of how to achieve that. Especially
  the fast ones are very important in computer science which results in higher
  overall throughput of the system.

  There are several computer science areas, where fast transfers are very
  important. Beside the computer-inside transfers among CPU, disks,
  DVD writers or main memory, we must take in account also the outside ones,
  such as moving huge data amounts from/to external disks, displays or data
  cards in cellphones for example, all over buses with specific data formats.
  Outstanding speed is important especially in the area of network
  communication; backbone network routers or monitoring probes are very
  dependent on how fast data transfer can be achieved in the end.

  Nowadays the equipment of operating system can do fast network transfers
  through well-known interfaces, but there exist several methods which are
  not so common, but might provide yet another significant improvement. For
  instance nowadays mainly used technique of \emph{direct memory access} in
  cooperation with ringbuffer data structure and memory mapping (so-called
  \emph{mmap}) might move big chunks of data without even processor
  assistance from network adapter towards application in userspace and vice
  versa without any real data copying. This results in very short transaction
  times.

  \section{Linux}
  Linux is an operating system based on kernel which was established in the
  early nineties of the 20th century by Linus Torvalds. There is over 3
  hundreds variants called distributions shipped by either communities or
  software giants.
  
  Linux kernel in its version 2.6 has no stable api inside the
  kernel, but still the api is well-evolved and easy to use, because the
  development is ridden and checked by a linux kernel community. We will
  see consequences of such development method in detailed description of some
  subsystems in detail later and in compatible layer implementation too.

  Major part of linux kernel is licensed under GPL version 2 \cite{gplv2}, but
  BSD and ISC code is not a rarity. Chosen license which will be the
  implementation distributed under is GPL version 2 like other drivers
  majority.

  \section{Liberouter project}
  Liberouter is a project supported by \emph{CESNET} (Czech academic network)
  and \emph{6NET} (European internet project). There were raised many projects
  under Liberouter mainly to transfer and scan high traffic on backbones.
  All those work on cards called \com\ or \comx\ (PCI-X and PCI express version)
  respectively. One of such card is for illustration depicted on figure
  \ref{fig.combo6}.

  \section{NetCOPE project and szedata2}
  One of Liberouter projects is \emph{NetCOPE}. The goal is to develop a
  \com\ cards based platform which is immutable against fast evolution of
  networking. \emph{NetCOPE} data flow might be seen in schematic on
  figure \ref{fig.netcope_schema}. As can one see there, this universal
  solution allows involving it in multiple areas
  such as network traffic filtering, routing packets, intrusion detection and
  others.

  \begin{figure}[h]
    \begin{center}
      \includegraphics[width=11cm]{images/netcope_color}
      \caption{NetCope schematic}
      \label{fig.netcope_schema}
    \end{center}
  \end{figure}

  We'll develop the software on the top of the \emph{NetCOPE} platform.
  \sze\ is chosen as a name after its predecessing first generation
  driver based on pre-\emph{NetCOPE} hardware. It stands for \emph{straight
  zero copy data} in the meaning of avoid copying data from memory to the
  application and vice versa by remapping it as we'll later.
