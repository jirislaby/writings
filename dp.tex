\begin{ThesisAbstract}
  In this thesis we are concerned in fast transfers from hardware to computer
  programs and vice versa. Above all we specialize to coherent direct
  memory access transfers from/to hardware through ring buffers. Also we
  discuss other transfer methods implemented in other projects and consider
  using them here. Finally we implement chosen method and test it with all
  available tools, describe the testing process and compare with already
  existing solutions.

\end{ThesisAbstract}

\chapter{Introduction}
  To even boot a computer, it is necessary to proceed many transfers inside the
  hardware among its components. It is a basic function which must work,
  however there are several methods of how to achieve that. Especially
  the fast transfers are very important in computer science which results in
  higher overall throughput of the system itself and also in faster
  communication with other ones.

%  There are many computer technology areas, where fast transfers are very
%  important. Beside the computer-inside transfers among CPU, disks,
%  DVD writers and main memory, we must take in account also the outside ones,
%  such as moving huge data amounts from/to external disks, monitors or data
%  cards in cellphones for example; all over buses with specific
%  data formats. Outstanding speed is important especially in the area of
%  network communication -- as an example backbone network routers or
%  monitoring probes are very dependent on how fast data transfers can be
%  achieved in the end.

%  Nowadays the equipment of operating systems can do fast network transfers
%  through well-known interfaces, but like in other areas, even here is a
%  space for improvements, for methods which are not so common, but might
%  provide yet another significant speedup.

  As an answer to permanent higher requirements in the networking area,
  Liberouter project has been established as a project supported by Czech
  national research and education network CESNET and European project 6NET.
  The original intention was to develop a router
  for high traffic networks. After some time, this idea was augmented to other
  network usages such as network filtering and monitoring. All these functions
  run on \com\ family PCI cards (see Figure~\ref{fig.combo6} for
  illustration).

    \end{center}
  \end{figure}

  This thesis is based on one of many researches that have been already done
  under Liberouter -- \ncope\ -- which we describe in
  chapter~\ref{chap.netcope}.
  In short, it is a hardware platform trying to achieve universality along with
  speed and it is fully programmable. Software running on them can do fast
  transfers if it is instructed to. Drivers are usually used for this purpose.
  Despite some are ready yet, a new \ncope\ software was developed and
  it needs a new driver.

  This thesis describes the evolution process of such driver from its
  requirements, considering variants to be as fast as possible but still be
  compliant with hardware specification \cite{netcope}, especially
  direct memory access (see \ref{chap.dma}) in combination with ring buffers
  (see \ref{chap.netcope}).

  After requirements settle down, we model the specification with help of UML
  and get towards final code implementation in C language (kernel part in GNU
  flavour, see \cite{GNUC}). As much as
  possible code parts (all corner cases at best) need testing, we figure out
  which testing techniques might be applied on drivers and finally comparing
  with older drivers and card softwares will be done.

  \paragraph{Linux}
  In the early stages, we decided to create a driver implementation only for
  Linux operating system. It is an operating system based on kernel which was
  established in the
  early nineties of the 20th century by Linus Torvalds. There is over three
  hundred variants called distributions shipped by either communities or
  software giants.

  Linux kernel in its version 2.6 has no stable application programming
  interface (API) inside the kernel, but still the API is well-evolved and
  easy to use, because the
  development is ridden and checked by a Linux kernel community. We will
  see consequences of such development method in detailed description of some
  subsystems later and in compatible layer implementation too.

  Major part of Linux kernel is licensed under GPL version 2\footnote{License
  is available at \url{http://www.gnu.org/licenses/gpl-2.0.html}.}.
  Chosen license which the implementation will be distributed under is GPL
  version 2 like most other drivers.

  \subsection{PCI}
  \label{chap.pci}
  We are concerned especially in \emph{peripheral component interconnect}
  (PCI) devices, because \com\ card families are PCI based. Currently we have
  a legacy PCI (32-bit PCI) \com\ card, \comx\ for PCI-X (64-bit PCI) and
  \come\ based on PCI-Express specification (64-bit packet based PCI). Each
  such device is physically connected to any PCI bus via PCI slot.
  PCI bus is attached to \emph{southbridge}, it communicates
  with \emph{northbridge} over internal bus and it is finally wired
  with processor via \emph{front-side bus} (FSB) and with memory. Nowadays
  it is still more common, that all southbridge, northbridge and memory
  controller are implemented as proper PCI devices directly connected to the
  main PCI bus.

  Older PCI buses have a \emph{control bus} supposed to send controlling
  signals and synchronize objects which are connected to it and further
  \emph{address} and \emph{data buses} multiplexed on one address/data bus.
  A transmission starts by sending address followed by data on the same wires.
  Legacy PCI address/data bus is only 32-bit wide and hence sending of 64-bit
  numbers over the bus to older \com\ cards is not possible. Unlike data,
  addresses are normally sent in so-called \emph{single address cycle} (SAC)
  mode, when 32-bit address is set on the bus and latched by the other side,
  but it is possible to send even 64-bit address via \emph{double} (sometimes
  \emph{dual}) \emph{address cycle} (DAC) by negotiating it and repeating SAC
  twice. The \com\ hardware supports SAC nowadays, but we plan to utilize DAC
  mechanism in the future to address more than 4\,GiB of physical memory, see
  chapter~\ref{chap.future}.

  These problems are completely avoided later in PCI-Express which communicates
  over a serial physical layer by packets routed by switches and hubs. Each
  request or reply is a packet which may contain both 32 and 64-bit addresses
  in the packet header and whatever data in the payload.

  Further detailed PCI information is in Legacy PCI, PCI-X\cite{pcispec} and
  PCI-Express\cite{pciespec} specifications.

  \paragraph{Configuration Space}
  Every modern bus supports a mechanism to autoconfigure devices connected
  to it. Also each PCI device has its configuration space, where all the
  needed settings are stored or may be set to. This space contains information
  about who developed the device and which model it is. These data are useful
  especially in a driver area, where the drivers want to bind only to a
  specific subset of devices.

  In our scope of interest are device, vendor and their subsystem variant
  IDs contained in this space, because these are mostly merely identifiers
  designating one concrete device family. It is also helpful in userspace
  scripts to create persistently named nodes in section~\ref{chap.udev}.

  \paragraph{Linux PCI Support}
  PCI is well supported in Linux since it became a standard. Concrete API is
  in a scope of the chapter~12 of \cite{ldd}. In short, a driver's hook is
  called whenever the PCI subsystem finds a suitable device,
  initialization is done by functions with a \emph{pci\_} prefix, especially
  device must be enabled and spaces
  remapped to be usable. Then the device is ready to work. Deinitialization
  and device switch off is done in driver's remove hook.

  Transfers are done by a processor performing standard read and writes over
  the remapped space, unit by unit. The unit is up to 4 bytes long on 32-bit,
  on 64-bit up to 8 bytes. These transfers are not too fast and need processor
  attention.

  \section{DMA and Bus Mastering}
  \label{chap.dma}
  \emph{Direct memory access} known as DMA is a method widely used to
  speed up transfers in computer engineering. It is based on technique called
  \emph{bus mastering}. A device which wants to move data from/to memory
  becomes a bus master and no other device or even processor cannot touch
  address and data bus. While being a master, it sets address on an address
  bus and sends or latches data on a data bus. When there are more than one
  data phases after address phase on multiplexed bus, the transfer is done in
  so-called \emph{burst mode}.

  On older machines, there was merely one device called \emph{DMA controller}
  (DMAC), which could become a master. Nowadays each PCI device (see also
  section~\ref{chap.pci}) may become a master, so that it can access memory
  directly by its own DMAC\footnote{Note that there is no common PCI DMAC, so
  if the device doesn't have one, it doesn't support DMA either.}. As we will
  see in chapter~\ref{chap.netcope}, their count is not limited to one per PCI
  device. An unit which takes care of fair bus accesses by more than one DMAC
  is called \emph{Arbiter}.

  \subsection{PCI DMA Interface}
  To become a master on legacy PCI bus, the device asks the Arbiter for a band
  and after it gets start signal on control bus, it starts sending. First,
  address is sent, then data part. If there is more data to send, burst mode
  is used until the Arbiter emits a stop
  signal. After that signal, the device must finish the transfer of near few
  bytes in the cache line\footnote{Its size is configurable through config
  space.} and acknowledge stopping. Next device gets start signal from the
  Arbiter
  at this moment and starts data transfer. The ancestral device may ask for a
  band again unless all data have not been sent. This method is implemented in
  PCI and PCI-X versions of \com\ cards.

  PCI-Express is a bit different, it asks the other device to get some amount
  of data in request packet and the device fills reply packet with the data.
  Sending is similar, the source device fills the data
  directly in the request and gets back completion packet. This is obviously
  implemented in PCI-Express based \com.

  The transfer is theoretically limited by the other device (including the
  main memory) boundary\footnote{It is 4096 bytes on the main memory with
  extensions disabled.}. It must not exceed this limit, the transfer is
  interrupted by the other side otherwise. Transfers of 1\,KiB per one
  transfer are not uncommon in \com\ case.

  \subsection{Linux DMA Interface}
  Linux has three groups of DMA APIs. ISA/LPC, PCI and generic. ISA/LPC is
  used only for old 8237 chips located on old ISA\footnote{Industry Standard
  Architecture} buses and on todays LPC\footnote{Low Pin Count} buses, neither
  one is in our scope of interest. See
  \emph{Documentation/DMA-ISA-LPC.txt} from Linux kernel sources for usage
  explanation.

  \paragraph{PCI interface} This interface supports coherent (also called
  consistent) and non-coherent DMA mappings, both documented in
  \emph{Documentation/DMA-mapping.txt}. We always must decide which one
  we should employ.
  
  Non-coherent mappings are often mapped exclusively for a transfer
  and then unmapped and underlying buffer freed. This is method widely used
  for socket buffers which are thrown away after successful transmission. We
  will probably utilize this interface for exactly this purpose out of scope
  of this thesis, see section~\ref{chap.future}.

  The other group are coherent mappings usually mapped at the driver load or
  the device start and unmapped and freed even after the device stop. Its
  domain are
  DMA ring buffers, firmwares used by some devices and executed from main
  memory meanwhile device operation and so on. This mapping hence perfectly
  fits our needs and will be implemented in the new module,
  see section~\ref{chap.netcope}.

  \paragraph{Generic interface} It is actually an abstraction of PCI interface.
  It is not bounded to any PCI device, it may be applied for non-PCI devices,
  however it obviously doesn't support PCI features completely. API of this
  type is taken down in \emph{Documentation/DMA-API.txt}.

  \section{Memory Management}
  \label{chap.mm}
  This chapter is a short introduction into memory management as used not
  only in Linux. Memory management in details on Intel processors is described
  in \cite{ia1} and also in \cite{gorman} and chapter~2 of \cite{ulk}.

  \subsection{Physical Memory}
  In the usual meaning we call main \emph{random access memory} as a physical
  memory and we usually\footnote{It's possible not to have this kind of memory
  by a processor in special computers like \emph{NUMA} (non-uniform memory
  access).} have one or more of them plugged in the motherboard. These modules
  can be
  addressed by several methods; Linux implements \emph{flat mode}, where
  each byte of memory is simply addressed by \emph{linear address}. First
  memory byte has address 0, on 32-bit without any extensions the last
  possible byte (if there is that much physical memory) has address
  $2^{32}-1$.

  \subsection{Virtual Memory}
  With just physical memory addressing problems arose early. Applications
  needed large amounts of continuous memory, but it was inconceivable to find
  such long block after a while system was running. It was caused by process
  called (\emph{external}) \emph{fragmentation}. Other tasks held memory
  blocks and there wasn't enough space in between them to return.

  Many algorithms were introduced to avoid this behavior, some of them are
  described in \cite{knuth_art} including \emph{buddy system} employed in
  Linux kernel for in-kernel allocations of larger blocks.

  \subsubsection{Paging}
  However, the algorithms only mitigate the risk of fragmentation
  being hit, we would still have problems with allocations bigger than some
  constant specific for each allocation mechanism.
  To really solve the situation with external fragmentation, method named
  \emph{paging} was introduced. Linear address space is divided into pages
  of fixed size\footnote{4096 bytes on x86 with page size extension
  disabled.} that might be mapped to physical memory or disks as swapped pages.

  CPU accesses physical memory by virtual addresses with help of \emph{memory
  management unit} (MMU) which performs transparent translation to physical
  address. Some modern mainboards contain another unit called IOMMU (MMU for
  I/O) between a device and the main memory
  path, so that devices address by logical address too. It allows creating a
  virtual space even for devices, but it is not widely spread so far
  so \com\ card developers must not take it for granted.

  The paging schema allows us to allocate huge contiguous memory blocks
  without a
  need of having contiguous physical memory. In fact there is no need to have
  as much physical memory as requested. Some memory may be, as written above,
  backed by a page stored on disk (swapped out) or even have no "physical"
  page mapped to it thanks to \emph{demand paging}.

  Although 64-bit kernels are able to address up to $2^{47}=128$\,TiB mapped
  to the bottom of address space on modern processors (the kernel is mapped
  in the top of the same size thank to 64-bit sign extension, see
  \cite{ia3a} for reference, with so far unused \emph{memory hole} between
  them), 32-bit ones are bounded to nowadays real
  limit of 3\,GiB of virtual space per process. The remaining addressable
  1\,GiB is used as kernelspace. This will be reflected while estimating
  size of buffers needed for userspace applications along with
  physical memory constraints for device memory buffers.

  \paragraph{Linux memory interface} To obtain a buffer smaller than a page
  is used \emph{kmalloc} function and \emph{kzalloc} returning a zeroed
  space (buffer filled with zeroes). They are
  calls of slab allocator (see \cite{gorman}, chapter~8). Freeing is provided
  by \emph{kfree}.

  For larger physically contiguous buffers (consequent pages)
  \emph{get\_free\_page} family should be employed -- variants for zeroed
  page(s), for single page, for functions which return \emph{struct page}
  pointer instead of logical address. Freeing is done by \emph{free\_page}.
  These page allocations might be theoretically used for DMA, its physical
  addresses could be gained by \emph{page\_to\_phys} (\emph{struct page}) or
  \emph{virt\_to\_phys} (logical address), however this is not clean, because
  the returned address needn't be device addressable. We should call
  \emph{dma\_alloc\_coherent} to get an addressable page. It gets
  \emph{struct device} as one of parameters where DMA capabilities are
  written down and so it "knows" which addresses are fine to return. Possibly
  a wrapper \emph{pci\_alloc\_consistent} might be used to pass \emph{struct
  pci\_dev} directly. \emph{dma\_free\_coherent} and
  \emph{pci\_free\_consistent} serves as clean-up calls.

  \emph{vmalloc} is utilized for in-kernel virtual contiguous memory
  allocations. Other wrappers like \emph{vmalloc\_32}, \emph{vmalloc\_user} and
  \emph{vmalloc\_32\_user} exist for allocating buffers addressable by 32-bit
  addresses and the two latter for buffers intended to be used in
  userspace. In such allocations are underlying pages marked as \emph{user}
  and zeroed not to leak any previously stored information, however they are
  present only in kernels as of 2.6.18. \emph{vfree} frees all these
  allocations.

  For the convenience of \com\ being PCI devices, we employ
  \emph{pci\_alloc\_consistent} and \emph{pci\_free\_consistent} for DMA
  allocations, \emph{vmalloc\_32\_user} for buffer to be mapped to userspace
  and obviously \emph{kmalloc} for driver state structures.

  \subsection{Memory Map}
  As we are able to set (and unset) page present bit in structures
  MMU traverses and an exception is thrown when the bit is not set while any
  of the structures is accessed, we can hook a page fault handler to this
  path which would install missing pages.

  This can be used for \emph{memory mapping} (MMAP). We register a page fault
  function by OS interface and when user application maps some space and
  accesses it, we may set up there any pages we want. According to exact
  address which was the root cause of the fault, we compute a number of page
  which should be in that particular memory place and install it there.
  Application sees this as a transparent process and it accesses the data from
  the installed page since that moment.

  \paragraph{Linux mmap interface}
  Many incompatible interfaces were introduced and removed meanwhile Linux
  2.6 development process. First versions support remapping of some sort of
  memory simply by implementing real fault handler called \emph{nopage}. This
  function is called for every page fault
  and must install the requested page (offset is passed in as a parameter) or
  fail to program be killed. \emph{nopage} hook was changed to \emph{fault}
  with slightly modified prototype in kernel 2.6.23.

  Not to implement \emph{fault} or \emph{nopage} handler for each mapped
  space, helpers exist. For remapping I/O space, \emph{io\_remap\_page\_range}
  was used in some variants, because it changed prototype time by time, then
  \emph{io\_remap\_pfn\_range} was implemented and it is still in use. While
  for remapping kernel memory obtained by page allocation mechanism (i.e.
  physically contiguous memory), \emph{remap\_pfn\_range} may be used, a
  helper called \emph{remap\_vmalloc\_range} for virtual memory exists even
  since kernel 2.6.18 and it remaps exclusively \emph{vmalloc\_*user} buffers.
  We'll employ
  all these mentioned interfaces, depending on kernel we compile for, to be
  backward compatible and to have complete functionality (and simpleness for
  later kernels).

  \subsubsection{Memory Mapped I/O}
  Memory map is sometimes inappropriately confused with \emph{memory mapped
  I/O} (MMIO). We are able to access device memory space in kernel by virtual
  address by MMIO. It is achieved by mapping I/O space into kernelspace.
  For example \fpga\ memory space of a \com\ card may be referred
  by a virtual address after being mapped this way.

  \paragraph{MMIO interface} Memory mapped I/O is done by invoking
  \emph{ioremap} or \emph{pci\_iomap} for PCI devices. Then, for the former,
  \emph{read} and \emph{write} are used for device space access. The latter
  interface is
  slower, because it calls \emph{ioremap} and all consequent read and writes
  (functions \emph{ioread}, \emph{iowrite}) are done through calling back
  \emph{read} and \emph{write} or \emph{in} and \emph{out} depending on what
  kind is the underlying mapping of. In our case we know, it is mapped memory,
  so that we use \emph{ioremap}/\emph{iounmap} and \emph{read} and
  \emph{write} functions.

  We refer to \cite{ldd}, chapter~9 and 15 for further reading about accessing
  device I/O and memory mapping.

  \section{Character Device}
  \label{chap.chardev}
  While we are able to pass data from device to memory available only to
  kernel via DMA so far, we still need to put the data to userspace
  application somehow. Several ways how to achieve that exist in Linux. Most
  of them are based on moving data through special files such as sockets,
  block and character devices.

  The basic and simplest idea is to \textbf{copy} the data by a processor
  from kernel buffer to an user one. It is easy to implement and least
  error-prone, on the other hand it is one of the slowest known methods and as
  we need to read high bandwidth data in userspace, it is unusable in our case.
  Memory map suits in such situation, this eliminates utilization of both
  sockets and
  block devices, since neither of them can be mapped, furthermore socket data
  flow through networking stack. The solely remaining are character devices,
  introduction to char subsystem internals follows further.

  \subsection{Device Nodes}
  \label{chap.devnodes}
  As an entry point from userspace to kernelspace serve \emph{device nodes}.
  The existing node in \emph{/dev} might be opened and used by userspace
  application, demanded operations are handled by the driver which has
  registered it before. It depends on what features are implemented in the
  driver and what operations are defined as an interface. Handlers for
  szedata2 will be discussed in section~\ref{chap.code} after review of
  operations done in specification.

  If the driver is loaded and requested node does not exist yet, it must be
  created to allow access to it. It used to be created manually by userspace
  program \emph{mknod} or wrapper scripts like \emph{MAKEDEV}, however this
  was not perfect and mostly causes problems, because of the need of every
  single node creation and so some dynamic principles described later in this
  section were introduced.

  \subsection{udev}
  \label{chap.udev}
  Userspace device (\emph{udev}) is a dynamic device node creation mechanism
  with support of running external programs depending on defined configuration
  rules. It's a successor of \emph{devfs}, a filesystem implemented directly
  in kernel and mounted on \emph{/dev}. Udev mounts only a temporary
  filesystem (tmpfs) on the same directory, but from its name, it does all its
  work in userspace by listening on a socket for userspace events
  (\emph{uevents}) from kernel and invoking \emph{mknod},
  \emph{modprobe}\footnote{kernel module insertion program} and other
  programs.

  For instance following example shows a node creation in \emph{/dev/combosix/}
  for \com\ device matched by IDs read from PCI configuration space which were
  sent by kernel in uevent. \vspace{0.5em}
\begin{verbatim}
KERNEL=="combosix*", ATTRS{idVendor}=="18ec" \
    ATTRS{idProduct}=="c058", NAME="combosix/%n"
\end{verbatim}
\vspace{0.5em}

  As device node names are random so far -- it depends on kernel PCI cards
  enumeration exclusively (and it may vary) -- and we want numbers in names to
  correspond with each other, e.g. to have \emph{combosix6} and
  \emph{szedataII6} for one
  concrete card in PCI slot 6, we need to write similar rules to generate
  persistent and consistent naming.

  Two separate files are needed to achieve that, another one with persistent
  rules will be created on the fly. Shell script augmenting both the persistent
  rules and a rules file for matching the \com\ cards which runs the
  script are described in detail in Appendix~\ref{app.udev}.

  \subsection{Character Device Linux Interface}
  First step to support character devices by a driver is to allocate a
  region of devices. This was done by \emph{register\_chrdev\_region} in the
  past since regions were statically assigned among certain drivers, nowadays
  is employed a variant for dynamic registration \emph{register\_chrdev} which
  registers 256 nodes immediately or \emph{alloc\_chrdev\_region} which
  registers none, but keeps a track of a region being virtually used. These
  two give a registered region descriptor as a return value and it is then
  passed to \emph{unregister\_chrdev} in cleanup path.

  Each character device has its own file operations structure with hooks to be
  called on each particular event (\emph{open}, \emph{close},
  \emph{mmap}\ldots). Entries left blank (except \emph{open} and \emph{close})
  means that this concrete function is unsupported.
  While we pass this structure directly to \emph{register\_chrdev},
  \emph{alloc\_chrdev\_region} needs further attention, each incoming hardware
  device must be added by \emph{cdev\_add} with operations as second
  parameter. \emph{cdev\_del} is used for removal.

  Although that is everything what is needed from Linux kernel view, this is
  not
  enough to support \emph{udev} node creation. Each driver creates a class by
  \emph{class\_create}, frees up by \emph{class\_destroy}. Then each device is
  created by \emph{device\_create} and after this call \emph{udev} receives a
  signal to create a node. Node is deleted after \emph{device\_destroy}. Both
  these functions are created in class namespace, hence the class must be
  unconditionally created before and freed after \emph{device\_*} calls.
  This API evolved even in 2.6 Linux kernel and is probably most unstable one
  in a scope of this thesis. Many function variants appeared, many changed
  prototypes and then disappeared during 2.6. Functions like
  \emph{class\_simple\_create}, \emph{class\_device\_create} and others will
  be mentioned in back-porting chapter~\ref{chap.backport}.

  We choose \emph{alloc\_chrdev\_region} method, since it is recommended and
  \emph{device\_create} as well. Further
  character device resources are available in \cite{ldd}, chapter~3 and in
  \cite{ulk}.

  \subsection{szedata2}
  Szedata2 is one of designs, the abbreviation stands for \emph{straight
  zero copy data} in the meaning of avoid copying data actively by processor
  from memory to the application and vice versa. The name was chosen after its
  predecessor szedata version 1.

  \subsubsection{Design Specification}
  Both szedata versions run in \emph{FPGA} on \ncope\ architecture.
  Version 1 utilizes \emph{PowerPC} as a DMA controller -- program running
  on that processor used to transfer data by scatter-gather DMA method
  (described in \cite{ldd}, chapter~15) over memory block pool.
  In the first phase, the data transferred at one turn contained only one
  packet of constant length. While \emph{PowerPC} was working with higher
  rates in this mode, we found out, that it is a bottleneck of the whole
  architecture, its control bus was too slow to achieve higher transfer rates.

  We tried to improve this behaviour by aggregating more data packets into one
  transfer block and we redefined data structure too -- data are now
  transferred in a packet of variable length with modified header in
  comparison to the older one.
  We had to redesign also the program running on the card a bit and define the
  packet structure in the specification.
  
  Packet header includes two sizes: segment size and header size. The former
  is size of whole segment, i.e. size of the header, padding and data in sum,
  while the latter is size of optional part of segment containing some control
  information. These control data are placed directly after the header,
  optional padding follows and finally the packet contents (payload) with
  start aligned to 8-byte boundary (thanks to padding). Whole structure is
  padded at the and again to 8-byte boundary. We refer to this as
  a~\emph{packet of variable length}, its format is shown in
  Figure~\ref{fig.netcope_pkt}.

  \begin{figure}[hbt]
    \begin{center}
      \includegraphics[width=10cm]{images/packet}
      \caption{szedata variable length packet}
      \label{fig.netcope_pkt}
    \end{center}
  \end{figure}

  Despite the improvements, this was still unsuitable due to grown complexity
  and still the transfers couldn't be performed at estimated (theoretically
  computed) speeds. We decided to redesign whole design architecture and fork
  \emph{szedata} project into the version~2. \emph{PowerPC} will be no longer
  used for DMA transfers,
  it'll be fully disconnected. Four separate DMA controllers for each
  interface will be implemented directly in the design instead. By that step
  we eliminate the impact of slow control bus, but we still have problems
  with packet transfers, the aggregation is too complex.

  We performed some analyses what could be suitable underlying medium for data
  transfers. Some graphics adapters, several networking (including wireless)
  and video capture cards use a ring buffer structure with success. We decided
  to utilize it too.

  First, we needed to create a design specification. Let's consider 10\,Gbit/s
  data flow and userspace task latency of 50\,ms. We need a buffer of size
  at least $\frac{10\cdot10^{3}\cdot0,05}{8}\doteq60$\,MiB per interface not
  to throw incoming data away. But as we noted in chapter~\ref{chap.mm}, we
  cannot allocate that much contiguous physical memory. This made us to split
  the buffer on the page basis and create a mechanism to cope with this.

  The idea is to allocate yet another memory for descriptor page with entries
  pointing to the pages of the interspersed ring buffer. We took in account
  64-bit systems, so that the pointers are represented by 8 bytes each. Since
  we allocate the descriptor memory on a page basis too, it allows us to have
  only $\frac{4096}{8}=512$ pointers to ring pages which gives
  $512\cdot4096=2$\,MiB sized ring buffer. This is not enough, so we
  introduced two kinds of descriptor entries. One is still a direct pointer to
  a single ring buffer page, the other one is a pointer to the next descriptor
  table. To distinguish them, we take to advantage of all page addresses being
  zeroed in low 12~bits (see \cite{gorman}, chapter~3) by using these few bits
  for flags. Pointers to the next descriptor table read by the hardware are
  logically or-ed with~1. We call these pointers \emph{type 1} in
  Figure~\ref{fig.ring}, ring buffer page pointers are denoted as
  \emph{type 0}, and for illustration we also connect ring buffer pages with
  dashed \emph{virtual link}.

  \begin{figure}[htb]
    \begin{center}
      \includegraphics[width=9cm]{images/ring_struct}
      \caption{szedata2 ring buffer structure}
      \label{fig.ring}
    \end{center}
  \end{figure}

  \paragraph{Registers and address space}
  The last thing we need to cover before implementation is how we declare
  communication channel between driver and newly specified design. For the
  convenience of MMIO, we define design address space as a memory area. Every
  DMA engine has its own subspace with its controlling registers within this
  area. Up to 16 engines are supported for future purposes, we use only 2 so
  far. Each space consist of these register groups:

  \begin{enumerate}
    \item pointer to the first page with descriptors
    \item start, end and size registers -- ring buffer state pointers, they
      are current head and tail and size of whole buffer
    \item control and status registers -- used to instruct the design and check
      actual state
    \item interrupt registers -- timeout, position or both may be used for
      interrupt generation; position is triggered when one of ring buffer
      registers reaches value stored in position register
  \end{enumerate}

  As the DMA engine needs to have information about ring buffer before it
  starts working, registers groups 1. and 2. shall be set prior to sending
  \emph{start} command to control register 3. Interrupt registers 4. are
  optional, they may be set and changed on the fly. Concrete steps of hardware
  initialization and deinitialization will be described in the driver
  specification in chapter~\ref{chap.spec}.

  After successful launch of the engine, we just need to update tail
  pointers for reception after we process data and head for transmission
  after we write data to the ring buffer. Any other processing is implemented
  in the engine and we needn't to care about it.

  \section{Requirements}
  \label{chap.requirements}
  We decided to use some methods yet in previous chapters, here we
  summarize them and add some other requirements we must
  ensure our implementation will satisfy:
  \begin{itemize}
    \setlength{\itemsep}{-4pt}
    \item multiple ring buffer support (per interface and direction)
    \item correct ring buffers operations (transmit head must not exceed tail)
    \item ring buffers and their descriptors setup in hardware
    \item hardware setup (another initialization, start, stop)
    \item export/remap data through memory map
    \item some controlling mechanism to communicate with application
    \item multiple applications support, multiple access to the data
    \item speed
    \item as highest as possible code correctness (successful testing)
    \item documentation (the more well\footnote{Avoid over-commenting, well
      comments are good, over-commented core is worse than non-commented, see
      chapter~8 of Documentation/CodingStyle document from kernel sources.}
      commented code the better)
    \item abstraction (userspace library would be good)
    \item last but not the least: clean code (fulfil kernel coding style)
  \end{itemize}

  The sole thing needs discussion is how and when we should allocate space
  for a ring buffer. Since the impact to the system might be significant due
  to large amount of memory requirements, we must consider following two
  variants. Either we can allocate the space right after the driver loads or
  alternatively even after first application opens the corresponding character
  device node.

  The former variant assures fast application startup, but we might risk that
  we'll never use the allocated memory just because no user opens the node.
  Since the driver is about to be used in environments where the application
  is expected to be executed if the driver was loaded yet, we proceed with
  this choice by default. Otherwise we would stall on every application start
  end exit potentially exposed to start-up failures due to momental memory
  exhaustion. In spite of all the threats, somebody might want the latter
  behaviour, so we'll let user to choose by adding an option for this.

  \section{Models And Design}
  \label{chap.spec}
  As we mentioned earlier, we use UML for modelling. Many UML tools are
  available for Linux, we chose one named \emph{Bouml}\footnote{
  Bouml is a free (under the terms of GPL) multiplatform project with support
  of UML including version 2. See its homepage at \url{http://bouml.free.fr/}
  for more info.}. It can do most of UML
  defined diagrams, however we'll use just \emph{use case} and
  \emph{sequence} diagrams. Complete diagrams and their documentation are in
  Appendix~\ref{app.models}, this chapter introduces the way how we got
  those diagrams as a result.

  \subsection{Use Case Diagram}
  Starting point is an use case diagram. We create only one use case module
  which will correspond to our driver and so we name it the same -- szedata2.
  Actors choice is simple, we have an \emph{OS}, \emph{hardware} and an
  \emph{application}. OS just \emph{inserts} and \emph{removes} the driver to
  and from the kernel and furthermore since there are no asynchronous actions
  doable by the hardware except optional interrupts, the sole use case
  regarding hardware is \emph{interrupt}.

  Insert and remove takes care of initialization, the module may allocate and
  also free ring buffers depending on allocate-on-modprobe option in these two
  use cases. HW interrupt use case only wakes up applications waiting for
  this event.

  Application is more complicated actor, it opens (and initializes) the
  device, process data (and communicate) and finally closes (and stops) it.
  All the operations must be atomic due to concurrent access, this is not
  mentioned neither in the specification nor further, because we take this for
  granted. Fine grained application use cases description follows.

  \subsubsection{Open And Initialization} We already know the order of
  operations
  that shall be done before using the device from chapter~\ref{chap.netcope}.
  We create one use case as a starting point and place following steps in it:
  \begin{enumerate}
    \item \emph{open} system call $\rightarrow$ initialize buffers if it was
      the first open (not to break yet started hardware), we use reference
      counter to assure only one start of the device
    \item \emph{mmap} system call $\rightarrow$ memory map buffers
    \item \emph{select/subscribe} command $\rightarrow$ select interfaces
      which are about to be used
    \item \emph{start} command $\rightarrow$ initialize and start selected
      hardware (allow exclusively one start per interface throughout all
      applications)
  \end{enumerate}
  Even though \emph{open} and \emph{mmap} are different calls from the driver
  view, application sees them as one basic operation, so we leave them in one
  use case. On the other hand, we find out later, that we want to allow the
  application to select and start interfaces on the fly, so we must place
  points 3 and 4 in separate use cases which gives three initialization use
  cases in the end.

  \subsubsection{Data Processing} Design supports full-duplex communication, so
  we split processing further in two categories -- receive (RX) and
  transmit (TX) and describe them separately.

  \paragraph{RX} Since hardware is started and it does RX processing fully
  standalone, we only check buffer tail pointer for changes in loop.
  This means calling kernel and getting back area with data (or no data if
  no change). After finished processing on those data, we free them
  up to tell the driver to move the tail. This gives us two separate use cases
  \emph{get\_data} and \emph{release\_data}, since the processing is in charge
  of application.

  After we gained access to the data, two receive processing methods raised
  altogether:
  \begin{itemize}
    \item \textbf{load balancing} -- several readers exist and data are
      divided among them. Every single application gets distinct data and do
      the same operation on them. It is very similar to \emph{single program
      multiple data}\footnote{see \url{http://en.wikipedia.org/wiki/SPMD} for
      reference} in processors theory.
    \item \textbf{multiprocessing} -- all readers want all data to do
      different tasks with them. We can approach this to \emph{multiple
      program single data}.
  \end{itemize}
  We implement only the latter so far, but it is possible that we augment the
  implementation to the former too, see future work in
  chapter~\ref{chap.future}.

  \paragraph{TX} Outbound direction is similar in data processing --
  application asks for area to write to (gets current transmit head), writes
  data and issues command to move the head. It corresponds to the two created
  use cases \emph{request\_space} and \emph{put\_data}. It differs in access
  to the buffer, because all applications shares one TX
  ring buffer and hence only one may access it at a time not to create holes
  of unused space\footnote{There is a plan to extend this behaviour to allow
  multiple writers too, however it has not been finished yet.}. We cope with
  this by setting up proper prerequisites in the use case descriptions.

  \subparagraph{Security Note} Note that each application can access data
  of any other writer caused by single shared ring buffer per interface.
  This is known issue but considered non-risky because of deployment area of
  the hardware. We expect correct permissions on the device node and so allow
  access only to authorized programs.

  Anyway, we consider implementing multiple TX ring buffers (per application)
  to avoid any potential security violation. In this case, we decide
  whether this will be implemented by the design or in the driver. The
  latter would mean data copying and hence slowdown. More investigation of how
  much speed impact will this have should be performed but it is not trivial
  and also out of scope of this thesis.

  \subsubsection{Close And Deinitialization} After open reference count drops
  to zero we do steps from initialization in reverse order. \emph{close\_node}
  use case consist of \emph{hardware stop} command, \emph{unmap} and
  allocated memory cleanup. We might alternatively consider placing
  \emph{hardware stop} command in separate use case to allow stopping out of
  line, anyway we must turn off the hardware on last close if the use case
  wasn't invoked before.

  \subsection{Sequence Diagrams}
  Sequence diagrams are modeled to show up how events happen during time. The
  topmost event is executed first and then all events below it towards the
  bottom one. We create two, depicting RX and TX, they will slightly
  differ in how application locks and unlocks data.

  \paragraph{RX sequence diagram} We specify better application code flow
  here. We establish three lifelines, \textbf{application}, \textbf{module}
  and \textbf{hardware} which obviously match use case actors.

  The work starts by \emph{open} message from \textbf{application} to
  \textbf{module}. The diagram copies actions described in use case
  initialization section above, so it nonsense to repeat the steps again. We
  just augment the use case specification further by splitting single use case
  steps into multiple messages here. For instance \emph{mmap} from use case is
  here as two messages: \emph{mmap status page} and \emph{mmap RX space}.
  That is because we try to be as much exact as possible to directly turn these
  information into the real code in the next chapter.

  \paragraph{TX sequence diagram} We reflect difference of RX and TX showed
  before in use case modeling here by not conditioning head move on
  \emph{put\_data} message and we also change all RX occurrences to TX and
  invoke proper messages, indeed. There is no other difference to consider
  from model view.

  \subsection{Combo6 Code Base}
  \label{chap.c6base}
  We have evolved base common to all Liberouter projects not to reinvent the
  wheel for each design. These base drivers are responsible for whole card
  initialization, an interrupt hook setup, loading firmware/design,
  controlling combosix device node which allows access to all spaces of the
  card and few other tasks. Since we have two types of \com\ cards, two
  card-specific drivers exist: \emph{combo6} and \emph{combo6x}; they
  are used by the \emph{combo6core} module consisting of common parts of
  both. Since this module provides encapsulation, design drivers needn't care
  about which particular card driver is driving the device, it uses
  \emph{combo6core} interface to call the proper one, so
  that the access is fully transparent.

  One more convenience is exported info in \emph{sysfs}\footnote{sysfs on
  Wikipedia: \url{http://en.wikipedia.org/wiki/Sysfs}} and
  \emph{procfs}\footnote{procfs on Wikipedia:
  \url{http://en.wikipedia.org/wiki/Procfs}} filesystems provided by the
  central driver. This info may be further used by application to recognize
  module version or tune some values and parameters.

  Design driver should define \emph{combo6\_device\_ops} structure with name,
  card type, IDs which it supports and hooks for attach, detach and interrupt
  and pass it to \emph{combo6\_device\_register} to register it by the core
  module. Deregistration is then invoked by \emph{combo6\_device\_unregister}
  call. Our structure looks like this code:
\begin{lstlisting}[language=C,label=lst.c6dops,name=lst.c6dops]
static struct combo6_device_ops szedata2x_ops = {
	.owner = THIS_MODULE,
	.name = "szedata2", /* name of this driver */
	.dhw = DHW_COMBO6X, /* combo6x card only */ (*@\label{lstl.c6dops_4}@*)
	.exclusive = 1, /* only one attached driver */
	.intfs = 2, /* limit interfaces */
	.id_lsw	= 0x41c10300, /* from id */
	.id_hsw = 0x41c103ff, /* to id */
	.id_text = "NIC_NetCOPE", /* design name */
	.attach	= szedata2_hw_attach,
	.detach	= szedata2_hw_detach,
	.interrupt = szedata2_hw_interrupt,
};
\end{lstlisting}
  Let's explain values in some entries:
  \begin{itemize}
    \item \emph{owner} is set to not allow used modules removal, core driver
      adds 1 to reference counter of the module and kernel forbids the
      module removal
    \item \emph{intfs} are limited to 2 of total 4, since the testing design
      supports only 2 interfaces so far, whereas the hardware reports 4
      present, so we would end up in initialization of non-existent hardware
    \item \emph{id\_*} entries are taken directly from the \sze\ design,
      0x41c1 is \ncope\ project, 0x03 is major version (0x02 was szedata1
      design) and finally 0x00 is a minor. As we want to bind this driver to
      all \sze\ designs, we grab all its potential minors up to 0xff.
  \end{itemize}
  We can see, that the structure also contains designator of combo6x card on
  the line~\ref{lstl.c6dops_4}.
  We must set that entry and that means creating two such structures with the
  same contents but \emph{dhw} to support both \com\ and \comx\ cards.

  \subsection{Implementation}
  We used term \emph{command} in the specification many times, but we haven't
  explained command passing mechanism. Although \emph{ioctl} system call was
  uttered as deprecated long time ago, nothing better was introduced so far,
  its kernel interface has been improved and it is still widely used even in
  new projects. Hence we'll use it for commands passing too.

  We split the implementation into two separate files. One is for
  non-hardware dependent part (\emph{szedata2.c} and \emph{szedata2.h}) and
  one with the rest depending on the hardware (\emph{hw.c} and \emph{hw.h}).
  The former will consist of ring buffer computations, initialization and DMA
  allocations and also character device node operations (incl. \emph{ioctls}
  and \emph{mmap}). The latter will contain the structure from previous
  section and registration to the \emph{combo6} base and all ring buffer
  operations done on the hardware (get and set head/tail pointers, ring size
  and descriptors setting, etc.) and last but not the least invocation of DMA
  allocation routines from the HW independent file.

  We can conveniently use this split
  for testing and early development, because we hadn't working design while
  writing the driver. We'll create dummy \emph{hw.c} with few timers and fake
  ring operations and develop szedata2 part as specified and test it by
  that dummy module. We'll get properly working and tested driver by evoking
  corner cases by this technique. More of this is in the testing
  chapter~\ref{chap.testI}.

  \subsubsection{Structures}
  As a first thing let's settle down data structures passed through the
  system. Many of them raised through the modeling and module specification,
  we summarize them here.

  Since 64-bit operations are expensive and non-atomic on 32-bit machines,
  we try to minimize 64-bit computations as much as possible. In fact we need
  only an mmap offset and size to be 64-bit to allow spaces be bigger in sum
  than 4\,GiB on 64-bit or 32-bit PAE\footnote{Physical Address Extension, it
  extends paging mechanism by one table so that it is possible to
  address 36-bits of physical address.} enabled machines. See following
  structure descriptions and notes about 64-bit needs.

  \paragraph{Application-driver interface}
  We decided to pass all read-only information in status page which is
  allocated per application on its first open. It is depicted on the top left
  in Figure~\ref{fig.status_page}.

  \begin{figure}[htb]
    \begin{center}
      \includegraphics[width=12cm]{images/status_page}
      \caption{\emph{mmap} status page structure}
      \label{fig.status_page}
    \end{center}
  \end{figure}

  First, from the specification, status page shall contain information about
  spaces to be mmaped. According to the manual page of mmap and Linux kernel
  requirements, only two parameters must be known -- offset and size. As
  mentioned earlier, offset is 64-bit value and so the size is. Eleven
  (indexed by 0 through 10) such pairs might be stored in the page, however we
  use only two\footnote{Note that it is not space wasting, page size is usually
  4\,KiB and it is enough space to reserve some entries for future purposes.},
  one for RX space, one for TX. These offsets point to so-called \emph{spaces}
  (on the right side) which might consist (exclusively ring buffer spaces do)
  of \emph{areas}. \emph{Area} is directly the ring buffer corresponding to
  one particular hardware interface in the \com\ hardware case, however we do
  not call it interface ring buffer, because area-interface one-to-one mapping
  needn't hold if we recall, that the part which assures the mapping is
  hardware independent and might be used for any hardware.

  The application then remaps all desired spaces indexed by \emph{SZE2\_MMIO}
  constants and may poll for their status in \emph{area descriptors} (another
  status page entries). It is a structure with 64-bit offset and 32-bit size
  pointing directly to the area and flags. Flags denote if area/ring buffer
  wrap occurred and where is the next \emph{area descriptor} for the wrapped
  data.

  There are also sanity checks (magic numbers), version number and set values
  for e.g. poll in the status page, it is not so important so we let reader to
  investigate exact descriptor structure. Status page is named as \emph{struct
  sze2\_instance\_info} in the source code.

  To not poll every area we use commands to get a number of the area ready to
  be read or written to. Four \emph{ioctls} corresponding to use cases
  exist for this purpose. RX lock and unlock are easy to define. We just ask
  for areas we want (bitmap in one parameter) and unlock all areas without
  a parameter. TX direction needs complicated passing of parameters. Beside
  the area we are interested in, we need to pass a size too. We define
  \emph{struct sze2\_tx\_lock} containing these two entries, is the same but
  the name -- \emph{struct sze2\_tx\_unlock}. All these sizes are sufficient
  to be 32-bit, since we don't expect somebody to read or write amounts larger
  than 4\,GiB.

  The remaining is the area subscription done by \emph{ioctl} too. It contains
  array of two members (RX and TX) with info about areas to subscribe and poll
  threshold (minimal ready-to-process size resuming the application) used for
  waiting in the driver via \emph{poll} system call.
  Documentation in Appendix~\ref{app.doxygen} deals with this interface.

  \paragraph{Internal interface} We've declared already, that we will split
  the driver into two parts.
  Interface between them is straightforward. Interrupt, tail and head pointers
  passing is needed in one direction, registration and DMA allocation in the
  other. This interface is described in the documentation contained in
  Appendix~\ref{app.kdoc}.

  \subsection{Phase I -- Hardware Independent Code}
  As we don't have fully working design yet, we start with design
  independent part. First we write module \emph{init} and \emph{exit}
  functions. We create a \emph{class} and reserve character device region
  (with at most \emph{SZEDATA2\_MAX\_MINORS}\footnote{currently the value is
  8, but it may be altered on compile time, since it is a macro}) as described
  in chapter~\ref{chap.devnodes} and store a number which we got from it to
  use later in \emph{exit} function when unregistering character device.

  We see, that we don't allocate any buffers or any other data, so we must
  export hooks for events when a device comes and leaves, even then we can
  create and destroy any structures. These operations are yielded to the
  hardware specific part, because it knows how many areas are needed and how
  large ring buffers should be allocated.

  Registration of incoming device is split into two parts -- allocation
  (\emph{alloc\_szedata2}) and registration (\emph{szedata2\_register})
  itself. After calling the former, callbacks shall be set up and eventually
  DMA allocations must be done now (or alternatively even on the first
  open). If we didn't the split, we might end up with kernel panic, because
  concrete character device is set in the register call and some so far zeroed
  callback may be invoked already. \emph{szedata2\_destroy} is created for
  cleanup purposes. We must take in account situations when latter register
  function hasn't been called at all. This is achieved by setting one bit in
  the register function and test for this bit in the destroy one.

  DMA allocation is performed via PCI interface described in
  chapter~\ref{chap.dma}. Allocation is done per page basis (recall
  chapter~\ref{chap.netcope}), so if we consider four interfaces, both RX and
  TX directions and 50\,MiB sized ring buffers, we have
  $\frac{4\cdot2\cdot50\cdot2^{20}}{4096}\doteq100000$ pages for 4096\,B pages.
  We need to store physical and virtual addresses for each page (to be able to
  work with the buffer and also free it later), worst case is on 64-bit
  machines, where both numbers are 8-bytes, so we further have
  $100000\cdot2\cdot8=1600000$. This information might be stored in
  $\frac{1600000}{4096}\doteq390$ pages. That is a number very close to
  $2^{MAX\_ORDER}$\footnote{\emph{MAX\_ORDER} is maximum allowed power of 2
  to allocate via \emph{get\_free\_page}, it used to be between 5 and 20
  depending on architecture and configuration}, so we would end up in
  failure if we didn't use virtual memory (\emph{vmalloc}) for those
  descriptors.

  Even if we're allocating ring buffer page by page, we might face
  failures. It is because of how allocations work in memory management
  subsystem. For implementation in Linux 2.4 see~\cite{gorman}, 2.6 differs
  slightly, but it is the same in principle. Most of inactive pages is freed
  even under high pressure, when priority is high. The priority is raised by
  each failure and the higher the priority is the more pages are tried to free
  (also the used ones might be swapped out if they belong to userspace). This
  means we should try to allocate failing pages at least several times
  to raise the priority with some delay between the tries to let swapper move
  some pages to the swap device. See \emph{szedata2\_alloc\_dmaspace} function
  for final implementation.

  \paragraph{Character Device Operations}
  We passed also \emph{struct file\_operations} when registering concrete
  character device which contains \emph{open}, \emph{release} (close),
  \emph{mmap}, \emph{poll} and \emph{ioctl} hooks. With use of reference
  counting we call HW open and close callbacks only once. The rest of
  \emph{open}, \emph{close} and \emph{poll} (wait for data) is implemented
  exactly as specified in model by recommended techniques from \cite{ldd} and
  later from Linux kernel mailing list.

  \emph{ioctls} need special handling, there are three types of \emph{ioctl}
  handling\footnote{Explanation is part of virtual filesystem documentation in
  Linux kernel tree at Documentation/filesystems/vfs.txt}. New drivers should
  use \emph{unlocked\_ioctl} and so this implementation. We also implement
  \emph{compat\_ioctl} for 32-bit emulation on 64-bit, however we only
  call \emph{unlocked\_ioctl} from it, because all \emph{ioctl} structure
  sizes and alignments are compatible. If we didn't implement it, commands
  wouldn't work, because 32-bit emulation ioctl code calls solely this
  function.

  As per specification and requirements in chapter~\ref{chap.requirements},
  commands must be atomic due to multiple application access, hence we use
  spin locks to avoid race conditions. We use spin locks rather than mutexes,
  because they are faster and there is not much code in critical sections to
  be feared of starving caused by long busy waiting loops.

  \emph{mmap} of status page is solved by \emph{remap\_vmalloc\_range} helper
  described in chapter~\ref{chap.mm}, on the other hand for ring buffer
  non-continuos memory must be implemented \emph{fault} method.

  \subsection{Phase I Testing}
  \label{chap.testI}
  As we have completed first increment, we need to do first tests. To check
  this piece of code we need to implement source which will behave as a fake
  hardware.

  Right on module load code we test forged illegal inputs by fault injection
  of \emph{NULL} pointers and error states encoded in a pointer (see
  \emph{ERR\_PTR} macro in \emph{linux/err.h}) passed to register functions.
  Then first available PCI device (of any type) is used as a parent
  parameter to allocation function to be able to allocate DMA space.
  After doing allocation tests, the device is registered along with other more
  than \emph{SZEDATA2\_MAX\_MINORS} to test proper error handling. Also
  another tests as closing used device or destroying opened device are
  presented in the \emph{szetest.c} file. See \emph{szetest\_test\_init}
  function for details.

  After initialization phase, some timers are fired to pretend real hardware
  work. Ring buffers are filled with random-sized packets with all reachable
  corner cases such as wraps of packet over page and whole ring buffer, split
  of packet header over the same boundaries and so on. \emph{szetest\_timer}
  and \emph{szetest\_timer1} implement this logic.

  To be able to utter the driver works, also reader/writer in user space must
  be implemented. \emph{szetest2} application is constructed in similar manner
  like the fake hardware code -- to test as much as possible. Some of these
  userspace tests were ready yet before the implementation itself, because we
  knew from specification how it should behave and how not. Implementation is
  further programmed to satisfy all the tests\footnote{This agile method is
  known as \emph{test driven development} (TDD), for reference see
  \url{http://en.wikipedia.org/wiki/Test-driven_development}.}.

  Tests for all implemented character device hooks are in \emph{szetest2.c}
  see for example \emph{test\_mmap} which tests correctness of \emph{mmap} by
  trying to forge invalid values (longer space, wrong offsets etc.). In
  addition we ought to add tests for commands and reading and writing of "real"
  data from ring buffer or to it respectively. These checks are implemented
  in \emph{read\_data} and \emph{write\_data} in the test application.

  \subsection{Phase II -- Hardware Dependent Code}
  After successful testing of the base, we started construction of hardware
  dependent part. The design was built meanwhile the specification and first
  part implementation and is fully working now. The code we build up here will
  be a bridge between currently finished szedata2 core and \emph{combo6} base.

  To be able to cooperate with the hardware behind the base, the driver needs
  to know constants understood by the hardware. They are described in the
  hardware specification and are transcribed to a header file named
  \emph{hw.h}.

  Then we start building the source file beginning with init and exit
  functions. Decision is so far to create a single module, so both module parts
  will be linked into one loadable object file. It means we can't add another
  \emph{module\_init} and \emph{module\_exit} functions in this part and we
  must solve it by calling those functions from the former code's init and
  exit functions. Their contents is straightforward from
  section~\ref{chap.c6base}, we simply take care of combo6 operations
  registration and in addition do module parameters alignment.

  \emph{block\_size} is aligned to whole page and \emph{block\_count} to
  nearest power of two as ring buffer size is multiple of these two numbers
  and design
  expect buffer size to be power of two\footnote{It's formally correct to
  assume the multiple is power of two, since page size is power of two too and
  $2^a\cdot2^b=2^{a+b}$ from algebra.}.

  Even when the attach function from the operations is called with real
  attached device, we might allocate a szedata2 device, set
  callbacks up and register it (all this is \emph{szedata2\_hw\_attach}). We
  also allocate a space for descriptor memory information storage just by
  passing size of this information to the szedata2 allocation call.

  Everything what \emph{open} and \emph{close} callbacks do is allocation
  and free of ring buffer if it isn't done at init and exit phases (the
  allocate-on-modprobe option from specification). Common functions
  \emph{szedata2\_hw\_alloc/free\_dma} are called from both code paths to not
  copy the code.

  Hardware initialization is done even in \emph{start} callback. We use
  \emph{combo6} core wrappers for 32-bit writes to PCI bus, so that
  64-bit descriptor must be send in two rounds, all other values to send are
  32-bit.
  After posting start to control register, the code inside the command waits
  until status register is either running or idle then the \emph{start}
  callback finishes and returns. It is possible, that hardware is locked up and
  it doesn't become ready in a reasonable time, so we must bound the waiting
  loop by some constant. According to a design specification, 100\,$\mu s$
  should be enough.

  \emph{Stop} callback is implemented in similar manner, except the fact we
  wait one whole second until DMA transfers get completed. When this timeout
  expires, we only report failure to logs and go on, because we don't have any
  chance to tell the kernel, that \emph{close} function (which issues this
  callback) failed.

  Pointer and interrupt callbacks are simple calls of PCI space reads and
  writes, a problem arises when computing sizes of spaces between read head
  and stored tail. When tail equals to head it is impossible to distinguish if
  the buffer is empty or full. Few
  techniques with less or more overhead exist\footnote{See
  \href{http://en.wikipedia.org/wiki/Circular_buffer\#Full_.2F_Empty_Buffer_Distinction}{http://en.wikipedia.org/wiki/Circular\_buffer}
  for description of each method.}. "Always Keep One Byte Open" method is
  implemented in the design, head and tail never equals in both cases, if and
  only if the buffer is empty. This is not a problem for receive pointers in
  the driver, hardware never sets them the same, it will ever use size-1 at
  most. But we must take care of transmit size, because we move with the head
  and we mustn't reach the tail. This is done by this code in
  \emph{szedata2\_head\_tail\_size}:

\begin{verbatim}
if (head == tail)
    return size - 1;
\end{verbatim}

  \subsection{Phase III -- Back-porting To Older Kernels}
  \label{chap.backport}
  So far the implementation has been run and tested exclusively on later
  kernels (2.6.24-rc8-mm1\footnote{release candidate with Andrew Morton's
  patch applied} and newer) and since we want the driver to be used on many
  configurations, we have to support all Linux~2.6 kernels in the best case.
  This is not feasible without a help of a programmer. Several APIs have
  changed during 2.6 kernel development as we mentioned in \emph{Linux
  interface} sections earlier.

  For the convenience of modularism, we needn't to care of IRQ handler or
  MMIO mapping changes, since we caught the changes in the \emph{combo6}
  modules already far before this design was decided to create. We have a
  file named \emph{kocompat.h} where compatibilities are solved by testing for
  kernel versions and using proper interfaces. However we must solve
  \emph{mmap}, \emph{ioctl} and \emph{class} changes for
  all drivers, because changes in them are too new and unresolved.

  While we were porting drivers from one versioning system to the another one
  in the past, we created two trees with drivers. One is \emph{kernel} with
  sourcecodes prepared for inclusion to mainline kernel and the another is
  \emph{build}. This is where we locally build the drivers and where another
  compatibility stuff lays. It has the same hierarchy and file names except
  the files contain only \emph{\#include} line to "load" the contents of the
  real file from \emph{kernel} subtree. We'll use this split for compatibility
  handling, because this issues need to be solved in the \emph{build} tree
  only.

  \paragraph{Mmap} After some investigation\footnote{The most powerful tools
  for such kind of research are \emph{cscope} to see, where the symbol is
  defined, \emph{git-blame} to figure out when and by what commit was each
  particular line (and function) changed and \emph{git-show} to see the patch
  which does the change. Finally \emph{git-describe} on the particular commit
  gives directly a kernel version the patch was committed after.}, we found
  out, that \emph{.fault} handler was introduced in kernel 2.6.23 and
  previously used \emph{.nopage} was removed along
  with that change. Also 2.6.18 was the first kernel, where is
  \emph{remap\_vmalloc\_range} available for public use. This gives three
  variants:
  \begin{CompactItemize}
    \item \textbf{before 2.6.18} -- we implement \emph{.nopage} handler for
      each space
    \item \textbf{between 2.6.18 and 2.6.23} -- only non-contiguos spaces
      need \emph{.nopage}, i.e. ring buffer; \emph{status\_page} is mapped by
      helper
    \item \textbf{after 2.6.23} -- ring buffer is switched to use \emph{.fault}
  \end{CompactItemize}

  Code for older variants is obviously located only in the \emph{build} tree
  in the \emph{szecore2.c} file with function prototypes before \emph{\#include}
  line and implementation after it. Unfortunately \emph{.fault} code in
  \emph{kernel} tree must be placed in conditional section too, but the test
  are easy to remove right before posting to mainline.

  \paragraph{Ioctl} \emph{unlocked\_ioctl} and \emph{compat\_ioctl} were
  introduced even in 2.6.11 kernel. Before that, merely \emph{ioctl} existed.
  We behave similarly as for \emph{compat\_ioctl}, simply call unlocked variant
  with proper parameters. It is possible thanks to difference being just in
  function prototype. Again this handler is present only in the \emph{build}
  tree.

  \paragraph{Class} With refer back to section~\ref{chap.chardev}, we already
  know this is most variable API in 2.6. Let's summarize what happened during
  2.6 development in following points:
  \begin{CompactItemize}
    \item \textbf{between 2.6.2 and 2.6.13} -- \emph{sysfs} filesystem
      created and
      \emph{class\_simple} defined in 2.6.2; \emph{class\_simple\_create}
      and \emph{class\_simple\_device\_add} are used (with their destroy
      opposites) to operate with classes and devices
    \item \textbf{between 2.6.13 and 2.6.15} -- \emph{class\_simple}
      disappeared with all its functions, \emph{class} structure which was not
      "simple" enough before is simplified and used instead.
      \emph{class\_create} and \emph{class\_device\_create} are functions of
      our interest in these two kernel releases.
    \item \textbf{between 2.6.15 and 2.6.18} -- \emph{class\_device\_create}
      changed prototype, it may have two parents reference in it (one
      \emph{device} and one \emph{class\_device}).
    \item \textbf{after 2.6.18} -- \emph{device\_create} should be used
      instead of \emph{class\_device\_create}. The older
      \emph{class\_device\_create}
      is going to be removed soon with \emph{class\_device} which is
      considered deprecated nowadays.
  \end{CompactItemize}

  We handle this incompatibilities in \emph{kocompat.h} header by testing
  which kernel release we are building against and providing correct function
  calls and structure definitions.

  \subsection{Phases II And III Testing}
  In fact, there isn't much new to test by our testing techniques thanks to
  test being ready before the driver programming. Only tests showed in phase~I
  must pass again both after phase~II completion and now. The tests should
  show no regressions on every last kernel we have available for phase~III
  tests though. However we might also use another techniques to catch bugs.

  Next steps usually lead to a static analysis of C language. Over hundred both
  commercial and free tools for this purpose exist\footnote{Non-complete list
  of checkers is on Wikipedia at \\
  \url{http://en.wikipedia.org/wiki/List_of_tools_for_static_code_analysis}.},
  one of the tools, \emph{sparse}, was designed directly for Linux kernel
  driver development needs. The checking is included in kernel makefile system
  also, \emph{make C=1} will invoke
  \emph{sparse} checker ever before each file compilation. The output
  is printed on standard error like usual compiler warnings and errors. We
  caught many signedness, cast and I/O operation bugs thanks to it.

  Linux kernel itself provides many options for development under \emph{Kernel
  hacking}
  configuration submenu. Most these configurations slow down kernel operation
  rapidly, but they still may be handful while seeking for deadlocks or
  missing lists initialization for example. All these options enabled allowed
  us to catch some section mismatches (problems when one function may call
  another from distinct, potentially already freed section and thus kernel
  crash may occur), unchecked return values of functions which must be checked
  and also it helped us to prove locking correctness.

  Sometimes we hit a design bug, because it is often very hard to emulate high
  traffic in simulations performed by hardware developers. This bug is in such
  case forwarded to a concrete hardware person via \emph{trac} system. After
  they solve the issue, we re-test and possibly change code if it is
  misunderstanding of specification details. We cooperated this way when we
  discovered a bug in stop hardware path both in the design and the driver for
  instance.

  \subsection{Userspace Library}
  Even though there is no library for szedata v.~1, we consider to have one
  for \sze. The main point to have it is to not propagate the kernel
  interface directly to the user, to simplify the interface (get rid of status
  page) and make it abstract.

  We create a directory for library in versioning system in software package,
  integrate with existing build system and create a file with where the
  library heart is going to be. The function bodies are exact copy of use
  cases from the specification, since the use case alone means exactly that it
  is a separate sequence of commands not being able to split further. If we
  carry over each use case contents to a library call, we reach the two
  aims from previous paragraph. Above all, if we create a shared library,
  we'll get easy to update part (module) of each application.

  If we look closer at the use cases, we'll see, that most of the library code
  may be extracted from \emph{szetest2.c} testing code thanks to its
  construction -- we though of creating a library from it later, so we wrote
  the code modularly. We just take away testing stuff and place atomic (from
  use case view) commands in one function call if not done already. Then this
  interface is exposed by library header file \emph{libsze2.h} and compiled
  and linked into \emph{libsze2.so} for further usage.

  \paragraph{Testing} Since we changed \emph{szetest2} only internally, the
  tests in
  it must work even now except the processing is delegated to the library.
  Its output should not change anyhow and it is checked in that manner.

  Also userspace application may be checked by some static code analyser. Even
  \emph{sparse} may be used for that purpose, but also
  \emph{lint}\footnote{\url{http://en.wikipedia.org/wiki/Lint_programming_tool}}
  successors such as
  \emph{splint}\footnote{\url{http://en.wikipedia.org/wiki/Splint_(programming_tool)}}.
  This helped us to fix some mostly coding style and non-intrusive errors.

  \section{Maintenance}
  Drivers for all designs and card types are deployed on each machine and
  kernel version after successful tests performed by testers. This must have
  been done for a first time due to conversion from one versioning system to
  another and dropping support of kernels older than 2.6. Meanwhile we have
  been creating this implementation other drivers were moved to the new
  system with many intrusive changes and it broke compatibility between the
  two versions from both systems, so they must be tested and distributed at
  once, now including \sze.

  Userspace library is not kernel dependent, so it needn't be necessarily
  tested on all kernels, however because of testing program dependency on this
  library, it is tested too. Deployment is \emph{strip}\footnote{Binaries
  are usually freed of unnecessary debugging sections and symbols not needed
  for running on UNIX systems by \emph{strip} utility (see \emph{man strip}).}
  followed by a copy to a directory shared between Liberouter machines.

  Thanks to modularity, further maintenance and updating are easy to realize.
  If we achieve the state, when all the drivers are from the new versioning
  system, changing one single module is conceivable by simply removing it and
  placing in the same directory an updated version either with fixes or some
  new features. After successful tests indeed.

\chapter{Conclusion And Future Work}
  \section{Conclusion}
  This thesis aimed at data transfers between network and an application in
  both directions and tried to find out a way how to do such transfers as fast
  as possible. As a
  first step we introduced hardware basics and limitations in fast transfer
  mechanisms such as DMA on PCI bus and then walk through possible data
  structures eligible rapid transmissions and implementation of ring buffer
  data structure on \com\ cards.

  Also several Linux interfaces for fast operations on the ring buffer were
  described and we decided to use DMA and memory map which gave us as a
  result zero CPU time consumed by data copying between hardware and
  userspace. All this information was instrumental to summarize
  requirements, model a design and implement the driver with reuse of current
  \emph{combo6} modules as a base and propose subtle interface between the
  driver and a library programmed as a part of this thesis too.

  We verified functionality during implementation with help of modern
  techniques. Each functional increment was tested, some code writing was
  ridden by
  tests thanks to requirements and specification being ready before the
  development itself. After all we were able to do integration testing when we
  finished whole module.

  Finally, developed module was compared with other available solutions
  orienting similarly in network environment. Measurements provided by
  an external vendor were compared with our solutions and whole package of
  drivers and library was deployed on Liberouter machines.

  \section{Future Work}
  \label{chap.future}
  Our implementation has been done, however some parts might still be
  improved. At least four kinds of improvements currently exist, but are out
  of scope of this thesis. We summarize them in these points:  
  \begin{itemize}
  \item
  We already introduced idea to support 64-bit PCI addressing in DAC
  mode in chapter~\ref{chap.pci}. After hardware being ready to do such
  transfers, our task would be to add the support to the \emph{combo6} base.
  Note that \sze\ driver is 64-bit ready and so the design registers, what
  blocks using address higher than $2^{32}$ is the design as a whole. Its DMA
  controllers can't do DAC transfers and so PCI core code is told to limit
  allocations by setting DMA mask. Memory management then return pages with
  physical addresses complying with the address mask.
  \item
  It would be worth to have an ability to use the \sze\ design also in a
  standard network adapter mode, i.e. with use of kernel networking stack.
  This needs slight changes of \emph{hw.c} and rewriting whole hardware
  independent part -- conversion from character device data passing to socket
  operations.
  \item
  Also we know, that we have another option in data processing in the
  application specific interface. Instead of multiprocessing of incoming data,
  we would want to balance load among many threads on multiprocessor machine
  to do more time-consuming jobs. Remember computation of ring buffer size,
  where we considered only 50\,ms latencies.
  \item
  Further optimizations may be introduced in the area of reading pointer from
  hardware. Reads and writes to slow (in comparison to front-side bus) PCI bus
  may take many hundreds of processor cycles until they finish. To
  avoid so long delays we might cache pointers for later use for example in
  \emph{poll}.
  \end{itemize}
