\begin{abstract}
  We present a generalisation of King's symbolic execution technique called
  \emph{compact symbolic execution}. It is based on a concept of templates:
  a template is a declarative parametric description of a program part,
  whose any execution has some regularities in program states along
  it. Typical sources of such regularities are program loops and recursive
  calls. Using the templates, we are able to build a symbolic execution tree
  where a single node represents arbitrarily many iterations of the
  corresponding program part without loss of any information. This leads to
  a considerable reduction of size of the symbolic execution tree. There are
  even programs for which the resulting compact symbolic execution trees are
  finite while the classic symbolic execution trees are infinite.
\end{abstract}

%{{{ Introduction

\section{Introduction} \label{sec:Introduction}

Classic symbolic execution as proposed by King in 1976~\cite{Kin76}
systematically explores all real execution paths in an analysed program.
% We commonly describe these paths in a tree structure called \emph{symbolic
% execution tree}. Each path in the tree from its root to any leaf
% represents one real path in the analysed program.
There is typically huge (or even infinite) number of execution paths even
for very small and simple programs. Therefore, exploration of all execution
paths is a serious problem known as \emph{path explosion problem}.

Compact symbolic execution also explores all real execution paths, but in a
compact manner. We analyse a given program \emph{before} we start its
symbolic execution. We look for those parts of the program, which might
produce execution paths with some regularities in program states along
them. Typically, program loops and recursion produce such regularities. We
analyse these program parts independently from the rest of the program. The
goal of the analysis is to find a \emph{template}, i.e.~a declarative
parametric description of the complete behaviour of the analysed part.
Later, whenever the symbolic execution enters a program part for which we
have a template, we add to the symbolic execution tree just a single node 
that reflects a cumulative effect of any execution of the program part as
represented by the template. This node has one successor for each edge
leaving the program part. Symbolic execution continues from these successors
in the standard way.

% This node corresponds to an application of the template which specifies
% changes of the symbolic state and path condition after leaving this
% program part, i.e.~after leaving the node of the symbolic tree.

% we instantiate the template into the end
% of the current path by adding a \emph{single} vertex, representing
% \emph{any} symbolic execution through the part, and then we simply
% \emph{jump} behind the part, where we continue with the classic symbolic
% execution again.

The worst case for compact symbolic execution occurs when we fail to
compute any template for a given program. Compact symbolic execution then
reduces to the classic one.

presents the basic idea of the compact symbolic execution using an
illustrative example. Section~\ref{sec:technique} provides a formal
description of the technique.

\section{Basic Idea}\label{sec:basic}

In this paper, we work with programs in form of flowgraphs. For example, the
flowgraph of Figure~\ref{fig:linSrch}(a) represents a simple linear search
program. The program sets \texttt{i} to \texttt{1} if some of the first
\texttt{n} elements of the array \texttt{A} has the value given by
\texttt{x}. Otherwise, the program sets \texttt{i} to \texttt{-1}.

We first briefly describe the classic symbolic execution as proposed by
King~\cite{Kin76}. Symbolic execution runs a program over symbols
representing arbitrary input values. We use symbol $\sym{v}$ to represent
any input value of variable $\var{v}$.
% $\sym{n},\sym{x},\sym{i},\sym{A}$ to represent input values of
% $\var{n},\var{x},\var{i},\var{A}$ respectively.
Symbolic execution maintains a \emph{symbolic memory} $\theta$ and a
\emph{path condition} $\varphi$. At each state of the symbolic execution,
$\theta$ assigns to each program variable its current value given as an
expression over the symbols. For example, if the first command of a program is
the assignment \texttt{i:=2*n+x}, then $\theta(\var{i})=2\sym{n}+\sym{x}$
after the first step. The path condition $\varphi$ is a formula representing
the necessary and sufficient condition on the symbols (i.e.~on input values)
to drive the execution along the current program path. $\varphi$ is initially
$\true$ and it can be updated at branching commands. For example, if we
execute a command \texttt{if~(x>n+5)} (in a flowgraph, this corresponds to a
location with two out-edges labelled by \texttt{x>n+5} and \texttt{x<=n+5}),
we instantiate the condition with use of $\theta$ and we check whether the
current path condition $\varphi$ implies validity of the instance or its
negation. In other words, we ask for validity of
\[
\varphi\implies(\theta(\var{x})>\theta(\var{n})+5)
~~~~~~~\textrm{and}~~~~~~~
\varphi\implies(\theta(\var{x})\le\theta(\var{n})+5).
\] 
If the first implication is valid, the symbolic execution continues along
the true branch. If the second implication is valid, the symbolic execution
continues along the false branch. If none of them is valid, it means that we
can follow either of the two branches. Hence, the symbolic execution forks
in order to execute both branches. In this case, we update the path
condition on the true branch to
$\varphi\wedge(\theta(\var{x})>\theta(\var{n})+5)$ and the one on the false
branch to $\varphi\wedge(\theta(\var{x})\le\theta(\var{n})+5)$. Due to the
forks\note{forks misto forkings. Forking ma jiny, sexualni vyznam.}, symbolic execution is traditionally represented by a tree called
\emph{symbolic execution tree}. Nodes of the tree (also called \emph{states}
of the symbolic execution) are triples $(l,\theta,\varphi)$ consisting of
the current program location $l$, symbolic memory $\theta$, and path
condition $\varphi$.

The symbolic execution tree for the program of Figure~\ref{fig:linSrch}(a)
is depicted in Figure~\ref{fig:linSrch}(b). Note that the figure does not
provide symbolic memories and path conditions of the nodes. However, the
figure contains instances of branching conditions on the corresponding
edges. For each node in the tree, the corresponding path condition $\varphi$
is a conjunction of the instances along the path from the root to the node.

\color{blue}
We first specify and then analyse a \emph{program part} consisting of
the locations $b,c,d,e,f$ and of all the edges between them. The part
is highlighted by a light grey region at Figure~\ref{fig:linSrch}~(a).
We can see, the locations' sequence $b,c,d,b$ forms a cyclic path in
the part. The sequence is the only source of the path in the grey
regions at Figure~\ref{fig:linSrch}~(b). In general, a part may
contain several cyclic paths, but we are only interested in those,
which are responsible for a production of those paths in classic
symbolic execution tree, we want to eliminate. Therefore, we call a
set of all such ``responsible" cyclic paths as a \emph{core} of the
part. In our example, the cyclic path $b,c,d,b$ is the core of the
part. It is highlighted by a grey region at
Figure~\ref{fig:linSrch}~(a). Next, we mark the location $b$ of the
part as an \emph{entry} location into the part, since we enter the
part by stepping into the location $b$. In general, there can be
several entries into the part. But we are always interested only in
that one, which is related to the entry vertex of that path in a
classic symbolic execution tree, we want to eliminate. The entry
location $b$ is highlighted by green colour in the figure. To complete
the part, we further need to specify \emph{exits} from the part.
Intuitively, an exit form a part is a location, into which we can
escape from an cyclic path of a core. There are two such exit
locations $e$ and $f$ in our example. They are highlighted by red
colour at the figure. Note that we call edges $(c,e)$ and $(b,f)$ as
\emph{exit edges} from the part.
\color{black}

Now we intuitively explain the idea of compact symbolic execution.
If we look at the program of Figure~\ref{fig:linSrch}(a), we immediately see
that locations $b,c,d$ form a cycle\note{v kontextu program bych pouzil spis
loop, cycle je to v grafech. anebo bych mozna nepsal nahore program, ale
flowgraph} (highlighted by the grey region). All
executions of this program part have a very regular structure: each
execution consists of $\kappa$ iterations along the cycle ($\kappa\ge0$)
followed either by the edge $(b,f)$ or by the edges $(b,c)$ and $(c,e)$.
% In other words, each excution follows a path of the form 
% $(bcd)^{\kappa}bf$ or $(bcd)^{\kappa}bce$.
We say that $b$ is an \emph{entry location} and $f,e$ are \emph{exit
  locations} of this program part.

We want to find a unified description of the effect of all executions of
this program part. We first study the effect of all iterating parts of these
executions. The effect of $\kappa$ iterations of the cycle on variable
values can be described by the following \emph{parametric symbolic memory}
$\theta_*[\kappa]$:
\[
\begin{array}{rclp{2ex}rcl}
  \theta_*[\kappa](\var{n})&=&\sym{n}&&\theta_*[\kappa](\var{x})&=&\sym{x}\\
  \theta_*[\kappa](\var{i})&=&\sym{i}+\kappa&&\theta_*[\kappa](\var{A})&=&\sym{A}\\
\end{array}
\]
Note that the symbols $\sym{n},\sym{x},\sym{i},\sym{A}$ now represent the
values of corresponding variables at the entry location of the program
part. 
% This is why $\theta_*[\kappa](\var{i})$ is $\sym{i}+\kappa$ and not just
% $\kappa$.

We can also formulate a \emph{parametric path condition} $\varphi_*[\kappa]$
representing all path conditions for executions looping around the cycle.
The necessary and sufficient condition to make $\kappa$ iterations of the
cycle is validity of both conditions on the cycle, \texttt{i<n} and
\texttt{A[i]!=x}, in each of $\kappa$ iterations. Using $\theta_*[\kappa]$,
one easily derives that instances of these conditions in the
$(\tau+1)$-st\note{to -st tady vypada divne}
iteration are $\sym{i}+\tau<\sym{n}$ and $\sym{A}(\sym{i}+\tau)\neq\sym{x}$,
respectively. Hence, $\varphi_*[\kappa]$ can be defined as
\[
  \varphi_*[\kappa]~=~\kappa\ge0~\wedge~
  \forall\tau(0\le\tau<\kappa\implies
  (\sym{i}+\tau<\sym{n}\,\wedge\,\sym{A}(\sym{i}+\tau)\neq\sym{x})).
\]

Finally, we use $\theta_*[\kappa]$ and $\varphi_*[\kappa]$ to define
symbolic memory $\theta_f[\kappa]$ and path condition $\varphi_f[\kappa]$
describing the effect of all executions of the considered program part
leading to exit location $f$ and $\theta_e[\kappa],\varphi_e[\kappa]$ with
the analogous information for the exit location $e$. As the edges
$(b,f),(b,c),(c,e)$ do not modify any variable, we can clearly see that
$\theta_f[\kappa]=\theta_e[\kappa]=\theta_*[\kappa]$. Further,
$\varphi_f[\kappa]$ and $\varphi_e[\kappa]$ are conjunctions of
$\varphi_*[\kappa]$ with the instances of the conditions on the edge $(b,f)$
or on the edges $(b,c),(c,e)$, respectively. Hence, the path conditions
$\varphi_f[\kappa],\varphi_e[\kappa]$ are defined as follows:
\[
\begin{array}{rcl}
  \varphi_f[\kappa]&~=~&\varphi_*[\kappa]~\wedge~\sym{i}+\kappa\ge\sym{n}\\
  \varphi_e[\kappa]&=&\varphi_*[\kappa]~\wedge~\sym{i}+\kappa<\sym{n}
    ~\wedge~\sym{A}(\sym{i}+\kappa)=\sym{x}
\end{array}
\]

The overall effect of the considered program part is now fully described by
a so-called \emph{template} consisting of the entry location $b$ and two
triples $(f,\theta_f[\kappa],\varphi_f[\kappa])$ and
$(e,\theta_e[\kappa],\varphi_e[\kappa])$.

The template is applied during \emph{compact symbolic execution} of the
program. The execution processes the first command \texttt{i:=0} in the
standard way. Then we reach the node $(b,\theta',\varphi')$ with the entry
location $b$ for which we have the template.
%  proceeds in the standard way until we reach the node
% $(b,\theta,\varphi)$ with entry location $b$. 
% where $\theta(\var{i})=0$ and it assigns to each other variable he
% corresponding symbol except for ...
At this point, we apply the template instead of executing the original
program. The node $(b,\theta',\varphi')$ gets one successor for each triple
of the template. The triple $(f,\theta_f[\kappa],\varphi_f[\kappa])$
generates the successor $(f,\theta'_f[\kappa],\varphi'_f[\kappa])$. Note
that we cannot use $(f,\theta_f[\kappa],\varphi_f[\kappa])$ directly as
$\theta_f[\kappa],\varphi_f[\kappa]$ describe executions starting just at
the entry location $b$, while $\theta'_f[\kappa],\varphi'_f[\kappa]$ have to
reflect the effect of the executions starting at $a$. We create
$\theta'_f[\kappa],\varphi'_f[\kappa]$ by composing
$\theta_f[\kappa],\varphi_f[\kappa]$ with $\theta',\varphi'$. The
composition is precisely described in the following section. In our simple
program, $\theta',\varphi'$ reflect only the effect of assignment
\texttt{i:=0}. Thus, $\theta'_f[\kappa]$ equals to $\theta_f[\kappa]$ where
$\sym{i}$ is replaced by $0$ and $\varphi'_f[\kappa]$ equals
$\varphi_f[\kappa]$ where $\sym{i}$ is replaced by $0$. The second triple
$(e,\theta_e[\kappa],\varphi_e[\kappa])$ of the template generates the
successor $(e,\theta'_e[\kappa],\varphi'_e[\kappa])$ analogously. The
symbolic execution then continues from these successor in the standard way.
The resulting \emph{compact symbolic execution tree} is depicted in
Figure~\ref{fig:linSrch}(c).

The compact symbolic execution tree is much smaller than the classic
tree. In particular, the infinite path in the classic tree (highlighted by
the grey region) corresponds to the single node $d$ in the compact
tree. However, the two trees keep the same information. For example, the
left leaf of the compact tree represents the following path condition
\[
\varphi[\kappa]~=~\kappa\ge0~\wedge~ \forall\tau(0\le\tau<\kappa\implies
(\tau<\sym{n}\,\wedge\,\sym{A}(\tau)\neq\sym{x}))~\wedge~\kappa<\sym{n}
~\wedge~\sym{A}(\kappa)=\sym{x}.
\]
Let us mark all leafs on the left-hand side of the classic tree as
$g_0,g_1,g_2,\ldots$ and let $\varphi_0,\varphi_1,\varphi_2,\ldots$ be the
corresponding path conditions. One can easily compute each path condition
$\varphi_i$ as a conjunction of the conditions along the corresponding path
in the figure of the tree and check that $\varphi_i$ is equivalent to
$\varphi[i]$ for each $i\ge0$. For example,
\[
\begin{array}{rcl}
\varphi_1&=
&0<\sym{n}~\wedge~\sym{A}(0)\neq\sym{x}~\wedge~1<\sym{n}~\wedge~\sym{A}(1)=\sym{x}\\[2mm]
\varphi[1]&\equiv&\forall\tau(0\le\tau<1\implies
(\tau<\sym{n}\,\wedge\,\sym{A}(\tau)\neq\sym{x}))~\wedge~1<\sym{n}
~\wedge~\sym{A}(1)=\sym{x}
\end{array}
\]
and hence $\varphi_1\equiv\varphi[1]$. Similarly, each symbolic memory of a
node $g_i$ is an instance $\theta[i]$ of the parametrized symbolic memory in
the left leaf of the compact tree. Analogous relations hold for leafs on the
right-hand sides of the trees.

Let us now have a look at Figure~\ref{fig:countIf}~(a) depicting a
program with a function \texttt{countIf}. The function counts number
of elements in array \texttt{A} having values equal to \texttt{x}. We
show the symbolic execution tree of the program at
Figure~\ref{fig:countIf}~(b). There we can see several sequences of
grey regions. According to our experience with the previous example we
can easily detect that all that paths in grey are generated by a
single program part consisting of locations $c,d,e,f,g$ and edges
between them. But there are two cyclic paths $\pi = c,d,e,f,c$ and
$\pi' = c,d,f,c$ inside the part. Nevertheless, the grey regions
highlight only the cycle $\pi$. So, we ignore the cycle $\pi'$ and
$\pi$ is therefore the only cycle we consider. The remainder is now
obvious. The locations $f$ and $g$ are exits from the part and $c$ is
the entry location into the part. The analysis of the path (discussed
later in Section~\ref{sec:Detection}) computes the following template
\[t = (c, 2, \{ (\theta_f, \varphi_f, [], f)\prm{\kappa}, (\theta_g,
\varphi_g, [], g)\prm{\kappa} \} )\] Compact symbolic execution with
the template $t$ computes compact symbolic execution tree depicted at
Figure~\ref{fig:countIf}~(c). The tree is basically a single link
list. Note that we instantiate the template each time we reach the
location $c$. But for each such instantiation we need a fresh
parameter to prevent parameter collisions from previous
instantiations. We assume we have infinitely many different names for
the parameters. Therefore, expressions and program states at
Figure~\ref{fig:countIf}~(c) are as follows: $\gamma_g^i  =
s_c^i.\theta\ese{t.\varphi_g\prm{\kappa_i}}$, $\gamma_f^i =
s_c^i.\theta\ese{t.\varphi_f\prm{\kappa_i}}$, $s_g^i = s_c^i.\theta
\circ (t.\theta_g, t.\varphi_g, [], g)\prm{\kappa_i}$ and $s_f^i =
s_c^i.\theta \circ (t.\theta_f, t.\varphi_f, [], f)\prm{\kappa_i}$.

The sequences of grey regions in the tree at
Figure~\ref{fig:countIf}~(b) goes bottom left. But imagine they would
go bottom right. Then each region would represent a sequence of
program locations $c,d,f,c$. If we analysed closer these sequences of
grey regions, we would realise that there is a part of the program
from Figure~\ref{fig:countIf}~(a) consisting of vertices $c,d,f,e,g$,
where $c,d,f,c$ is the only cycle in the part, $c$ is the entry
location into the part and locations $e$ and $g$ are exits from the
part. If we further built a template from the part and run compact
symbolic execution with it, we would also receive a compact symbolic
execution tree forming basically a single linked list.

To summarise, a general scheme for compact symbolic execution of the
examples above is as follows. We enumerate cyclic paths in a given
program one by one. For each enumerated cycle we first complete
related program part (determining entry and exit locations) and then
we compute a template for the part. Then we run compact symbolic
execution with the computed templates.

\section{Description of the Technique}\label{sec:technique}

This section describes the compact symbolic execution formally. For
simplicity, we formulate the algorithm for programs manipulating only
integer variables and read-only multidimensional integer arrays, and with no
function calls. Nevertheless, the algorithm can be extended to handle
variables of other types, function calls, etc.

  %{{{ Preliminaries

\subsection{Preliminaries}

\color{blue}
\begin{dfn}[Program] \label{def:Program}
A \emph{program} is a collection of functions and global variables.
Each function has its own local variables. All program variables and
functions have different names. Exactly one function is marked as a
starting one. Each function is represented as an oriented graph.
Vertices in a graph identify program locations, while edges define
transitions between them. We distinguish a single entry and an exit
location in each graph. There is no in-edge to an entry location and
there is no out-edge from an exit one. We label edges by actions to be
taken when moving between connected locations. An action can be:
\begin{itemize}
\item[(1)] An assignment of the form $<$variable$>${\tt
:=}$<$expression$>$,

\item[(2)] Call by value statements\\(a) $<$variable$>${\tt
:=}$<$function-name$>${\tt (}$<$arg-list$>${\tt )}, or\\(b)
$<$function-name$>${\tt (}$<$arg-list$>${\tt )}

\item[(3)] A return value statement {\tt ret }$<$expression$>$,

\item[(4)] {\tt skip} statement, which does nothing, or

\item[(5)] A boolean expression over program variables, which
either enables or disables its edge.
\end{itemize}
If an edge $e = (u,v)$ is labelled by one of the actions (1)-(4), then
out-degree of $u$ is $1$. Otherwise, label of $e$ is an action (5),
out-degree of $u$ is $2$ and its out-edges are labelled by boolean
expressions $\gamma$ and $\neg \gamma$. No action (2) can reference
the starting function. And for simplicity we do not consider pointer
arithmetic nor heap allocations. We prevent invalid operations in
actions (like division by zero, etc.) by branchings into error
locations. An error location is any location with a single out-edge
heading back to that location and labelled with {\tt skip} action.
\end{dfn}

We assume $P$ is a program throughout whole
Section~\ref{sec:technique}.

An injective function from a set of all program variables of $P$ to a
set of \emph{symbols} $\{ \alpha_0, \alpha_1, \alpha_2, \ldots \}$ is
an \emph{initial symbolic memory} of $P$ denoted by $\Theta$. For each
program variable \var{a} its symbol $\Theta(\var{a})$ represents some
yet unknown value of that variable. So, $\Theta(\var{a})$ must belong
to a domain of \var{a} (i.e.~$\Theta(\var{a})$ is of \var{a}'s type).
We further distinguish a set $\{ \kappa_1, \tau_1, \kappa_2, \tau_2,
\ldots \}$ of variables, called \emph{parameters}, ranging over
non-negative integers. A \emph{numeric symbolic expression} is an
application of denoted operators of $P$'s language to numeric
constants, symbols, or parameters. A \emph{boolean symbolic
expression} is either an equality or inequality predicate over numeric
symbolic expressions, or an application of quantifiers or logical
connectives to other boolean symbolic expressions. We require for any
boolean symbolic expression $\varphi$ that all symbols in it have only
free occurrences there. On the other hand, we put no restriction to
occurrences of parameters in $\varphi$. A \emph{symbolic expression}
is either numeric or boolean symbolic expression. \emph{Symbolic
memory} is a function $\theta$ from program variables to a set of
symbolic expressions. For each local variable \var{a} \emph{not}
defined at a location $l$, we have $\theta(\var{a}) =
\Theta(\var{a})$.   A \emph{call stack} $\Xi$ is a finite sequence of
stack records of the form $(\sigma,l)$, where $\sigma$ is a
restriction of a symbolic memory to local variables and $l$ is a
return location. A \emph{program state} is a tuple $s = (\theta,
\varphi, \Xi, l)$, where $\theta$ is a symbolic memory, $\varphi$ is a
path condition, $\Xi$ is a call stack and $l$ is a current program
location. When we want to emphasise that a set $\bld{\kappa}$ contains
all the parameters having a free occurrence in any of symbolic
expressions in a program state  $s$, we denote it as
$s\prm{\bld{\kappa}}$, and we commonly abbreviate that $\bld{\kappa}$
is a set of parameters in $s$. The notation also applies to the
components of $s$, i.e.~we write $\theta\prm{\bld{\kappa}}$,
$\varphi\prm{\bld{\kappa}}$, and $\Xi\prm{\bld{\kappa}}$. We now
describe a substitution of parameters. Each function $\bld{\nu}$ from
a finite set of parameters to non-negative integers is a
\emph{valuation} of the parameters. If $\bld{\kappa}$ is a set of
parameters in a state $s$, and $\bld{\nu}$ is a valuation of these
parameters, then $s\prm{\bld{\nu}}$ is the program state
$s\prm{\bld{\kappa}}$, where each free occurrence of each parameter
$\kappa \in \bld{\kappa}$ anywhere in $s\prm{\bld{\kappa}}$ is
substituted by the number $\bld{\nu}(\kappa)$. We typically compute
$s\prm{\bld{\nu}}$ form $s\prm{\bld{\kappa}}$ per component. In case
$\bld{\kappa} = \{ \kappa \}$ and $\bld{\nu}(\kappa) = \nu$ we
commonly abbreviate $s\prm{\bld{\nu}}$ and $s\prm{\{\kappa\}}$ to
$s\prm{\nu}$ and $s\prm{\kappa}$ respectively. Once we have all the
basic terms defined, we can finally define the key properties of
compact symbolic execution.

We often use the following dot-notation to access elements of tuples.
If $s = (\theta, \varphi, \Xi, l)$ is a program state, then $s.\theta$
denotes its symbolic memory, $s.\varphi$ denotes its path condition,
$s.\Xi$ is its call stack and $s.l$ is its current program location.
Further, if $u$ is a vertex of a symbolic execution tree, then $u.s$
denotes a program state labelling the vertex. And instead of
$u.s.\theta$, $u.s.\varphi$, $u.s.\Xi$, and $u.s.l$ we simply write
$u.\theta$, $u.\varphi$, $u.\Xi$, and $u.l$ respectively. Finally, if
$\Xi$ is a call stack, then we use the dot-notation to access a record
at the top of the call stack. For example, $\Xi.l$ denotes a return
location of a record at the top of $\Xi$.

\begin{dfn}[Composition]\label{def:Composition}
Let $\Xi = [r_0, \ldots, r_m]$ and $\Xi' = [r_0', \ldots, r_n']$ be
call stacks, and $s = (\theta, \varphi, \Xi, l)$ and $s' = (\theta',
\varphi', \Xi', l')$ be program states. Then a composite program state
$s \circ s' = (\theta \circ \theta',~\varphi \wedge
\theta\ese{\varphi'}, \Xi \circ (\theta \circ \Xi'), l')$, where
$\theta\ese{\varphi'}$ is a symbolic expression constructed from
$\varphi'$ such that all symbols $\alpha_i$ in $\varphi'$ are
simultaneously substituted by symbolic expressions
$\theta(\Theta^{-1}(\alpha_i))$, $\theta \circ \theta'$ is a symbolic
memory such that for each variable \var{a} we have $(\theta \circ
\theta')(\var{a}) = \theta\ese{\theta'(\var{a})}$, $\theta \circ \Xi'
= [\bar{r}_0', \ldots, \bar{r}_n']$, where each $\bar{r}_i'$ is equal
to $r_i$ except for the first component being $\bar{r}_i'.\sigma =
\theta \circ r_i'.\sigma$, and $\Xi \circ (\theta \circ \Xi') = [r_0,
\ldots,$ $ r_m, \bar{r}_0', \ldots, \bar{r}_n']$.
\end{dfn}

\begin{dfn}[Equivalence]\label{def:Equivalence}
Let $\varphi$, $\varphi'$ be symbolic expressions, $\theta$, $\theta'$
be symbolic memories, $\Xi = [r_0, \ldots, r_m]$, $\Xi' = [r_0',
\ldots, r_n']$ be call stacks, and $s$, $s'$ be program states. Then
$\varphi \equiv \varphi'$, if $\varphi$ and $\varphi'$ are either
logically equivalent boolean symbolic expressions or numeric symbolic
expressions such that $(\varphi = \varphi') \equiv \true$. $\theta
\equiv \theta'$, if for each variable \var{a} we have $\theta(\var{a})
\equiv \theta'(\var{a})$. $\Xi \equiv \Xi'$, if $m = n$ and for each
$i \in \{0, \ldots, m\}$ we have $r_i.\sigma$ and $r_i'.\sigma$ are
defined for the same variables with equivalent values and $r_i.l =
r_i'.l$. Finally, $s \equiv s'$, if both $s$ and $s'$ have equal or
equivalent components.
\end{dfn}

\begin{lem}[Equivalent Compositions]\label{lem:EquivalentCompositions}
Let $s, s'$, and $s''$ be program states, $\bld{\nu}$ and $\bld{\nu}'$
be valuations of parameters in $s$ and $s'$ respectively such that
$\bld{\nu} \cup \bld{\nu}'$ is also a valuation, $\theta$, $\theta'$,
and $\theta''$ be symbolic memories, and $\varphi$ and $\psi \wedge
\psi'$ be symbolic expressions. Then $s \circ (s' \circ s'') \equiv (s
\circ s') \circ s''$, $s\prm{\bld{\nu}} \circ s'\prm{\bld{\nu}'}
\equiv (s \circ s')\prm{\bld{\nu} \cup \bld{\nu}'}$, $\theta \circ
(\theta' \circ \theta'') \equiv (\theta \circ \theta') \circ
\theta''$, $(\theta \circ \theta')\ese{\varphi} \equiv
\theta\ese{\theta'\ese{\varphi}}$, and $\theta\ese{\psi} \wedge
\theta\ese{\psi'} \equiv \theta\ese{\psi \wedge \psi'}$.
\end{lem}

% For a loop $C$ with an entry node $v$, a \emph{flowgraph induced by the loop},
% denoted as $P(C,v)$, is the subgraph of the original flowgraph induced by $C$
% where $v$ is marked as the start node, a fresh node $v'$ is added and marked
% as the target node, and every edge $(u,v)\in E$ leading to $v$ is replaced
% by an edge $(u,v')$ labelled with the same instruction as one of the edge $(u, v)$.

A flowgraph is a tuple $P=(V,E,l_s,\iota)$, where $(V,E)$ is a finite
oriented graph, elements of $V$ are called \emph{locations}, $l_s\in V$ is a
\emph{start} location, and $\iota:E\rightarrow\mathcal{I}$ is a function
assigning to each edge $e$ an \emph{instruction} $\iota(e)$. A location is
\emph{branching} if its out-degree is $2$. All other locations have
out-degree at most $1$. We use two kinds of instruction: an assignment
instruction $\var{a}\texttt{:=}e$ for some scalar variable $\var{a}$ and
some expression $e$, and an assumption $\texttt{assume}(\gamma)$ for some
quantifier-free formula $\gamma$ over program variables. Out-edges of any
branching location are labelled with assumptions $\texttt{assume}(\gamma)$
and $\texttt{assume}(\neg\gamma)$ for some $\gamma$. We usually omit the
keyword $\texttt{assume}$ in our flowgraphs. We consider only instructions
operating on \emph{scalar variables} $\var{a},\var{b},\ldots$ of type
\texttt{Int} and multi-dimensional \emph{array variables}
$\var{A},\var{B},\ldots$ of type
\texttt{Int}$^k\rightarrow$\texttt{Int}. Flowgraphs are also called
\emph{programs}.

\medskip By \emph{symbolic expressions} we mean all expressions build with
integers, standard integer operations and functions, and
\begin{itemize}
\item a \emph{symbol} $\sym{a}$ for each scalar variable $\var{a}$,
\item a \emph{function symbol} $\sym{A}$ for each array variable $\var{A}$,
  where arity of $\sym{A}$ corresponds to the dimension of array $\var{A}$,
\item a countable set $\{\kappa_1,\tau_1,\kappa_2,\tau_2,\ldots\}$ of
  \emph{parameters}, and
\item a construct $\ite(\varphi,e_1,e_2)$, of meaning
  ``\textbf{i}f-\textbf{t}hen-\textbf{e}lse'', where $e_1,e_2$ are symbolic
  expressions and $\varphi$ is a first order formula over symbols and
  parameters. The value of $\ite(\varphi,e_1,e_2)$ is $e_1$ if $\varphi$
  holds and $e_2$ otherwise.
\end{itemize}

% Let $f,e_1,\ldots,e_n$ be symbolic expressions and $x_1,\ldots,x_n$ be path
% counters or symbols corresponding to scalar variables. Then
% $f[x_1/e_1,\ldots,x_n/e_n]$ is a symbolic expression $f$ where all
% occurrences of all $x_i$ are simultaneously replaced by expressions
% $e_i$. To shorten the notation, we write $f[\vec{x}/\vec{e}]$ when the
% meaning is clearly given by a context. We also use the notation
% $\varphi[\vec{x}/\vec{e}]$ with the analogous meaning for formulae
% $\varphi$.

A \emph{symbolic memory} is a function $\theta$ assigning to each scalar
variable $\var{a}$ a symbolic expression and to each array variable
$\var{A}$ the function symbol $\sym{A}$.
% Let $\var{a}$ be a scalar variable and $e$ be a symbolic expression. Then
% $\theta[\var{a}\rightarrow e]$ is a symbolic memory equal to $\theta$
% except for variable $\var{a}$, where $\theta[\var{a}\rightarrow
% e](\var{a})=e$.
The notation $\theta(\cdot)$ is used in a more general way. It always
denotes the operation of replacing each (scalar or array) variable $\var{a}$
by $\theta(\var{a})$. $\theta\langle\cdot\rangle$ denotes the operation on
symbolic expressions or formulae, where all occurrences of all symbols
$\sym{a}$ are simultaneously replaced by $\theta(\var{a})$. Additionally,
$\theta_1\langle\theta_2\rangle$ denotes composition of two symbolic
memories $\theta_1$ and $\theta_2$ satisfying
$\theta_1\langle\theta_2\rangle(\var{a})=\theta_1\langle\theta_2(\var{a})\rangle$
for each variable $\var{a}$. Intuitively, the symbolic memory
$\theta_1\langle\theta_2\rangle$ represents an overall effect of a code with
effect $\theta_1$ followed by a code with effect $\theta_2$.

% Finally, for vectors $\vec{u}=(u_1,\ldots,u_n)$ and
% $\vec{v}=(v_1,\ldots,v_n)$ we use $\vec{u}\le\vec{v}$ as an abbreviation
% for $u_1\le v_1\wedge\ldots\wedge u_n\le v_n$.

\medskip \emph{Symbolic state} (or simply a \emph{state}) is a triple
$s=(l,\theta,\varphi)$ of a location $l$, symbolic memory $\theta$ and first
order formula over symbols and parameters $\varphi$ called \emph{path
  condition}. \emph{Initial state} is the triple $(l_s,\theta_s,\true)$
where $l_s$ is the start location and $\theta_s$ is the symbolic memory
assigning to each variable $\var{a}$ the corresponding symbol $\sym{a}$. We
assume that the reader is familiar with standard symbolic execution (we
refer to~\cite{Kin76} if the intuition provided in the previous section is
not sufficient).

\subsection{Templates and their computation}

\color{blue}
\begin{dfn}[Program Part]\label{def:Part}
Let $P$ be a program, $(u,e)$ be an edge of $P$, $C = \{ \pi_1,
\ldots, \pi_n \}$ be a non-empty set of cyclic paths in $P$ such that
$e$ belongs to each $\pi_i$, while $u$ belongs to none of them, and
let $X = \{ (u_1,x_1), \ldots, (u_m,x_m) \}$ be a non-empty set of all
those edges of $P$, which do not belong to any of the paths $\pi_i$,
but each of their start vertices $u_j$ belongs to some $\pi_i$. Then,
a triple $(C, e, X) $ is a \emph{part} of $P$, where the set $C$ is a
\emph{core} of the part, $e$ is an \emph{entry} into the part, each
edge $(u_i,x_i) \in X$ is an \emph{exit edge} from the part, and each
location $x_i$ is called an \emph{exit} from the part.
\end{dfn}

\begin{dfn}[Loop template] \label{def:LoopTemplate}
Let $T$ be a classic symbolic execution tree of $P$, $n>0$ be an
integer, $l, l_1, \ldots, l_n$ be locations in $P$, $\kappa$ be a
parameter, $\theta_1\prm{\kappa}, \ldots,$ $\theta_n\prm{\kappa}$ be
symbolic memories, $\varphi_1\prm{\kappa}, \ldots,
\varphi_n\prm{\kappa}$ be satisfiable boolean symbolic expressions
such that for each $i,j \in \{ 1, \ldots, n \}, i \neq j$, we have
$(\varphi_i \wedge \varphi_j) \equiv \false$, and let
$\Xi_1\prm{\kappa}, \ldots, \Xi_n\prm{\kappa}$ be call stacks. Then a
tuple $t = (l, n, \{ (\theta_1, \varphi_1, \Xi_1, l_1), \ldots,
(\theta_n,\varphi_n, \Xi_n, l_n) \})$ is a \emph{loop template} in $P$, if
\begin{itemize}
\item[(L1)] For each path $\pi = u \omega$ in $T$ from any vertex $u$
satisfying $u.l = t.l$ to a leaf, there is a vertex $w \in \omega$, an
index $i \in \{ 1, \ldots, n \}$, and an integer $\nu \geq 0$ such
that $w.s \equiv u.s \circ
(t.\theta_i,~t.\varphi_i,~t.\Xi_i,~t.l_i)\prm{\nu}$.

\item[(L2)] For each vertex $u$ of $T$, an index $i \in \{ 1, \ldots,
n \}$, and non-negative integer $\nu$ such that $u.l = t.l$ and
$(u.\varphi \wedge u.\theta\ese{t.\varphi_i\prm{\nu}})$ is
satisfiable, there is a successor $w$ of $u$ in $T$ such that $w.s
\equiv u.s \circ (t.\theta_i,~t.\varphi_i,~t.\Xi_i,~t.l_i)\prm{\nu}$.
\end{itemize}
\end{dfn}

In this section we present an algorithm computing a template with one
parameter for such a program part $(C,e,X)$ of a given program $P$,
whose core $C$ consist of a \emph{single} cyclic path. It is
sufficient to show how to compute a parametrised symbolic state
$(\theta_x, \varphi_x, \Xi_x, x)\prm{\kappa}$ for an exit $x$ from the
part. The computation of the remainder of the resulting template is
then straightforward. Without loss of generality we may assume that
$X$ only contains a single exit edge with the exit $x$ as its end
location and we may also assume that the exit $x$ does not belong to
the core $C$ (since we can always consider its copy outside).

The algorithm proceeds in two steps. First, we compute a program state
$(\theta, \varphi, [], e)$ resulting from a classic symbolic execution
around the core from the entry $e$ back to it exactly once, and a
program state $(\hat{\theta}, \hat{\varphi}, \hat{\Xi}, x)$ resulting
from a classic symbolic execution from $e$ to $x$. The second step is
to express the state $(\theta_x, \varphi_x, \Xi_x, x)\prm{\kappa}$ in
terms of the program states computed in the first step.

The computation of the program states $(\theta, \varphi, [], e)$ and
$(\hat{\theta}, \hat{\varphi}, \hat{\Xi}, x)$ requires to run classic
symbolic execution on the part. But
Algorithm~\ref{alg:executeSymbolically} can only execute programs
satisfying Definition~\ref{def:Program}. Therefore, we create a new
program, say $P'$, representing the part.
%
We start with the program $P'$ consisting of all variables of $P$ and
of all those functions of $P$ having at least one location of the
part. Note that the core may traverse several functions through call
sites. We now remove all the locations and edges in $P'$, which do not
belong to the part. Next, we mark the function containing the entry
location $e$ as the starting one, and we set $e$ as its exit location.
Then we create a new location $e'$ representing an entry location of
the starting function. Now we break the cyclic path of the core in $e$
such that all out-edges of $e$ become the out-edges of $e'$. Then we
transform $x$ to an error location by adding a loop edge with
\texttt{skip} action. It remains to set entry and exit locations of
the remaining functions of $P'$. This is straightforward, as their
graphs are just rooted trees.
%
$P'$ is now a program according to Definition~\ref{def:Program}, so we
run Algorithm~\ref{alg:executeSymbolically} (without $\Box$ lines, of
course) on it. Note that the run must always terminate. Let $E$ be a
set of all program states labelling all the leaf vertices of the
resulting classic symbolic execution tree. Obviously, $|E| \leq 2$. If
there is no $s \in E$ such that $s.l = e$, then we stop with failure,
since there is no real path going around the core. If there is no
state $s \in E$ such that $s.l = x$, then we discard our exit edge
from the consideration for the template, since it is impossible to
leave the core through it. Otherwise, $E$ contains exactly two program
states, which are the states we are looking for.

Now we show how to express the state $(\theta_x, \varphi_x, \Xi_x,
x)\prm{\kappa}$ in terms of the program states computed above. Let $T$
be classic symbolic execution tree of $P$, computed by
Algorithm~\ref{alg:executeSymbolically} without $\Box$ lines. Further,
let $u$ be a vertex of $T$ such that $u.l = e$, and let $\pi = u_0
\ldots u_1 \ldots u_2 \ldots u_{\nu} \ldots w$ be a path in $T$
starting at $u$ (i.e.~$u_0 = u$), then iterating the core exactly $\nu
\geq 0$ times (i.e.~all the vertices $u_i \in \pi$ are exactly those
having $u_i.l = e$), and then $\pi$ finally leaves the core by
following the path towards the vertex $w$, satisfying $w.l = x$. We
use the memory composition operation to express memories of the
vertices $u_i$ along $\pi$ as follows.
\begin{align*}
u_1.\theta &= u.\theta \circ \theta
\\
u_2.\theta &= u_1.\theta \circ \theta = u.\theta \circ (\theta \circ
\theta)
\\
\cdots
\\
u_{\nu}.\theta &= u_{\nu-1}.\theta \circ \theta = u.\theta \circ
(\underset{\nu}{\underbrace{\theta \circ \cdots \circ \theta}}).
\end{align*}
If we denote the composition of $i$ symbolic memories $\theta$ by
$\theta^i$, where $\theta^0 = \Theta$, $\theta^1 = \theta$, and
$\theta^i = \theta \circ \theta^{i-1}$, then we have $u_{i}.\theta =
u.\theta \circ \theta^i$ and we get
\begin{align*}
w.\theta = u.\theta \circ (\theta^{\nu} \circ \hat{\theta}).
\end{align*}
We proceed similarly to express path conditions of the vertices along
$\pi$.
\begin{align*}
u_1.\varphi &\equiv u.\varphi \wedge u.\theta\ese{\varphi} \equiv
u.\varphi \wedge (u.\theta \circ \theta^0)\ese{\varphi} \equiv
u.\varphi \wedge u.\theta\ese{\theta^0\ese{\varphi}}
\\
u_2.\varphi &\equiv u_1.\varphi \wedge u_1.\theta\ese{\varphi} \equiv
u.\varphi \wedge u.\theta\ese{\theta^0\ese{\varphi}} \wedge (u.\theta
\circ \theta^1)\ese{\varphi} \\ &\equiv u.\varphi \wedge
u.\theta\ese{\theta^0\ese{\varphi} \wedge \theta^1\ese{\varphi}}
\\
\cdots
\\
u_{\nu}.\varphi &\equiv u_{\nu-1}.\varphi \wedge
u_{\nu-1}.\theta\ese{\varphi} \equiv u.\varphi \wedge
u.\theta\ese{\underset{\nu}{\underbrace{\theta^0\ese{\varphi} \wedge
\ldots \wedge \theta^{\nu-1}\ese{\varphi}}}}.
\end{align*}
Using the following equivalence
\[\theta^0\ese{\varphi} \wedge \ldots \wedge
\theta^{\nu-1}\ese{\varphi} \equiv 0 \leq \nu \wedge \forall \tau~(0
\leq \tau < \nu \rightarrow \theta^{\tau}\ese{\varphi}),\]
we can write
\begin{align*}
w.\varphi &\equiv u_{\nu}.\varphi \wedge
u_{\nu}.\theta\ese{\hat{\varphi}} \\ &\equiv u.\varphi \wedge
u.\theta\ese{\theta^0\ese{\varphi} \wedge \ldots \wedge
\theta^{\nu-1}\ese{\varphi} \wedge \theta^{\nu}\ese{\hat{\varphi}}}
\\
&\equiv u.\varphi \wedge u.\theta\ese{0 \leq \nu \wedge \forall
\tau~(0 \leq \tau < \nu \rightarrow \theta^{\tau}\ese{\varphi}) \wedge
\theta^{\nu}\ese{\hat{\varphi}}}.
\end{align*}
But SMT solvers do not support the memory composition operation
appearing in the formula $w.\varphi$. Therefore, we need an equivalent
\emph{declarative} description of the operation. Such a description is
a parametrised symbolic memory $\theta\prm{\kappa}$, for which we
require $\theta\prm{\kappa} \equiv \theta^{\kappa}$, for each $\kappa
\geq 0$. For a given symbolic memory $\theta$ we compute content of
$\theta\prm{\kappa}$ per variable by applying the following two rules
\begin{gather*}
\frac{\theta(\var{a}) = \Theta(\var{a}) + c,~~~\var{a}~\textrm{is of a
numeric type},~~~c \textrm{~is a numeric constant of \var{a}'s
type}}{\theta\prm{\kappa}(\var{a}) = \Theta(\var{a}) + c \cdot
\texttt{typeOf<a>(}\kappa\texttt{)}},
\\ \\
\frac{\theta(\var{A}) = \Theta(\var{A}),~~~\var{A}~\textrm{is of an
array type}}{\theta\prm{\kappa}(\var{A}) = \Theta(\var{A})},
\end{gather*}
where the expression $\texttt{typeOf<a>(}\kappa\texttt{)}$ represent a
casting operation of $\kappa$ to a type of the variable \var{a}. If
there is a variable, which does not match any of the rules, then we
fail to compute $\theta\prm{\kappa}$. And we thus fail to compute the
resulting program state. Obviously, one can provide more rules for
more complex symbolic memories. The presented rules are only supposed
to illustrate the process.

Having $\theta\prm{\kappa}$ we define
\begin{align*}
\theta_x\prm{\kappa} &= \theta\prm{\kappa} \circ \hat{\theta}
\\
\varphi_x\prm{\kappa} &= 0 \leq \kappa \wedge \forall \tau~(0 \leq
\tau < \kappa \rightarrow \theta\prm{\tau}\ese{\varphi}) \wedge
\theta\prm{\kappa}\ese{\hat{\varphi}}
\\
\Xi_x\prm{\kappa} &= \theta\prm{\kappa} \circ \hat{\Xi},
\end{align*}
and we get $w.\theta \equiv u.\theta \circ \theta_x\prm{\nu}$,
$w.\varphi \equiv u.\varphi \wedge u.\theta\ese{\varphi_x\prm{\nu}}$
and $w.\Xi \equiv u.\Xi \circ (u.\theta \circ \Xi_x\prm{\nu})$. Using
these equivalences we write $w.s \equiv u.s \circ (\theta_x,
\varphi_x, \Xi_x, x)\prm{\nu}$, which is exactly the equivalence used
in Definition~\ref{def:LoopTemplate}~(L1) and (L2).
\color{black}

Let us consider a fixed program $(V,E,l_s,\iota)$. A \emph{template
(with   one parameter)} is a pair $(l,M)$, where $l$ is a location
called \emph{entry location} and $M$ is a finite set of triples. Each
triple in $M$ has the form $(e,\theta[\kappa],\varphi[\kappa])$, where
$e$ is a location, $\theta[\kappa]$ is a symbolic memory using a
single parameter $\kappa$, and $\varphi[\kappa]$ is a satisfiable
first order formula over symbols and parameters with at most one free
parameter $\kappa$.

Moreover, each template has to correctly describe some prefix of each
possible execution starting in the entry node. Formally, each template
$(l,M)$ has to satisfy the following two conditions.  Let us consider a
symbolic execution tree of the program $(V,E,l,\iota)$, i.e.~the original
program with $l$ as start location.
\begin{enumerate}
\item Each path in the tree leading from the root to a leaf contains a state
  $(e,\theta,\varphi)$ such that there exists a triple
  $(e,\theta'[\kappa],\varphi'[\kappa])\in M$ and a number $n\ge0$
  satisfying $\theta=\theta'[n]$ and $\varphi\equiv\varphi'[n]$.
\item For each triple $(e,\theta'[\kappa],\varphi'[\kappa])\in M$ and each
  number $n\ge0$ such that $\varphi'[n]$ is satisfiable there exists a
  state $(e,\theta,\varphi)$ in the tree satisfying $\theta=\theta'[n]$ and
  $\varphi\equiv\varphi'[n]$.
\end{enumerate}

The definition of a template is quite general and there are many ways of
obtaining a template. In the rest of this subsection, we present an
algorithm that tries to derive templates from cycles in flowgraphs.

Let $c_0,c_1,\ldots,c_{n-1}$ be locations forming a cycle in a program. We
consider $c_0$ as its entry location. Algorithm~\ref{alg:template} describes
the computation of a template for this cycle. 

\begin{algorithm}[!tb]
  \caption{computeTemplate}\label{alg:template}
  \KwIn{a program cycle $c_0,c_1,\ldots,c_{n-1}$}
  \KwOut{the corresponding template}
  \BlankLine
  \DontPrintSemicolon
Set $\theta_*[\kappa](\var{a})=\bot$ for each scalar variable $\var{a}$ and
\mbox{~~~~~$\theta_*[\kappa](\var{A})=\sym{A}$ for each
  array variable $\var{A}$}\\
$(\theta,\varphi)\aset\texttt{executePath(}c_0,c_1,\ldots,c_{n-1},c_0\texttt{)}$\;
\If{$\varphi\equiv\false$}{\Return{template computation failed}}
\Repeat{$\var{change} = \false$}{ \label{alg:theta:repeat}
  $\var{change} \aset \false$\;
  \ForEach{{\rm scalar variable} $\var{a}$}{ \label{alg:theta:foreach}
    \If{$\theta_*[\kappa](\var{a})=\bot$}{
      \If{$\theta(\var{a})=\sym{a}+c$ for some constant $c$}{
        $\theta_*[\kappa](\var{a}) \aset \sym{a}+\kappa\cdot c$\;
        $\var{change} \aset \true$\;
      }  
      \If{$\theta(\var{a})=\sym{a}\cdot c$ for some constant $c$}{
        $\theta_*[\kappa](\var{a}) \aset \sym{a}\cdot c^\kappa$\;
        $\var{change} \aset \true$\;
      }  
      \If{$\theta(\var{a})=e$ for some symbolic expression $e$ such that
        \mbox{~~~$\theta_*[\kappa](\var{b})\neq\bot$ for each symbol $\sym{b}$ in $e$}}{
        $\theta_*[\kappa](\var{a}) \aset
        \ite(\kappa>0,\theta_*[\kappa-1]\langle e\rangle,\sym{a})$\;
        $\var{change} \aset \true$\;
      }  
    }
  }
}
\If{$\theta_*[\kappa](\var{a})=\bot$ for some variable $\var{a}$}{
  \Return{template computation failed}
}
\Else{
  $\varphi_*[\kappa]\aset\kappa\ge 0~\wedge~
  \forall\tau(0\le\tau<\kappa\implies\theta_*[\tau]\langle\varphi\rangle)$\;
  $M\aset\emptyset$\;
  \ForEach{program path $\pi=c_0,c_1,\ldots,c_i,v$ such that 
    \mbox{~~~~~~~~~~~$i<n$ and $v\neq c_{(i+1)\bmod n}$}}{
    $(\theta,\varphi)\aset\texttt{executePath(}\pi\texttt{)}$\;
    \If{$\varphi\not\equiv\false$}{
      $M\aset M\cup\{(v,\theta_*[\kappa]\langle\theta\rangle,\varphi_*[\kappa]\wedge
      \theta_*[\kappa]\langle\varphi\rangle)\}$\;
    }
  }
  \Return{template $(c_0,M)$}
}
\end{algorithm}

Recall that $\theta\langle\cdot\rangle$ denotes the operation where all
occurences of all symbols $\sym{a}$ are simultaneously replaced by
$\theta(\var{a})$. The algorithm uses a function $\texttt{executePath}(\pi)$
that symbolically executes all instructions along a path $\pi$ in a
flowgraph and returns the resulting symbolic memory and path condition.

The first part of the algorithm (lines 1--18) tries to derive a parametric
symbolic memory $\theta_*[\kappa]$ uniformly describing symbolic memory
after $\kappa$ iterations over the cycle for each $\kappa\ge 0$. Initially,
we set $\theta_*[\kappa]$ to be undefined for all scalar variables. As we
consider only programs that do not modify arrays, we can directly set
$\theta_*[\kappa](\var{A})=\sym{A}$ for each array variable. Then we compute
the effect of one iteration of the cycle and check whether the iteration is
feasible. If it is not, it does not make sense to compute the template and
thus the algorithm stops. In the \textbf{repeat}-\textbf{until} cycle (lines
5--18) we try to derive a parametric description $\theta_*[\kappa]$ of
symbolic memory after $\kappa$ iteration based on the information about
symbolic memory $\theta$ after one iteration. We repeatedly try to apply one
of three simple rules. The first rule (lines 9--11) says that if one
iteration increases the value of a variable $\var{a}$ by a constant $c$, then
$\kappa$ iterations increases the value by $\kappa\cdot c$. Note that this
rule covers also the case when the cycle does not modify $\var{a}$
(i.e.~$c=0$). The second rule (lines 12--14) is similar. The last rule
(lines 15--17) observes that if the cycle sets the the value of $\var{a}$ to
$e$, then the value of $\var{a}$ after $\kappa$ iterations is the same as
the value of $e$ after $\kappa-1$ iterations. Moreover, if we do not perform
any iteration, the value of $\var{a}$ is unchanged (i.e.~$\sym{a}$). If this
process does not lead to a complete description of the symbolic memory after
$\kappa$ iterations, the algorithm fails to find a template for the cycle.

If we have this description, we can compute the template in two
steps. First, we derive a parametric description $\varphi_*[\kappa]$ of the
path condition after $\kappa$ iterations (line 22): using
$\theta_*[\kappa]$, we construct a parametric description of each instance
of conditions on the cycle and we say that $\kappa$ is non-negative and each
of the $\kappa$ instances holds. The template should represent all
executions iterating the cycle for some time and eventually leaving the
cycle by the last step. To detect all ways of leaving the cycle, we find all
acyclic paths $c_0,c_1,\ldots,c_i,v$ starting in the entry location and
leaving the cycle by their last step. Each such a path corresponds to one
triple in the template. To construct the triple, we symbolically execute the
acyclic path to obtain $\theta$ and $\varphi$. We check feasibility of the
acyclic path. The triple represents all executions iterating the cycle
$\kappa$-times and then leaving the cycle along the acyclic path
$c_0,c_1,\ldots,c_i,v$. Hence, each such an execution has the symbolic
memory $\theta_*[\kappa]\langle\theta\rangle$ and its path condition is a
conjunction of $\varphi_*[\kappa]$ and $\varphi$ with values of symbols
defined by $\theta_*[\kappa]$.

We emphasize that one can add other rules to the part of the algorithm
computing the parametric symbolic memory $\theta_*[\kappa]$. For example, a
rule handling the case when $\theta(\var{a})=c\cdot\sym{a}+d$ for some
constants $c,d$.

Besides cyclic paths, recursive calls also produce real program paths
with regularities in program states along them. At
Figure~\ref{fig:linSrchRec}~(a) there is a recursive function
\texttt{linSrchRec} which is equivalent to the function
\texttt{linSrch} discussed before. Symbolic execution tree of the
function is depicted at Figure~\ref{fig:linSrchRec}~(b). The root of
the tree is the left-most vertex referencing program location $a$.
There are two sequences of grey regions. The top sequence represents
recursive calls, while the bottom sequence represents returning from
the calls. We see that top sequence goes from left to the right. The
bottom sequence goes in the opposite direction. We can further see
there is one to one correspondence between regions of both sequences.
Below each region in the top sequence, there is a single region of
bottom sequence. Paths in both sequences of regions are connected in
the tree. But this is not shown in the figure. The connection happens,
when all the recursive calls are done and some basic case is executed
in the recursive function. Then we get to the path of the bottom
regions.

Let us first focus on the path at the top sequence of regions.
Vertices in each region are related to the same sequence of program
locations: $a,b,c,a$. Moreover, we enter the path in a vertex
referencing location $a$ and we can leave the path either by stepping
into a vertex referencing location $f$ or into a vertex referencing
location $e$. If we look at the program (at
Figure~\ref{fig:linSrchRec}~(a)), the sequence $a,b,c,a$ forms a
cyclic path in it. Of course, the edge $(c,a)$ is not explicit in the
program. But we consider it as a meta-edge labelled by an action
simulating the effect of the function call, as defined by action of
edge $(c,d)$. We now define a program part, say $P_1$, consisting of
the cyclic path, the entry location $a$ and two exit locations $f$ and
$e$. The part represent the phase of recursive calls of the function
\texttt{linSrch}.

Now we similarly analyse the path in bottom sequence of regions. Each
region repeats the same sequence of program locations $g,d,g$. The
path is entered in vertex referencing location $g$, but there is no
exit from the path. The sequence $g,d,g$ of locations forms a cyclic
path in the program (at Figure~\ref{fig:linSrchRec}~(a)). Note, that
we assume there is an artificial edge $(g,d)$ enclosing the cycle.
Action of this edge is supposed to simulate the effect of return from
the function call, as defined by action of edge $(c,d)$. We want to
define a program part $P_2$ representing the phase of returning from
recursive calls. We have the cyclic path and we have the entry
location $g$ to the part. But there is no exit from the part.
Obviously, the recursive calls ends in location $g$, where we leave
the function. Therefore, our exit location is $g$ and we have the
program part $P_2$. Note that if we want to formally match the exit
location detection algorithm introduced for previous examples, we may
imagine there is an edge from $g$ back to $g$ and labelled by
\texttt{skip} action.

For the program parts $P_1$ and $P_2$ we compute the following
templates $t_1$ and $t_1$ as described in the previous examples.
\begin{align*}
t_1 &= (a, 2, \{ (\theta_f, \varphi_f, [], f)\prm{\kappa}, (\theta_e,
\varphi_e, [], e)\prm{\kappa} \})
\\
t_2 &= (g, 1, \{ (\theta_g\prm{\kappa}, \true, [], g) \}).
\end{align*}
Note that the path condition of $t_2$ is simply $\true$, since we
cannot escape from the path. In other words, as there is no branching
along the path, the path condition cannot be updated from its initial
value $\true$. It is important to note, that both templates use
exactly the same parameter. The use of the same parameter creates a
link between the number of recursive calls and number of returns from
them. Having the templates we are able to formulate the template $t$
for the recursive function \texttt{linSrchRec}. \[ t = (a, 2, \{
(\theta_f, \varphi_f, [], f)\prm{\kappa}, (\theta_e, \varphi_e, [],
e)\prm{\kappa} \}, \theta_g\prm{\kappa}, g) \] The template $t$
contains whole the template $t_1$, but it took only symbolic memory
$\theta_g$ and the exit location $g$ from the template $t_2$.

We are ready to start compact symbolic execution with the template
$t$. Symbolic execution tree for the program is depicted at
Figure~\ref{fig:linSrchRec}~(c). First we step into the program
location $a$. The tree contains only the root vertex referencing
location $a$. The location $a$ is the entry location of $t$. Hence, we
instantiate the first part of $t$ (related to phase of recursive
calls, i.e.~related to $t_1$) into the tree. The number $2$ in $t$
identifies, that the root will have two successor vertices referencing
locations $f$ and $e$ and they will be labelled by program states $s_f
= (\theta_f, \varphi_f, [(t, \kappa)] \circ [], f)\prm{\kappa}$ and
$s_e = (\theta_e, \varphi_e, [(t, \kappa)] \circ [], e)\prm{\kappa}$
respectively. Note that we omitted composition of these states with
the initial program state labelling the root. We could do that, since
composition of initial program state with any other state produces the
other state again. Also note that call stacks of both states
(i.e.~$[]$) are composed with a call stack containing a single special
record of the form $(t, \kappa)$. This type of call stack record is
introduced only for templates of recursive functions. First of all,
this single record represents any number of subsequent recursive calls
done by classic symbolic execution. And the record also saves
reference to the template $t$ and the parameter $\kappa$ used in the
instantiation. We note that edges from the root to its successors are
labelled by expressions $\gamma_f \equiv t.\varphi_f\prm{\kappa}$ and
$\gamma_e \equiv t.\varphi_e\prm{\kappa}$. Having computed successors
of the root, we continue by classic symbolic execution independently
from both of these vertices, until we reach the location $g$. For both
the executions we do the same think at the location $g$. Let us
consider execution continuing from the successor referencing location
$f$. We need to instantiate the second part of the template
representing returns from the recursive calls. So, we remove the
record $(t, \kappa)$ from the top of the call stack, but we take the
template $t$ and the parameter $\kappa$ stored in the record $(t,
\kappa)$. In general, between both instantiation parts of a given
template, there might be executed any code, there can be instantiated
many other templates and there can even be instantiated the same
template several times always with different (fresh) parameters. That
is why we save the template and the parameter in the stack record. Let
$s_1$ be a program state of the current leaf vertex of the tree. We
create its only successor vertex labelled with program state $s_g^1 =
(s_1.\theta \circ t.\theta\prm{\kappa_1}, s_1.\varphi, [], g)$. We see
that there are two differences between states $s_1$ and $s_g^1$. First
of all call stack of $s_g^1$ does not contain the special record $(t,
\kappa)$ as we have popped it from the stack. And second, the symbolic
memory of $s_g^1$ is the composition $s_1.\theta \circ
t.\theta\prm{\kappa_1}$. Further classic symbolic execution form the
vertex terminates, since we are leaving exit location of the starting
function. We proceed similarly for the other run of symbolic execution
(from the second successor of the root), where we get the final
program state $s_g^2 = (s_2.\theta \circ t.\theta\prm{\kappa},
s_2.\varphi, [], g)$.

When returning from a function call, values of local variables are
discarded. Therefore, if we have two program states at the same exit
location of a function, we may restrict equivalence between symbolic
memories of these states only to global variables. Therefore we define
also the following equivalence between program states.

\begin{dfn}[Equivalence on Global Variables]
Let $s$ and $s'$ be program states. Then $s$ is equivalent on global
variables with $s'$, written by $s \overset{g}{\equiv} s'$, if they
have equal or equivalent components except one with symbolic memories,
where for each global variable \var{a} we require $s.\theta(\var{a})
\equiv s'.\theta(\var{a})$.
\end{dfn}

\begin{dfn}[Recursion template] \label{def:RecursionTemplate}
Let $T$ be symbolic execution tree of $P$ computed by $*$-version of
Algorithm~\ref{alg:executeSymbolically}, $n>0$ be an integer, $l, l',
l_1, \ldots, l_n$ be locations in $P$, $\kappa$ be a parameter,
$\theta\prm{\kappa}, \theta_1\prm{\kappa}, \ldots,$
$\theta_n\prm{\kappa}$ be symbolic memories $\varphi_1\prm{\kappa},
\ldots, \varphi_n\prm{\kappa}$ be satisfiable boolean symbolic
expressions such that for each $i,j \in \{ 1, \ldots, n \}, i \neq j$
we have $(\varphi_i \wedge \varphi_j) \equiv \false$ and let
$\Xi_1\prm{\kappa}, \ldots, \Xi_n\prm{\kappa}$ be call stacks.

A tuple $t = (l, n, \{ (\theta_1, \varphi_1, \Xi_1, l_1), \ldots,
(\theta_n,\varphi_n, \Xi_n, l_n) \})$ is a \emph{template with one
parameter $\kappa$} in $P$, if
\begin{itemize}
\item[(L1)] All the locations $l, l_1, \ldots, l_n$ in $t$ are neither
entry nor exit ones.

\item[(L2)] For each path $\pi = u \omega$ in $T$ from any vertex $u$
satisfying $u.l = t.l$ to a leaf, there is a vertex $w \in \omega$, an
index $i \in \{ 1, \ldots, n \}$ and an integer $\nu \geq 0$, such
that $w.s \equiv u.s \circ
(t.\theta_i,~t.\varphi_i,~t.\Xi_i,~t.l_i)\prm{\nu}$.

\item[(L3)] For each vertex $u$ of $T$, an index $i \in \{ 1, \ldots,
n \}$  and non-negative integer $\nu$ such that $u.l = t.l$ and
$(u.\varphi \wedge u.\theta\ese{t.\varphi_i\prm{\nu}})$ is
satisfiable, there is a successor $w$ of $u$ in $T$ such that $w.s
\equiv u.s \circ (t.\theta_i,~t.\varphi_i,~t.\Xi_i,~t.l_i)\prm{\nu}$.
\end{itemize}

A tuple $t = (l, n, \{ (\theta_1, \varphi_1, \Xi_1, l_1), \ldots,
(\theta_n,\varphi_n, \Xi_n, l_n) \}, \theta, l')$ is a 
\emph{recursion template with one parameter $\kappa$} in $P$, if
\begin{itemize}
\item[(R1)] $t.l$ and $t.l'$ are entry and exit locations of the same
function respectively and $t.l'$ is the target vertex of an edge with
a call action of that function. All the locations $l_1, \ldots, l_n$
in $t$ are neither entry nor exit ones.

\item[(R2)] For each path $\pi = u \omega$ in $T$ from any vertex $u$
satisfying $u.l = t.l$ to a leaf, there is a non-leaf vertex $w \in
\omega$, an index $i \in \{ 1, \ldots, n \}$ and an integer $\nu \geq
0$, such that $w.s \equiv u.s \circ
(t.\theta_i,~t.\varphi_i,~[(t,\kappa)] \circ
t.\Xi_i,~t.l_i)\prm{\nu}$.

Further, if there is the first successor $\bar{w}$ of $w$ in $\pi$
such that $\bar{w}.l = t.l'$ and $\bar{w}.\Xi = w.\Xi$, then there is
a non-leaf vertex $\bar{u}$ in a suffix of $\pi$ starting with
$\bar{w}$ such that $\bar{u}.s \overset{g}{\equiv} (\bar{w}.\theta
\circ t.\theta\prm{\nu}, \bar{w}.\varphi, u.\Xi, t.l')$.

\item[(R3)] For each vertex $u$ of $T$, an index $i \in \{ 1, \ldots,
n \}$  and non-negative integer $\nu$ such that $u.l = t.l$ and
$(u.\varphi \wedge u.\theta\ese{t.\varphi_i\prm{\nu}})$ is
satisfiable, there is a successor $w$ of $u$ in $T$ such that $w.s
\equiv u.s \circ (t.\theta_i,~t.\varphi_i,~[(t,\kappa)] \circ
t.\Xi_i,~t.l_i)\prm{\nu}$.
\end{itemize}
\end{dfn}

Note that requirements (L2) and (R2) guarantees that no path in $T$
with vertices $u$ and $v$ such that $u.l = l$ and $v.l = l_i$ is
suppressed by the state
$(t.\theta_i,~t.\varphi_i,~t.\Xi_i,~t.l_i)\prm{\kappa}$. And
requirement (L3) and (R3) guarantees that program state
$(t.\theta_i,~t.\varphi_i,~t.\Xi_i,~t.l_i)\prm{\kappa}$ does not
produce unreal paths. Also note that in requirement (R2) there we use
restriction of equivalence to global variables for the phase of
returning from recursive calls. Since values of local variables are
not important when returning from a function call, the restriction may
help to simplify detection of a recursion template.

            \If(/* returning from recursion */)
            {\rm \texttt{top(}$s.\Xi$\texttt{)} $ = (t, \kappa)
                    \wedge s.l = s.\Xi.t.l'$}{
                    \label{alg:executeSymbolically:BoxesBeginRec}
                \lmark{$\Box$}
                $t \asgn s.\Xi.t$ \;
                    \label{alg:executeSymbolically:Unwinding}
                \lmark{$\Box$}
                $\kappa \asgn s.\Xi.\kappa$ \;
                    \label{alg:executeSymbolically:UnwindingGetKappa}
                \lmark{$\Box$}
                Replace all occurrences of the former parameter in
                    $t$ by $\kappa$\;
                \lmark{$\Box$}
                $s' \asgn (s.\theta \circ t.\theta\prm{\kappa}, s.\varphi,
                    \texttt{pop(}s.\Xi\texttt{)}, t.l')$ \;
                    \label{alg:executeSymbolically:SuccessorInUnwinding}
                    \label{alg:executeSymbolically:Pop}
                \lmark{$\Box$}
                Insert $s'$ into $S$\;
            }
            \lmark{$\Box$}
            \Else{
                \lmark{$\Box$}
                $p' \asgn \texttt{getTemplatesAt(}s.l, p\texttt{)}$ \;
                    \label{alg:executeSymbolically:GetTemplatesAtRec}
                \lmark{$\Box$}
                \If {$p' \neq \emptyset$}{
                    \lmark{$\Box$}
                    $t \asgn \texttt{chooseTemplate(}p'\texttt{)}$ \;
                        \label{alg:executeSymbolically:ChooseTemplateRec}
                    \lmark{$\Box$}
                    $\kappa \asgn \texttt{getFreshParam()}$ \;
                        \label{alg:executeSymbolically:FreshKappa}
                    \lmark{$\Box$}
                    Replace all occurrences of the former parameter in
                        $t$ by $\kappa$\;
                    \lmark{$\Box$}
                    \If(/* recursive calling */)
                    {\rm $t$ is a recursion template}{
                        \lmark{$\Box$}
                        \ForEach{$i = 1, \ldots, t.n$}{
                                \label{alg:executeSymbolically:Calling}
                            \lmark{$\Box$}
                            $s' \asgn s \circ (t.\theta_i, t.\varphi_i,
                                    [(t,\kappa)] \circ t.\Xi_i,
                                    t.l_i)\prm{\kappa}$\;
                              \label{alg:executeSymbolically:SuccessorInCalling}
                              \label{alg:executeSymbolically:Push}
                            \lmark{$\Box$}
                            Insert $s'$ into $S$\;
                        }
                    }
                    \lmark{$\Box$}
                    \Else(/* $t$ is a general template with one parameter */){
                        \lmark{$\Box$}
                        \ForEach{$i = 1, \ldots, t.n$}{
                                \label{alg:executeSymbolically:UnrollingRec}
                            \lmark{$\Box$}
                            $s' \asgn s \circ (t.\theta_i, t.\varphi_i,
                                               t.\Xi_i, t.l_i)\prm{\kappa}$\;
                            \label{alg:executeSymbolically:SuccessorInUnrollingRec}
                            \lmark{$\Box$}
                            Insert $s'$ into $S$\;
                        }
                    }
                }
                \lmark{$\Box$}
                \Else(/* applying classic symbolic execution step */){
                        \label{alg:executeSymbolically:BoxesEndRec}
                    $S \asgn \texttt{computeClassicSuccessors(}P,s\texttt{)}$ \;
                        \label{alg:executeSymbolically:ClassicStepRec}
                }
            }
            \lmark{$*$}
            Let $u$ be a leaf of $T$ whose label is $s$ \;
                    \label{alg:executeSymbolically:GetLeafRec}
%             \lmark{$*$}
%             /* Destroy label $s$ of vertex $u$ and leave $u$ unlabelled */ \;
            \ForEach{\rm program state $s' \in S$ such that
                            $s'.\varphi$ is satisfiable}{
                    \label{alg:executeSymbolically:AddSATSuccessorsRec}
                Insert $s'$ at the end of $Q$ \;
                \lmark{$*$}
                Insert a new vertex $v$ labeled with $s'$ into $T$ \;
                    \label{alg:executeSymbolically:CreateVertexRec}
                \lmark{$*$}
                Insert an edge $(u,v)$ into $T$ \;
                    \label{alg:executeSymbolically:InsertEdge}
            }
        }
    } \label{alg:executeSymbolically:LoopEndRec}
    \Return{$E$ \\ \lmark{$*$} \hspace{1.05cm} $T$} \;
\end{algorithm}

stack of $i$-th successor state is of the form $s.\Xi \circ [(t,
\kappa)] \circ t.\Xi_i$. It means that the special record is at the
position in the stack, when we entered the recursive function. The
only special record $(t, \kappa)$ in the call stack represents
\emph{any} possible number of subsequent recursive calls in classic

we have to return from recursive calls. We first retrieve the

of the default parameter by the retrieved one, we finish the
instantiation of $t$ by computing the only successor of the actual
state. The successor state represents the effect of all the returns
from recursive calls done previously. This is ensured by using of the
same parameter form both phases of the instantiation of the template
$t$. A number of recursive calls therefore matches the number of
returns form them. Also note that call stack of the successor does not
contain the special record. We finish the description of the algorithm

line~\ref{alg:executeSymbolically:SuccessorInCallingRec}. And this
program state had to be related to the entry location of a function
causing the recursive calls.


\subsubsection{Recursion Templates Computation}

Let $P$ be a program, $f$ be a recursive function of $P$, $e$ and $x$
be entry and exit locations of $f$ respectively and let $h = (u,v)$ be
an edge of $f$ with an action representing recursive call of $f$. We
transform computation of recursion template for recursive calling of
$f$ into analysis of two program parts $P_1$ and $P_2$ with cyclic
paths. The cycle of $P_1$ starts at location $e$ and leads to $u$. We
then enclose the cycle by an artificial edge whose action simulate an
effect of any call of $f$. Let $e$ be entry location of $P_1$ and let
$x_1, \ldots, x_n$ be its exit locations. We compute a template $t_1 =
(e, n, \{ (\theta_1, \varphi_1, \Xi_1, x_1)\prm{\kappa}, \ldots,
(\theta_n, \varphi_n, \Xi_n, x_n)\prm{\kappa} \})$ for $P_1$ according
to algorithm from Section~\ref{sec:Detection:Cycles}. Having $t_1$ we
can express the resulting recursive template $t$ as follows. \[ t =
(e, n, \{ (\theta_1, \varphi_1, \Xi_1, x_1)\prm{\kappa}, \ldots,
(\theta_n, \varphi_n, \Xi_n, x_n)\prm{\kappa} \}, \theta\prm{\kappa},
x), \] where $\theta\prm{\kappa}$ is the only unknown component in
$t$. We compute the symbolic memory $\theta$ from analysis of the
second program part $P_2$. The cycle of $P_2$ starts at $x$. There we
add an artificial edge, whose action simulate an effect of return from
any call of $f$. The artificial edge gets us to location $v$. Then we
enclose the cycle by following a path from $v$ to $x$. We set $x$ to
be the entry location of $P_2$ and we further set $x$ to also be the
only exit location from $P_2$. As you can see, here we have introduced
an assumption that there is no branching along the path from $v$ to
$x$, i.e.~we cannot escape from the path. We discuss the case, when
there is some branching (escape edges) along the path later. Since we
have defined the program part $P_2$, we compute its template $t_2 =
(x, 1, \{ (\theta\prm{\kappa}, \true, [], x) \})$ according to
algorithm from Section~\ref{sec:Detection:Cycles}. Then we take the
symbolic memory $\theta\prm{\kappa}$ and we complete the recursion
template $t$.

\subsection{Extensions of the technique}

For clarity of presentation, the compact symbolic execution has been
described in a very simple settings of programs working only with integer
variables, read-only integer arrays, and without function calls. However,
one can extend the technique to programs with more data types, function
calls allowing recursion etc. Further, one can improve the presented
technique for computing templates (e.g.~by addition of new rules) or suggest
a completely new algorithms for template construction. The definition of
templates can be also extended. For example, one can also consider templates
with more than one parameter.

In~\cite{Trt12} we present a version of this technique working with function
calls. Moreover, we introduce there a notion of paired templates that can
capture some kinds of (unbounded) recursion.

\section{Experimental Results}\label{sec:experiments}

We have implemented both classic and compact symbolic execution in an
experimental tool to empirically evaluate their effectiveness in
exploration of program paths. In particular, we want to measure, for
each technique, a speed in which they explore program paths. Given the
same fixed exploration method of program paths
% (e.g.~in breadth-first manner)
for both techniques, we may ask how quickly they are able to
reach a particular location in the analysed program.

Therefore, we fixed the breadth-first search of program paths for both
techniques and we measured their running time on ten benchmark
programs with a marked target location, until the location is reached
or until 15 minutes timeout is exceeded. Note that for compact symbolic
execution we also have to fix a template-selection strategy to be
instantiated at a particular program location, since there can
generally be more than one template related to the same program
location. We always choose randomly between candidate templates. All
the benchmarks have a huge number of real program paths, generated by
loops. The target location is chosen to be difficult to reach. A
technique must therefore explore a lot of program paths to get to the
right one. The first seven benchmarks have reachable target location,
while the last tree do not. For these three benchmark all the
program paths must be explored to give an answer. The individual
benchmarks can be described as follows (\texttt{C} listing of majority
of the benchmarks can be found in the Appendix).

The first three benchmarks Hello, HW, and HWM are similar. The
benchmark HWM was taken from \cite{AGT08} and it accepts a \texttt{C}
string as an input and scans the string for the presence of substrings
\texttt{"Hello"},\texttt{"World"}, \texttt{"At"} and
\texttt{"Microsoft!"}. It does not matter at which position and in
which order the words occur in the string. The target location is
reached when all four substrings are found. Benchmark HW searches an
input string only for substrings \texttt{Hello} and \texttt{World},
while benchmark Hello searches only for the first one.

Benchmark MatrIR scans upper triangle of an input matrix. The target
location is reached if the matrix is bigger than 20 $\times$ 20 and it
contains a line with more than 15 scanned elements with values between 10 and
100. We took this benchmark from~\cite{ST12}.

Benchmark DOIF scans an input string and counts number of characters
\texttt{U} and \texttt{V}. The target location is reached when the counts
have the exact values $10$ and $15$ respectively.

Benchmark DOIFex extends DOIF by another loop scanning the input
stream again. Occurrences of characters \texttt{u} and \texttt{v}
decreases the counts computed in the first loop for the characters
\texttt{U} and \texttt{V} respectively. The target location is
reached when the resulting counts are the same as described in DOIF
benchmark.

Benchmark EQCNT originate from \cite{OT11}. It consists of two nested
loops with branching, where a variable defined in the outermost scope
is incremented in the innermost loop. The benchmark accepts two
strings, \texttt{A} and \texttt{B}. For each character in string
\texttt{A} we count the number of times this character occurs in
\texttt{B}. The target is reached when the overall count is exactly
20.

Benchmark EQCNTex extends the benchmark EQCNT. It contains two additional
nested loops in row. Nevertheless, the bounds of all loops in the benchmarks
are set to fixed constants. The huge number of program paths is generated by
the symbolic arrays \texttt{A} and \texttt{B} described above. The limits of
the loops makes the target location unreachable.

Benchmarks OneLoop and TwoLoops are also designed such that their target
locations are not reachable. Both benchmarks contain a loop iterated
\texttt{n}–times. In each iteration, the variable \texttt{i} (initially set
to \texttt{0}) is increased by \texttt{4}. The target location is then
guarded by an assertion \texttt{i==15} in OneLoop and by a loop
\texttt{while (i != j + 7) j += 2} in TwoLoops (\texttt{j} is initialised to
\texttt{0} before the loop).

\section{Related Work}\label{sec:related}

Compact symbolic execution is tightly related to the work of King in
1976~\cite{Kin76}, where the author introduced the original concept of
classic symbolic execution. Nevertheless, performance issues related to the
path explosion problem were not tackled. The original concept was
generalised in \cite{KPV03} for programs with heap by introducing lazy
initialisation of dynamically allocated data structures.  The lazy
initialisation algorithm was further improved and formally defined
in~\cite{DLR11}. Another generalisation step was done in~\cite{KS05}, where
the authors attempt to avoid symbolic execution of library code (called from
an analysed program), since such code can be assumed as well defined and
properly tested.

In \cite{SPmCS09,GL11} the path explosion problem is tackled by focusing on
program loops. The information inferred from a loop allows to talk about
multiple program paths through that loop. But the goal is to explore classic
symbolic execution tree in some effective manner: more interesting paths
sooner. Approaches \cite{G07,AGT08} shares the same goal as the previous
ones. Only, they focus on a computation of function summaries rather than on
program loops. Our goal is completely different: Instead of guiding
exploration of paths in classic symbolic execution tree, we collapse paths
(with regularities in states along them) into single vertices in compact
symbolic execution tree. Paths in compact symbolic execution tree can thus
be exponentially shorter than corresponding paths in the classic one.
% The main disadvantage of the approaches \cite{SPmCS09,GL11,G07,AGT08} over
% ours is the exploration of paths of \emph{classic} symbolic execution
% tree. These paths can be exponentially longer then corresponding ones in
% compact symbolic execution tree. The same negative effect therefore
% applies to lengths of the corresponding path conditions of these paths.

Further, techniques~\cite{QNR11,SH10} group paths of classic symbolic
execution tree according to an a priori given constraints to a program
output, while exploring only one path per group. We do not require any such
constraints.

Note that the compact symbolic execution has a different objective than
standard summarization techniques~\cite{G07,AGT08,GL11}. Summarization
basically ``caches'' results of some finite part of symbolic execution. The
presented technique wants to generalize the precomputed information and
replace potentially infinite parts of symbolic executions by a single node.

%{{{ Conclusion 

\section{Conclusion}\label{sec:conclusion}

We have introduced a generalization of classic symbolic execution. The new
technique called compact symbolic execution can analyze a program and derive
templates representing an overall effect of some program parts to executions
leading through the program part. Using such a template, one can construct
substantially smaller tree that the classic symbolic execution tree without
loss of information. In particular, one node of the compact tree can
represent even infinitely many nodes of the classic tree.

Our experimental results show that construction of compact trees instead of
classic trees often leads to significantly faster detection of errors in
programs.

\section{Proofs of Theorems}\label{sec:Proofs}

In this section we prove soundness and completeness theorems presented
at the end of Section~\ref{sec:Definition}. We assume for the
remainder of this section that $P$ is a program, and further that $T$
and $T'$ are classic and compact symbolic execution trees of the
program $P$ computed by Algorithm~\ref{alg:executeSymbolically}
without and with $\Box$ lines respectively.

\setcounter{thm}{0}

\begin{thm}[Soundness]
For each leaf vertex $e \in T$ there is a leaf vertex $e' \in T'$ and
a valuation  $\bld{\nu}$ of parameters in $e'.s$ such that $e.s \equiv
e'.s\prm{\bld{\nu}}$.
\end{thm}
\begin{proof}
Let $\pi$ be the path in $T$ from the root to the leaf vertex $e$. We
prove the theorem by the following induction:

\emph{Basic case}: The root vertices $r$ and $r'$ of $T$ and $T'$
respectively are labelled by the same program state $s_0$ (see lines
\ref{alg:executeSymbolically:InitialState} and
\ref{alg:executeSymbolically:Root} of
Algorithm~\ref{alg:executeSymbolically}). Therefore, $r.s \equiv
r'.s\prm{\bld{\nu}}$, for $\bld{\nu} = \emptyset$.

\emph{Inductive step}: Let $u \in \pi$, $u \neq e$, $u'$ be a vertex
of $T'$, and $\bld{\nu}$ be a valuation such that $u.s \equiv
u'.s\prm{\bld{\nu}}$. We show, there is a successor $w$ of $u$ in
$\pi$, a successor vertex $w'$ of $u'$ in $T'$, and a valuation
$\bld{\nu}'$ such that $w.s \equiv w'.s\prm{\bld{\nu}'}$. There are
two possible cases in Algorithm~\ref{alg:executeSymbolically} for
$u'.s$:

(1) We reach line~\ref{alg:executeSymbolically:Unrolling}: According
to Definition~\ref{def:LoopTemplate}~(L1), there is a successor
vertex $w$ of $u$ in $\pi$, an index $i$, and a non-negative integer
$\nu$ for $\kappa$ such that
\begin{align*}
w.s &\equiv u.s \circ (t.\theta_i\prm{\kappa},
t.\varphi_i\prm{\kappa}, t.\Xi_i\prm{\kappa},
t.l_i)\prm{\{(\kappa,\nu)\}}
\\
&\equiv u'.s\prm{\bld{\nu}} \circ (t.\theta_i\prm{\kappa},
t.\varphi_i\prm{\kappa}, t.\Xi_i\prm{\kappa},
t.l_i)\prm{\{(\kappa,\nu)\}}
\\
&\equiv (u'.s \circ (t.\theta_i\prm{\kappa}, t.\varphi_i\prm{\kappa},
t.\Xi_i\prm{\kappa}, t.l_i))\prm{\bld{\nu} \cup \{(\kappa,\nu)\} }
\\
&\equiv s'\prm{\bld{\nu}'},
\end{align*}
where $s'$ is the $i$-th direct successor of $u'.s$ computed at
line~\ref{alg:executeSymbolically:SuccessorInUnrolling}. And since $w
\in T$, we have $w.\varphi$ is satisfiable. Therefore, there is a
direct successor $w'$ of $u'$ in $T'$ with $w'.s = s'$.

(2) Otherwise, we reach
line~\ref{alg:executeSymbolically:ClassicStep}: Since $u.s \equiv
u'.s\prm{\bld{\nu}}$ and we apply classic symbolic execution step for
$u'.s$, there must be a direct successor $w$ of $u$ and a direct
successor $w'$ of $u'$ such that $w.s \equiv w'.s\prm{\bld{\nu'}}$,
where $\bld{\nu'} = \bld{\nu}$.
\qed
\end{proof}

\begin{thm}[Completeness]
For each leaf vertex $e' \in T'$ there is a leaf vertex $e \in T$ and
a valuation $\bld{\nu}$ of parameters in $e'.s$ such that $e.s \equiv
e'.s\prm{\bld{\nu}}$.
\end{thm}
\begin{proof}
Let $\pi'$ be the path in $T'$ from the root to the leaf vertex $e'$.
We prove the theorem by the following induction:

\emph{Basic case}: The root vertices $r$ and $r'$ of $T$ and $T'$
respectively are labelled by the same program state $s_0$ (see lines
\ref{alg:executeSymbolically:InitialState} and
\ref{alg:executeSymbolically:Root}). Let us construct a non-empty set
$U$ of vertices of $T$ such that for each valuation $\bld{\nu}$ of
parameters in $r'.s$ such that $r'.\varphi\prm{\bld{\nu}}$ is
satisfiable, there is $u \in U$ such that $u.s \equiv
r'.s\prm{\bld{\nu}}$. Obviously $U = \{ r \}$, because $r'.\varphi$
contains no parameter (so $r.s \equiv r'.s\prm{\bld{\nu}}$, for each
$\bld{\nu})$.

\emph{Inductive step}:  Let $u' \in \pi'$, $u' \neq e'$ and $U$ be a
non-empty set of vertices of $T$ such that for each valuation
$\bld{\nu}$ of parameters in $u'.s$ such that
$u'.\varphi\prm{\bld{\nu}}$ is satisfiable, there is $u \in U$ such
that $u.s \equiv u'.s\prm{\bld{\nu}}$. We show, there is a successor
$w'$ of $u'$ in $\pi'$ and a non-empty set $W$ of vertices of $T$ such
that for each valuation $\bld{\nu}'$ of parameters in $w'.s$ such that
$w'.\varphi\prm{\bld{\nu}'}$ is satisfiable, there is $w \in W$ such
that $w.s \equiv w'.s\prm{\bld{\nu}'}$. And we further show that each
$w \in W$ is a successor of some $u \in U$. There are two possible
cases in Algorithm~\ref{alg:executeSymbolically} for $u'.s$:

(1) We reach line~\ref{alg:executeSymbolically:Unrolling}: Let $w'$ be
a direct successor of $u'$ in $\pi'$. Obviously, $w'.s$ is one of the
states $s'$ computed at
line~\ref{alg:executeSymbolically:SuccessorInUnrolling}. Let $i$ be
the index, for which $w'.s = s'$. The formula $w'.\varphi$ is
satisfiable, since $w'$ is in $T'$ (see condition at
line~\ref{alg:executeSymbolically:AddSATSuccessors}). Let $\bld{\nu}$
be a valuation for which $w'.\varphi$ is satisfiable. And let
$\bld{\nu}' = \bld{\nu} \smallsetminus \{ (\kappa,\nu) \}$, where
$\nu$ is an integer assigned in $\bld{\nu}$ to the fresh parameter
$\kappa$ introduced at
line~\ref{alg:executeSymbolically:UnrollingFreshKappa}. From
line~\ref{alg:executeSymbolically:SuccessorInUnrolling} we see that
$u'.\varphi\prm{\bld{\nu}'}$ is satisfiable. Therefore, there is a
vertex $u \in U$ such that $u.s \equiv u'.s\prm{\bld{\nu}'}$.
According to Definition~\ref{def:LoopTemplate}~(L2) there is a
successor $w$ of $u$ in $T$ such that
\begin{align*}
w.s &\equiv u.s \circ (t.\theta_i\prm{\kappa},
t.\varphi_i\prm{\kappa}, t.\Xi_i\prm{\kappa},
t.l_i)\prm{\{(\kappa,\nu)\}}
\\
&\equiv u'.s\prm{\bld{\nu}'} \circ (t.\theta_i\prm{\kappa},
t.\varphi_i\prm{\kappa}, t.\Xi_i\prm{\kappa},
t.l_i)\prm{\{(\kappa,\nu)\}}
\\
&\equiv (u'.s \circ (t.\theta_i\prm{\kappa}, t.\varphi_i\prm{\kappa},
t.\Xi_i\prm{\kappa}, t.l_i))\prm{\bld{\nu}}
\\
&\equiv w'.s\prm{\bld{\nu}}.
\end{align*}
Therefore, $w \in W$.

(2) Otherwise, we reach
line~\ref{alg:executeSymbolically:ClassicStep}: Let $u$ be any vertex
in $U$. Since $u.s \equiv u'.s\prm{\bld{\nu}}$ for some valuation
$\bld{\nu}$ for which $u'.\varphi\prm{\bld{\nu}}$ is satisfiable, and
since all direct successors of both $u$ and $u'$ are computed by
classic symbolic execution step, there must be a direct successor $w$
of $u$ in $T$ and a direct successor $w'$ of $u'$ in $T'$ such that
$w.s \equiv w'.s\prm{\bld{\nu'}}$, where $\bld{\nu'} = \bld{\nu}$.
% Note that both $u'.s$ and $w'.s$ have exactly the same parameters.
Therefore, $w \in W$.
\qed
\end{proof}

\begin{lem}[Equivalent Compositions]
Let $s, s'$, and $s''$ be program states, $\bld{\nu}$ and $\bld{\nu}'$
be valuations of parameters in $s$ and $s'$ respectively such that
$\bld{\nu} \cup \bld{\nu}'$ is also a valuation, $\theta$, $\theta'$,
and $\theta''$ be symbolic memories, and $\varphi$ and $\psi \wedge
\psi'$ be symbolic expressions. Then $s \circ (s' \circ s'') \equiv (s
\circ s') \circ s''$, $s\prm{\bld{\nu}} \circ s'\prm{\bld{\nu}'}
\equiv (s \circ s')\prm{\bld{\nu} \cup \bld{\nu}'}$, $\theta \circ
(\theta' \circ \theta'') \equiv (\theta \circ \theta') \circ
\theta''$, $(\theta \circ \theta')\ese{\varphi} \equiv
\theta\ese{\theta'\ese{\varphi}}$, and $\theta\ese{\psi} \wedge
\theta\ese{\psi'} \equiv \theta\ese{\psi \wedge \psi'}$.
\end{lem}
\begin{proof}
We split the proof of the lemma into separate points related to a
particular equivalences.

(1) $(\theta \circ \theta')\ese{\varphi} \equiv
\theta\ese{\theta'\ese{\varphi}}$. The expression $(\theta \circ
\theta')\ese{\varphi}$ simultaneously substitutes each symbol
$\alpha_i$ in $\varphi$ by a symbolic expression $(\theta \circ
\theta') \ese{\Theta^{-1}(\alpha_i)} =
\theta\ese{\theta'(\Theta^{-1}(\alpha_i))}$ This follows directly from
Definition~\ref{def:Composition}. In the expression
$\theta\ese{\theta'\ese{\varphi}}$, there we have to apply the
substitution twice. First we simultaneously substitute each symbol
$\alpha_i$ in $\varphi$ by a symbolic expression
$\theta'(\Theta^{-1}(\alpha_i))$. If the resulting formula contains
any symbol, then it must necessarily lie in some of the substituted
expressions $\theta'(\Theta^{-1}(\alpha_i))$. Therefore, it is
sufficient to apply the second substitution only to the substituted
expressions $\theta'(\Theta^{-1}(\alpha_i))$. In other words, it is
sufficient to apply only one simultaneous substitution of symbols
$\alpha_i$ in $\varphi$ by symbolic expressions
$\theta\ese{\theta'(\Theta^{-1}(\alpha_i))}$.

(2) $\theta\ese{\psi} \wedge \theta\ese{\psi'} \equiv \theta\ese{\psi
\wedge \psi'}$. The operation $\theta\ese{\cdot}$ only applies symbol
substitutions inside formulae $\psi$ and $\psi'$.

(3) $s \circ (s' \circ s'') \equiv (s \circ s') \circ s''$. Let $s$,
$s'$, and $s''$ be program states $(\theta, \varphi, \Xi, l)$,
$(\theta', \varphi', \Xi', l')$, and $(\theta'', \varphi'', \Xi'',
l'')$ respectively. According to Definition~\ref{def:Composition}, we
have $s \circ (s' \circ s'') = (\theta \circ (\theta' \circ
\theta''),~\varphi \wedge \theta\ese{\varphi' \wedge
\theta'\ese{\varphi''}}, \Xi \circ (\theta \circ (\Xi' \circ (\theta'
\circ \Xi''))), l'')$ and $(s \circ s') \circ s'' = ((\theta \circ
\theta') \circ \theta'',~(\varphi \wedge \theta\ese{\varphi'}) \wedge
(\theta \circ \theta')\ese{\varphi''}, (\Xi \circ (\theta \circ \Xi'))
\circ ((\theta \circ \theta') \circ \Xi'')), l'')$. We prove the
equivalence of the composed states per component in points (3a), (3b),
and (3c).

(3a) $\theta \circ (\theta' \circ \theta'') \equiv (\theta \circ
\theta') \circ \theta''$. According to
Definition~\ref{def:Composition} we have for each variable \var{a} the
following: $(\theta \circ (\theta' \circ \theta''))(\var{a}) \equiv
\theta\ese{(\theta' \circ \theta'')(\var{a})} \equiv
\theta\ese{\theta'\ese{\theta''(\var{a})}} \equiv (\theta \circ
\theta')\ese{\theta''(\var{a})} \equiv ((\theta \circ \theta') \circ
\theta'')(\var{a})$.

(3b) $\varphi \wedge \theta\ese{\varphi' \wedge
\theta'\ese{\varphi''}} \equiv (\varphi \wedge \theta\ese{\varphi'})
\wedge (\theta \circ \theta')\ese{\varphi''}$. According to points (1)
and (2) we have $(\varphi \wedge \theta\ese{\varphi'}) \wedge (\theta
\circ \theta')\ese{\varphi''} \equiv \varphi \wedge
\theta\ese{\varphi'} \wedge \theta\ese{\theta'\ese{\varphi''}} \equiv
\varphi \wedge \theta\ese{\varphi' \wedge \theta'\ese{\varphi''}}$.

(3c) $\Xi \circ (\theta \circ (\Xi' \circ (\theta' \circ \Xi'')))
\equiv (\Xi \circ (\theta \circ \Xi')) \circ ((\theta \circ \theta')
\circ \Xi''))$. Let $\Xi = [r_0, \ldots, r_m]$, $\Xi' = [r_0', \ldots,
r_n']$, and $\Xi'' = [r_0'', \ldots, r_k'']$. Then a symbolic memory
of the $i$-th record ($i \in \{0, \ldots, m \}$) in a call stack
$(\theta \circ \theta') \circ \Xi$ has a form $(\theta \circ \theta')
\circ r_i.\sigma = (\theta \circ (\theta' \circ r_i.\sigma)$ according
to Definition~\ref{def:Composition} and point (3a). But $(\theta \circ
(\theta' \circ r_i.\sigma)$ is actually a symbolic memory of the
$i$-th record of a call stack $\theta \circ (\theta' \circ \Xi)$.
Therefore, from Definition~\ref{def:Composition} we infer $(\theta
\circ \theta') \circ \Xi'' \equiv \theta \circ (\theta' \circ \Xi'')$.
Further, $(\theta \circ \Xi) \circ (\theta \circ \Xi') \equiv [(\theta
\circ r_0.\sigma, r_0.l), \ldots, (\theta \circ r_m.\sigma, r_m.l),
(\theta \circ r_0'.\sigma, r_0'.l), \ldots, (\theta \circ r_n'.\sigma,
r_n'.l) ] \equiv \theta \circ (\Xi \circ \Xi')$. Finally, from
Definition~\ref{def:Composition}  immediately follows associativity
$\Xi \circ (\Xi' \circ \Xi'') \equiv (\Xi \circ \Xi') \circ \Xi''$.
Using the equivalences above,  we can finally prove the equivalence
(3c) as follows: $(\Xi \circ (\theta \circ \Xi')) \circ ((\theta \circ
\theta') \circ \Xi'') \equiv \Xi \circ (\theta \circ \Xi') \circ
(\theta \circ (\theta' \circ \Xi'')) \equiv \Xi \circ (\theta \circ
(\Xi' \circ (\theta' \circ \Xi'')))$.

(4) $s\prm{\bld{\nu}} \circ s'\prm{\bld{\nu}'} \equiv (s \circ
s')\prm{\bld{\nu} \cup \bld{\nu}'}$. The equivalence follows from
these two facts: (a) The composition of states operates on symbols,
while the parameter substitution operates on parameters. (b)
$\bld{\nu} \cup \bld{\nu}'$ is supposed to be a valuation. Therefore,
if there is $(\kappa, \nu) \in \bld{\nu}$ and $(\kappa, \nu') \in
\bld{\nu}'$, then $\nu = \nu'$ must be valid.
\qed
\end{proof}

\begin{lem} \label{lem:GEquivToEquiv}
Let $u \in T$, $u' \in T'$, $u'.\Xi \neq []$, {\tt top(}$u'.\Xi${\tt
)} $ = (t, \kappa)$, $u'.l$ is an exit location and $u.s

\end{lem}
\begin{proof}
Follows directly from Lemma~\ref{lem:NonAdjacentPushes} and from the

\begin{thm}[Soundness]
    For each leaf vertex $e \in T$ there is a leaf vertex $e' \in T'$ and a valuation  $\bld{\nu}$ of all parameters in $e'.s$ such that $e.s \equiv e'.s\prm{\bld{\nu}}$.
\end{thm}
\begin{proof}
    Let $\pi$ be the path in $T$ from the root to the leaf vertex $e$. We prove the theorem by the following induction:

    \emph{Basic case}: The root vertices $r$ and $r'$ of $T$ and $T'$ respectively are labelled by the same program state $s_0$ (see lines \ref{alg:executeSymbolically:InitialState} and \ref{alg:executeSymbolically:Root}). So, $r.s \equiv r'.s\prm{\bld{\nu}}$, for $\bld{\nu} = \emptyset$.

    \emph{Inductive step}: Let $u \in \pi$, $u \neq e$, $u'$ be a vertex of $T'$ and $\bld{\nu}$ be a valuation such that $u.s \equiv u'.s\prm{\bld{\nu}}$. We show, there is a successor $w$ of $u$ in $\pi$, a successor vertex $w'$ of $u'$ in $T'$ and a valuation $\bld{\nu}'$ such that $w.s \equiv w'.s\prm{\bld{\nu}'}$. And we further show there is no vertex $v'$ in the path between $u'$ and $w'$ in $T'$ such that successors of $v'.s$ are computed at line~\ref{alg:executeSymbolically:SuccessorInCalling}. There are four possible cases in Algorithm~\ref{alg:executeSymbolically} for $u'.s$:

    (1) We reach line~\ref{alg:executeSymbolically:Unrolling}: According to Definition~\ref{def:Templates}~(L2), there is a successor vertex $w$ of $u$ in $\pi$, an index $i$ and a non-negative integer $\nu$ for $\kappa$ such that
    \begin{align*}
        w.s &\equiv u.s \circ (t.\theta_i\prm{\kappa}, t.\varphi_i\prm{\kappa}, t.\Xi_i\prm{\kappa}, t.l_i)\prm{\{(\kappa,\nu)\}}
        \\
        &\equiv u'.s\prm{\bld{\nu}} \circ (t.\theta_i\prm{\kappa}, t.\varphi_i\prm{\kappa}, t.\Xi_i\prm{\kappa}, t.l_i)\prm{\{(\kappa,\nu)\}}
        \\
        &\equiv (u'.s \circ (t.\theta_i\prm{\kappa}, t.\varphi_i\prm{\kappa}, t.\Xi_i\prm{\kappa}, t.l_i))\prm{\bld{\nu} \cup \{(\kappa,\nu)\} }
        \\
        &\equiv s'\prm{\bld{\nu}'},
    \end{align*}
    where $s'$ is the $i$-th direct successor of $u'.s$ computed at line~\ref{alg:executeSymbolically:SuccessorInUnrolling}. And since $w \in T$, we have $w'.\varphi$ is satisfiable. Therefore, there is be a direct successor $w'$ of $u'$ in $T'$ with $w.s = s'$.

    (2) We reach line~\ref{alg:executeSymbolically:Calling}: According to Definition~\ref{def:Templates}~(R2), there is a successor vertex $w$ of $u$ in $\pi$, an index $i$ and a non-negative integer $\nu$ for $\kappa$ such that
    \begin{align*}
        w.s &\equiv u.s \circ (t.\theta_i\prm{\kappa}, t.\varphi_i\prm{\kappa}, [(t,\kappa)] \circ t.\Xi_i\prm{\kappa}, t.l_i)\prm{\{(\kappa,\nu)\}}
        \\
        &\equiv u'.s\prm{\bld{\nu}} \circ (t.\theta_i\prm{\kappa}, t.\varphi_i\prm{\kappa}, [(t,\kappa)] \circ t.\Xi_i\prm{\kappa}, t.l_i)\prm{\{(\kappa,\nu)\}}
        \\
        &\equiv (u'.s \circ (t.\theta_i\prm{\kappa}, t.\varphi_i\prm{\kappa}, [(t,\kappa)] \circ t.\Xi_i\prm{\kappa}, t.l_i))\prm{\bld{\nu} \cup \{(\kappa,\nu)\} },
        \\
        &\equiv s'\prm{\bld{\nu}'},
    \end{align*}
    where $s'$ is the $i$-th direct successor of $u'.s$ computed at line~\ref{alg:executeSymbolically:SuccessorInCalling}. And since $w \in T$, we have $w'.\varphi$ is satisfiable. Therefore, there is a direct successor $w'$ of $u'$ in $T'$ with $w.s = s'$.

    (3) We reach line~\ref{alg:executeSymbolically:Unwinding}: 
    Let $\pi'$ be a path in $T'$ from the root to the vertex $u'$. According to connections between vertices $u'$ constructed for vertices $u$ along $\pi$, there is a predecessor $x'$ of $u'$ in $\pi'$, which pushed (at line~\ref{alg:executeSymbolically:Push}) the record being at the top of $u'.\Xi$. Obviously, successors of $x'.s$ are computed at line~\ref{alg:executeSymbolically:SuccessorInCalling}. Therefore, there is $x \in \pi$ such that $x.s \equiv x'.s\prm{\bld{\nu}}$. According to case (2) there is a successor $y$ of $x$ in $\pi$ and a direct successor $y'$ of $x'$ in $\pi'$ such that $y.s \equiv y'.s\prm{\bld{\nu}}$. Note that $y'.s$ uses the parameter $\kappa$ retrieved from stack $u'.\Xi$ at line~\ref{alg:executeSymbolically:UnwindingGetKappa}. Therefore, valuation $\bld{\nu}$ defines an integer $\nu = \bld{\nu}(\kappa)$. Also note that $u$ is the first successor of $y$ in $\pi$ with $u.l$ being an exit location and $u.\Xi = y.\Xi$. Otherwise we would apply this case (3) for some other vertex lying between $y'$ and $u'$ in $\pi'$. Therefore, from Definition~\ref{def:Templates}~(R2) there is a non-leaf vertex $v$ in a suffix of $\pi$ starting with $u$ such that
    \begin{align*}
        v.s &\overset{g}{\equiv} (u.\theta \circ t.\theta\prm{\kappa}, u.\varphi, x.\Xi, t.l')\prm{\{(\kappa,\nu)\}}
        \\
        &\overset{g}{\equiv} (u'.\theta\prm{\bld{\nu}} \circ t.\theta\prm{\kappa}, u'.\varphi\prm{\bld{\nu}}, \texttt{pop(}u'.\Xi\texttt{)}\prm{\bld{\nu}}, t.l')\prm{\{(\kappa,\nu)\}}
        \\
        &\overset{g}{\equiv} (u'.\theta \circ t.\theta\prm{\kappa}, u'.\varphi, \texttt{pop(}u'.\Xi\texttt{)}, t.l')\prm{\bld{\nu}}
        \\
        &\overset{g}{\equiv} s'\prm{\bld{\nu}},
    \end{align*}
    where $s'$ is the only successor state of $u'.s$ computed at line~\ref{alg:executeSymbolically:SuccessorInUnwinding}. Since $v \in T$, then $s'.\varphi$ is satisfiable and there is a direct successor $v'$ of $u'$ in $T'$ with $v'.s = s'$. And finally Lemma~\ref{lem:GEquivToEquiv} ensures there are the only direct successors $w$ and $w'$ of $v$ and $v'$ respectively, such that $w.s \equiv w'.s\prm{\bld{\nu}}$.

    (4) Otherwise, we reach line~\ref{alg:executeSymbolically:ClassicStep}: Since $u.s \equiv u'.s\prm{\bld{\nu}}$ and we apply classic symbolic execution step for $u'.s$, there must be a direct successor $w$ of $u$ and a direct successor $w'$ of $u'$ such that $w.s \equiv w'.s\prm{\bld{\nu}}$.
\end{proof}

The formula $w'.\varphi$ is satisfiable, since $w'$ is in $T'$ (see condition at line~\ref{alg:executeSymbolically:AddSATSuccessors}). Let $\bld{\nu}$ be a valuation for which $w'.\varphi$ is satisfiable. And let $\bld{\nu}' = \bld{\nu} \smallsetminus \{ (\kappa,\nu) \}$, where $\nu$ is an integer assigned in $\bld{\nu}$ to the fresh parameter $\kappa$ introduced at line~\ref{alg:executeSymbolically:FreshKappa}. From line~\ref{alg:executeSymbolically:SuccessorInCalling} we see that $u'.\varphi\prm{\bld{\nu}'}$ is satisfiable. Therefore, there is a vertex $u \in U$ such that $u.s \equiv u'.s\prm{\bld{\nu}'}$. According to Definition~\ref{def:Templates}~(R3) there is a successor $w$ of $u$ in $T$ such that
    \begin{align*}
        w.s &\equiv u.s \circ (t.\theta_i\prm{\kappa}, t.\varphi_i\prm{\kappa}, [(t,\kappa)] \circ t.\Xi_i\prm{\kappa}, t.l_i)\prm{\{(\kappa,\nu)\}}
        \\
        &\equiv u'.s\prm{\bld{\nu}'} \circ (t.\theta_i\prm{\kappa}, t.\varphi_i\prm{\kappa}, [(t,\kappa)] \circ t.\Xi_i\prm{\kappa}, t.l_i)\prm{\{(\kappa,\nu)\}}
        \\
        &\equiv (u'.s \circ (t.\theta_i\prm{\kappa}, t.\varphi_i\prm{\kappa}, [(t,\kappa)] \circ t.\Xi_i\prm{\kappa}, t.l_i))\prm{\bld{\nu}}
        \\
        &\equiv w'.s\prm{\bld{\nu}}.
    \end{align*}
    Therefore, $w \in W$.

    (3) We reach line~\ref{alg:executeSymbolically:Unwinding}: Let $x'$ be a predecessor of $u'$ in $\pi'$, which pushed (at line~\ref{alg:executeSymbolically:Push}) the record being at the top of $u'.\Xi$. Obviously, successors of $x'.s$ are computed at line~\ref{alg:executeSymbolically:SuccessorInCalling}. Further, let $y'$ and $v'$ be direct successors of $x'$ and $u'$ in $\pi'$ respectively. The formula $v'.\varphi$ is satisfiable, since $v'$ is in $T'$ (see condition at line~\ref{alg:executeSymbolically:AddSATSuccessors}). Note that $v'$ is the only successor of $u'$ in $T'$. Let $\bld{\nu}$ be a valuation for which $v'.\varphi$ is satisfiable. Note that $\bld{\nu}$ defines an integer $\nu = \bld{\nu}(\kappa)$ for the parameter $\kappa$ retrieved from stack $u'.\Xi$ at line~\ref{alg:executeSymbolically:UnwindingGetKappa}, since $y'.s$ must have already used it. From line~\ref{alg:executeSymbolically:SuccessorInUnwinding} we see that $u'.\varphi\prm{\bld{\nu}}$ is satisfiable. Therefore, there is a vertex $u \in U$ such that $u.s \equiv u'.s\prm{\bld{\nu}}$. Let $\pi$ be a path in $T$ from the root to a leaf vertex and going through $u$. According to connections between vertices of sets $U$ constructed for vertices $u'$ along $\pi'$, there is a predecessor $x$ of $u$ in $\pi$, such that $x.s \equiv x'.s\prm{\bld{\nu}}$. Since $y'$ is the direct successor of $x$ in $\pi$ (i.e.~there was computed a set $W$ for $y'$), there must also exist a vertex $y \in \pi$ lying between $x$ and $u$ and $y.s \equiv y'.s\prm{\bld{\nu}}$. Note that $u$ is the first successor of $y$ in $\pi$ with $u.l$ being an exit location and $u.\Xi = y.\Xi$. Otherwise we would apply this case (3) for some other vertex lying between $y'$ and $u'$ in $\pi'$. Therefore, from Definition~\ref{def:Templates}~(R2) there is a non-leaf vertex $v$ in a suffix of $\pi$ starting with $u$ such that
    \begin{align*}
        v.s &\overset{g}{\equiv} (u.\theta \circ t.\theta\prm{\kappa}, u.\varphi, x.\Xi, t.l')\prm{\{(\kappa,\nu)\}}
        \\
        &\overset{g}{\equiv} (u'.\theta\prm{\bld{\nu}} \circ t.\theta\prm{\kappa}, u'.\varphi\prm{\bld{\nu}}, \texttt{pop(}u'.\Xi\texttt{)}\prm{\bld{\nu}}, t.l')\prm{\{(\kappa,\nu)\}}
        \\
        &\overset{g}{\equiv} (u'.\theta \circ t.\theta\prm{\kappa}, u'.\varphi, \texttt{pop(}u'.\Xi\texttt{)}, t.l')\prm{\bld{\nu}}
        \\
        &\overset{g}{\equiv} v'.s\prm{\bld{\nu}}.
    \end{align*}
    And finally Lemma~\ref{lem:GEquivToEquiv} ensures there are the only direct successors $w$ and $w'$ of $v$ and $v'$ respectively, such that $w.s \equiv w'.s\prm{\bld{\nu}}$. Therefore, $w \in W$.

    (4) Otherwise, we reach line~\ref{alg:executeSymbolically:ClassicStep}: Let $u$ be any vertex in $U$. Since $u.s \equiv u'.s\prm{\bld{\nu}}$ for some valuation $\bld{\nu}$ for which $u'.\varphi\prm{\bld{\nu}}$ is satisfiable and since all direct successors of both $u$ and $u'$ are computed by classic symbolic execution step, there must be a direct successor $w$ of $u$ in $T$ and a direct successor $w'$ of $u'$ in $T'$ such that $w.s \equiv w'.s\prm{\bld{\nu}}$. Note that both $u'.s$ and $w'.s$ have exactly the same parameters. Therefore, $w \in W$.
\end{proof}
