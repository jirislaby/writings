\begin{abstract}
  We present a generalisation of King's symbolic execution technique called
  \emph{compact symbolic execution}. It is based on a concept of templates:
  a template is a declarative parametric description of a cycle in the
  control flow graph of a program. 
  % program part, whose any execution has some regularities in program
  % states along it. Typical sources of such regularities are program loops
  % and recursive calls.
  Using the templates, we are able to build a symbolic execution tree where
  a single node represents arbitrarily many iterations of the corresponding
  cycle without loss of any information. This leads to a considerable
  reduction of size of the symbolic execution tree. There are even programs
  for which the resulting compact symbolic execution trees are finite while
  the classic symbolic execution trees are infinite.
\end{abstract}

\section{Introduction} \label{sec:Introduction}

Classic symbolic execution as proposed by King in 1976~\cite{Kin76}
systematically explores all execution paths in an analysed program.
% We commonly describe these paths in a tree structure called \emph{symbolic
% execution tree}. Each path in the tree from its root to any leaf
% represents one real path in the analysed program.
There is typically a huge (or even infinite) number of execution paths even
for very small and simple programs. Therefore, exploration of all execution
paths is a serious problem known as \emph{path explosion problem}.

Compact symbolic execution also explores all execution paths, but in a
compact manner. We analyse a given program \emph{before} we start its
symbolic execution. We look for cycles in the control flow graph of the
program. %  which might produce execution paths with some regularities in
% program states along them. Typically, program loops and recursion produce
% such regularities.
We analyse these cycles independently from the rest of the program. The goal
of the analysis is to find a \emph{template}, i.e.~a declarative parametric
description of the complete behavior of the cycle. Later, whenever the
symbolic execution enters a cycle for which we have a template, we either
proceed in the standard way, or add to the symbolic execution tree only one
successor node for each edge leaving the cycle. Each such a node reflects a
cumulative effect of any execution of the cycle eventually leaving the cycle
by the corresponding edge. Symbolic execution continues from these
successors in the standard way. 

If our algorithm fails to compute any template for a given program, then the
compact symbolic execution reduces to the classic one. However, if a
template for a cycle is found and applied, one node in the compact symbolic
execution tree can represent all execution paths that differ only in number
of iterations around the cycle. Such a node corresponds to (potentially
infinitely) many nodes of the standard symbolic execution tree. Therefore,
the compact symbolic execution typically produces much smaller trees
comparing to the standard symbolic execution. Also, application of a
template leads to a faster exploration of execution paths (in one step, we
explore all iterations of the cycle). As a results, test-generation
techniques based on symbolic execution can be more successful (cover more
locations in a given time limit) if they use compact symbolic execution.

On the negative side, each application of a template adds one quantifier to
path conditions. As a result, SMT solvers fail to decide satisfiability of
our path conditions more often than in the standard case. Experimental
evaluation of our technique indicates that the benefits of our technique are
in practice considerably higher than its drawbacks.

We first briefly describe the classic symbolic execution as proposed by
King~\cite{Kin76}. Symbolic execution runs a program over symbols
representing arbitrary input values. For each variable $\var{v}$, we use a
symbol $\sym{v}$ to represent its input value.
% $\sym{n},\sym{x},\sym{i},\sym{A}$ to represent input values of
% $\var{n},\var{x},\var{i},\var{A}$ respectively.
Symbolic execution maintains a \emph{symbolic memory} $\theta$ and a
\emph{path condition} $\varphi$. At each state of the symbolic execution,
$\theta$ assigns to each program variable its current value given as an
expression over the symbols. For example, if the first command of a program
is the assignment \texttt{i:=2*n+x}, then $\theta(\var{i})=2\sym{n}+\sym{x}$
after its execution. The path condition $\varphi$ is a formula representing
the necessary and sufficient condition on the symbols (i.e.~on input values)
to drive the execution along the current execution path. $\varphi$ is
initially $\true$ and it can be updated at branching commands. For example,
if we execute a command \texttt{if~(x>n+5)} (in a flowgraph, this
corresponds to a location with two out-edges labelled by \texttt{x>n+5} and
\texttt{x<=n+5}), we instantiate the condition with use of $\theta$ and we
check whether the current path condition $\varphi$ implies validity of the
instance or its negation. In other words, we ask for validity of
\[
\varphi\implies(\theta(\var{x})>\theta(\var{n})+5)
~~~~~~~\textrm{and}~~~~~~~
\varphi\implies(\theta(\var{x})\le\theta(\var{n})+5).
\] 
If the first implication is valid, the symbolic execution continues along
the true branch. If the second implication is valid, the symbolic execution
continues along the false branch. If none of them is valid, it means that we
can follow either of the two branches. Hence, the symbolic execution forks
in order to execute both branches. In this case, we update the path
condition on the true branch to
$\varphi\wedge(\theta(\var{x})>\theta(\var{n})+5)$ and the one on the false
branch to $\varphi\wedge(\theta(\var{x})\le\theta(\var{n})+5)$. Due to the
forks, symbolic execution is traditionally represented by a tree called
\emph{(classic) symbolic execution tree}. Nodes of the tree (also called
\emph{states} of the symbolic execution) are triples $(l,\theta,\varphi)$
consisting of the current program location $l$, symbolic memory $\theta$,
and path condition $\varphi$.

The symbolic execution tree for the program of Figure~\ref{fig:linSrch}(a)
is depicted in Figure~\ref{fig:linSrch}(b). Note that the figure does not
provide symbolic memories and path conditions of the nodes. However, the
figure contains instances of branching conditions on the corresponding
edges. For each node in the tree, the corresponding path condition $\varphi$
is a conjunction of the instances along the path from the root to the node.

location $b$) proceed in the same way: each execution performs $\kappa$
iterations along the cycle (for some $\kappa\ge0$) and continues either
along the edge $(b,f)$ or along the edges $(b,c)$ and $(c,e)$ to leave the
cycle. The described parts of executions are called \emph{cycle executions}.
% In other words, each excution follows a path of the form 
% $(bcd)^{\kappa}bf$ or $(bcd)^{\kappa}bce$.
We say that $b$ is an \emph{entry location} and $f,e$ are \emph{exit
  locations} of this cycle. The edges $(b,f)$ and $(c,e)$ are \emph{exit
  edges} of the cycle.

Compact symbolic execution aims to effectively exploit the uniformity of all
cycle executions. To do so, we need to find a unified declarative
description of the \emph{effect} of all cycle executions on a symbolic
memory and a path condition. Hence, we analyze the cycle separately from the
rest of the flowgraph. In the following, the symbols
$\sym{n},\sym{x},\sym{i},\sym{A}$ represent the values of the corresponding
variables at the entry location $b$, before the first iteration.

We first study the effect of all iterating parts of the cycle
executions. One can see that each iteration increases the value of $\var{i}$
by one while values of the other variables keep unchanged. Hence, after
$\kappa$ iterations, the value of $\var{i}$ is $\sym{i}+\kappa$. Formally,
the effect of $\kappa$ iterations of the cycle on values of all variables is
described by the following \emph{parametric symbolic memory}
$\theta_*\prm{\kappa}$:
\[
\setlength{\arraycolsep}{2pt}
\begin{array}{rclp{3ex}rcl}
  \theta_*\prm{\kappa}(\var{n})&=&\sym{n}&&
  \theta_*\prm{\kappa}(\var{x})&=&\sym{x}\\
  \theta_*\prm{\kappa}(\var{i})&=&\sym{i}+\kappa&&
  \theta_*\prm{\kappa}(\var{A})&=&\sym{A}\\
\end{array}
\]
% This is why $\theta_*[\kappa](\var{i})$ is $\sym{i}+\kappa$ and not just
% $\kappa$.

Further, we formulate a \emph{parametric path condition}
$\varphi_*\prm{\kappa}$ representing the path condition after $\kappa$
iterations along the cycle. The necessary and sufficient condition to perform
these $\kappa$ iterations is validity of both conditions on the cycle,
\texttt{i<n} and \texttt{A[i]!=x}, in each of $\kappa$ iterations. % As the
% conditions are at the beginning of the cycle (i.e.~not preceded by any
% assignment), the variables $\var{i},\var{n},var{A},var{x}$ in these
% conditions during $\tau$-th iteration (where $1\le\tau\le\kappa$) have the
% same values as they have after $\tau-1$ iterations, which is given by the
% parametric symbolic memory $\theta_*\prm{\tau-1}$.
Therefore, the path condition after $\kappa$ iterations has the form
\begin{align*}
  \sym{i}<\sym{n}&~~\wedge~~\sym{A}(\sym{i})\not=\sym{x}~~\wedge\\
  \wedge~~\sym{i}+1<\sym{n}&~~\wedge~~\sym{A}(\sym{i}+1)\not=\sym{x}~~\wedge\\
  &~~~~\vdots\\
  \wedge~~\sym{i}+(\kappa-1)<\sym{n}&~~\wedge~~\sym{A}(\sym{i}+(\kappa-1))\not=\sym{x}
%  \textrm{,}
\end{align*}
where $\tau$-th line, $\tau\in\{0,1,\ldots,\kappa-1\}$, consists of two
predicates resulting from evaluation of the conditions \texttt{i<n} and
\texttt{A[i]!=x} respectively after $\tau$ iterations of the cycle,
i.e.~during the $(\tau+1)$-st iteration. Unfortunately, the conjunction
above is not a first order formula as its length depends on the parameter
$\kappa$, whose value can be arbitrary. The conjunction can be equivalently
expressed by the following quantified formula:
% To uniformly describe all such path
% conditions for all values of $\kappa$, we have to use a quantified formula:
\[
  \forall\tau(0\le\tau<\kappa\implies
  (\sym{i}+\tau<\sym{n}\,\wedge\,\sym{A}(\sym{i}+\tau)\neq\sym{x}))
\]
If we now add to the formula above the obvious fact, that we cannot
iterate the cycle negative number of times (i.e.~$\kappa \geq 0$), we
get the resulting the parametric path condition $\varphi_*\prm{\kappa}$ as
\begin{align*}
  \varphi_*\prm{\kappa}~=~&\kappa\ge0~~\wedge\\
  \wedge~&~\forall\tau(0\le\tau<\kappa\implies
  (\sym{i}+\tau<\sym{n}~\wedge~\sym{A}(\sym{i}+\tau)\neq\sym{x})).
\end{align*}

Finally, we use $\theta_*\prm{\kappa}$ and $\varphi_*\prm{\kappa}$ to define
symbolic memory $\theta_{b\!f}\prm{\kappa}$ and path condition
$\varphi_{b\!f}f\prm{\kappa}$ describing the effect of all cycle executions
leaving of the cycle by exit edge $(b,f)$ and
$\theta_{ce}\prm{\kappa},\varphi_{ce}\prm{\kappa}$ with the analogous information
for the exit edge $(c,e)$. As the edges $(b,f),(b,c),(c,e)$ do not modify
any variable, we can clearly see that
$\theta_{b\!f}\prm{\kappa}=\theta_{ce}\prm{\kappa}=\theta_*\prm{\kappa}$. Further,
$\varphi_{b\!f}\prm{\kappa}$ and $\varphi_{ce}\prm{\kappa}$ are conjunctions of
$\varphi_*\prm{\kappa}$ with the instances of the conditions on the edge
$(b,f)$ or on the edges $(b,c),(c,e)$, respectively. Hence, the path
conditions $\varphi_{b\!f}\prm{\kappa},\varphi_{ce}\prm{\kappa}$ are defined as
follows:
\[
\setlength{\arraycolsep}{0pt}
\begin{array}{rcl}
  \varphi_{b\!f}\prm{\kappa}&~=~&\varphi_*\prm{\kappa}~\wedge~\sym{i}+\kappa\ge\sym{n}\\
  \varphi_{ce}\prm{\kappa}&=&\varphi_*\prm{\kappa}~\wedge~\sym{i}+\kappa<\sym{n}
    ~\wedge~\sym{A}(\sym{i}+\kappa)=\sym{x}
\end{array}
\]

The overall effect of the considered cycle is now fully described by a
so-called \emph{template} consisting of the entry location $b$ and two
triples $((cdb)^\kappa
f,\theta_{b\!f}\prm{\kappa},\varphi_{b\!f}\prm{\kappa})$ and $((cdb)^\kappa
ce,\theta_{ce}\prm{\kappa},\varphi_{ce}\prm{\kappa})$, one for each exit
edge. The first elements of these triples are expressions $(cdb)^\kappa f$
and $(cdb)^\kappa ce$ representing sequences of locations passed in
correspoding cycle executions. Technically, we only need to remember the
correspoding exit locations $f$ and $e$ to know where to continue the
symbolic execution from. We include the whole expressions only for clarity
of presentation (and we will leave this practice in the formal definition).

The template is applied during \emph{compact symbolic execution} of the
program. The execution processes the first command \texttt{i:=0} in the
standard way. Then we reach the node $(b,\theta',\varphi')$ with the entry
location $b$ for which we have the template.
%  proceeds in the standard way until we reach the node
% $(b,\theta,\varphi)$ with entry location $b$. 
% where $\theta(\var{i})=0$ and it assigns to each other variable he
% corresponding symbol except for ...
At this point, we can apply the template instead of executing the original
program. The node $(b,\theta',\varphi')$ gets one successor for each triple
of the template. The triple $((cdb)^\kappa
f,\theta_f\prm{\kappa},\varphi_f\prm{\kappa})$ generates the successor
$((cdb)^\kappa
f,\theta'_{b\!f}\prm{\kappa},\varphi'_{b\!f}\prm{\kappa})$. Note that we
cannot use $((cdb)^\kappa
f,\theta_{b\!f}\prm{\kappa},\varphi_{b\!f}\prm{\kappa})$ directly as
$\theta_{b\!f}\prm{\kappa},\varphi_{b\!f}\prm{\kappa}$ describe executions
starting just at the entry location $b$, while
$\theta'_{b\!f}\prm{\kappa},\varphi'_{b\!f}\prm{\kappa}$ have to reflect the
effect of the executions starting at $a$. We create
$\theta'_{b\!f}\prm{\kappa},\varphi'_{b\!f}\prm{\kappa}$ by composing
$\theta_{b\!f}\prm{\kappa},\varphi_{b\!f}\prm{\kappa}$ with
$\theta',\varphi'$. The composition is precisely described in the following
section. In our simple program, $\theta',\varphi'$ reflect only the effect
of assignment \texttt{i:=0}. Thus, $\theta'_{b\!f}\prm{\kappa}$ equals to
$\theta_{b\!f}\prm{\kappa}$ where $\sym{i}$ is replaced by $0$ and
$\varphi'_{b\!f}\prm{\kappa}$ equals $\varphi_{b\!f}\prm{\kappa}$ where
$\sym{i}$ is replaced by $0$. The second triple $((cdb)^\kappa
ce,\theta_{ce}\prm{\kappa},\varphi_{ce}\prm{\kappa})$ of the template
generates the successor $((cdb)^\kappa
ce,\theta'_{ce}\prm{\kappa},\varphi'_{ce}\prm{\kappa})$ analogously. Recall
that the expressions $(cdb)^\kappa f$ and $(cdb)^\kappa ce$ only support
intuitive meaning of the nodes in the tree. Only the last locations $f$ and
$e$ of these expressions are technically significant. The symbolic execution
then continues from these successor in the standard way. The resulting
\emph{compact symbolic execution tree} is depicted in
Figure~\ref{fig:linSrch}(c).

The compact symbolic execution tree is much smaller than the classic
tree. In particular, the infinite path in the classic tree (highlighted by
the grey region) corresponds to the single node $d$ in the compact
tree. However, the two trees keep the same information. For example, the
left leaf of the compact tree represents the following path condition
\begin{align*}
  \varphi\prm{\kappa}~=~&\kappa\ge0~~\wedge\\
  \wedge~&~\forall\tau(0\le\tau<\kappa\implies
  (\tau<\sym{n}~\wedge~\sym{A}(\tau)\neq\sym{x}))~~\wedge\\
  \wedge~&~\kappa<\sym{n}~~\wedge~~\sym{A}(\kappa)=\sym{x}
\end{align*}
% \[
% \varphi\prm{\kappa}~=~\kappa\ge0~\wedge~ \forall\tau(0\le\tau<\kappa\implies
% (\tau<\sym{n}\,\wedge\,\sym{A}(\tau)\neq\sym{x}))~\wedge~\kappa<\sym{n}
% ~\wedge~\sym{A}(\kappa)=\sym{x}.
% \]
Let us mark all leafs on the left-hand side of the classic tree as
$g_0,g_1,g_2,\ldots$ and let $\varphi_0,\varphi_1,\varphi_2,\ldots$ be the
corresponding path conditions. One can easily compute each path condition
$\varphi_i$ as a conjunction of the conditions along the corresponding path
in the figure of the tree and check that $\varphi_i$ is equivalent to
$\varphi\prm{i}$ for each $i\ge0$. For example,
\[
\setlength{\arraycolsep}{0pt}
\begin{array}{rcl}
  \varphi_1&~=~
  &0<\sym{n}~\wedge~\sym{A}(0)\neq\sym{x}
   ~\wedge~1<\sym{n}~\wedge~\sym{A}(1)=\sym{x}\\[2mm]
  \varphi\prm{1}&\equiv&\forall\tau(0\le\tau<1\implies
  (\tau<\sym{n}~\wedge~\sym{A}(\tau)\neq\sym{x}))~\wedge\\
  &\wedge&1<\sym{n}~\wedge~\sym{A}(1)=\sym{x}
\end{array}
\]
and hence $\varphi_1\equiv\varphi\prm{1}$. Similarly, each symbolic memory
of a node $g_i$ is an instance $\theta\prm{i}$ of the parametrized symbolic
memory in the left leaf of the compact tree. Analogous relations hold for
leafs on the right-hand sides of the compact and classic symbolic execution
trees. To sum up, the compact symbolic execution tree carries in its leaves the
same information as the classic symbolic execution tree.

\section{Description of the Technique}\label{sec:technique}

This section describes the compact symbolic execution formally. For
simplicity, we formulate the algorithm for programs manipulating only
integer variables and read-only multidimensional integer arrays, and with no
function calls. The algorithm can be extended to handle other program
features including modifiable arrays, function calls, variables of other
types, dynamically allocated memory etc. The first two mentioned features
are actually supported in our experimental implementation mentioned in the
next section.

\subsection{Preliminaries}

First of all, we present the considered model of programs and define
composition and equivalence of symbolic program states.

Programs are represented in the form of flowgraphs. A \emph{flowgraph} is a
tuple $P=(V,E,l_s,l_e,\iota)$, where $(V,E)$ is a finite oriented graph,
elements of $V$ are called \emph{locations}, $l_s,l_e\in V$ are \emph{start}
and \emph{end locations}, and $\iota:E\rightarrow\mathcal{I}$ is a function
assigning to each edge $e$ an \emph{instruction} $\iota(e)$. A location is
\emph{branching} if its out-degree is $2$. All other locations have
out-degree at most $1$. The in-degree of start location $l_s$ and the
out-degree of end location $l_e$ is 0. We use two kinds of instruction: an
assignment instruction $\var{a}\texttt{:=}e$ for some scalar variable
$\var{a}$ and some expression $e$, and an assumption
$\texttt{assume}(\gamma)$ for some quantifier-free formula $\gamma$ over
program variables. Out-edges of any branching location are labelled with
assumptions $\texttt{assume}(\gamma)$ and $\texttt{assume}(\neg\gamma)$ for
some $\gamma$. We usually omit the keyword $\texttt{assume}$ in our
flowgraphs. We consider only instructions operating on \emph{scalar
  variables} $\var{a},\var{b},\ldots$ of type \texttt{Int} and
multi-dimensional \emph{array variables} $\var{A},\var{B},\ldots$ of type
\texttt{Int}$^k\rightarrow$\texttt{Int}. We assume that a flowgraph prevents
invalid operations (like division by zero, etc.) by branchings into error
locations. An error location is any location without any outgoing edge which
is not the end location. In the rest of the paper, $P$ is always a
flowchart.

\medskip By \emph{symbolic expressions} we mean all expressions build with
integers, standard integer operations and functions, and
\begin{itemize}
\item a \emph{symbol} $\sym{a}$ for each scalar variable $\var{a}$,
\item a \emph{function symbol} $\sym{A}$ for each array variable $\var{A}$,
  where arity of $\sym{A}$ corresponds to the dimension of array $\var{A}$,
\item a countable set $\{\kappa_1,\tau_1,\kappa_2,\tau_2,\ldots\}$ of
  \emph{parameters}, and
\item a construct $\ite(\varphi,e_1,e_2)$, of meaning
  ``\textbf{i}f-\textbf{t}hen-\textbf{e}lse'', where $e_1,e_2$ are symbolic
  expressions and $\varphi$ is a first order formula over symbols and
  parameters. The value of $\ite(\varphi,e_1,e_2)$ is $e_1$ if $\varphi$
  holds and $e_2$ otherwise.
\end{itemize}

% Let $f,e_1,\ldots,e_n$ be symbolic expressions and $x_1,\ldots,x_n$ be path
% counters or symbols corresponding to scalar variables. Then
% $f[x_1/e_1,\ldots,x_n/e_n]$ is a symbolic expression $f$ where all
% occurrences of all $x_i$ are simultaneously replaced by expressions
% $e_i$. To shorten the notation, we write $f[\vec{x}/\vec{e}]$ when the
% meaning is clearly given by a context. We also use the notation
% $\varphi[\vec{x}/\vec{e}]$ with the analogous meaning for formulae
% $\varphi$.

A \emph{symbolic memory} is a function $\theta$ assigning to each scalar
variable $\var{a}$ a symbolic expression and to each array variable
$\var{A}$ the function symbol $\sym{A}$. \emph{Initial symbolic memory}
$\Theta$ assigns to each scalar variable $\var{a}$ the corresponding symbol
$\sym{a}$.
% Let $\var{a}$ be a scalar variable and $e$ be a symbolic expression. Then
% $\theta[\var{a}\rightarrow e]$ is a symbolic memory equal to $\theta$
% except for variable $\var{a}$, where $\theta[\var{a}\rightarrow
% e](\var{a})=e$.
We sometimes use the notation $\theta\prm{\kappa}$ to emphasize that
$\theta$ is a symbolic memory using a single parameter $\kappa$.  We use an
analogous notation $\varphi\prm{\kappa}$ to emphasize that the first order
formula $\varphi$ has only one free parameter $\kappa$. For an integer
$\nu>0$, $\theta\prm{\nu}$ and $\varphi\prm{\nu}$ denote the symbolic memory
$\theta\prm{\kappa}$ and the formula $\varphi\prm{\kappa}$ where all (free)
occurrences of $\kappa$ are replaced by~$\nu$.
 
The notation $\theta(\cdot)$ is used in a more general way. It always
denotes the operation of replacing each (scalar or array) variable $\var{a}$
by $\theta(\var{a})$. $\theta\langle\cdot\rangle$ denotes the operation on
symbolic expressions or formulae, where all occurrences of all symbols
$\sym{a}$ are simultaneously replaced by $\theta(\var{a})$. Additionally,
$\theta_1\compose\theta_2$ denotes \emph{composition of symbolic memories}
$\theta_1$ and $\theta_2$ satisfying
$(\theta_1\compose\theta_2)(\var{a})=\theta_1\langle\theta_2(\var{a})\rangle$
for each variable $\var{a}$. Intuitively, the symbolic memory
$\theta_1\compose\theta_2$ represents an overall effect of a code with
effect $\theta_1$ followed by a code with effect $\theta_2$.

% Finally, for vectors $\vec{u}=(u_1,\ldots,u_n)$ and
% $\vec{v}=(v_1,\ldots,v_n)$ we use $\vec{u}\le\vec{v}$ as an abbreviation
% for $u_1\le v_1\wedge\ldots\wedge u_n\le v_n$.

\medskip \emph{Symbolic program state} (or simply a \emph{state}) is a
triple $s=(l,\theta,\varphi)$ of a location $l$, symbolic memory $\theta$
and first order formula over symbols and parameters $\varphi$ called
\emph{path condition}. \emph{Initial state} is the triple
$(l_s,\Theta,\true)$ where $l_s$ is the start location and $\Theta$ is the
initial symbolic memory.

We define \emph{composition of states} $s_1=(l_1,\theta_1,\varphi_1)$ and
$s_2=(l_2,\theta_2,\varphi_2)$ to be the state $s_1\compose
s_2=(l_2,\theta_1\compose\theta_2,\varphi_1\wedge\theta_1\langle\varphi_2\rangle)$. The
composed state corresponds to the symbolic state resulting from symbolic
execution of the code that produced $s_1$ immediately followed by the code
that produced $s_s$.

\begin{lem}[Equivalent Compositions]\label{lem:EquivalentCompositions}
Let $s, s'$, and $s''$ be program states, $\bld{\nu}$ and $\bld{\nu}'$
be valuations of parameters in $s$ and $s'$ respectively such that
$\bld{\nu} \cup \bld{\nu}'$ is also a valuation, $\theta$, $\theta'$,
and $\theta''$ be symbolic memories, and $\varphi$ and $\psi \wedge
\psi'$ be symbolic expressions. Then $s \compose (s' \compose s'') \equiv (s
\compose s') \compose s''$, $s\prm{\bld{\nu}} \compose s'\prm{\bld{\nu}'}
\equiv (s \compose s')\prm{\bld{\nu} \cup \bld{\nu}'}$, $\theta \compose
(\theta' \compose \theta'') \equiv (\theta \compose \theta') \compose
\theta''$, $(\theta \compose \theta')\ese{\varphi} \equiv
\theta\ese{\theta'\ese{\varphi}}$, and $\theta\ese{\psi} \wedge
\theta\ese{\psi'} \equiv \theta\ese{\psi \wedge \psi'}$.
\begin{proof}
We split the proof of the lemma into separate points related to a
particular equivalences.

(1) $(\theta \compose \theta')\ese{\varphi} \equiv
\theta\ese{\theta'\ese{\varphi}}$. The expression $(\theta \compose
\theta')\ese{\varphi}$ simultaneously substitutes each symbol
$\alpha_i$ in $\varphi$ by a symbolic expression $(\theta \compose
\theta') \ese{\Theta^{-1}(\alpha_i)} =
\theta\ese{\theta'(\Theta^{-1}(\alpha_i))}$ This follows directly from
Definition~\ref{def:Composition}. In the expression
$\theta\ese{\theta'\ese{\varphi}}$, there we have to apply the
substitution twice. First we simultaneously substitute each symbol
$\alpha_i$ in $\varphi$ by a symbolic expression
$\theta'(\Theta^{-1}(\alpha_i))$. If the resulting formula contains
any symbol, then it must necessarily lie in some of the substituted
expressions $\theta'(\Theta^{-1}(\alpha_i))$. Therefore, it is
sufficient to apply the second substitution only to the substituted
expressions $\theta'(\Theta^{-1}(\alpha_i))$. In other words, it is
sufficient to apply only one simultaneous substitution of symbols
$\alpha_i$ in $\varphi$ by symbolic expressions
$\theta\ese{\theta'(\Theta^{-1}(\alpha_i))}$.

(2) $\theta\ese{\psi} \wedge \theta\ese{\psi'} \equiv \theta\ese{\psi
\wedge \psi'}$. The operation $\theta\ese{\cdot}$ only applies symbol
substitutions inside formulae $\psi$ and $\psi'$.

(3) $s \compose (s' \compose s'') \equiv (s \compose s') \compose s''$. Let $s$,
$s'$, and $s''$ be program states $(\theta, \varphi, \Xi, l)$,
$(\theta', \varphi', \Xi', l')$, and $(\theta'', \varphi'', \Xi'',
l'')$ respectively. According to Definition~\ref{def:Composition}, we
have $s \compose (s' \compose s'') = (\theta \compose (\theta' \compose
\theta''),~\varphi \wedge \theta\ese{\varphi' \wedge
\theta'\ese{\varphi''}}, \Xi \compose (\theta \compose (\Xi' \compose (\theta'
\compose \Xi''))), l'')$ and $(s \compose s') \compose s'' = ((\theta \compose
\theta') \compose \theta'',~(\varphi \wedge \theta\ese{\varphi'}) \wedge
(\theta \compose \theta')\ese{\varphi''}, (\Xi \compose (\theta \compose \Xi'))
\compose ((\theta \compose \theta') \compose \Xi'')), l'')$. We prove the
equivalence of the composed states per component in points (3a), (3b),
and (3c).

(3a) $\theta \compose (\theta' \compose \theta'') \equiv (\theta \compose
\theta') \compose \theta''$. According to
Definition~\ref{def:Composition} we have for each variable \var{a} the
following: $(\theta \compose (\theta' \compose \theta''))(\var{a}) \equiv
\theta\ese{(\theta' \compose \theta'')(\var{a})} \equiv
\theta\ese{\theta'\ese{\theta''(\var{a})}} \equiv (\theta \compose
\theta')\ese{\theta''(\var{a})} \equiv ((\theta \compose \theta') \compose
\theta'')(\var{a})$.

(3b) $\varphi \wedge \theta\ese{\varphi' \wedge
\theta'\ese{\varphi''}} \equiv (\varphi \wedge \theta\ese{\varphi'})
\wedge (\theta \compose \theta')\ese{\varphi''}$. According to points (1)
and (2) we have $(\varphi \wedge \theta\ese{\varphi'}) \wedge (\theta
\compose \theta')\ese{\varphi''} \equiv \varphi \wedge
\theta\ese{\varphi'} \wedge \theta\ese{\theta'\ese{\varphi''}} \equiv
\varphi \wedge \theta\ese{\varphi' \wedge \theta'\ese{\varphi''}}$.

(3c) $\Xi \compose (\theta \compose (\Xi' \compose (\theta' \compose \Xi'')))
\equiv (\Xi \compose (\theta \compose \Xi')) \compose ((\theta \compose \theta')
\compose \Xi''))$. Let $\Xi = [r_0, \ldots, r_m]$, $\Xi' = [r_0', \ldots,
r_n']$, and $\Xi'' = [r_0'', \ldots, r_k'']$. Then a symbolic memory
of the $i$-th record ($i \in \{0, \ldots, m \}$) in a call stack
$(\theta \compose \theta') \compose \Xi$ has a form $(\theta \compose \theta')
\compose r_i.\sigma = (\theta \compose (\theta' \compose r_i.\sigma)$ according
to Definition~\ref{def:Composition} and point (3a). But $(\theta \compose
(\theta' \compose r_i.\sigma)$ is actually a symbolic memory of the
$i$-th record of a call stack $\theta \compose (\theta' \compose \Xi)$.
Therefore, from Definition~\ref{def:Composition} we infer $(\theta
\compose \theta') \compose \Xi'' \equiv \theta \compose (\theta' \compose \Xi'')$.
Further, $(\theta \compose \Xi) \compose (\theta \compose \Xi') \equiv [(\theta
\compose r_0.\sigma, r_0.l), \ldots, (\theta \compose r_m.\sigma, r_m.l),
(\theta \compose r_0'.\sigma, r_0'.l), \ldots, (\theta \compose r_n'.\sigma,
r_n'.l) ] \equiv \theta \compose (\Xi \compose \Xi')$. Finally, from
Definition~\ref{def:Composition}  immediately follows associativity
$\Xi \compose (\Xi' \compose \Xi'') \equiv (\Xi \compose \Xi') \compose \Xi''$.
Using the equivalences above,  we can finally prove the equivalence
(3c) as follows: $(\Xi \compose (\theta \compose \Xi')) \compose ((\theta \compose
\theta') \compose \Xi'') \equiv \Xi \compose (\theta \compose \Xi') \compose
(\theta \compose (\theta' \compose \Xi'')) \equiv \Xi \compose (\theta \compose
(\Xi' \compose (\theta' \compose \Xi'')))$.

(4) $s\prm{\bld{\nu}} \compose s'\prm{\bld{\nu}'} \equiv (s \compose
s')\prm{\bld{\nu} \cup \bld{\nu}'}$. The equivalence follows from
these two facts: (a) The composition of states operates on symbols,
while the parameter substitution operates on parameters. (b)
$\bld{\nu} \cup \bld{\nu}'$ is supposed to be a valuation. Therefore,
if there is $(\kappa, \nu) \in \bld{\nu}$ and $(\kappa, \nu') \in
\bld{\nu}'$, then $\nu = \nu'$ must be valid.
\qed
\end{proof}
\end{lem}

We have seen in Section \ref{sec:basic} that checking satisfiability
of path condition is essential for the classic symbolic execution (see
also \cite{Kin76}). And we will see, that such checks are also
essential for the compact one. Unfortunately, programs commonly
operates in undecidable theories (like Peano arithmetic). We therefore
assume in this paper that each satisfiability query always terminates
with a result either SAT, UNSAT, or FAILURE, with the obvious
meanings.


We often use the following dot-notation to access elements of a state: if
$s$ is a symbolic state, then $s.l$ denotes its current program location,
$s.\theta$ denotes its symbolic memory, and $s.\varphi$ denotes its path
condition. We use an analogous notation also for other tuples. The meaning

\subsection{Templates and their computation}

We start with formal definitions of the terms \emph{cycle} and
\emph{template}, since they are essential for this section. We assume
$P$ is a program throughout the section.

\begin{dfn}[Cycle]\label{def:Part}
Let $(u,e)$ be an edge of a flowgraph $f$ of $P$, $\pi = e \omega e$
be a cyclic path in $P$ such that $\omega$ does not contain neither
the entry location of $f$ nor the location $u$, and let $X = \{
(u_1,x_1), \ldots, (u_n,x_n) \}$ be a non-empty set of all those edges
of $P$, which do not belong to the path $\pi$, but each of their start
vertices $u_i$ does. Then, a triple $C = (\pi, e, X) $ is a
\emph{cycle} in $P$, where the path $\pi$ is a \emph{core} of $C$, $e$
is an \emph{entry} into $C$, each edge $(u_i,x_i)$ is an \emph{exit
edge} from $C$, and each location $x_i$ is called an \emph{exit} from
$C$.
\end{dfn}

\begin{dfn}[Template] \label{def:LoopTemplate}
Let $T$ be a classic symbolic execution tree of $P$, $n>0$ be an
integer, $l, l_1, \ldots, l_n$ be locations in $P$, $\kappa$ be a
parameter, $\theta_1\prm{\kappa}, \ldots,$ $\theta_n\prm{\kappa}$ be
symbolic memories, $\varphi_1\prm{\kappa}, \ldots,
\varphi_n\prm{\kappa}$ be satisfiable boolean symbolic expressions
such that for each $i,j \in \{ 1, \ldots, n \}, i \neq j$, we have
$(\varphi_i \wedge \varphi_j) \equiv \false$, and let
$\Xi_1\prm{\kappa}, \ldots, \Xi_n\prm{\kappa}$ be call stacks. Then a
pair $t = (l,M)$, where $M = \{ (\theta_1, \varphi_1, \Xi_1, l_1),
\ldots, (\theta_n,\varphi_n, \Xi_n, l_n) \})$ is a \emph{template} in
$P$, if
\begin{itemize}
\item[(L1)] For each path $\pi = u \omega$ in $T$ from any vertex $u$
satisfying $u.l = t.l$ to a leaf, there is a vertex $w \in \omega$, an
index $i \in \{ 1, \ldots, n \}$, and an integer $\nu \geq 0$ such
that $w.s \equiv u.s \compose
(t.\theta_i,~t.\varphi_i,~t.\Xi_i,~t.l_i)\prm{\nu}$.

\item[(L2)] For each vertex $u$ of $T$, an index $i \in \{ 1, \ldots,
n \}$, and non-negative integer $\nu$ such that $u.l = t.l$ and
$(u.\varphi \wedge u.\theta\ese{t.\varphi_i\prm{\nu}})$ is
satisfiable, there is a successor $w$ of $u$ in $T$ such that $w.s
\equiv u.s \compose (t.\theta_i,~t.\varphi_i,~t.\Xi_i,~t.l_i)\prm{\nu}$.
\end{itemize}
\end{dfn}

Now we focus on the computation of a template $t = (e, M)$ from a
cycle $C = (\pi,e,X)$ of a program $P$. Algorithm~\ref{alg:template}
describes the computation. It uses a function
$\texttt{executePath}(P,\rho)$ which applies classic symbolic
execution to actions of edges along the passed path $\rho$ in the
passed program $P$ and returns the resulting symbolic state
$(\theta,\varphi,\Xi,u)$, where $u$ is the last location in $\rho$.

The first part of the algorithm (lines
\ref{alg:template:begin}--\ref{alg:template:endMemory}) tries to
derive a parametric symbolic memory $\theta_*\prm{\kappa}$ and a
parametric path condition $\varphi_*\prm{\kappa}$, which together
describe symbolic state after $\kappa$ iterations over the core $\pi$
of the cycle $C$, for any $\kappa \geq 0$. Initially, at
line~\ref{alg:template:begin}, we compute the effect of a
\emph{single} iteration of the core $\pi$ and then we check whether
the iteration is feasible. If it is not, it does not make sense to
compute the template and thus the algorithm terminates with failure,
i.e. it returns \textbf{null}. Otherwise, we get a symbolic state
$(\theta,\varphi,[],e)$, whose elements $\theta$ and $\varphi$ form a
basis for the computation of $\theta_*\prm{\kappa}$ and
$\varphi_*\prm{\kappa}$. First we compute $\theta_*\prm{\kappa}$. We
initialise $\theta_*\prm{\kappa}$ such that each variable is assigned
an undefined value $\bot$. Then, in the loop at lines
\ref{alg:template:repeatBegin}--\ref{alg:template:repeatEnd} we try to
increase precision of $\theta_*\prm{\kappa}$. The loop terminates, if
we are unable to improve value of any variable any further. We improve
variables separately one by one in the nested loop at lines
\ref{alg:template:repeat:foreachBegin}--\ref{alg:template:repeat:foreachEnd}
Observe, that we improve only those variables which are still
uninitialised. In other words, any variable can be improved at most
once; form the undefined value $\bot$ to an exact symbolic value for
the variable after all $\kappa$ iterations. Variables are improved
according to a content of $\theta$. Branching statements at lines
\ref{alg:template:repeat:foreach:BranchAritSeq}, and
\ref{alg:template:repeat:foreach:BranchGeomSeq} are supposed to
recognise, whether symbolic value $\theta(\texttt{a})$ for an improved
variable \texttt{a} is in one of exact forms. Only for these exact
forms we know how to express values of variables exactly after any
$\kappa$ iterations. For example, at line
\ref{alg:template:repeat:foreach:BranchAritSeq} there we recognise
arithmetic progressions, and at line
\ref{alg:template:repeat:foreach:BranchGeomSeq} there we recognise
geometric ones. Note that case when value of a scalar variable is not
changed at all along $\pi$ is covered at line
\ref{alg:template:repeat:foreach:BranchAritSeq}, because constant
progression is a special arithmetic progression ($c = 0$). Obviously,
one can add recognition of other progressions of values. Here we
present only those, which are the most common in majority of programs.
Once we cannot improve $\theta_*\prm{\kappa}$ any more, we get to line
\ref{alg:template:isMemoryPrecise}. There we check whether we
succeeded to express \emph{all} the variables exactly. If we failed
for at least one variable, then we fail to compute a template for $C$,
and we return \textbf{null}. Otherwise, at line
\ref{alg:template:endMemory} there we use $\theta_*\prm{\kappa}$ to
compute the formula $\varphi_*\prm{\kappa}$ from $\varphi$. We already
gave an intuition about the structure of the formula in
Section~\ref{sec:basic}.

In the second part of the algorithm (lines
\ref{alg:template:secondPartBegin}--\ref{alg:template:end}) there we
compute elements of the set $M$ of the resulting template. As we
already know from Section~\ref{sec:basic}, we try to compute a single 
element of $M$ for each exit edge $(u, x) \in X$. At line
\ref{alg:template:pathToExit} there we compute a path $\rho$ form the
entry location $e$ to $u$ (along $\pi$), where we escape from $\pi$
into the location $x$. Next we symbolically execute the path $\rho x$
at line \ref{alg:template:exitState}. Only in case the path $\rho x$
is feasible, we can escape $\pi$ by taking the exit edge $(u,x)$.
Therefore, only in this case we add a new element into $M$ at line
\ref{alg:template:extendM}. We already understand the structure of the
added element based on the intuition we get in
Section~\ref{sec:basic}.

We finish this section by formulating and proving correctness and
termination theorems for Algorithm~\ref{alg:template}.

\begin{thm}[Correctness]
Whenever Algorithm~\ref{alg:template} returns a template $(e,M)$ for a
passed program $P$ and a cycle $(\pi,e,X)$ in $P$, then the template
satisfies Definition~\ref{def:LoopTemplate}.
\begin{proof}
Our goal is to show that all tuples inserted into $M$ at line
\ref{alg:template:extendM} together satisfy
Definition~\ref{def:LoopTemplate}. Formulae $\varphi_*\prm{\kappa}
\wedge \theta_*\prm{\kappa} \langle\varphi\rangle$ constructed at line
\ref{alg:template:extendM} are trivially satisfiable for $\kappa = 0$,
since $\theta_*\prm{0}(\var{a}) = \Theta(\var{a})$ and satisfiability of
$\varphi$ is ensured at line \ref{alg:template:satCheck2}. Further,
for any two different  elements of $X$ the function
\texttt{executePath} returns two formulae $\varphi$ and $\varphi'$
satisfying $\varphi \wedge \varphi' \equiv \false$ (this is a
consequence of the same statement for leaves of classic symbolic
execution tree; see \cite{Kin76} for details). Therefore,
\begin{align*}
(&\varphi_*\prm{\kappa} \wedge \theta_*\prm{\kappa}
\langle\varphi\rangle) \wedge (\varphi_*\prm{\kappa} \wedge
\theta_*\prm{\kappa} \langle\varphi'\rangle)~\equiv\\
&\varphi_*\prm{\kappa} \wedge \theta_*\prm{\kappa}\langle\varphi
\wedge \varphi'\rangle~\equiv\\
&\varphi_*\prm{\kappa} \wedge \false~\equiv\\
&\false.
\end{align*}
It remains to prove properties (L1) and (L2) of
Definition~\ref{def:LoopTemplate}. We start with (L1).

Let $T$ be classic symbolic execution tree of $P$, $(\theta, \varphi,
[], e)$ be a symbolic state computed at line \ref{alg:template:begin},
and $(\hat{\theta}, \hat{\varphi}, \hat{\Xi}, x)$ be a symbolic state
computed at line \ref{alg:template:exitState} (for some exit edge from
$X$). Further, let $u$ be a vertex of $T$ such that $u.l = e$, and let
$\delta = u_0 \ldots u_1 \ldots u_2$ $\ldots u_{\nu} \ldots w$ be a path
in $T$ starting at $u$ (i.e.~$u_0 = u$), then iterating the core $\pi$
exactly $\nu \geq 0$ times (i.e.~all the vertices $u_i \in \pi$ are
exactly those having $u_i.l = e$), and then $\delta$ finally leaves
the core $\pi$ by following the path towards the vertex $w$,
satisfying $w.l = x$. We use the memory composition operation to
express memories of the vertices $u_i$ along $\delta$ as follows.
\begin{align*}
u_1.\theta &= u.\theta \compose \theta
\\
u_2.\theta &= u_1.\theta \compose \theta = u.\theta \compose (\theta \compose
\theta)
\\
\cdots
\\
u_{\nu}.\theta &= u_{\nu-1}.\theta \compose \theta = u.\theta \compose
(\underset{\nu}{\underbrace{\theta \compose \cdots \compose \theta}}).
\end{align*}
If we denote the composition of $i$ symbolic memories $\theta$ by
$\theta^i$, where $\theta^0 = \Theta$, $\theta^1 = \theta$, and
$\theta^i = \theta \compose \theta^{i-1}$, then we have $u_{i}.\theta =
u.\theta \compose \theta^i$ and we get
\begin{align*}
w.\theta = u.\theta \compose (\theta^{\nu} \compose \hat{\theta}).
\end{align*}
We proceed similarly to express path conditions of the vertices along
$\delta$.
\begin{align*}
u_1.\varphi &\equiv u.\varphi \wedge u.\theta\ese{\varphi} \equiv
u.\varphi \wedge (u.\theta \compose \theta^0)\ese{\varphi} \equiv
u.\varphi \wedge u.\theta\ese{\theta^0\ese{\varphi}}

u_2.\varphi &\equiv u_1.\varphi \wedge u_1.\theta\ese{\varphi} \equiv
u.\varphi \wedge u.\theta\ese{\theta^0\ese{\varphi}} \wedge (u.\theta

u.\theta\ese{\underset{\nu}{\underbrace{\theta^0\ese{\varphi} \wedge
\ldots \wedge \theta^{\nu-1}\ese{\varphi}}}}.
\end{align*}
Using the following equivalence
\[\theta^0\ese{\varphi} \wedge \ldots \wedge

\leq \tau < \nu \rightarrow \theta^{\tau}\ese{\varphi}),\]

\end{align*}
But SMT solvers do not support the memory composition operation
appearing in the formula $w.\varphi$. Therefore, we need an equivalent
\emph{declarative} description of the operation. Such a description is

g~.~\theta_*\prm{\kappa}(\var{b})\neq\bot}{\theta_*\prm{\kappa}(\var{a}) =
\ite(\kappa>0,\theta_*[\kappa-1]\langle g\rangle,\Theta(\var{a}))},
\\ \\
\frac{\theta(\var{A}) = \Theta(\var{A})%,~~~\var{A}~\textrm{is of an array type}
}{\theta_*\prm{\kappa}(\var{A}) = \Theta(\var{A})},
\end{gather*}

%casting operation of $\kappa$ to a type of the variable \var{a}.

Let $u.\varphi \wedge u.\theta\ese{\varphi_x\prm{\nu}}$ be satisfiable

\begin{thm}[Termination]
Algorithm~\ref{alg:template} terminates for any program $P$ and any
cycle $(\pi,e,X)$ in $P$.
\begin{proof}
The algorithm calls the function \texttt{executePath} at lines
\ref{alg:template:begin} and \ref{alg:template:exitState}. The
function always accepts a finite path and it therefore always
symbolically evaluates only a finite number of actions labelling edges
along the path. And according to our assumption about termination of
satisfiability queries the function \texttt{executePath} must always
terminate.

Similarly, satisfiability queries at lines
\ref{alg:template:satCheck1} and \ref{alg:template:satCheck2} must
also always terminate, again according to our assumption.

Since evaluation of all the remaining expressions in the algorithm is
finite, we may focus on termination of loops. Obviously, both
\textbf{foreach} loops at lines \ref{alg:template:repeat:foreachBegin}
and \ref{alg:template:foreach2begin} always iterate over finite number
of elements. And the termination of the \textbf{repeat-until} loop at lines
\ref{alg:template:repeatBegin}--\ref{alg:template:repeatEnd} follows
from the fact that number of variables having the value $\bot$ in
$\theta_*\prm{\kappa}$ can only decrease and the variable
\texttt{change} is set to $\true$ if and only if some variable has
changed its value in $\theta_*\prm{\kappa}$ from $\bot$ to a proper
symbolic expression. Indeed, the value $\bot$ is not written anywhere inside the loop and according to line \ref{alg:template:repeatIsBot} only variables with the value $\bot$ in $\theta_*\prm{\kappa}$ can be updated to a proper
symbolic expression.
\end{proof}
\end{thm}

\subsection{Compact symbolic execution}

The compact symbolic execution is formally defined in
Algorithm~\ref{alg:executeSymbolically}. If we ignore the lines marked
by $\Box$, then we get classic symbolic execution. But, as we focus on
compact symbolic execution, we describe the algorithm with $\Box$
lines included. The algorithm accepts a program $P$ and a finite set
$p$ of templates resulting from analyses of some cycles in $P$. At
lines~\ref{alg:executeSymbolically:InitialState}--\ref{alg:executeSymbolically:InitialQ},
there we create an initial state and insert it into a queue $Q$. The queue
$Q$ keeps all the program states waiting for their processing in the
\textbf{repeat}-\textbf{until} loop (lines
\ref{alg:executeSymbolically:LoopBegin}--\ref{alg:executeSymbolically:LoopEnd}). The
key part of the loop's body begins at
line~\ref{alg:executeSymbolically:GetTemplatesAt}, where we select at
most one of templates in $p$, whose entry locations match the actual
program location $s.l$. Note that there can be more then one template
available at $s.l$. We do not put any constraints to the selection
strategy. We may for example choose randomly. Also note that we may
choose none of the templates (i.e. we select \textbf{null}), even if
there are available templates at $s.l$.
% all those
%templates from the set $p$, whose entry locations match an actual program
%location $l$. If the selection is not empty, we choose one template $(l,M)$
%of the selected templates.
% This single template $t$ is chosen at
% line~\ref{alg:executeSymbolically:ChooseTemplate}.
%We do not put any constraints to the selection strategy. We may for example
%choose randomly.
If the selection is a template $t$, then we first get a fresh
parameter at line~\ref{alg:executeSymbolically:UnrollingFreshKappa}
and then we replace an original parameter in all tuples of $t.M$ by
the fresh one. This replacement is supposed to prevent parameter
collisions with those in already applied templates. In the
\textbf{foreach} loop at
lines~\ref{alg:executeSymbolically:Unrolling}--\ref{alg:executeSymbolically:UnrollingEnd}
we create a successor state $s'$ for each tuple of $t.M$.
% (see line~\ref{alg:executeSymbolically:SuccessorInUnrolling}).
If the selection computed at
line~\ref{alg:executeSymbolically:GetTemplatesAt} is \texttt{null}, we
proceed to line~\ref{alg:executeSymbolically:ClassicStep}, where we
compute successor states of the state $s$ by classic symbolic
execution. The computed successor states are inserted into the queue
$Q$ and into computed compact symbolic execution $T$ in the
\textbf{foreach} loop at
lines~\ref{alg:executeSymbolically:AddSATSuccessors}--\ref{alg:executeSymbolically:CreateVertex}.
Note that only successor states with satisfiable path conditions are
considered. And finally, the \textbf{foreach} loop at lines
\ref{alg:executeSymbolically:LailedLeafBegin}--\ref{alg:executeSymbolically:LailedLeafEnd}
is supposed to collect those successors $s'$ of the state $s$ for
which we are unable to decide satisfiability of the path condition
$s'.\varphi$. We insert these states into the resulting tree as so
called \emph{failed leaves}. A presence of a failed leaf in the
resulting tree indicates that applied symbolic execution has
failed to explore whole path-space of the executed program.



\begin{algorithm}[!t]
    \caption{\texttt{executeSymbolically}} \label{alg:executeSymbolically}
    \KwIn{a program $P$ to be executed 
        \KwInSep \hspace{-1.1cm}\lmark{$\Box$}\hspace{0.9cm}
         a finite set $p$ of templates computed for cycles in $P$
    }
    \KwOut{a symbolic execution tree $T$ of $P$ (compact tree in $\Box$--version)}
    \BlankLine
    \DontPrintSemicolon
    $s_0 \aset (\Theta,\true,[],\text{the entry location of the
    starting function})$\;        
    \label{alg:executeSymbolically:InitialState}
    Let $Q$ be a queue of states initially containing only $s_0$ \;
        \label{alg:executeSymbolically:InitialQ}
    Into the empty tree $T$ insert the root vertex labelled by $s_0$\;
        \label{alg:executeSymbolically:Root}
    \Repeat{\rm $Q$ becomes empty}{ \label{alg:executeSymbolically:LoopBegin}
        Extract the first state $s$ from $Q$ \;
        \If{\rm $s.l$ is either exit from the starting function or an error location}{
          \textbf{continue}
        }
            $S \aset \emptyset$ \;
            \lmark{$\Box$}
            $t \aset \texttt{chooseTemplate(}s.l,p\texttt{)}$ \;
                \label{alg:executeSymbolically:BoxesBegin}
                \label{alg:executeSymbolically:GetTemplatesAt}
                \label{alg:executeSymbolically:ChooseTemplate}
            \lmark{$\Box$}
            \If {\rm $t \neq $ \bf{null}}{
                \lmark{$\Box$}
                $\kappa \aset \texttt{getFreshParam()}$ \;
                    \label{alg:executeSymbolically:UnrollingFreshKappa}
                \lmark{$\Box$}
                Replace all occurrences of the former parameter in
                    all tuples in $t.M$ by $\kappa$\;
                \lmark{$\Box$}
                \ForEach{$(\theta,\varphi,\Xi,l)\prm{\kappa} \in t.M$}{
                        \label{alg:executeSymbolically:Unrolling}
                    \lmark{$\Box$}
                    $s' \aset s \compose (\theta,\varphi,\Xi,l)\prm{\kappa}$\;
                    \label{alg:executeSymbolically:SuccessorInUnrolling}
                    \lmark{$\Box$}
                    Insert $s'$ into $S$\;
                        \label{alg:executeSymbolically:UnrollingEnd}
                }
            }
            \lmark{$\Box$}
            \Else(~~/* apply classic symbolic execution step */){
                    \label{alg:executeSymbolically:BoxesEnd}
                $S \aset \texttt{computeClassicSuccessors(}P,s\texttt{)}$ \;
                    \label{alg:executeSymbolically:ClassicStep}
            }
            Let $u$ be the leaf of $T$ whose label is $s$ \;
                    \label{alg:executeSymbolically:GetLeaf}
            \ForEach{\rm state $s' \in S$ s.t. $s'.\varphi$ is satisfiable}{
                    \label{alg:executeSymbolically:AddSATSuccessors}
                    Insert $s'$ at the end of $Q$ \; 
                    Insert a new vertex $v$ labelled with $s'$ and a new edge $(u,v)$ both into $T$ \;
                    \label{alg:executeSymbolically:CreateVertex}
            }
            \ForEach{\rm state $s' \in S$ s.t. satisfiability query of $s'.\varphi$ results in FAILURE}{
                    \label{alg:executeSymbolically:LailedLeafBegin}
                    Insert a new vertex $v$ labelled with $s'$ and a new edge $(u,v)$ both into $T$ \;
                    Mark the vertex $v$ in $T$ as failed leaf.
                    \label{alg:executeSymbolically:LailedLeafEnd}
            }
    } \label{alg:executeSymbolically:LoopEnd}
    \Return{$T$} \;
\end{algorithm}

Observe, that we may apply an optimisation at lines
\ref{alg:executeSymbolically:AddSATSuccessors} and
\ref{alg:executeSymbolically:LailedLeafBegin}, where we check for
satisfiability of $s'.\varphi$ only if it differs from $s.\varphi$.

We finish this section by formulating and proving soundness and
completeness theorems for compact symbolic execution. We assume that
$T$ and $T'$ are classic and compact symbolic execution trees of the
program $P$ computed by Algorithm~\ref{alg:executeSymbolically}
without and with $\Box$-lines respectively. We further assume that
neither $T$ nor $T'$ contain failed leaves.

\begin{thm}[Soundness]
For each leaf vertex $e \in T$ there is a leaf vertex $e' \in T'$ and
a valuation  $\bld{\nu}$ of parameters in $e'.s$ such that $e.s \equiv
e'.s\prm{\bld{\nu}}$.
\begin{proof}
Let $\pi$ be the path in $T$ from the root to the leaf vertex $e$. We
prove the theorem by the following induction:

\emph{Basic case}: The root vertices $r$ and $r'$ of $T$ and $T'$
respectively are labelled by the same program state $s_0$ (see lines
\ref{alg:executeSymbolically:InitialState} and
\ref{alg:executeSymbolically:Root} of
Algorithm~\ref{alg:executeSymbolically}). Therefore, $r.s \equiv
r'.s\prm{\bld{\nu}}$, for $\bld{\nu} = \emptyset$.

\emph{Inductive step}: Let $u \in \pi$, $u \neq e$, $u'$ be a vertex
of $T'$, and $\bld{\nu}$ be a valuation such that $u.s \equiv
u'.s\prm{\bld{\nu}}$. We show, there is a successor $w$ of $u$ in
$\pi$, a successor vertex $w'$ of $u'$ in $T'$, and a valuation
$\bld{\nu}'$ such that $w.s \equiv w'.s\prm{\bld{\nu}'}$. There are
two possible cases in Algorithm~\ref{alg:executeSymbolically} for
$u'.s$:

(1) We reach line~\ref{alg:executeSymbolically:Unrolling}: According
to Definition~\ref{def:LoopTemplate}~(L1), there is a successor
vertex $w$ of $u$ in $\pi$, an index $i$, and a non-negative integer
$\nu$ for $\kappa$ such that
\begin{align*}
w.s &\equiv u.s \compose (t.\theta_i\prm{\kappa},
t.\varphi_i\prm{\kappa}, t.\Xi_i\prm{\kappa},
t.l_i)\prm{\{(\kappa,\nu)\}}
\\
&\equiv u'.s\prm{\bld{\nu}} \compose (t.\theta_i\prm{\kappa},
t.\varphi_i\prm{\kappa}, t.\Xi_i\prm{\kappa},
t.l_i)\prm{\{(\kappa,\nu)\}}
\\
&\equiv (u'.s \compose (t.\theta_i\prm{\kappa}, t.\varphi_i\prm{\kappa},
t.\Xi_i\prm{\kappa}, t.l_i))\prm{\bld{\nu} \cup \{(\kappa,\nu)\} }
\\
&\equiv s'\prm{\bld{\nu}'},
\end{align*}
where $s'$ is the $i$-th direct successor of $u'.s$ computed at
line~\ref{alg:executeSymbolically:SuccessorInUnrolling}. And since $w
\in T$, we have $w.\varphi$ is satisfiable. Therefore, there is a
direct successor $w'$ of $u'$ in $T'$ with $w'.s = s'$.

(2) Otherwise, we reach
line~\ref{alg:executeSymbolically:ClassicStep}: Since $u.s \equiv
u'.s\prm{\bld{\nu}}$ and we apply classic symbolic execution step for
$u'.s$, there must be a direct successor $w$ of $u$ and a direct
successor $w'$ of $u'$ such that $w.s \equiv w'.s\prm{\bld{\nu'}}$,
where $\bld{\nu'} = \bld{\nu}$.
\qed
\end{proof}
\end{thm}

\begin{thm}[Completeness]
For each leaf vertex $e' \in T'$ there is a leaf vertex $e \in T$ and
a valuation $\bld{\nu}$ of parameters in $e'.s$ such that $e.s \equiv
e'.s\prm{\bld{\nu}}$.
\begin{proof}
Let $\pi'$ be the path in $T'$ from the root to the leaf vertex $e'$.
We prove the theorem by the following induction:

\emph{Basic case}: The root vertices $r$ and $r'$ of $T$ and $T'$
respectively are labelled by the same program state $s_0$ (see lines
\ref{alg:executeSymbolically:InitialState} and
\ref{alg:executeSymbolically:Root}). Let us construct a non-empty set
$U$ of vertices of $T$ such that for each valuation $\bld{\nu}$ of
parameters in $r'.s$ such that $r'.\varphi\prm{\bld{\nu}}$ is
satisfiable, there is $u \in U$ such that $u.s \equiv
r'.s\prm{\bld{\nu}}$. Obviously $U = \{ r \}$, because $r'.\varphi$
contains no parameter (so $r.s \equiv r'.s\prm{\bld{\nu}}$, for each
$\bld{\nu})$.

\emph{Inductive step}:  Let $u' \in \pi'$, $u' \neq e'$ and $U$ be a
non-empty set of vertices of $T$ such that for each valuation
$\bld{\nu}$ of parameters in $u'.s$ such that
$u'.\varphi\prm{\bld{\nu}}$ is satisfiable, there is $u \in U$ such
that $u.s \equiv u'.s\prm{\bld{\nu}}$. We show, there is a successor
$w'$ of $u'$ in $\pi'$ and a non-empty set $W$ of vertices of $T$ such
that for each valuation $\bld{\nu}'$ of parameters in $w'.s$ such that
$w'.\varphi\prm{\bld{\nu}'}$ is satisfiable, there is $w \in W$ such
that $w.s \equiv w'.s\prm{\bld{\nu}'}$. And we further show that each
$w \in W$ is a successor of some $u \in U$. There are two possible
cases in Algorithm~\ref{alg:executeSymbolically} for $u'.s$:

(1) We reach line~\ref{alg:executeSymbolically:Unrolling}: Let $w'$ be
a direct successor of $u'$ in $\pi'$. Obviously, $w'.s$ is one of the
states $s'$ computed at
line~\ref{alg:executeSymbolically:SuccessorInUnrolling}. Let $i$ be
the index, for which $w'.s = s'$. The formula $w'.\varphi$ is
satisfiable, since $w'$ is in $T'$ (see condition at
line~\ref{alg:executeSymbolically:AddSATSuccessors}). Let $\bld{\nu}$
be a valuation for which $w'.\varphi$ is satisfiable. And let
$\bld{\nu}' = \bld{\nu} \smallsetminus \{ (\kappa,\nu) \}$, where
$\nu$ is an integer assigned in $\bld{\nu}$ to the fresh parameter
$\kappa$ introduced at
line~\ref{alg:executeSymbolically:UnrollingFreshKappa}. From
line~\ref{alg:executeSymbolically:SuccessorInUnrolling} we see that
$u'.\varphi\prm{\bld{\nu}'}$ is satisfiable. Therefore, there is a
vertex $u \in U$ such that $u.s \equiv u'.s\prm{\bld{\nu}'}$.
According to Definition~\ref{def:LoopTemplate}~(L2) there is a
successor $w$ of $u$ in $T$ such that
\begin{align*}
w.s &\equiv u.s \compose (t.\theta_i\prm{\kappa},
t.\varphi_i\prm{\kappa}, t.\Xi_i\prm{\kappa},
t.l_i)\prm{\{(\kappa,\nu)\}}
\\
&\equiv u'.s\prm{\bld{\nu}'} \compose (t.\theta_i\prm{\kappa},
t.\varphi_i\prm{\kappa}, t.\Xi_i\prm{\kappa},
t.l_i)\prm{\{(\kappa,\nu)\}}
\\
&\equiv (u'.s \compose (t.\theta_i\prm{\kappa}, t.\varphi_i\prm{\kappa},
t.\Xi_i\prm{\kappa}, t.l_i))\prm{\bld{\nu}}
\\
&\equiv w'.s\prm{\bld{\nu}}.
\end{align*}
Therefore, $w \in W$.

(2) Otherwise, we reach
line~\ref{alg:executeSymbolically:ClassicStep}: Let $u$ be any vertex
in $U$. Since $u.s \equiv u'.s\prm{\bld{\nu}}$ for some valuation
$\bld{\nu}$ for which $u'.\varphi\prm{\bld{\nu}}$ is satisfiable, and
since all direct successors of both $u$ and $u'$ are computed by
classic symbolic execution step, there must be a direct successor $w$
of $u$ in $T$ and a direct successor $w'$ of $u'$ in $T'$ such that
$w.s \equiv w'.s\prm{\bld{\nu'}}$, where $\bld{\nu'} = \bld{\nu}$.
% Note that both $u'.s$ and $w'.s$ have exactly the same parameters.
Therefore, $w \in W$.
\qed
\end{proof}
\end{thm}

Note that in both theorems there we consider only all \emph{finite}
execution paths in the trees $T$ and $T'$.

\section{Related Work}\label{sec:related}

Compact symbolic execution is tightly related to the work of King in
1976~\cite{Kin76}, where the author introduced the original concept of
classic symbolic execution. Nevertheless, performance issues related to the
path explosion problem were not tackled. The original concept was
generalised in \cite{KPV03} for programs with heap by introducing lazy
initialisation of dynamically allocated data structures.  The lazy
initialisation algorithm was further improved and formally defined
in~\cite{DLR11}. Another generalisation step was done in~\cite{KS05}, where
the authors attempt to avoid symbolic execution of library code (called from
an analysed program), since such code can be assumed as well defined and
properly tested.

In \cite{SPmCS09,GL11} the path explosion problem is tackled by focusing on
program loops. The information inferred from a loop allows to talk about
multiple program paths through that loop. But the goal is to explore classic
symbolic execution tree in some effective manner: more interesting paths
sooner. Approaches \cite{G07,AGT08} shares the same goal as the previous
ones. Only, they focus on a computation of function summaries rather than on
program loops. Our goal is completely different: Instead of guiding
exploration of paths in classic symbolic execution tree, we collapse paths
(with regularities in states along them) into single vertices in compact
symbolic execution tree.
%Paths in compact symbolic execution tree can thus be exponentially
%shorter than corresponding paths in the classic one.
% The main disadvantage of the approaches \cite{SPmCS09,GL11,G07,AGT08} over
% ours is the exploration of paths of \emph{classic} symbolic execution
% tree. These paths can be exponentially longer then corresponding ones in
% compact symbolic execution tree. The same negative effect therefore
% applies to lengths of the corresponding path conditions of these paths.

Note that templates of compact symbolic execution has a different
objective than summarization used in~\cite{G07,AGT08,GL11}. As
summarization basically caches results of some finite part of symbolic
execution for later fast reuse, our templates are supposed to
\emph{replace} potentially infinite parts of symbolic executions by a
single node.

Techniques~\cite{QNR11,SH10} group paths of classic symbolic
execution tree according to an a priori given constraints to a program
output, while exploring only one path per group. We do not require any such
constraints.

Finally, in our previous work~\cite{ST12} there we compute, for a
given program and a target location in it, a non-trivial necessary
condition for reaching the target location. In other words, the result
of the analysis is a first order logic formula. In this work, instead,
we show how to explore all execution paths of an analysed program in a
very compact manner. The result is therefore a compact symbolic
execution tree. Moreover, we do not require any target location, since
we do not focus on program location reachability here. Nevertheless,
to achieve our goal here, we adopted a part of a technical stuff
introduced in~\cite{ST12}. Namely, lines
\ref{alg:template:memoryBegin}--\ref{alg:template:repeatEnd} of
Algorithm~\ref{alg:template} are basically a simplified version of a
computation of a so called iterated memory, which is in~\cite{ST12} an
over-approximation of memory content after several iterations in a
program loop. But in this technique, the memory content must always be
absolutely precise and we may, in addition, analyse only parts of
program loops.

%{{{ Conclusion 

\section{Conclusion}\label{sec:conclusion}

Compact symbolic execution is a generalisation of the classic one. It
takes advantage over the classic symbolic execution by accepting
information about an executed program in advance, in form of
templates.
% We have also presented the algorithm for templates
%computation for a given program.
Using the templates, one can construct substantially smaller tree that
the classic symbolic execution tree without loss of information. In
particular, one finite path of the compact tree can represent even
infinitely many finite paths of the classic tree. Our experimental
results further show that compact symbolic execution explores
execution paths faster then the classic one. On the other hand,
compact symbolic execution is more vulnerable to SAT failures.

%We have introduced a generalization of classic symbolic execution. The new
%technique called compact symbolic execution can analyze a program and derive
%templates representing an overall effect of some program parts to executions
%leading through the program part. Using such a template, one can construct
%substantially smaller tree that the classic symbolic execution tree without
%loss of information. In particular, one node of the compact tree can
%represent even infinitely many nodes of the classic tree.
%
%Our experimental results show that construction of compact trees instead of
%classic trees often leads to significantly faster detection of errors in
%programs. 

% Compact symbolic execution is sound and complete generalisation of the
% classic one. It takes advantage over the classic symbolic execution by
% accepting information about an executed program in advance. The correctness
% of the passed information is formulated by the concept of templates. We also
% showed one possible approach to computation of templates for simple program
% parts.
