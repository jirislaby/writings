\section{Introduction}
TO BE REWRITTEN ACCORDING TO THE BODY
\\[1cm] \noindent
Static analysis is currently considered to be a powerful approach used for
finding errors in a source code. The first real application was performed by
Engler et al. in \textsc{Metal}~\cite{metal} in 2000.

Static analysis might have a form of finding some patterns in the code in some
specified manner (similarly to \textsc{Metal}) or test cases generation which
try to cover the most of the code (covering both true and false branches of
each branching).

Many tools were developed based on either approach however they are commercial
with closed source code, limited in what they can do or both. We are currently
working under Institute of Theoretical Informatics at the Faculty of
Informatics on a free (open-source) tool called \stan. It aims at finding
errors with a use of static analysis.

\subsection{Symbolic Execution}
One method of static analysis, \emph{Symbolic execution} is currently widely
used for checking code properties. It stems from~\cite{select} and many
studies tried to apply the principle on a variety of userspace programs.
\textsc{Klee}~\cite{klee} primarily aims at tests generation that would cover
as many program instructions as possible. \textsc{Pex}~\cite{pex} uses
symbolic execution in cooperation with standard execution on processor
to address program coverage as well, but from another point of view.
Section \ref{current-se} is deserved for detailed description of the work done
in this area.

To the best of my knowledge nobody has ever tried to apply symbolic execution
on an operation system kernel (further only \emph{kernel}) -- WRONG. This task
incorporates not only the execution itself, but some kind of code preparation,
compilation, linking and dynamic linking. These terms will be discussed
further in the text.

\subsection{Justification}
Symbolic execution has shown its power in finding errors in userspace
programs. The idea of applying the approach to the kernel is straightforward
extension, however it is not as easy task as it might seem. The kernel needs
to be processed, the result then executed and checked for some properties.
All of
\begin{itemize}
\item processing the kernel code,
\item finding the right properties to check and
\item finding them in the processed result
\end{itemize}
was not done by anybody yet. The dissertation should lead to complete
implementation of the mentioned and finding errors with minimum false
positives.

\section{Report on Current Results}
Static analysis is a form of software testing and/or finding errors. It
incorporates many branches such as \emph{model checking} and \emph{abstract
interpretation}. Further in the text I will aim specifically at abstract
interpretation, because it is a main field of my doctoral study.

\subsection{Abstract Interpretation}
Abstract interpretation is a relatively old approach, first summarized and
formalized by Cousot \& Cousot in~\cite{cousot} in 1977. There they specify a
formal model used in interpretation to (a) construct a correct abstraction of
the code; (b) be able to compute a solution of some properties defined on the
abstracted code (by fixpoints). Note that checking properties in programs is
in general undecidable, hence they also define narrowing and widening
techniques if the system of equations does not approach convergence when
computing a fixpoint.

Cousots laid down the foundation for others, however the principle was brought
to interest even after 2000. It was thanks to work of Engler et al. on a meta
compilation based technique described in~\cite{metacomp}. System developers
there specify properties of the system in \textsc{Metal}~\cite{metal} language,
compile it into internal representation and the tool applies the specification
on all executable paths in a particular function. Violation of the
specification on a path is then reported as error.

By improving the technique\footnote{According to the press release at
\href{http://www.coverity.com/html/press_story04_02_02_05.html}%
{http://www.coverity.com/html/press\_story04\_02\_02\_05.html}, it produces
less false positives, can run in parallel on the code and contains a
statistical engine.} they developed a commercial derivation of \textsc{Metal}
named \textsc{Coverity Prevent}~\cite{coverity}, the current world leader in
static analysis, according to developers. Currently it analyses many
production systems including the Linux kernel on a daily
basis\footnote{\url{http://marc.info/?l=linux-kernel&m=125856033627134&w=2}}.

Programming languages supported by the mentioned Engler's tools are C and C++.

Aside from Engler's work, Klocwork~\cite{klocwork} started to develop their
bug-finding static analysis tools in 2003. Currently they, as same as
Coverity, find errors in many production projects. In addition to C/C++,
Klocwork products are able to parse Java and C\# programs as well. They seem
to be world number two in code analysis.

\paragraph{Open-source}
All the mentioned tools till now are commercial products without an access to
their source code. However there are also many open-source tools available.
First, let me mention \textsc{Lint} derivatives. They belong to the
\emph{first static analysis tools} used whatsoever. \textsc{Lint} used to find
suspicious and non-portable code constructs. Mostly, the checks are currently
superseded by compilers which perform the checking during standard compilation
phase, so that there is no reason to use \textsc{Lint} anymore.

There was also a popular successor \textsc{LCLint}~\cite{lclint} later renamed
to \textsc{Splint}. The development stalls for more than five
years\footnote{\url{http://www.cs.virginia.edu/pipermail/%
splint-discuss/2007-July/000994.html}} now though. Again \textsc{LCLint}
performs many checks which are done by compiler nowadays. Besides that, it can
catch buffer overflow vulnerabilities. It also interprets \emph{annotated
code} and tries to avoid false positives that way (it allows turning on more
checks which would, without the annotations, generate many false positives).

Later, tools dedicated for a \emph{single purpose or project} emerged.
Specifically for the Linux kernel purposes, \textsc{Sparse}~\cite{sparse} was
developed.  For example it watches pointer usage on user- and kernel-space
interface (they must not be intermixed) or uses of remapped I/O memory
(have to be dereferenced via special helpers) and similar -- kernel specific
-- properties. All the mentioned has no use other than in the kernel.
\textsc{Sparse} is widely used by kernel developers and many sparse errors are
fixed in each kernel release cycle\footnote{For instance, over 20 of sparse
errors were fixed (since 2.6.31) in 2.6.32.}. Let me note, that since this
tool is able to check whole kernel, it is capable to parse most of GNU~C
extensions described in~\cite{gcc}.

From a different point of view, static checkers need not to be only
\emph{dedicated programs}. \textsc{Clang}~\cite{clang} compiler contains a
built-in non-configurable static analyzer. With a use of internal compiler
structures, it looks for typical programming errors like buffer overruns,
using of deprecated (insecure) functions, a sort of memory leaks etc.

All previous open-source tools are configurable only to some extent. There is
no way of specifying new check but changing the sources.
\textsc{Uno}~\cite{uno} is a \emph{fully configurable} tool. It allows user to
specify for what type of bugs should the checker look for. Of course there are
predefined ones (e.g. use of uninitialized variables), but they can be easily
augmented. \textsc{Uno} is the only open-source tool from these with a support
of \emph{inter-procedural analysis}. Unfortunately its parser does not even
support full ANSI C99~\cite{ansi_c}\footnote{Minimal input to demonstrate that
is \texttt{\_Bool ib;}.}.

Still, some tools are hard to use. There is one exception called
\textsc{FindBugs}~\cite{findbugs}. To the best of my knowledge it is the most
known code-checking tool for Java sources. \textsc{FindBugs} is capable to
find hundreds types\footnote{\url{http://www.ibm.com/developerworks/%
java/library/j-findbug1/}} of errors in whole Java project. It has a
\emph{graphical user interface} (GUI) and in fact it is a single button tool.

\paragraph{Stanse}\label{stanse} I slightly described \stan\ in the
Introduction. It is developed at ITI by J. Obdržálek, J. Slabý and M. Trtík
(alphabetical order, I will narrow down my role later in
Section~\ref{results}) with a help of master students. For the ease of
programming, the tool is written mostly in Java. Further, some parts are pure
C and the rest (scripts) needs Perl. All these parts are open-source.

\stan\ tries to pick all the pros from previously named tools and also adds
some novelty. It consists of three checkers:
\begin{itemize}
\item \textbf{AutomatonChecker} -- technique based on finite automata. I will
describe it in more detail in Section~\ref{automata}.
\item \textbf{ThreadChecker} -- Jan Kučera developed the checker in the scope
of his master thesis. The checker looks for thread-creation functions and
statically follows the threads. Namely, it watches lock operations and builds
\emph{locksets}~\cite{lockset}, represented as resource allocation graphs
(RAG). The checker combines RAGs of all threads and tries to find cycles
there. If it is successful with the search, an error is reported\footnote{For
example one report is available at \url{http://lkml.org/lkml/2009/4/14/527}.}.
\item \textbf{ReachabilityChecker} -- It is a short checker (below 200 lines
of code) which performs reachability analysis of statements to reveal
unintentional dead code. It catches errors like omitted semicolons after the
if condition, so that the branching is not done properly\footnote{Such bad
behaviour was fixed e.g. by \url{http://patchwork.kernel.org/patch/56386/}.}
(\texttt{if (cond)\underline{;} true\_statement;}). The true branch here would
be an empty statement.
\end{itemize}

For simplicity, the last checker needs no configuration. The former two are
fully configurable though. User can specify properties to check, which
functions to watch and importance of found errors. All expressed by XML
structures.

Before the checking is started, files to check need to be known. \stan\
supports several methods of specifying input files. The most convenient way is
to use batch files generated by build system of code to be checked. After
specifying project's \texttt{Makefile}, the checking will be performed
automatically on all its files.

On the file basis, C language with most GNU extensions is supported, hence
it is successful in parsing whole Linux kernel. A speedup was made to be able
to parse such big code bases. Each preprocessed file is split and one part
which originates from external files (\texttt{\#include}s) is thrown away after
remembering typedefs. They are the only information needed for the second
part, which is the original file itself after preprocessing.

Sometimes preprocessor expands some code constructs unintentionally. For
example \texttt{spin\_lock} call in kernel may be expanded to many different
variants depending on configuration. \stan\ avoids that by renaming these
functions to disable the expansion.

While having internal structures parsed from a source file, inter-procedural
analysis is performed. It is implemented as binding a CFG node of a call to a
CFG of a called function. A summaries computation similar to~\cite{metal} is
yet to be implemented.

Strictly speaking, \stan\ is not bound to a single language, not even
to a single compiler version (cf. \textsc{Metal}). The language frontend may
be easily changed thanks to modular structure of whole tool.

Inspired by \textsc{FindBugs}, everything what can be done on a command line
is also available in the \stan\ GUI. It renders the tool into one-button
checker. From another point of view, the tool may be instructed on
command-line to dump results into XML to later inspection in GUI.

\paragraph{Stanse Framework} \stan\ is not only a checker. It provides
easily extensible framework for checking. All code structures such as AST and
CFG are available to the programmer. For the convenience \stan\ offers
traversal functions over these structures as well. Also unified error
reporting is exposed, so that checker programmers have to implement only
relevant parts of their new checker. A good example of simplicity is
ReachabilityChecker.

\subsection{Algorithms over Abstraction}
The previous chapter introduced solely to the tools currently known and used.
This chapter will cover the main algorithms implemented over abstract
interpretation in some of them.

\subsubsection{Automata}
\label{automata}
\textsc{Metal} uses state machines (SM) as an abstraction. The machine specifies
some rules which should ever hold for the code. See Figure~\ref{fig.auto_mem}
for example of SM representing states of pointers. Transitions are performed
by code matching against patterns (alloc, free etc.).

For instance, if the automaton is in the initial/freed (I/F) state and there
is a pointer dereference in the code, the error state is reached and the
incident reported. Allocations may fail and hence there are two \emph{alloc}
transitions -- when it succeeds (valid pointer returned) and when it does not
(NULL pointer returned).

The automaton creates another report on memory leak, i.e. valid memory is lost
by another allocation (the bottom-down arrow in the right). Note that for
simplicity the automaton does not cover the case of allocation from NULL
state.

Much code behavior may be described by automata. \stan\ contains thorough
automata descriptions for memory, locking (lock, unlock, try\_lock, nested
locks up to 5 levels), operation pairing (mostly for reference counting) and
atomic functions (ones make atomic contexts, others de-atomize, they should be
balanced).

\subsubsection{Symbolic Execution}
\label{current-se}
So far I have not described one special code interpretation method called
symbolic execution (SE). As mentioned in the Introduction, it is based on
\cite{select}. The principle is that chosen variables are tracked symbolically
rather than concretely.

\begin{figure}[htb]
\begin{center}
\begin{tabular}{c} % listing refuses to be in the center otherwise
\begin{lstlisting}[language=C]
x = random();
if (x < 8) (*@\label{lst_symb1}@*)
	lock();
compute();
if (x < 8) (*@\label{lst_symb2}@*)
	unlock();
\end{lstlisting}
\end{tabular}
\caption{Symbolic execution example}
\label{fig.symbolic}
\end{center}
\end{figure}

Let us consider the imaginary code in Figure~\ref{fig.symbolic}. \texttt{x} is
initialized to some random integer value and tested at lines \ref{lst_symb1}
and \ref{lst_symb2} for the same condition (\texttt{x < 8}). If \texttt{x} was
not handled symbolically, the execution might run through the true branch
(\texttt{lock}) first and then through the false one (no \texttt{unlock}).
This would generate a locking imbalance error in the code even though it's a
clear false positive.

Now, if we handled \texttt{x} symbolically and store the path condition
\texttt{x < 8} at line \ref{lst_symb1}, the executor is not allowed to
traverse the code via the false branch at line~\ref{lst_symb2} since the path
condition would be violated. Hence the code is properly executed such that
either the lock is both locked and unlocked or not touched at all.

The next chapter will cover evolution of SE and current state of the art.

\subsection{Current Status of SE}
Symbolic execution was already successfully deployed in software testing. The
very first tools with SE were \textsc{Exe}~\cite{exe} and
\textsc{Dart}~\cite{dart}.

\paragraph{Dart and Pex} \textsc{Dart} is designed to use SE in cooperation
with standard (concrete) execution on a processor. This principle, first
mentioned in~\cite{first_symb_dyn}, is used here to generate tests. The tests
are intended to cover as much code as possible. In fact, the method leads to
almost 100\,\% code coverage\footnote{The problem is undecidable in general,
approximations may lead to imprecision.} and hence it generates tests for
almost all possible code paths. \textsc{Dart} works on the top of C language.

In detail, \textsc{Dart} first scans the source code by simple static analysis
tool to find so-called \emph{interface}. It is a description of external
program inputs such as input specified by user on command-line (parameters of
top-level functions, e.g. of \texttt{main}), external functions etc. This
interface is then used as a potential symbolic input.

Once the interface is known, a \emph{test driver} is created. It is a small
code hunk written in C which represents the interface. It initializes
top-level function parameters with random values and also implements external
functions found in the previous step. These stubs return random values as
well.

With all the code known, \textsc{Dart} starts SE, initially with random input.
Since it wants to cover both true and false branches of every branching, it
executes the code repeatedly with the input corresponding to the opposite
branch. To demonstrate it on our example in Figure~\ref{fig.symbolic},
consider \textsc{Dart} goes through the true branch first time, i.e. with path
condition \texttt{x < 8}. Next time it will add \texttt{x >= 8} to a path
condition and generates the program input accordingly to cover even the false
branch.

Sometimes it may be infeasible for several reasons\footnote{Non-linearity in
expressions, dereferences of user-provided pointers etc.} to obtain some values
symbolically. In such a situation \textsc{Dart} falls back to the value from
dynamic execution. In this case, testing completeness is of course lost.

\textsc{Pex}~\cite{pex} is a successor of \textsc{Dart} working with .NET
applications. The same as \textsc{Dart}, \textsc{Pex} executes the code
dynamically and statically in parallel. Its difference lays in parametrization
of tests specified by a user here. It might seem similar to the standard unit
testing known from \emph{extreme programming}, but exactly the parametrization
makes the difference.

Each method (test itself) has several parameters, which specify inputs. Let us
have a look at slightly modified example taken from~\cite{pex}:
\begin{lstlisting}[language=Java]
public void TestAdd(int capacity, object element) {
	ArrayList a = new ArrayList(capacity);
	a.Add(element);
	Assert.IsTrue(a[0] == element);
}
\end{lstlisting}
With standard unit testing, one would need to specify several objects and
capacities to test \texttt{ArrayList} implementation. However, given
\texttt{TestAdd}, \textsc{Pex} will exercise all \texttt{capacity} and
\texttt{element} combinations which branches \texttt{ArrayList} implementation
differently. In turn it covers more code.

\textsc{Pex}, with one of its later extensions (\cite{pex_ddse}), may look at
the source code differently. Unlike generating tests for all possible paths,
a location to explore may be known in advance. One example for all is
\texttt{assert} and its location is easily obtainable from the code. In such
situations a \emph{demand-driven analysis} may be incorporated. This approach
reduces time spent by analysis significantly, because paths which leads to the
target are really taken into computations.

The very same paper also mentions \emph{compositional} analysis. It uses
already mentioned summaries to represent functions behaviour, so that
functions with same environment (same constraints on parameters and heap) are
not executed twice. A value returned in previous run is used instead.

\paragraph{Cute} Also \textsc{Cute}~\cite{cute} is based on parallel execution
principle. However it tries to address pointer parameters of test functions.
A structure behind the pointer is represented as a \emph{memory graph} and its
contents is changed according to branches with pointer accesses in the code.

For example if the code contains \texttt{if (ptr->member == 1)}, for the first
time, \textsc{Cute} will construct random memory map for structure behind
\texttt{ptr}. Second, the map will contain 1 on the \texttt{ptr->member}
position (or some integer other than 1 in case the random input was 1
already).

\paragraph{Exe and Klee} Contrary to the previous work, a tool from Engler et
al. called \textsc{Exe}~\cite{exe} executes the code solely symbolically.
\textsc{Exe} works as follows.

User \emph{specifies variables} which should be handled symbolically by
\texttt{make\_symbolic} function. This is further compiled by \texttt{exe-cc}
compiler and later compiled along with the code by standard compiler.

Before the compilation, each branch, assignment and expression from the
original code must be \emph{instrumented}. It has two reasons. First,
operands of these operations are checked whether they should be handled
symbolically (at least one operand is symbolic) or as in uninstrumented code
(all are concrete). Second, each branch contains a \texttt{fork} call to
(natively, on OS level) split the execution.

Each cloned child contains negated last element from path condition and hence
walks the true or false branch respectively.

With the instrumentation described, resulting binary is normally \emph{run}.
It may finish either by standard way (\texttt{main} returned, \texttt{exit}
called etc.) or with an error (crash, \texttt{assert} parameter is false,
\textsc{Exe} detected an error such as zero division, \ldots). In all cases
inputs corresponding to each path constraint are dumped. When from the latter
case, it is an "input of death" in their language.

They designed also their own tailor-made solver. This allows \textsc{Exe} to
perform certain operations and optimizations which cannot be done by other
tools when using external solvers. \textsc{Exe}'s key improvements are
constraint caching, independent constraint optimization, bitvector arithmetic,
and tracking indirect memory accesses symbolically. This allows them to
speedup checking significantly, since solvers take huge amount of time from
checking procedure. See~\cite{exe} for technical details.

With this tool, apart from others, they checked three filesystems in Linux
kernel by writing special block device which injected "corrupted" filesystem
structure. They were also successful in finding bugs in firewall
implementations/interpreters of BSD and Linux. This problematics is also
covered in Section~\ref{exe_kernel}.

However there is still one notable bottleneck in the algorithm: \texttt{fork}.
It does not belong to the fastest system calls. Hence
\textsc{Klee}~\cite{klee}, next Engler's work, tries to tackle this
disadvantage.

Instead of \texttt{fork}, they wrote their own \emph{process scheduler}. There
is one immediate outcome -- scheduling is in their hands and can thus be
parametrized and tuned. Created children are scheduled by heuristics to not
unroll infinite loops and to cover as much code as possible early.

Another improvements in \textsc{Klee} are further simplification of path
conditions before solving, the solving itself and also lower memory
consumption targeted by condensed representation of created children.
Intermediate representation was changed too. LLVM is used for symbolic
execution. LLVM is described in more detail in Section~\ref{llvm}.

\textsc{Klee} is distributed with a \emph{C library implementation} so that
external C functions are not black boxes anymore. Those which still cannot be
resolved (including system calls) are tried run dynamically with reconstructed
environment and parameters back from symbolic path constraints. If that fails
too, \textsc{Klee} refuses to check the program and forces user to specify
sources of that unknown function.

Similar to \textsc{Exe}, it dumps input for programs for every possible code
branch, allowing near 100\,\% code coverage.

With a use of these methods, they were able to find three severe bugs in
\textsc{Coreutils}, the package of tools used daily on most Linux machines.

\paragraph{Sage} All previous work was based on some intermediate language
representation. \textsc{Sage}~\cite{sage} is unique in that it symbolically
executes native x86 instructions. Immediate advantage is that it need not
source codes. The program is linked in a standard way and all code is
available. This allows even close-source code to be exercised.

What is worse are the instruction themselves. According to the Intel processor
manual\footnote{Available at \url{intel.com}.}, there is almost 400 basic
instructions. Most of them has to be supported by the tool\footnote{Some are
often omitted in the implementations. For instance floating point operations
have no support, since checkers do not track floating point variables
either.}. And with new processors, this count grows.

One more \textsc{Sage} novelty is \emph{generational search} which maximizes
number of generated inputs on each execution. It creates inputs for each
negated component of a path constraint at once. I.e. if the path condition was
$a \wedge b \wedge c$ and first run satisfies all of them (true branches),
next run will generate inputs with $!a$, $a \wedge !b$ and $a \wedge b \wedge
!c$ path conditions at once (if satisfiable indeed).

\subsubsection{LLVM}
\label{llvm}
LLVM~\cite{llvm} seems to be a future of compilers. The support of languages
in it is already large. Currently, C, C++, Obj-C, Ocaml and Java are
supported.  Others are planned, it is a matter of implementing a front-end.

There is a specially designed instruction set which every front-end should
compile into. The set is reduced, with only tens of instructions. As it is
language-independent, it allows several language-independent optimizations,
separately from front-ends.

The language may be also emitted to a file (as plain text or bytecode), so
that it can be used arbitrarily e.g. for symbolic execution. A big advantage
is that LLVM framework provides also libraries for LLVM code traversing. This
renders "parser" for symbolic execution quite easy, because only few
instructions have to be supported.

Although this may seem perfect, inline assembly thrashes the whole idea of few
instructions. The assembly is hard-copied from source code to the LLVM code
and hence tools have to take care of it properly. A simple solution is to
ignore it altogether, however for kernel, it may lead to inconsistencies due
to high inline assembly occurrence rate.

What is worse, none of contemporary C front-ends (\texttt{llvm-gcc} and
\texttt{clang}) is capable of parsing kernel code seamlessly. GNU C compiler
(\texttt{gcc}) is still a superset of both of them in what language constructs
are supported. I personally already reported two bugs from this area in a hope
of fixing. Both of them are a matter of unsupported inline assembly operations
found in kernel.

Still, LLVM turns out to be eligible choice nowadays because it is under heavy
development with release schedules. Hence it looks like missing features will
be added in the near future.

In addition, this framework was used in \textsc{Klee}. Hence, due to
licensing, it may be freely obtained at LLVM site.

\paragraph{Toolkit} I decided to use LLVM as a base for my dissertation and
hence I will briefly introduce the toolkit here. Current stable version is
2.6, all facts drawn here are with regard to this release.

From the beginning, compilers have a \texttt{--emit-llvm} parameter which
gives us a LLVM code. Each file will be generated to a separate file. However
it is still possible to have the code all-in-one. For that purpose
\texttt{llvm-link} and \texttt{llvm-ld}. They differ only in details. When all
files are linked together it is one single LLVM code which external tools
accept as input.

The input is handled through LLVM framework. It is written and exported in C,
C++ and Ocaml languages which designates the tool to be written in one of
those languages.

\subsubsection{Exe and Kernel}
\label{exe_kernel}
\textsc{Exe} was used for testing picked parts of Linux kernel. Specifically,
three filesystems were checked in~\cite{exe_fs}. A disadvantage of
\textsc{Exe} is that it has to run as a standard system process to be
clone-able by \texttt{fork} (compare to the description earlier). They tried
to exempt pieces of code relevant to the filesystem and use them in userspace
first, but they failed. It is because the code was still dependent on the rest
of the kernel in some way\footnote{Either environment, assembly, allocators or
other peculiar issues.}.

Finally they felt back to an
UML\footnote{\url{user-mode-linux.sourceforge.net.}} adaptation. It runs as a
userspace process which exactly matches \textsc{Exe}'s needs. The adaptation
is a reuse of their past work on kernel Model Checking (see
Section~\ref{mc_kernel}).

There was still a problem to cope with. Minimal input to a filesystem is
16\,MiB. The solution is to handle the filesystem image symbolically only
on demand. I.e. when the kernel asks for a block, it is made symbolic. This
allows the tool to run quickly, as relevant metadata are stored only in few
blocks.

\subsection{Related Work}
Static analysis is indeed a wide area and I have not named all tools and
algorithms currently available and known. This section provides quick overview
of other approaches and techniques.

\paragraph{Dynamic Testing}
Dynamic testing was here since first programs were written. They are fed by
random inputs, a kind of black-box testing~\cite{random1,random2,random3}.
Unit testing is a clear opposite. Programmer tries to use the knowledge of
what the code does and writes a testbed for his implementation to test as many
paths as possible.

Programmers often instrument their code with conditional (debug-only) fault
injections. For example allocators may return an error instead of returning
correctly allocated memory even if there is enough of memory. It helps to
cover corner cases in the code.

Specifications may reveal many bugs when converted to grammars and utilized to
generate inputs like in~\cite{grammar1,grammar2,grammar3}.

\paragraph{Model Checking} Opposing to checkers, model checking (MC) tries
to \emph{prove}, that checked programs satisfy a particular property according
to specification. We are not trying to achieve such goals, because MC (a)
suffers from state space explosion and hence lasts long often; (b) need
precise models. On the other hand, it reports no false positive errors.

A comprehensive overview of Microsoft MC tools with further links and
technical details about \textsc{Slam} and \textsc{SDV} is in~\cite{slam}. It
has found many errors in Windows Device Drivers. \textsc{Slam}-based tool that
is supplied with lazy abstraction is \textsc{Blast}~\cite{blast}. A model
there is very coarse at the beginning and is narrowed even after it is
necessary and only at places which needs that. It lowers time and space spent
on run.

The two were picked of plenty abstraction-based model checkers because of
summary papers provided. Reader will be directed for further reading there.

There exist many of execution-based tools. Let us mention
\textsc{Spin}~\cite{spin}, a widely used formal verification tool in present.

\paragraph{Kernel and Model Checking} \label{mc_kernel} Similarly to
Section~\ref{exe_kernel}, three filesystem implementations were verified. This
time with a use of MC in~\cite{mc_kernel_fs}. Linux TCP/IP stack was under MC
algorithms inspection in~\cite{mc_kernel_ip}. Both works are based upon
a~\cite{cmc}.
